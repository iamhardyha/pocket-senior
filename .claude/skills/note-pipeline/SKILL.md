# Note Pipeline — Pocket Senior 노트 제작 오케스트레이터

사용자가 궁금해하는 백엔드 주제를 **검증된 노트 + 슬라이드 덱**으로 만들어 사이트에 통합하는 전체 절차. 핵심 철학: **바로 노트로 직행하지 않고, 먼저 대화로 궁금증을 구체화한 뒤, 확인받고 자동 파이프라인을 돌린다.**

**실행 모드: 하이브리드** — Phase 1·2는 메인 에이전트가 사용자와 직접 대화, Phase 3은 서브에이전트 파이프라인(파일 핸드오프). 모든 Agent 호출은 `model: "opus"`.

## Phase 0: 컨텍스트 확인
시작 시 기존 산출물을 확인해 실행 모드를 정한다:
- `_workspace/` 존재 + 사용자가 부분 수정 요청 → **부분 재실행**(해당 에이전트만 재호출)
- `_workspace/` 존재 + 새 주제 → 기존을 `_workspace_prev/`로 옮기고 **새 실행**
- 기존 노트 수정 요청(이미 docs/에 있는 글) → 해당 노트를 입력으로 **수정 파이프라인**
- 그 외 → **초기 실행**

## Phase 1: 구체화 (메인 에이전트 ↔ 사용자 대화)
**바로 답하거나 노트를 쓰지 않는다.** 먼저 사용자가 *무엇이* 막히는지 끌어낸다.
1. 되묻는다: "구체적으로 어떤 점이 궁금한가요 / 어디가 이해 안 되나요?"
2. 사용자가 막힌 지점을 말하면, **스코프 합의에 필요한 만큼만** 설명하거나 다룰 수 있는 앵글을 제시한다(완전한 답은 노트에서 푼다).
3. 왕복 대화로 **"이 노트가 다룰 범위·관점·깊이 + 카테고리"**를 합의한다.

산출(머릿속/메모): 주제, 카테고리, 궁금한 지점 목록, 깊이/관점.

**Why**: 사용자의 진짜 막힌 지점을 겨냥해야 헛다리 노트가 안 나온다. 이 단계가 노트 품질의 8할이다.

## Phase 2: 확인 게이트
스코프가 굳으면 사용자에게 묻는다: **"이대로 노트로 작성할까요?"**
- 아니오/더 논의 → Phase 1로 돌아간다.
- 예 → Phase 3.

## Phase 3: 풀 파이프라인 (서브에이전트, 파일 핸드오프)
`_workspace/`를 작업 공간으로 쓰고, 각 단계는 `model: "opus"`로 호출한다.

1. **노트 초안** — `Agent(note-author, model:opus)`에 합의된 스코프를 넘긴다. `note-authoring` 스킬 규약으로 `_workspace/note-draft.md` 작성. 반환: 제목·카테고리·order·검증 필요 주장.

2. **도메인 검증** — `Agent(domain-verifier, model:opus)`에 초안+카테고리를 넘긴다. `domain-verification` 스킬로 사실 검증, `_workspace/verify-report.md` 작성.
   - 카테고리가 여러 도메인에 걸치면 페르소나별로 **병렬** 호출(`run_in_background`).

3. **원문 수정 (검증 결과 반영)** — verify-report에 HIGH/MED 오류가 있으면, **원문(초안)을 먼저 고친다** (`Agent(note-author)` 재호출 또는 직접 Edit). 검증가가 웹으로 기각한 의심 지적은 반영하지 않는다.

4. **슬라이드 덱** — 검증·수정된 노트로 `Agent(slide-deck-builder, model:opus)` 호출. `slide-deck-build` 스킬로 덱 생성 + `slides: true` 추가.
   - 빌더가 "생략 권장"을 반환하면 사용자에게 사유를 전달하고 슬라이드는 건너뛴다(노트만 저장).

5. **저장 & 빌드 확인** — 노트를 `docs/<category>/<제목>.md`로, 덱을 `docs/public/slides/<category>/<note>.html`로 확정. `npm run build`로 빌드 통과 확인(빌드는 **`npm run check` 정합성 검사를 먼저 돌리고 실패 시 중단**한다). **카테고리 index 페이지·사이드바·홈 카운트·갤러리·MetaBar 토글은 전부 `notes.data.ts` 단일 출처에서 자동 반영 — index.md를 손으로 고치지 말 것**(`CategoryNotes.vue` 자동 렌더). 상세 규약·체크리스트는 `note-authoring` 스킬의 "통합 정합성" 참고.

6. **배포 안내** — 빌드가 통과하면 **사용자에게 commit & push를 안내**한다(자동 푸시하지 않음). 푸시 시 GitHub Actions 자동 배포.

## 데이터 전달
- 중간 산출물: `_workspace/{단계}_{산출물}` (note-draft.md, verify-report.md). 사후 추적용으로 보존.
- 최종 산출물만 `docs/`에 확정.

## 에러 핸들링
- 서브에이전트 실패 시 1회 재시도, 재실패면 그 결과 없이 진행하되 사용자에게 누락을 명시.
- 검증가의 지적과 원문이 상충하면 삭제하지 말고 **웹 1차 출처로 판정** → 채택/기각 사유를 남긴다.
- 빌드 실패 시 배포로 넘어가지 않는다 — 원인 수정 후 재빌드.

## 테스트 시나리오
- **정상 흐름**: "Redis 분산 락이 헷갈려" → 구체화 대화(어떤 점이? SETNX? 펜싱 토큰? Redlock?) → 확인 → note-author 초안 → domain-verifier(concurrency=Concurrency specialist) 검증 → 원문 수정 → slide-deck-builder 덱 → 빌드 → 커밋 안내.
- **에러 흐름**: domain-verifier가 "Redlock은 안전하지 않다"는 강한 주장을 지적 → 웹 교차검증 → 논쟁적 사안임을 확인 → 원문에 양쪽 관점 병기로 수정, 일방 단정 제거.

## 후속 요청
"이 노트 슬라이드만 다시" → Phase 3의 4단계만. "내용 보완" → note-author 재호출(이전 초안 기반) → 재검증 → 덱 갱신. Phase 0에서 부분 재실행으로 분기.
