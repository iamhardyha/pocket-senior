# Domain Verification — 노트 사실 검증 규율

노트가 슬라이드로 전파되기 전에 **핵심 기술 주장의 사실 정확성**을 점검한다. 스타일·문장력은 대상이 아니다.

## 1. 페르소나 선택 (카테고리 → 전문가)

| 카테고리 | 페르소나 | 검증 초점 |
|---|---|---|
| database | Senior DBA | 쿼리 정확성, 격리 수준, 인덱스 동작 |
| concurrency | Concurrency specialist | 락 시맨틱, 레이스 컨디션, 정합성 |
| architecture | System architect | 패턴 적용성, 트레이드오프, 확장성 |
| traffic / failure | SRE/Reliability engineer | 부하 처리, 페일오버, 장애 모드, 복구 |
| infra | DevOps/Platform engineer | 마이그레이션 리스크, 호환성, 운영 |

여러 도메인이 걸치면 해당 페르소나를 모두 적용한다(병렬 검증 가능).

## 2. 무엇을 검증하나
- 구체적 수치·기본값(프레임워크 설정 기본값, 클라우드 기본값, 버전별 동작)
- 단정 표현("항상", "반드시", "N배")의 정당성
- 패턴/알고리즘 설명의 정확성(예: 해시 분배, 격리 수준, 합의 프로토콜)
- 코드·설정·다이어그램의 기술적 오류
- note-author가 ❓로 표시한 주장은 **집중 검증**

## 3. 핵심 규율 두 가지

### (a) 오류 발견 → 원문 먼저 수정 → 그 다음 덱
틀린 내용을 고치기 전에 슬라이드를 만들면 오류가 그대로 전파된다. **항상 원문 .md를 먼저 고치고**, 검증 통과한 노트로만 덱을 만든다.

### (b) 의심스러운 자기 지적은 웹으로 교차검증
기본값·표준 동작에 대한 지적은 **1차 출처(`WebSearch`/`WebFetch`)로 확인한 뒤에만 확정**한다.

**Why**: 그럴듯하지만 틀린 지적이 정확한 원문을 망친다. 실제 사례:
- "Resilience4j aspect 순서가 틀렸다"는 지적 → 공식 문서 확인 결과 **원문이 맞았음**(CircuitBreaker 바깥·Bulkhead 안쪽). 그대로 적용했다면 정확한 노트를 오염시켰을 것.
- "ALB Health Check 기본값 Healthy 5회가 틀렸다(3회여야)"는 지적 → 확인 결과 **5회가 ALB 기준 정확**(3회는 NLB). 기각.

확인되면 채택, 반박되면 **기각하고 그 사실을 함께 보고**한다.

## 4. 보고 형식

```
## <파일/섹션>
- [HIGH/MED/LOW] 원문 인용: "<틀린 문장 그대로>"
  → 문제: <무엇이 왜 틀렸는지 한 줄>
  → 수정안: "<교체할 정확한 문장>"
  → 근거: <웹 확인 시 URL, 아니면 도메인 지식>
```

- 심각도: HIGH(명백한 사실 오류·슬라이드 전파 차단 필수) / MED(오해 소지·정밀도) / LOW(표현 강도).
- 오류가 없으면 **솔직히 "검증 통과 — 수정 불필요"**. 억지 지적 금지.
- 기각한 의심 지적도 "확인 결과 원문이 맞음(근거 URL)"으로 남겨 재검증 낭비를 막는다.
