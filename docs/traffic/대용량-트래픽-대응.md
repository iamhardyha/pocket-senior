# 대용량 트래픽 대응

> **핵심 질문**: 많은 트래픽에 대비하여 어떻게 하는가?

---

## 한 줄 요약

**"트래픽을 줄이고, 분산하고, 버틸 수 있게 만든다."**

대용량 트래픽 대응은 크게 3가지 전략으로 나뉜다:
1. **트래픽 자체를 줄인다** (Caching, CDN)
2. **트래픽을 분산한다** (Load Balancing, Sharding)
3. **시스템이 버틸 수 있게 만든다** (Scaling, Async, Rate Limiting)

---

## 1. 트래픽을 줄인다 — Caching

서버까지 요청이 안 오게 하는 것이 가장 효과적이다.

### Multi-Layer Caching

```
[Client] → [CDN] → [API Gateway] → [Application Cache] → [DB Cache] → [DB]
   ↑          ↑                          ↑                    ↑
 브라우저    정적 리소스              Redis/Local           Query Cache
 캐시       (이미지,JS,CSS)          (비즈니스 데이터)       (DB 레벨)
```

| 계층 | 위치 | 예시 | TTL |
|------|------|------|-----|
| Browser Cache | 클라이언트 | Cache-Control 헤더 | 분~일 |
| CDN | Edge Server | CloudFront, Cloudflare | 분~시간 |
| Application Cache | 서버 메모리/Redis | 상품 정보, 사용자 세션 | 초~분 |
| DB Query Cache | DB 레벨 | MySQL Query Cache (8.0에서 제거됨), Materialized View | 초~분 |

### 실무 포인트

- **Cache Hit Rate**가 핵심 지표. 읽기 요청의 Hit Rate가 90% 이상이면 읽기 DB 부하가 약 1/10로 줄어든다
- **Cache Invalidation**이 가장 어려운 문제 — "컴퓨터 과학에서 어려운 2가지: 캐시 무효화와 이름 짓기"
- 캐시 전략 선택:
  - **Cache-Aside**: 애플리케이션이 직접 캐시 관리 (가장 일반적)
  - **Write-Through**: 쓰기 시 캐시도 같이 업데이트
  - **Write-Behind**: 캐시에만 쓰고, 나중에 DB에 반영 (위험하지만 빠름)

```
// Cache-Aside 패턴 (가장 많이 씀)
fun getProduct(id: Long): Product {
    // 1) 캐시 조회
    val cached = redis.get("product:$id")
    if (cached != null) return cached    // Cache Hit

    // 2) DB 조회
    val product = db.findById(id)

    // 3) 캐시 저장
    redis.set("product:$id", product, TTL = 5.minutes)
    return product
}
```

## 2. 트래픽을 분산한다 — Load Balancing & Sharding

### Load Balancing

하나의 서버가 아닌 여러 서버가 트래픽을 나눠 받는다.

```
                    ┌─── Server 1
Client → LB ───────┼─── Server 2
                    ├─── Server 3
                    └─── Server 4
```

**L4 vs L7 로드밸런서**

| 구분 | L4 (Transport) | L7 (Application) |
|------|----------------|-------------------|
| 기준 | IP, Port | URL, Header, Cookie |
| 속도 | 빠름 | 상대적으로 느림 |
| 예시 | NLB (AWS) | ALB (AWS), Nginx |
| 용도 | 단순 분산 | 경로 기반 라우팅 |

**분산 알고리즘**

| 알고리즘 | 설명 | 사용 시점 |
|----------|------|-----------|
| Round Robin | 순서대로 배분 | 서버 스펙이 동일할 때 |
| Weighted Round Robin | 가중치 기반 배분 | 서버 스펙이 다를 때 |
| Least Connections | 연결 적은 서버로 | 요청 처리 시간이 다양할 때 |
| IP Hash | 같은 IP → 같은 서버 | 세션 유지 필요 시 (비추, Sticky Session) |

### DB 분산 — Read Replica

읽기 트래픽이 80% 이상인 서비스가 대부분이다.

```
         ┌─── Write ──→ Primary (Master)
Client ──┤                   │
         └─── Read  ──→ Replica 1  (복제)
                         Replica 2  (복제)
```

- **Write**: Primary에만
- **Read**: Replica에서 분산 처리
- **주의**: Replication Lag (복제 지연) — 쓰기 직후 읽기 시 이전 데이터가 보일 수 있음

→ 자세한 내용: [샤딩과-레플리카](../database/샤딩과-레플리카.md)

## 3. 시스템이 버틸 수 있게 만든다

### 3-1. Scaling

| 구분 | Vertical Scaling (Scale-Up) | Horizontal Scaling (Scale-Out) |
|------|-----------------------------|-------------------------------|
| 방법 | 서버 스펙 올리기 (CPU, RAM) | 서버 수 늘리기 |
| 장점 | 간단, 코드 변경 없음 | 이론적으로 무한 확장 |
| 한계 | 물리적 한계, 단일 장애점 | 상태 관리 복잡, 데이터 정합성 |
| 비용 | 고스펙일수록 비선형 증가 | 선형 증가 |

**실무에서는?**
- 처음엔 Scale-Up으로 시작 (간단하니까)
- 한계가 오면 Scale-Out으로 전환
- Scale-Out 시 **Stateless** 설계가 필수 — 세션은 Redis로, 파일은 S3로

### 3-2. Auto Scaling

트래픽에 따라 서버 수를 자동으로 조절한다.

```
평상시:  Server x 2
피크:    Server x 8  (자동 증가)
새벽:    Server x 1  (자동 감소)
```

- **기준 지표**: CPU 사용률, 메모리, 요청 수, 응답 시간
- **쿨다운(Cooldown)**: 스케일링 후 일정 시간 대기 (불필요한 반복 방지)
- **주의**: 새 인스턴스가 뜨는 데 시간이 걸림 (Cold Start) → [스파이크-트래픽-대처](./스파이크-트래픽-대처.md)에서 다룸

### 3-3. 비동기 처리 — Message Queue

모든 요청을 즉시 처리할 필요는 없다.

```
동기 (Before):
Client → API → 결제 처리 → 재고 차감 → 알림 발송 → Response (3초)

비동기 (After):
Client → API → 결제 처리 → Response (0.5초)
                  ↓
              Message Queue → 재고 차감 (비동기)
                            → 알림 발송 (비동기)
```

- 사용자 응답 시간이 줄어든다
- 각 처리가 독립적으로 실패/재시도 가능
- 대표 도구: [Kafka](../architecture/Kafka.md), RabbitMQ, AWS SQS

→ 자세한 내용: [이벤트-리스너-vs-비동기](../architecture/이벤트-리스너-vs-비동기.md)

### 3-4. Rate Limiting

서버가 처리할 수 있는 양 이상의 요청은 거부한다.

```
정상 요청 (100 req/s 이하) → 처리
과도한 요청 (100 req/s 초과) → 429 Too Many Requests
```

**알고리즘**

| 알고리즘 | 특징 |
|----------|------|
| Token Bucket | 일정 속도로 토큰 충전, 토큰 있을 때만 요청 허용. 버스트 허용 |
| Sliding Window | 시간 창 내 요청 수 제한. 정확하지만 메모리 사용 |
| Fixed Window | 고정 시간 단위로 카운트. 간단하지만 경계에서 2배 허용 가능 |

## 전체 아키텍처 조감도

```
                        ┌──────────────┐
                        │     CDN      │ ← 정적 리소스 캐싱
                        └──────┬───────┘
                               │
                        ┌──────┴───────┐
                        │ API Gateway  │ ← Rate Limiting, 인증
                        └──────┬───────┘
                               │
                        ┌──────┴───────┐
                        │ Load Balancer│ ← L7 라우팅
                        └──────┬───────┘
                    ┌──────────┼──────────┐
                    │          │          │
                ┌───┴──┐  ┌───┴──┐  ┌───┴──┐
                │ App 1│  │ App 2│  │ App 3│  ← Stateless, Auto Scaling
                └───┬──┘  └───┬──┘  └───┬──┘
                    │         │         │
              ┌─────┴─────────┴─────────┴─────┐
              │           Redis Cluster        │ ← Application Cache
              └─────────────┬──────────────────┘
                            │
                  ┌─────────┴─────────┐
                  │                   │
            ┌─────┴─────┐     ┌──────┴──────┐
            │  Primary  │     │  Replica x2 │ ← Read 분산
            │  (Write)  │     │  (Read)     │
            └───────────┘     └─────────────┘
                  │
            ┌─────┴──────┐
            │ Message Queue│ ← 비동기 처리
            └─────┬──────┘
              ┌───┴───┐
              │Worker  │ ← 알림, 정산, 로그 등이어
              └───────┘
```

## 실무에서 순서

**단계적으로 적용한다.** 처음부터 다 하지 않는다.

| 단계 | 조치 | 효과 |
|------|------|------|
| 1단계 | 캐싱 (Redis, CDN) | DB 부하 급감 |
| 2단계 | Read Replica | 읽기 분산 |
| 3단계 | Auto Scaling | 트래픽 대응 탄력성 |
| 4단계 | 비동기 처리 (MQ) | 응답 시간 단축 |
| 5단계 | 샤딩 | 쓰기 분산 (최후 수단) |

> **냉정한 현실**: 대부분의 서비스는 1~2단계면 충분하다. 샤딩까지 필요한 서비스는 극소수.

---

## 면접에서 이렇게 답하라

> "대용량 트래픽 대응은 3가지 축으로 접근합니다.
> 첫째, **캐싱**으로 트래픽 자체를 줄입니다. CDN으로 정적 리소스를, Redis로 비즈니스 데이터를 캐싱합니다.
> 둘째, **로드밸런싱과 Read Replica**로 트래픽을 분산합니다.
> 셋째, **Auto Scaling과 비동기 처리**로 시스템의 처리 용량을 늘립니다.
> 중요한 건 단계적으로 적용하는 것이고, 캐싱만 잘해도 대부분의 트래픽 문제는 해결됩니다."
