# 대용량 데이터 스트리밍 처리

> **핵심 질문**: 대용량 엑셀 생성이나 DB 전체 조회 시 OOM이 발생하는데, 어떻게 해결하는가?

---

## 한 줄 요약

**"전부 메모리에 올리지 말고, 흘려보내라. DB → 애플리케이션 → 파일/응답 전 구간을 스트림으로 연결하면 메모리 사용량이 O(N)에서 O(1)로 바뀐다."**

---

## 문제의 본질

### OOM이 발생하는 전형적인 코드

```java
// ❌ 전형적인 OOM 패턴
public void exportExcel(HttpServletResponse response) {
    // 1단계: DB에서 전체 데이터를 List로 가져옴 → 메모리 점유 시작
    List<Order> orders = orderRepository.findAll();  // 500만 건 → ~2GB

    // 2단계: 엑셀 워크북을 메모리에 생성 → 메모리 2배 점유
    Workbook workbook = new XSSFWorkbook();  // Apache POI 기본 모드
    Sheet sheet = workbook.createSheet("Orders");
    for (int i = 0; i < orders.size(); i++) {
        Row row = sheet.createRow(i);
        row.createCell(0).setCellValue(orders.get(i).getId());
        row.createCell(1).setCellValue(orders.get(i).getAmount());
        // ...
    }

    // 3단계: 응답으로 쓰기
    workbook.write(response.getOutputStream());
}
```

```
메모리 사용량:
┌─────────────────────────────────────────────────┐
│ DB 전체 조회 결과 (List)          ~2GB          │
│ + 엑셀 워크북 (메모리 내 생성)    ~2GB          │
│ = 총 ~4GB                                       │
│                                                  │
│ JVM Heap: -Xmx2g                                │
│ → java.lang.OutOfMemoryError: Java heap space   │
└─────────────────────────────────────────────────┘
```

**문제는 두 군데에서 발생한다:**

| 구간 | 문제 | 메모리 사용 |
|------|------|-------------|
| **DB → 애플리케이션** | `findAll()`로 전체 데이터를 List에 적재 | O(N) — 데이터 건수에 비례 |
| **애플리케이션 → 파일** | `XSSFWorkbook`이 전체 엑셀을 메모리에 구성 | O(N) — 행 수에 비례 |

## 해결 전략: 전 구간 스트리밍

```
[기존 — 배치 적재]
DB ──전체 로드──▶ List<Order> (메모리) ──전체 생성──▶ Workbook (메모리) ──출력──▶ 응답

[개선 — 스트리밍]
DB ──커서/스트림──▶ 1건씩 읽기 ──즉시 쓰기──▶ 파일/응답 스트림
                    └── 메모리: 현재 처리 중인 1건(또는 청크)만 점유
```

## 1단계: DB 조회를 스트림으로 변경

### 문제: `findAll()`은 ResultSet 전체를 메모리에 올린다

JDBC의 기본 동작은 쿼리 결과를 **한꺼번에 메모리에 적재**한다. 500만 건을 조회하면 500만 건이 전부 힙에 올라간다.

### 방법 1: Spring Data JPA `Stream` 반환

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @QueryHints(@QueryHint(name = HINT_FETCH_SIZE, value = "1000"))
    @Query("SELECT o FROM Order o WHERE o.createdAt BETWEEN :from AND :to")
    Stream<Order> streamByCreatedAtBetween(
        @Param("from") LocalDateTime from,
        @Param("to") LocalDateTime to
    );
}
```

```java
@Transactional(readOnly = true)  // 반드시 트랜잭션 내에서 사용
public void exportWithStream(LocalDateTime from, LocalDateTime to, OutputStream out) {
    try (Stream<Order> stream = orderRepository.streamByCreatedAtBetween(from, to)) {
        stream.forEach(order -> {
            writeRow(out, order);
            entityManager.detach(order);  // 영속성 컨텍스트에서 제거 → GC 대상
        });
    }
}
```

**핵심 포인트:**
- `Stream<T>` 반환 시 JPA는 내부적으로 Hibernate `ScrollableResults`를 사용한다
- **⚠️ MySQL 주의**: JPA Stream을 쓴다고 자동으로 스트리밍되지 않는다. MySQL JDBC 드라이버는 기본적으로 전체 ResultSet을 메모리에 적재하므로, 반드시 아래 MySQL 전용 설정이 필요하다 (아래 테이블 참고)
- **PostgreSQL**: `fetchSize > 0` + `@Transactional`이면 자동으로 서버 사이드 커서 동작
- `@QueryHint(HINT_FETCH_SIZE)`로 네트워크 라운드트립을 줄인다 (1000건씩 가져옴)
- 반드시 `@Transactional` 안에서 사용해야 한다 (커서가 트랜잭션 생명주기에 묶임)
- `entityManager.detach(order)` — 처리 완료 후 영속성 컨텍스트에서 제거하지 않으면 1차 캐시에 계속 쌓여 결국 OOM

### 방법 2: JDBC 커서 직접 사용 (더 낮은 레벨, 더 정밀한 제어)

```java
@Transactional(readOnly = true)
public void exportWithCursor(LocalDateTime from, LocalDateTime to, OutputStream out) {
    jdbcTemplate.query(
        connection -> {
            PreparedStatement ps = connection.prepareStatement(
                "SELECT id, amount, status, created_at FROM orders WHERE created_at BETWEEN ? AND ?",
                ResultSet.TYPE_FORWARD_ONLY,       // 순방향 전용
                ResultSet.CONCUR_READ_ONLY          // 읽기 전용
            );
            ps.setFetchSize(1000);                  // MySQL: Integer.MIN_VALUE로 설정하면 row-by-row 스트리밍
            ps.setTimestamp(1, Timestamp.valueOf(from));
            ps.setTimestamp(2, Timestamp.valueOf(to));
            return ps;
        },
        (ResultSet rs) -> {
            while (rs.next()) {
                writeRow(out, rs.getLong("id"), rs.getBigDecimal("amount"),
                         rs.getString("status"), rs.getTimestamp("created_at"));
            }
        }
    );
}
```

### MySQL vs PostgreSQL 커서 동작 차이

**MySQL에는 두 가지 스트리밍 방식이 있다. 혼동하지 말 것:**

| 방식 | 설정 | 동작 원리 | 특징 |
|------|------|-----------|------|
| **Row-by-row 스트리밍** | `fetchSize = Integer.MIN_VALUE` + `TYPE_FORWARD_ONLY` + `CONCUR_READ_ONLY` | 서버가 결과를 보내고, 클라이언트가 한 행씩 네트워크에서 읽음 | 클라이언트 사이드 처리. 가장 간단 |
| **서버 사이드 커서** | `useCursorFetch=true` + 양수 `fetchSize` | MySQL 서버에 실제 커서 생성, fetchSize만큼씩 가져옴 | 진짜 서버 커서. 더 정밀한 제어 |

| 항목 | MySQL | PostgreSQL |
|------|-------|------------|
| 스트리밍 설정 | 위 두 방식 중 택 1 (기본: 전체 메모리 적재) | `fetchSize > 0`이면 자동 서버 사이드 커서 |
| 기본 동작 | **전체 ResultSet 메모리 적재** (추가 설정 없이 Stream 써도 OOM) | `fetchSize` 설정 시 커서 사용 |
| 주의사항 | JDBC URL 또는 fetchSize 설정 필수 | `autoCommit=false` 필수 (Spring `@Transactional`이면 자동) |

```yaml
# MySQL 서버 사이드 커서 방식
spring:
  datasource:
    url: jdbc:mysql://host:3306/db?useCursorFetch=true&defaultFetchSize=1000

# 또는 Row-by-row 스트리밍 방식 (JDBC 코드에서 fetchSize = Integer.MIN_VALUE 설정)
```

## 2단계: 엑셀 생성을 스트림으로 변경

### 문제: Apache POI `XSSFWorkbook`은 전체 엑셀을 메모리에 구성한다

`XSSFWorkbook`(.xlsx)은 XML 기반으로 전체 워크시트를 메모리의 DOM 트리로 보유한다. 100만 행이면 수 GB를 소비한다.

### 방법 1: Apache POI SXSSF (Streaming Usermodel)

```java
// ✅ SXSSFWorkbook — 메모리에 100행만 유지, 나머지는 임시 파일로 플러시
public void exportExcelStreaming(OutputStream out, Stream<Order> orders) {

    // rowAccessWindowSize = 100: 메모리에 최근 100행만 유지
    try (SXSSFWorkbook workbook = new SXSSFWorkbook(100)) {
        workbook.setCompressTempFiles(true);  // 임시 파일 압축 → 디스크 절약

        Sheet sheet = workbook.createSheet("Orders");

        // 헤더
        Row header = sheet.createRow(0);
        header.createCell(0).setCellValue("주문ID");
        header.createCell(1).setCellValue("금액");
        header.createCell(2).setCellValue("상태");
        header.createCell(3).setCellValue("주문일시");

        AtomicInteger rowNum = new AtomicInteger(1);

        orders.forEach(order -> {
            Row row = sheet.createRow(rowNum.getAndIncrement());
            row.createCell(0).setCellValue(order.getId());
            row.createCell(1).setCellValue(order.getAmount().doubleValue());
            row.createCell(2).setCellValue(order.getStatus().name());
            row.createCell(3).setCellValue(order.getCreatedAt().toString());
        });

        workbook.write(out);

    } finally {
        // ⚠️ 반드시 호출 — 임시 파일 정리
        // SXSSFWorkbook은 /tmp에 임시 파일을 생성하므로 미호출 시 디스크 누수
        // POI 4.x+ 에서는 close()가 dispose()를 호출하지만, 명시적 호출 권장
        workbook.dispose();
    }
}
```

```
메모리 사용량 비교:

XSSFWorkbook (기본):
500만 행 × ~400byte/행 = ~2GB 메모리

SXSSFWorkbook (스트리밍):
100행 × ~400byte/행 = ~40KB 메모리 + 임시 파일 (디스크)
```

### 방법 2: Alibaba EasyExcel (더 가볍고 간편)

```java
// EasyExcel — POI보다 메모리 효율적, API가 단순
public void exportWithEasyExcel(OutputStream out, Stream<Order> orders) {

    ExcelWriter writer = EasyExcel.write(out, OrderExcelDto.class).build();
    WriteSheet sheet = EasyExcel.writerSheet("Orders").build();

    List<OrderExcelDto> buffer = new ArrayList<>(1000);

    orders.forEach(order -> {
        buffer.add(OrderExcelDto.from(order));

        if (buffer.size() >= 1000) {
            writer.write(buffer, sheet);  // 1000건씩 플러시
            buffer.clear();
        }
    });

    if (!buffer.isEmpty()) {
        writer.write(buffer, sheet);  // 잔여 데이터 플러시
    }

    writer.finish();
}

@Data
public class OrderExcelDto {
    @ExcelProperty("주문ID")
    private Long id;

    @ExcelProperty("금액")
    private BigDecimal amount;

    @ExcelProperty("상태")
    private String status;

    @ExcelProperty("주문일시")
    private String createdAt;

    public static OrderExcelDto from(Order order) {
        OrderExcelDto dto = new OrderExcelDto();
        dto.setId(order.getId());
        dto.setAmount(order.getAmount());
        dto.setStatus(order.getStatus().name());
        dto.setCreatedAt(order.getCreatedAt().toString());
        return dto;
    }
}
```

| 라이브러리 | 100만 행 메모리 | 특징 |
|------------|----------------|------|
| **XSSFWorkbook** | ~2GB | 전체 메모리 적재, 소규모만 |
| **SXSSFWorkbook** | ~10MB | 슬라이딩 윈도우, 임시 파일 사용 |
| **EasyExcel** | ~5MB | Alibaba, 가장 가벼움, API 간단 |

## 전체 파이프라인 — 조합

```java
@GetMapping("/orders/export")
@Transactional(readOnly = true)
public void exportOrders(
    @RequestParam LocalDateTime from,
    @RequestParam LocalDateTime to,
    HttpServletResponse response
) throws IOException {

    // 1. 응답 헤더 설정
    response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet");
    response.setHeader("Content-Disposition", "attachment; filename=orders.xlsx");

    // 2. DB → Stream → Excel → 응답 (전 구간 스트리밍)
    try (Stream<Order> orders = orderRepository.streamByCreatedAtBetween(from, to)) {
        exportExcelStreaming(response.getOutputStream(), orders);
    }
    // Stream이 닫히면 커서도 닫힘
    // 트랜잭션이 끝나면 DB 커넥션도 반환
}
```

```
메모리 사용 흐름:

DB ──커서(1000건씩)──▶ Stream<Order>
                        │
                        ├── 1건 읽기
                        ├── Excel 행 생성 (SXSSFWorkbook, 100행 윈도우)
                        ├── detach(order) → GC 대상
                        └── 반복

메모리에 동시 존재하는 데이터:
- DB 페치 버퍼: ~1000건 × 400byte = ~400KB
- Excel 윈도우: ~100행 = ~40KB
- 합계: ~500KB (데이터 500만 건이어도 동일)
```

## 추가 고려사항

### 1. 영속성 컨텍스트 메모리 누수

```java
// ❌ detach 안 하면 영속성 컨텍스트가 계속 커짐
stream.forEach(order -> writeRow(out, order));
// 500만 건 처리 후 → 영속성 컨텍스트에 500만 개 엔티티 → OOM

// ✅ 처리 후 즉시 detach
stream.forEach(order -> {
    writeRow(out, order);
    entityManager.detach(order);
});

// ✅ 또는 StatelessSession 사용 (Hibernate)
// 영속성 컨텍스트 자체를 사용하지 않음
Session session = entityManagerFactory.unwrap(SessionFactory.class).openStatelessSession();
```

### 2. 커넥션 점유 시간

```
스트리밍 = DB 커넥션을 오래 점유한다

500만 건 처리 × 행당 0.01ms = ~50초 커넥션 점유
→ 커넥션 풀이 10개인데 동시에 5명이 다운로드하면?
→ 나머지 5개 커넥션으로 전체 서비스 운영

대응:
1. 다운로드 전용 DataSource 분리 (Replica 사용)
2. 동시 다운로드 수 제한 (Semaphore / Rate Limiting)
3. 비동기 처리: 요청 → 백그라운드에서 파일 생성 → S3 업로드 → 다운로드 링크 제공
```

### 3. 대용량이면 비동기 + S3 패턴이 정석

```
[동기 스트리밍 — 10만 건 이하]
사용자 ──요청──▶ 서버 ──스트리밍──▶ 브라우저 다운로드

[비동기 + S3 — 10만 건 초과]
사용자 ──요청──▶ API (202 Accepted, "생성 중")
                  │
           백그라운드 Worker
                  │
            파일 생성 → S3 업로드
                  │
            완료 알림 (이메일/Push/Polling)
                  │
사용자 ──다운로드──▶ S3 Pre-signed URL (서버 부하 없음)
```

```java
// 비동기 엑셀 생성
// ⚠️ 주의: @Async 메서드는 같은 클래스 내 호출 시 프록시를 타지 않아 비동기가 안 됨 (self-invocation)
//         @Async void의 예외는 호출자에게 전파되지 않으므로 AsyncUncaughtExceptionHandler 설정 필요
@Async
@Transactional(readOnly = true)
public void generateExcelAsync(Long requestId, LocalDateTime from, LocalDateTime to) {
    Path tempFile = Files.createTempFile("export-", ".xlsx");

    try (Stream<Order> orders = orderRepository.streamByCreatedAtBetween(from, to);
         OutputStream out = new BufferedOutputStream(Files.newOutputStream(tempFile))) {

        exportExcelStreaming(out, orders);

        // S3 업로드
        String s3Key = "exports/" + requestId + ".xlsx";
        s3Client.putObject(bucket, s3Key, tempFile.toFile());

        // Pre-signed URL 생성 (1시간 유효)
        // ⚠️ 보안 주의: URL 유출 시 누구나 다운로드 가능 (인증 우회)
        //   - 민감 데이터면 만료 시간을 짧게 (5~15분)
        //   - 다운로드 로그/감사 추적 필수
        //   - HTTPS 강제
        String downloadUrl = s3Client.generatePresignedUrl(bucket, s3Key, Duration.ofHours(1));

        // 사용자에게 알림
        notificationService.notify(requestId, downloadUrl);

    } finally {
        Files.deleteIfExists(tempFile);  // 임시 파일 정리
    }
}
```

### 4. CSV가 답일 수 있다

```
엑셀(.xlsx)이 반드시 필요한가?

CSV 장점:
- 스트리밍이 자연스럽다 (한 줄씩 쓰면 됨)
- 라이브러리 불필요 (문자열 조합만)
- 메모리 사용 거의 없음
- 생성 속도 엑셀 대비 5~10배 빠름

CSV 단점:
- 셀 서식, 수식, 시트 분리 불가
- 한글 인코딩 주의 (BOM 필요: UTF-8 with BOM)

실무 판단:
- 운영팀 내부용 → CSV 충분
- 고객/파트너 전달용 → 엑셀 필요
```

```java
// CSV 스트리밍 — 가장 가벼운 방법
@GetMapping("/orders/export-csv")
@Transactional(readOnly = true)
public void exportCsv(
    @RequestParam LocalDateTime from,
    @RequestParam LocalDateTime to,
    HttpServletResponse response
) throws IOException {
    response.setContentType("text/csv; charset=UTF-8");
    response.setHeader("Content-Disposition", "attachment; filename=orders.csv");

    PrintWriter writer = response.getWriter();
    writer.write('\uFEFF');  // UTF-8 BOM (엑셀에서 한글 깨짐 방지)
    writer.println("주문ID,금액,상태,주문일시");

    try (Stream<Order> orders = orderRepository.streamByCreatedAtBetween(from, to)) {
        orders.forEach(order -> {
            writer.printf("%d,%s,%s,%s%n",
                order.getId(), order.getAmount(),
                order.getStatus(), order.getCreatedAt());
            entityManager.detach(order);
        });
    }
}
```

## 면접에서 이렇게 답하자

### Q. 대용량 엑셀 다운로드에서 OOM이 발생합니다. 어떻게 해결하시겠습니까?

> "OOM의 원인은 두 가지입니다. 첫째, DB 조회 결과를 `findAll()`로 전체 List에 적재하는 것. 둘째, Apache POI `XSSFWorkbook`이 전체 엑셀을 메모리에 구성하는 것입니다.
>
> 해결은 **전 구간을 스트림으로 연결**합니다. DB 조회는 JPA `Stream<T>` 반환과 `fetchSize` 힌트로 서버 사이드 커서를 사용하고, 엑셀 생성은 `SXSSFWorkbook`이나 EasyExcel로 슬라이딩 윈도우 방식으로 변경합니다. 처리한 엔티티는 `entityManager.detach()`로 영속성 컨텍스트에서 즉시 제거합니다.
>
> 이렇게 하면 500만 건이어도 메모리 사용량이 수백 KB 수준으로 일정합니다.
>
> 다만 스트리밍은 DB 커넥션을 오래 점유하므로, 10만 건 이상이면 **비동기 + S3 패턴**을 사용합니다. 백그라운드 Worker가 파일을 생성하여 S3에 업로드하고, Pre-signed URL로 다운로드하게 하면 서버 부하도 없습니다."

### Q. MySQL에서 스트리밍 조회 시 주의할 점은?

> "MySQL JDBC 드라이버는 기본적으로 ResultSet 전체를 메모리에 적재합니다. 서버 사이드 커서를 사용하려면 `fetchSize = Integer.MIN_VALUE`로 설정하거나, JDBC URL에 `useCursorFetch=true`를 추가하고 `fetchSize`를 양수로 설정해야 합니다. PostgreSQL은 `fetchSize > 0`이면 자동으로 커서를 사용하지만, `autoCommit=false` 상태여야 합니다. Spring의 `@Transactional`이면 자동으로 해결됩니다."
