# 스파이크 트래픽 대처

> **핵심 질문**: 트래픽이 갑자기 폭증하는 스파이크 상황에 어떻게 대처할 것인가?

---

## 한 줄 요약

**"예측 가능한 스파이크는 준비하고, 예측 불가능한 스파이크는 버리면서 버틴다."**

[대용량-트래픽-대응](./대용량-트래픽-대응.md)이 "평소 많은 트래픽"에 대한 설계라면, 스파이크 대처는 **순간적인 폭증**에 살아남는 전략이다.

---

## 스파이크 트래픽 vs 일반 대용량 트래픽

| 구분 | 일반 대용량 | 스파이크 |
|------|-------------|----------|
| 패턴 | 점진적 증가 | 수초~수분 내 10~100배 폭증 |
| 예측 | 가능 | 부분적 가능 or 불가 |
| 대응 시간 | 여유 있음 | 거의 없음 |
| 예시 | 일 1000만 PV 서비스 | 자정 선착순 쿠폰, 수능 성적 발표 |

```
일반 대용량:          스파이크:
    ───────────           │
   /                      │
  /                 ──────┘
 /
─                   ──────
  (점진적 증가)       (순간 폭증)
```

## 전략 1: 사전 준비 — 예측 가능한 스파이크

### 1-1. Pre-Warming (사전 워밍업)

Auto Scaling의 치명적 약점: **새 인스턴스가 뜨는 데 시간이 걸린다.**

```
스파이크 발생 → Auto Scaling 감지 (30초) → 인스턴스 기동 (1~10분+, 환경별 상이) → 헬스체크 (30초)
              ↑
              이 2분 동안 기존 서버가 죽을 수 있다
```

**해결: 미리 서버를 띄워놓는다.**

```
// Scheduled Scaling (AWS)
// 자정 쿠폰 이벤트 → 23:50에 미리 서버 증설
{
  "ScheduledActions": [{
    "ScheduledActionName": "coupon-event-scale-out",
    "StartTime": "2026-03-22T23:50:00Z",
    "DesiredCapacity": 20,   // 평소 4대 → 20대
    "MinSize": 20
  }]
}
```

- 이벤트, 프로모션, 정기 패턴(출퇴근 시간) → **Scheduled Scaling**으로 미리 준비
- JVM 기반이면 JIT 컴파일 워밍업도 필요 (첫 요청들이 느림)
- Redis, DB 커넥션 풀도 미리 채워두기 (Eager Initialization)

### 1-2. Connection Pool Pre-warming

```
평소: DB Connection Pool = min 10, max 50
스파이크 직전: min 50, max 100으로 미리 올려놓기

// HikariCP 예시
spring:
  datasource:
    hikari:
      minimum-idle: 50      # 미리 50개 연결 확보
      maximum-pool-size: 100
      connection-timeout: 3000  # 3초 내 연결 못하면 빠르게 실패
```

커넥션을 새로 만드는 건 비싸다 (TCP handshake + DB 인증). 스파이크 때 커넥션 생성이 몰리면 DB가 먼저 죽는다.

## 전략 2: 트래픽 제어 — 들어오는 양을 조절한다

### 2-1. Rate Limiting (요청 제한)

서버가 감당할 수 있는 양만 받고, 나머지는 거부한다.

```
처리 가능: 10,000 req/s
실제 유입: 100,000 req/s

→ 10,000개 처리, 90,000개는 429 Too Many Requests
→ 서버는 살아있다. 일부만 실패한다.
→ 서버가 죽으면? 100% 실패한다.
```

**핵심 원칙: 일부를 버려서 전체를 살린다.**

### 2-2. 대기열 (Queue-Based Load Leveling)

선착순 이벤트의 정석 패턴.

```
[100만 사용자] → [대기열 진입] → [순서대로 처리] → [결과 반환]
                     │
                "현재 32,847번째입니다"
                 (WebSocket/Polling으로 순번 안내)
```

```
// 구현 흐름
1. 사용자 요청 → Redis Sorted Set에 timestamp로 입장
   ZADD waiting_queue {timestamp} {userId}

2. 순번 확인
   ZRANK waiting_queue {userId}  → 32847

3. Worker가 앞에서부터 꺼내서 처리
   ZPOPMIN waiting_queue 100  → 100명씩 처리

4. 처리 완료 → 결과 알림
```

- **네이버 예약**, **티켓팅**, **쿠팡 로켓와우** 등이 이 패턴
- 사용자 경험도 좋다: "기다리는 중"이 "접속 불가"보다 100배 낫다
- → [선착순-쿠폰-유실-대응](../failure/선착순-쿠폰-유실-대응.md)에서 더 깊이 다룸

### 2-3. Throttling (처리율 조절)

Rate Limiting은 "거부", Throttling은 "속도 조절".

| 구분 | Rate Limiting | Throttling |
|------|---------------|------------|
| 초과 시 | 거부 (429) | 지연 처리 or 품질 저하 |
| 사용자 경험 | 에러 | 느리지만 동작 |
| 적합 상황 | API, 봇 방어 | 사용자 대면 서비스 |

**Graceful Degradation** (품질 저하 허용):
```
정상: 상품 상세 + 리뷰 + 추천 + 실시간 재고
스파이크: 상품 상세 + 캐시된 리뷰 (추천, 실시간 재고 생략)
```

핵심 기능만 살리고 부가 기능은 끈다.

## 전략 3: 비동기 버퍼링 — 순간을 평탄화한다

### Message Queue로 Peak Shaving

```
Without MQ:
    ┃█████████┃  ← 순간 10만 req → DB 직접 타격 → DB 사망
    ┃         ┃
────┗━━━━━━━━━┛────

With MQ:
    ┃█████████┃  ← 순간 10만 req → MQ에 적재
    ┃ ░░░░░░░ ┃  ← MQ에서 1만 req/s씩 꺼내서 처리
    ┃ ░░░░░░░ ┃
────┗━━━━━━━━━┛────
```

**스파이크를 MQ가 흡수하고, Consumer가 일정한 속도로 처리한다.**

```
// Producer (API 서버)
// 즉시 응답: "요청 접수됨" (202 Accepted)
kafkaTemplate.send("coupon-request", CouponEvent(userId, couponId))
return ResponseEntity.accepted().body("요청이 접수되었습니다")

// Consumer (Worker)
// 일정 속도로 처리
@KafkaListener(topics = ["coupon-request"], concurrency = "5")
fun processCoupon(event: CouponEvent) {
    couponService.issue(event.userId, event.couponId)
}
```

→ 자세한 내용: [Kafka](../architecture/Kafka.md)

## 전략 4: 인프라 레벨 방어

### 4-1. Circuit Breaker

장애가 전파되지 않도록 차단한다.

```
상태 전이:
CLOSED (정상) ──실패율 50% 초과 (Resilience4j 기본값; 실무에서는 5~20%로 조정)──→ OPEN (차단)
                                     │
                                  일정 시간 후
                                     │
                                     ↓
                                HALF-OPEN (시험)
                                     │
                              ┌──────┴──────┐
                           성공 →          실패 →
                          CLOSED           OPEN
```

```java
// Resilience4j 예시
@CircuitBreaker(name = "productService", fallbackMethod = "getProductFallback")
public Product getProduct(Long id) {
    return productClient.getProduct(id);  // 외부 호출
}

// 장애 시 대체 응답
public Product getProductFallback(Long id, Exception e) {
    return cachedProductService.get(id);  // 캐시에서 반환
}
```

- 외부 서비스 장애가 내 서비스를 죽이지 않도록
- → [외부-API-장애-대응](../failure/외부-API-장애-대응.md)에서 심화

### 4-2. Bulkhead (격벽 패턴)

서비스 내부를 격리해서, 하나가 죽어도 전체가 안 죽게 한다.

```
Before (공유 스레드풀):
결제 API ──┐
상품 API ──┼── Thread Pool (100) ← 결제 지연 → 100개 다 점유 → 상품도 죽음
리뷰 API ──┘

After (격리 스레드풀):
결제 API ── Thread Pool (40) ← 결제 지연 → 40개만 점유
상품 API ── Thread Pool (40) ← 정상 동작
리뷰 API ── Thread Pool (20) ← 정상 동작
```

### 4-3. Timeout 설정

**모든 외부 호출에 Timeout은 필수다.**

```yaml
# 적정 Timeout 설정 예시
connection-timeout: 1s     # 연결 수립
read-timeout: 3s           # 응답 대기
write-timeout: 3s          # 요청 전송

# 절대 하면 안 되는 것
timeout: 30s  # 이러면 스레드가 30초간 점유 → 스레드 고갈 → 서버 사망
timeout: 0    # 무한 대기 → 확실한 죽음
```

## 전략 5: 프론트엔드 협력

백엔드만으로 해결하지 않는다.

| 방법 | 설명 |
|------|------|
| 버튼 중복 클릭 방지 | 클릭 후 비활성화 (debounce) |
| 대기열 UI | "현재 N번째" 안내 → 새로고침 폭탄 방지 |
| 정적 페이지 전환 | 장애 시 S3 정적 페이지로 라우팅 |
| 클라이언트 캐싱 | SWR, React Query의 stale-while-revalidate |

## 실전 체크리스트

스파이크가 예상될 때:

```
□ 사전 준비
  ├─ [ ] Scheduled Scaling으로 서버 미리 증설
  ├─ [ ] Connection Pool 사이즈 확대
  ├─ [ ] Redis 캐시 워밍업 (주요 데이터 미리 적재)
  └─ [ ] JVM Warm-up (트래픽 소량 흘려보내기)

□ 트래픽 제어
  ├─ [ ] Rate Limiting 임계치 설정
  ├─ [ ] 대기열 시스템 준비 (선착순 이벤트)
  └─ [ ] Graceful Degradation 시나리오 정의

□ 인프라 방어
  ├─ [ ] Circuit Breaker 설정 확인
  ├─ [ ] Timeout 값 점검 (너무 길지 않은지)
  ├─ [ ] Bulkhead 격리 확인
  └─ [ ] 모니터링 대시보드 & 알람 설정

□ 비상 대응
  ├─ [ ] 수동 스케일링 절차 문서화
  ├─ [ ] 긴급 기능 OFF 스위치 (Feature Flag)
  └─ [ ] 롤백 계획 수립
```

## 면접에서 이렇게 답하라

> "스파이크 트래픽은 일반적인 Auto Scaling으로는 대응이 늦습니다.
> 먼저 **예측 가능한 스파이크**는 Scheduled Scaling으로 미리 서버를 띄우고, 커넥션 풀과 캐시를 워밍업합니다.
> **예측 불가능한 스파이크**는 Rate Limiting으로 과도한 요청을 거부하고, Message Queue로 피크를 평탄화합니다.
> 선착순 이벤트라면 대기열 패턴으로 순서를 보장하면서 시스템을 보호합니다.
> 그리고 Circuit Breaker와 Bulkhead로 장애 전파를 차단하고, Graceful Degradation으로 핵심 기능만이라도 살립니다.
> 핵심은 **일부를 버려서 전체를 살리는 것**입니다."
