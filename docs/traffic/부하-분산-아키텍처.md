# 부하 분산 아키텍처

> **핵심 질문**: 부하 분산은 어떻게 설계하는가?

---

## 한 줄 요약

**"한 곳에 몰리는 부하를 여러 곳으로 나누는 모든 기법의 총칭."**

부하 분산은 단순히 로드밸런서 하나 두는 게 아니다. **네트워크 → 애플리케이션 → 데이터베이스** 각 계층마다 분산 전략이 다르다.

---

## 전체 구조 — 계층별 부하 분산

```
[사용자]
   │
   ▼
┌──────────┐
│   DNS    │  ← ① DNS 레벨 분산 (지역별 라우팅)
└────┬─────┘
     │
┌────▼─────┐
│   CDN    │  ← ② 정적 리소스 분산 (Edge 캐싱)
└────┬─────┘
     │
┌────▼─────┐
│ L4 / L7  │  ← ③ 네트워크 레벨 분산 (로드밸런서)
│    LB    │
└────┬─────┘
     │
┌────▼──────────────────┐
│  App Server 1,2,3...  │  ← ④ 애플리케이션 레벨 분산 (Scale-Out)
└────┬──────────────────┘
     │
┌────▼──────────────────┐
│  Cache (Redis Cluster) │  ← ⑤ 캐시 레벨 분산
└────┬──────────────────┘
     │
┌────▼──────────────────┐
│  DB Primary + Replica  │  ← ⑥ DB 레벨 분산 (Read/Write 분리)
│  + Sharding            │     ⑦ 데이터 레벨 분산 (샤딩)
└───────────────────────┘
```

---

## ① DNS 레벨 분산

사용자와 가장 가까운 서버로 보낸다.

### Route 53 라우팅 정책 (AWS)

| 정책 | 설명 | 용도 |
|------|------|------|
| Simple | 단일 IP 반환 | 소규모 서비스 |
| Weighted | 가중치로 트래픽 비율 조절 | 카나리 배포 (신규 서버 10%만) |
| Latency-based | 지연시간 가장 낮은 리전 | 글로벌 서비스 |
| Geolocation | 사용자 위치 기반 | 법적 요구 (데이터 주권) |
| Failover | 헬스체크 실패 시 백업으로 | DR (Disaster Recovery) |

```
한국 사용자  → ap-northeast-2 (서울)
미국 사용자  → us-east-1 (버지니아)
유럽 사용자  → eu-west-1 (아일랜드)
서울 리전 장애 → 자동으로 도쿄 리전으로 Failover
```

## ② CDN — Edge 분산

원본 서버까지 안 가게 한다. 이게 가장 효과적인 부하 분산이다.

```
Without CDN:
서울 사용자 ──────────────────────→ 미국 원본 서버 (200ms)
부산 사용자 ──────────────────────→ 미국 원본 서버 (200ms)

With CDN:
서울 사용자 ──→ 서울 Edge (5ms)  [캐시 HIT]
부산 사용자 ──→ 부산 Edge (5ms)  [캐시 HIT]
                                  ↓ 캐시 MISS일 때만
                               미국 원본 서버
```

| CDN 서비스 | 특징 |
|------------|------|
| CloudFront (AWS) | AWS 생태계 통합 |
| Cloudflare | DDoS 방어 강력, 무료 플랜 |
| Akamai | 엔터프라이즈, 가장 큰 네트워크 |

**CDN에 태울 수 있는 것**: 이미지, JS/CSS, 폰트, API 응답(짧은 TTL)
**CDN에 태우면 안 되는 것**: 사용자별 다른 데이터, 실시간 데이터

## ③ 로드밸런서 심화

### L4 vs L7 선택 기준

```
L4 (Transport Layer):
┌────────────────────────────┐
│ TCP/UDP 패킷 기준으로 분산  │
│ IP + Port만 봄             │
│ 빠르다, 단순하다           │
│ SSL Termination 불가       │
└────────────────────────────┘
→ 용도: 내부 서비스 간 통신, TCP 기반 프로토콜 (gRPC, DB)

L7 (Application Layer):
┌────────────────────────────┐
│ HTTP 요청 내용 기준으로 분산 │
│ URL, Header, Cookie 등 파싱 │
│ 느리지만 유연하다           │
│ SSL Termination 가능        │
└────────────────────────────┘
→ 용도: API 라우팅, A/B 테스트, 카나리 배포
```

### L7 경로 기반 라우팅 실전 예시

```
/api/v1/products/*  → Product Service (서버 그룹 A)
/api/v1/orders/*    → Order Service (서버 그룹 B)
/api/v1/payments/*  → Payment Service (서버 그룹 C)
/static/*           → S3 + CloudFront (서버 필요 없음)
```

서비스별로 독립적으로 스케일링 가능하다. 결제만 트래픽이 몰리면 Payment 서버만 늘리면 된다.

### 로드밸런서 알고리즘 심화

| 알고리즘 | 동작 | 장점 | 단점 |
|----------|------|------|------|
| **Round Robin** | 순서대로 | 단순, 균등 | 서버 상태 무시 |
| **Weighted RR** | 가중치 비례 | 서버 스펙 차이 반영 | 수동 가중치 관리 |
| **Least Connections** | 연결 적은 서버로 | 처리시간 다양할 때 유리 | 새 서버에 몰림 |
| **Least Response Time** | 응답 빠른 서버로 | 실제 성능 반영 | 측정 오버헤드 |
| **IP Hash** | 같은 IP → 같은 서버 | 세션 유지 | 서버 추가/제거 시 재분배 |
| **Consistent Hashing** | 해시링 기반 | 서버 변경 시 최소 재분배 | 구현 복잡 |

**실무 선택 가이드**:
- 대부분: **Round Robin** or **Least Connections**
- Stateful 서비스(WebSocket): **IP Hash** or **Sticky Session** (가급적 피하라)
- 캐시 서버 분산: **Consistent Hashing** (→ [샤딩과-레플리카](./database/샤딩과-레플리카.md)에서 다룸)

### Health Check

로드밸런서가 죽은 서버에 트래픽을 보내지 않도록.

```
ALB Health Check 설정:

Target:     GET /health
Interval:   10초
Timeout:    5초
Healthy:    연속 3회 성공 → 정상
Unhealthy:  연속 2회 실패 → 제외
```

```java
// /health 엔드포인트 예시
@GetMapping("/health")
public ResponseEntity<Map<String, String>> health() {
    // 단순 200 OK가 아니라 핵심 의존성도 체크
    boolean dbHealthy = checkDatabase();
    boolean redisHealthy = checkRedis();

    if (dbHealthy && redisHealthy) {
        return ResponseEntity.ok(Map.of("status", "UP"));
    }
    return ResponseEntity.status(503)
        .body(Map.of("status", "DOWN"));
}
```

**주의**: Health Check에 무거운 로직 넣지 마라. DB 커넥션 확인 정도만. 비즈니스 로직 넣으면 Health Check 자체가 부하가 된다.

## ④ 애플리케이션 레벨 분산

### Stateless 설계 — Scale-Out의 전제 조건

```
Stateful (Scale-Out 불가):
Server 1: 세션에 장바구니 저장 ← 이 서버 죽으면 장바구니 사라짐
Server 2: 이 서버에는 장바구니 없음

Stateless (Scale-Out 가능):
Server 1 ─┐
Server 2 ─┼── Redis에 세션 저장 ← 어느 서버로 가도 동일
Server 3 ─┘
```

**외부로 빼야 할 것들**:

| 상태 | 저장소 |
|------|--------|
| 세션 | Redis |
| 파일 업로드 | S3 |
| 설정 | 환경변수 or Config Server |
| 캐시 | Redis (Local Cache는 보조만) |
| 스케줄링 | 분산 스케줄러 (ShedLock, Quartz Cluster) |

### 서비스별 독립 스케일링

```
평소:
Product Service: 3대
Order Service:   3대
Payment Service: 2대

프로모션 기간:
Product Service: 10대  ← 상품 조회 폭증
Order Service:   8대   ← 주문 폭증
Payment Service: 2대   ← 변화 없음 (PG사가 병목이라 늘려봐야 의미 없음)
```

각 서비스가 독립적으로 스케일링되어야 자원을 효율적으로 쓴다. → [MSA와-메시지큐](./architecture/MSA와-메시지큐.md)

## ⑤ 캐시 레벨 분산

### Redis Cluster

```
단일 Redis:
모든 요청 → Redis 1대 → 메모리 한계, 단일 장애점

Redis Cluster:
요청 A → Shard 1 (key hash 0~5460)
요청 B → Shard 2 (key hash 5461~10922)
요청 C → Shard 3 (key hash 10923~16383)
```

- 16384개 슬롯(slot)을 여러 노드에 분배
- 각 노드에 Replica가 있어서 Master 죽으면 자동 승격
- 클라이언트가 어느 노드에 키가 있는지 알아야 함 (MOVED redirect)

### Local Cache + Remote Cache 조합

```
요청 → Local Cache (Caffeine) → Redis → DB
         (1ms, 서버 메모리)      (3ms)   (10ms+)

- Hot Data: Local Cache에 짧은 TTL (10~30초)
- Warm Data: Redis에 중간 TTL (5~30분)
- Cold Data: DB 직접 조회
```

**주의**: Local Cache는 서버마다 다른 값을 가질 수 있다 (정합성 문제). 짧은 TTL로 허용 가능한 범위 내에서만 사용.

## ⑥⑦ DB 레벨 분산

### Read/Write 분리

```java
// Spring에서 AbstractRoutingDataSource로 구현
public class RoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
            ? "replica"
            : "primary";
    }
}

// 사용
@Transactional(readOnly = true)  // → Replica로 라우팅
public Product getProduct(Long id) { ... }

@Transactional  // → Primary로 라우팅
public void createOrder(Order order) { ... }
```

## 실전 아키텍처 패턴

### 패턴 1: 중소규모 (DAU 10만 이하)

```
CloudFront → ALB → App x2~4 → Redis → RDS (Primary + Replica x1)
```

- Auto Scaling 설정
- 이 정도면 대부분의 서비스가 충분하다

### 패턴 2: 중대규모 (DAU 100만)

```
CloudFront → ALB ──→ API Gateway
                        ├── Product Service (x5)
                        ├── Order Service (x5)
                        └── Payment Service (x3)
                              │
                    Redis Cluster (3 shard)
                              │
                    Aurora (Primary + Replica x2)
                              │
                    Kafka → Worker (비동기 처리)
```

### 패턴 3: 대규모 (DAU 1000만+)

```
Multi-Region DNS → CDN → API Gateway
                           │
            ┌──────────────┼──────────────┐
       Region A        Region B       Region C
     (서울)           (도쿄)          (버지니아)
         │               │               │
    Service Mesh     Service Mesh    Service Mesh
         │               │               │
    DB (Shard 1~3)  DB (Shard 4~6)  DB (Shard 7~9)
```

- 글로벌 분산
- Cross-Region Replication
- 이 수준이 필요한 서비스는 한국에 손가락으로 꼽는다

## 흔한 실수들

| 실수 | 왜 문제인가 | 올바른 접근 |
|------|-------------|-------------|
| Sticky Session 남용 | Scale-Out 효과 반감, 서버 죽으면 세션 유실 | Stateless + Redis 세션 |
| Health Check 없이 LB 운영 | 죽은 서버에 트래픽 전송 | 반드시 Health Check 설정 |
| 모든 것을 샤딩 | 운영 복잡도 폭증, Join 불가 | 캐싱 → Replica → 샤딩 순서 |
| Local Cache만 의존 | 서버 간 데이터 불일치 | Remote Cache 메인, Local은 보조 |
| DB Connection Pool 기본값 사용 | 부하 시 커넥션 고갈 | 서비스 특성에 맞게 튜닝 |

## 면접에서 이렇게 답하라

> "부하 분산은 계층별로 접근합니다.
> **DNS 레벨**에서 지역 기반 라우팅으로 가장 가까운 서버로 보내고,
> **CDN**으로 정적 리소스를 Edge에서 처리해 원본 서버 부하를 줄입니다.
> **L7 로드밸런서**로 서비스별 경로 기반 라우팅을 하고,
> **애플리케이션**은 Stateless로 설계해서 자유롭게 Scale-Out합니다.
> **캐시**는 Local + Redis 조합으로 DB 부하를 줄이고,
> **DB**는 Read/Write 분리로 읽기를 분산합니다.
> 중요한 건 순서입니다. CDN → 캐싱 → Replica → 서버 증설 → 샤딩.
> 앞단에서 해결할수록 비용 대비 효과가 큽니다."
