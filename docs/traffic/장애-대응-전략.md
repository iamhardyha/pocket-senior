# 장애 대응 전략

> **핵심 질문**: 장애 대응 어떻게 하는지?

---

## 한 줄 요약

**"감지 → 공유 → 완화 → 복구 → 회고. 이 순서를 지키면 된다."**

장애 대응은 기술이 아니라 **프로세스**다. 개인 역량이 아니라 **팀 체계**로 해결한다.

---

## 장애 대응 타임라인

```
장애 발생
  │
  ▼ (0~1분)
① 감지 (Detection)
  │  - 모니터링 알람 발동
  │  - 사용자 신고
  │
  ▼ (1~5분)
② 공유 (Communication)
  │  - 온콜 담당자 확인
  │  - 장애 채널 오픈 (Slack #incident-xxx)
  │  - 관련팀 소집
  │
  ▼ (5~15분)
③ 완화 (Mitigation)
  │  - 원인 분석보다 서비스 복구 우선
  │  - 롤백, 트래픽 차단, 기능 OFF
  │
  ▼ (15분~)
④ 복구 (Recovery)
  │  - 근본 원인 파악
  │  - 수정 배포 or 우회
  │  - 정상 확인
  │
  ▼ (1~3일 이내)
⑤ 회고 (Post-mortem)
     - 타임라인 정리
     - 근본 원인 분석
     - 재발 방지 액션 아이템
```

---

## ① 감지 (Detection) — 장애를 빨리 아는 것이 절반이다

### 모니터링 4대 지표 (Golden Signals)

Google SRE 책에서 정의한 4가지 핵심 지표:

| 지표 | 설명 | 알람 기준 예시 |
|------|------|----------------|
| **Latency** | 응답 시간 | p99 > 3초 |
| **Traffic** | 요청량 | QPS 급등/급락 |
| **Errors** | 에러율 | 5xx > 1% |
| **Saturation** | 자원 포화도 | CPU > 80%, Memory > 85% (예시, 서비스별 조정) |

```
정상:  Error Rate 0.1% / Latency p99 200ms / CPU 40%
경고:  Error Rate 1%   / Latency p99 1s    / CPU 70%   ← 슬랙 알림
위험:  Error Rate 5%   / Latency p99 5s    / CPU 90%   ← 전화/PagerDuty
장애:  Error Rate 30%  / Latency p99 30s   / CPU 100%  ← 전원 소집
```

### 모니터링 스택 예시

```
[Application]
    │
    ├── Metrics → Prometheus → Grafana (대시보드)
    │                            └── AlertManager → Slack/PagerDuty
    │
    ├── Logs → ELK (Elasticsearch + Logstash + Kibana)
    │          or Loki + Grafana
    │
    ├── Traces → Jaeger or Zipkin (분산 추적)
    │
    └── APM → Datadog / New Relic (올인원)
```

**실무 포인트**:
- 알람은 **Actionable**해야 한다. 알람이 울리면 누군가가 뭔가를 해야 하는 상황이어야 한다
- 알람이 너무 많으면 무시하게 된다 (Alert Fatigue) → 진짜 중요한 것만 남겨라
- **급락도 장애다**. 트래픽이 갑자기 0이 되면? → 앞단 로드밸런서나 DNS 문제

---

## ② 공유 (Communication) — 혼자 해결하려 하지 마라

### 장애 대응 역할

| 역할 | 책임 |
|------|------|
| **Incident Commander (IC)** | 전체 지휘, 의사결정 |
| **Tech Lead** | 기술적 원인 분석, 수정 방향 결정 |
| **Communicator** | 이해관계자/고객에게 상황 공유 |
| **Operator** | 실제 조치 실행 (배포, 설정 변경) |

```
장애 채널 첫 메시지 템플릿:

🚨 [Incident] 결제 서비스 장애
- 시작 시각: 2026-03-22 14:30 KST
- 영향 범위: 결제 실패율 30%
- 현재 상태: 조사 중
- IC: @홍길동
- 관련 대시보드: [링크]
```

**실무 포인트**:
- 5분 안에 공유하지 않으면 늦은 것이다
- 상태 업데이트는 10분마다
- "모르겠다"도 공유다. 침묵이 최악이다

---

## ③ 완화 (Mitigation) — 원인보다 복구가 먼저다

### 즉시 실행 가능한 완화 조치들

| 조치 | 상황 | 소요 시간 |
|------|------|-----------|
| **롤백** | 최근 배포 후 장애 | 1~5분 |
| **Feature Flag OFF** | 특정 기능 문제 | 즉시 |
| **트래픽 차단** | 특정 API 과부하 | 즉시 |
| **서버 증설** | 자원 부족 | 3~10분 |
| **Read Replica 전환** | Primary DB 장애 | 1~5분 |
| **캐시 우회** | Redis 장애 | 즉시 (설정 변경) |
| **외부 API 차단** | 외부 서비스 장애 | 즉시 (Circuit Breaker) |

### 의사결정 트리

```
장애 발생
  │
  ├─ 최근 배포 있었나? ──YES──→ 즉시 롤백
  │
  ├─ 특정 기능만 문제? ──YES──→ Feature Flag OFF
  │
  ├─ 트래픽 급증? ──YES──→ Rate Limiting + 서버 증설
  │
  ├─ 외부 서비스 문제? ──YES──→ Circuit Breaker 발동 + Fallback
  │
  ├─ DB 문제? ──YES──→ Slow Query Kill + 커넥션 정리
  │
  └─ 모르겠다 ──→ 일단 최근 배포 롤백 (가장 흔한 원인)
```

**냉정한 원칙**:
> "원인을 찾고 있는 동안 서비스는 계속 죽어있다.
> 완벽한 수정보다 빠른 완화가 100배 중요하다."

---

## ④ 복구 (Recovery) — 정상 확인까지가 복구다

### 복구 확인 체크리스트

```
□ Error Rate가 정상 수준으로 돌아왔는가?
□ Latency가 정상 수준으로 돌아왔는가?
□ 밀린 데이터 처리(MQ 적체 등)가 완료되었는가?
□ 관련 팀에 복구 완료를 공유했는가?
□ 추가 모니터링이 필요한 구간에 알람을 설정했는가?
```

### 데이터 정합성 확인

장애 중 발생한 데이터 불일치를 반드시 확인한다.

```
- 결제는 됐는데 주문이 안 만들어진 건?
- 쿠폰이 차감됐는데 사용 기록이 없는 건?
- MQ에 적재됐지만 처리되지 않은 메시지?
```

→ [정합성-무결성-멱등성-설계](../concurrency/정합성-무결성-멱등성-설계.md)에서 상세 다룸

## ⑤ 회고 (Post-mortem) — 같은 장애를 두 번 겪지 않는다

### Blameless Post-mortem

**사람을 탓하지 않는다. 시스템을 고친다.**

```
Post-mortem 문서 구조:

## 개요
- 장애 시간: 2026-03-22 14:30 ~ 15:10 (40분)
- 영향 범위: 결제 서비스, 주문 서비스
- 심각도: SEV-1 (서비스 전체 영향)

## 타임라인
- 14:30 - 결제 서비스 Error Rate 급증 (알람 발동)
- 14:32 - IC 지정, 장애 채널 오픈
- 14:35 - 14:20 배포 확인, 롤백 시작
- 14:40 - 롤백 완료
- 14:50 - Error Rate 정상화
- 15:10 - 밀린 결제 처리 완료, 장애 종료

## 근본 원인
- 결제 API에서 외부 PG사 타임아웃 설정이 30초로 되어 있어
  스레드 풀 고갈 발생

## 재발 방지
- [ ] PG사 타임아웃 3초로 변경 (담당: 김개발, 3/25까지)
- [ ] Circuit Breaker 추가 (담당: 박서버, 3/27까지)
- [ ] 스레드 풀 모니터링 알람 추가 (담당: 이데옵, 3/24까지)

## 교훈
- 외부 API 호출 시 Timeout 기본값을 믿지 않는다
- Timeout * 동시 요청 수 = 필요 스레드 수 계산 필수
```

---

## 장애 등급 분류 (Severity)

| 등급 | 정의 | 대응 |
|------|------|------|
| **SEV-1** | 서비스 전체 불가 or 매출 영향 | 전원 소집, 10분 내 완화 시작 |
| **SEV-2** | 주요 기능 장애 | 온콜 + 관련팀, 30분 내 완화 |
| **SEV-3** | 부분 기능 저하 | 온콜이 업무 시간 내 처리 |
| **SEV-4** | 경미한 이슈 | 다음 스프린트에 수정 |

---

## 면접에서 이렇게 답하라

> "장애 대응은 5단계 프로세스로 진행합니다.
> **감지** — Golden Signals(Latency, Traffic, Errors, Saturation) 기반 모니터링으로 빠르게 인지합니다.
> **공유** — 5분 내 장애 채널을 열고, IC를 지정해서 역할을 분담합니다.
> **완화** — 원인 분석보다 서비스 복구가 먼저입니다. 최근 배포 롤백, Feature Flag OFF, Rate Limiting이 가장 빠른 조치입니다.
> **복구** — 데이터 정합성까지 확인하고, 밀린 처리가 모두 끝나야 복구 완료입니다.
> **회고** — Blameless Post-mortem으로 사람이 아닌 시스템을 고칩니다.
> 가장 중요한 원칙은 **완벽한 수정보다 빠른 완화**입니다."
