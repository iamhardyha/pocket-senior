# MSA와 메시지큐 아키텍처

> **핵심 질문**: MSA 구조와 메시지큐의 역할은 무엇인가?

---

## 한 줄 요약

**"MSA는 서비스를 나누는 것이고, 메시지큐는 나눈 서비스들을 느슨하게 연결하는 것이다."**

---

## Monolith vs MSA

```
Monolith:
┌──────────────────────────────────┐
│ 주문 + 결제 + 상품 + 알림 + 정산  │ ← 하나의 배포 단위
│           단일 DB                 │
└──────────────────────────────────┘

MSA:
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ 주문   │ │ 결제   │ │ 상품   │ │ 알림   │ ← 독립 배포
│ DB-A   │ │ DB-B   │ │ DB-C   │ │ DB-D   │ ← 독립 DB
└───┬────┘ └───┬────┘ └───┬────┘ └───┬────┘
    └──────────┴──────────┴──────────┘
              메시지큐 (Kafka)
```

| 항목 | Monolith | MSA |
|------|----------|-----|
| 배포 | 전체 한 번에 | 서비스별 독립 |
| 확장 | 전체 Scale | 서비스별 Scale |
| 장애 | 하나 죽으면 전체 죽음 | 하나 죽어도 나머지 동작 |
| 개발 | 초기 빠름 | 초기 느림, 장기적 유리 |
| 트랜잭션 | 단일 DB 트랜잭션 | 분산 트랜잭션 (복잡) |
| 디버깅 | 쉬움 | 어려움 (분산 추적 필요) |
| 팀 구조 | 하나의 코드베이스 | 서비스별 팀 (Conway's Law) |

### MSA로 가야 하는 시점

```
이런 상황이면 MSA를 고려:
- 팀이 10명 이상으로 커졌다
- 배포 주기가 팀 간 충돌한다
- 특정 기능만 스케일링하고 싶다
- 기술 스택을 서비스별로 다르게 쓰고 싶다

이런 상황이면 Monolith 유지:
- 팀 5명 이하
- 서비스 초기 단계
- 도메인 경계가 아직 불명확
```

**냉정한 현실**: "MSA를 해야 한다"는 기술적 판단이 아니라 **조직적 판단**이다.

## 서비스 간 통신

### 동기 통신 (Request-Response)

```
주문 서비스 ──HTTP/gRPC──→ 결제 서비스
              ←── 응답 ────
```

- REST API, gRPC, GraphQL
- **장점**: 직관적, 즉시 결과
- **단점**: 강결합, 호출 대상 장애 시 같이 장애

### 비동기 통신 (Event-Driven)

```
주문 서비스 ──이벤트 발행──→ 메시지큐 ──→ 결제 서비스
                                       ──→ 알림 서비스
                                       ──→ 정산 서비스
```

- Kafka, RabbitMQ, AWS SQS
- **장점**: 약결합, 장애 격리, 확장성
- **단점**: 복잡도, 디버깅 어려움, 결과적 일관성

### 언제 동기, 언제 비동기?

| 상황 | 방식 | 예시 |
|------|------|------|
| 즉시 결과 필요 | 동기 | 상품 조회, 결제 결과 |
| 실패 시 재시도 필요 | 비동기 | 알림, 이메일 |
| 여러 서비스에 전파 | 비동기 | 주문 생성 → 재고/알림/정산 |
| 처리 순서 중요 | 비동기 (Kafka) | 선착순, 이벤트 소싱 |

## 메시지큐 비교

| 항목 | Kafka | RabbitMQ | AWS SQS |
|------|-------|----------|---------|
| 모델 | Log (Pull) | Queue (Push) | Queue (Pull) |
| 처리량 | 매우 높음 (100만+ msg/s) | 높음 (수만 msg/s) | 높음 |
| 메시지 보관 | 영구 (설정에 따라) | 소비 후 삭제 | 최대 14일 |
| 순서 보장 | 파티션 내 보장 | 보장 안 됨 (기본) | FIFO 큐로 가능 |
| 재처리 | offset으로 자유롭게 | 불가 (소비 후 삭제) | 제한적 |
| 운영 | 직접 관리 or MSK | 직접 관리 | 완전 관리형 |
| 적합 | 이벤트 스트리밍, 대용량 | 작업 큐, RPC | 간단한 비동기 |

### 선택 가이드

```
대용량 이벤트 스트리밍, 재처리 필요 → Kafka
작업 분배, 라우팅 복잡 → RabbitMQ
간단한 비동기, 관리 부담 최소 → SQS
```

## 핵심 MSA 패턴

### API Gateway

```
Client → API Gateway → 주문 서비스
                     → 상품 서비스
                     → 사용자 서비스

역할:
- 라우팅
- 인증/인가
- Rate Limiting
- 요청/응답 변환
- 로깅
```

### Service Discovery

```
서비스가 동적으로 늘고 줄 때, 서로의 위치를 어떻게 알까?

Client-side Discovery:
서비스 → Service Registry(Eureka)에 등록
호출자 → Registry에서 주소 조회 → 직접 호출

Server-side Discovery:
호출자 → Load Balancer → 서비스
LB가 Registry 역할 겸임 (AWS ALB, Kubernetes Service)
```

## MSA의 고통

```
1. 분산 트랜잭션: @Transactional 하나로 안 됨 → Saga 필요
2. 분산 추적: 로그가 서비스별로 흩어짐 → Jaeger/Zipkin 필요
3. 데이터 조회: JOIN 불가 → CQRS 또는 API Composition
4. 테스트: 통합 테스트가 어려움 → Contract Testing
5. 배포: 서비스별 CI/CD → Kubernetes 거의 필수
6. 모니터링: 서비스 수 × 메트릭 수 → 관측성(Observability) 도구 필수
```

## 면접에서 이렇게 답하라

> "MSA는 **조직 확장**과 **독립 배포**가 필요할 때 전환합니다.
> 서비스 간 통신은 즉시 결과가 필요하면 **동기(REST/gRPC)**, 느슨한 결합이 필요하면 **비동기(Kafka)**를 사용합니다.
> 대부분의 서비스 간 연동은 비동기가 유리한데, 장애 격리와 확장성 때문입니다.
> MSA의 가장 큰 고통은 **분산 트랜잭션**이고, 이는 Saga + Outbox 패턴으로 해결합니다.
> 중요한 건 MSA는 기술 선택이 아니라 **조직 선택**이라는 것입니다. 팀이 작으면 Monolith가 오히려 빠릅니다."
