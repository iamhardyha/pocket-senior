# 벌크헤드 패턴 (Bulkhead Pattern)

> **핵심 질문**: 벌크헤드 패턴이란 무엇이고, 왜 필요한가?

---

## 한 줄 요약

**"서비스의 자원을 격리해서, 한 곳이 터져도 전체가 죽지 않게 만드는 패턴."**

---

## 벌크헤드란

벌크헤드(Bulkhead)는 **선박의 격벽**에서 유래한 용어다.

```
[격벽 없는 배]
┌─────────────────────────────┐
│          하나의 공간          │  ← 한 곳에 물이 들어오면
│     💧💧💧💧💧💧💧💧💧      │     배 전체가 침몰
└─────────────────────────────┘

[격벽 있는 배]
┌────────┬────────┬────────┬────────┐
│ 구역 1  │ 구역 2  │💧구역 3│ 구역 4  │  ← 한 구역에 물이 들어와도
│  안전   │  안전   │ 침수   │  안전   │     나머지 구역은 무사
└────────┴────────┴────────┴────────┘
```

소프트웨어에서도 동일한 원리다. **자원(스레드, 커넥션, 메모리)을 서비스별로 격리**하면, 한 서비스의 장애가 전체 시스템으로 전파되지 않는다.

---

## 왜 필요한가 — 격리 없는 시스템의 문제

### 스레드 풀을 공유하는 경우

```
[공유 스레드 풀: 200개]

주문 서비스 → ─┐
결제 서비스 → ─┤→ [Thread Pool: 200]
배송 서비스 → ─┤
알림 서비스 → ─┘
```

외부 배송 API가 응답을 안 주면:

```
1. 배송 서비스 호출 스레드들이 타임아웃까지 대기 (30초)
2. 스레드 200개가 전부 배송 API 대기에 묶임
3. 주문, 결제, 알림 서비스도 스레드를 못 받음
4. 전체 서비스 다운 ← 배송 하나 때문에 전부 죽음
```

이것이 **연쇄 장애(Cascading Failure)**다. 벌크헤드 패턴은 이걸 막는다.

---

## 벌크헤드의 3가지 격리 방식

### 1. 스레드 풀 격리 (Thread Pool Isolation)

서비스별로 **전용 스레드 풀**을 할당한다.

```
주문 서비스 → [Thread Pool A: 80개]
결제 서비스 → [Thread Pool B: 60개]
배송 서비스 → [Thread Pool C: 40개]
알림 서비스 → [Thread Pool D: 20개]
```

배송 API가 먹통이 되어도:

```
Thread Pool C: 40개 전부 대기 (배송만 영향)
Thread Pool A: 80개 정상 가동 (주문은 문제없음)
Thread Pool B: 60개 정상 가동 (결제도 문제없음)
```

**장점**: 완전한 격리. 한 풀이 소진되어도 다른 풀은 영향 없음
**단점**: 스레드 수 총합이 늘어남 → 메모리 사용량 증가, 컨텍스트 스위칭 비용

### 2. 세마포어 격리 (Semaphore Isolation)

스레드를 새로 만들지 않고, **동시 호출 수를 제한**한다.

```
주문 서비스 → [Semaphore: max 80]  ← 동시 80개까지만 허용
결제 서비스 → [Semaphore: max 60]
배송 서비스 → [Semaphore: max 40]  ← 40개 차면 즉시 거부 (Fast Fail)
알림 서비스 → [Semaphore: max 20]
```

**장점**: 스레드 풀보다 가벼움. 추가 스레드 생성 없음
**단점**: 호출자 스레드에서 실행되므로, 느린 호출이 호출자 스레드를 점유

### 3. 커넥션 풀 격리 (Connection Pool Isolation)

DB나 외부 서비스별로 **전용 커넥션 풀**을 분리한다.

```
주문 DB     → [HikariPool-order: max 20]
결제 DB     → [HikariPool-payment: max 15]
외부 API    → [HttpClient-external: max 10]
```

```yaml
# Spring Boot - 다중 데이터소스 분리
spring:
  datasource:
    order:
      hikari:
        maximum-pool-size: 20
        pool-name: HikariPool-order
    payment:
      hikari:
        maximum-pool-size: 15
        pool-name: HikariPool-payment
```

## Spring Boot에서 구현하기

### 1. Resilience4j Bulkhead (세마포어 방식)

```java
// build.gradle
implementation 'io.github.resilience4j:resilience4j-spring-boot3:2.2.0'
implementation 'io.github.resilience4j:resilience4j-bulkhead:2.2.0'
```

```yaml
# application.yml
resilience4j:
  bulkhead:
    instances:
      orderService:
        maxConcurrentCalls: 50       # 동시 최대 50개
        maxWaitDuration: 500ms       # 대기 최대 500ms, 초과 시 거부
      deliveryService:
        maxConcurrentCalls: 20       # 배송은 20개로 제한
        maxWaitDuration: 100ms       # 빨리 실패 (Fast Fail)
```

```java
@Service
public class OrderService {

    private final DeliveryClient deliveryClient;

    @Bulkhead(name = "deliveryService", fallbackMethod = "deliveryFallback")
    public DeliveryResult requestDelivery(Long orderId) {
        return deliveryClient.request(orderId);  // 외부 API 호출
    }

    // 벌크헤드가 가득 찼을 때 실행되는 폴백
    private DeliveryResult deliveryFallback(Long orderId, BulkheadFullException e) {
        log.warn("배송 벌크헤드 포화 — 주문 {} 대기열로 전환", orderId);
        return DeliveryResult.queued(orderId);   // 큐에 넣고 나중에 재시도
    }
}
```

### 2. Resilience4j ThreadPoolBulkhead (스레드 풀 방식)

```yaml
resilience4j:
  thread-pool-bulkhead:
    instances:
      deliveryService:
        maxThreadPoolSize: 20        # 전용 스레드 20개
        coreThreadPoolSize: 10       # 기본 10개
        queueCapacity: 50            # 대기 큐 50개
        keepAliveDuration: 60s
```

```java
@Bulkhead(name = "deliveryService", type = Bulkhead.Type.THREADPOOL)
public CompletionStage<DeliveryResult> requestDeliveryAsync(Long orderId) {
    return CompletableFuture.supplyAsync(() -> deliveryClient.request(orderId));
}
```

### 3. 세마포어 vs 스레드 풀 선택 기준

| 기준 | 세마포어 | 스레드 풀 |
|------|---------|----------|
| **리소스 비용** | 가벼움 (카운터만) | 무거움 (스레드 생성) |
| **격리 수준** | 호출자 스레드 공유 | 완전 격리 |
| **타임아웃 제어** | 단독 불가 (`@TimeLimiter` 조합으로 보완) | 독립적 제어 가능 |
| **적합한 상황** | 빠른 내부 호출 | 느린 외부 API |
| **반환 타입** | 동기 | CompletionStage (비동기) |

**실무 가이드**:
- 내부 서비스 간 호출 → **세마포어** (가볍고 빠름)
- 외부 API, DB 호출 → **스레드 풀** (완전 격리)

## 벌크헤드 + 서킷브레이커 조합

벌크헤드만 쓰면 **느린 호출이 제한된 슬롯을 계속 점유**한다. 서킷브레이커와 함께 써야 한다.

```
요청 → [CircuitBreaker: 실패율 50% 넘으면 차단]
         ↓
      [Bulkhead: 동시 20개 제한]
         ↓
      [TimeLimiter: 3초 타임아웃]
         ↓
      외부 API 호출
```

```yaml
resilience4j:
  bulkhead:
    instances:
      deliveryService:
        maxConcurrentCalls: 20
        maxWaitDuration: 100ms

  circuitbreaker:
    instances:
      deliveryService:
        failureRateThreshold: 50       # 실패율 50% 넘으면 OPEN
        waitDurationInOpenState: 10s   # 10초 후 HALF_OPEN
        slidingWindowSize: 10          # 최근 10개 요청 기준

  timelimiter:
    instances:
      deliveryService:
        timeoutDuration: 3s            # 3초 안에 응답 없으면 타임아웃
```

```java
@CircuitBreaker(name = "deliveryService", fallbackMethod = "deliveryFallback")
@Bulkhead(name = "deliveryService")
@TimeLimiter(name = "deliveryService")
public CompletionStage<DeliveryResult> requestDelivery(Long orderId) {
    return CompletableFuture.supplyAsync(() -> deliveryClient.request(orderId));
}
```

**실행 순서** (Resilience4j 기본 Aspect 우선순위): `CircuitBreaker → Bulkhead → TimeLimiter → 실제 호출`

> CircuitBreaker가 가장 바깥. 서킷이 OPEN이면 Bulkhead 체크도 하지 않고 즉시 거부.

```
서킷 OPEN 상태  → CircuitBreaker가 즉시 차단 (CallNotPermittedException)
동시 21번째 요청 → Bulkhead가 거부 (BulkheadFullException)
3초 초과        → TimeLimiter가 취소 (TimeoutException)
```

> `@TimeLimiter`는 `CompletionStage` 반환 타입에서만 동작한다. 동기 메서드에 적용하면 타임아웃이 작동하지 않으므로 주의.

## 벌크헤드 크기 산정

### 공식

```
필요 스레드 수 = 초당 요청 수(RPS) × 평균 응답 시간(초)

예: 배송 API
  - RPS: 100 req/s
  - 평균 응답: 200ms (0.2초)
  - 필요: 100 × 0.2 = 20개

  - 여유 버퍼 (1.5배): 20 × 1.5 = 30개
  → maxConcurrentCalls: 30
```

### 크기 설정 시 고려사항

| 항목 | 설명 |
|------|------|
| **너무 작으면** | 정상 트래픽도 거부됨. 불필요한 폴백 발생 |
| **너무 크면** | 격리 효과 없음. 장애 시 자원 고갈 |
| **피크 트래픽** | 평소의 2~3배를 기준으로 설정 |
| **다운스트림 용량** | 상대방 서비스가 감당 가능한 수준 이내로 |

## MSA에서의 벌크헤드 적용 패턴

```
┌─────────────┐
│  API Gateway │
│  Bulkhead    │ ← 서비스별 동시 요청 수 제한
└──────┬───────┘
       │
  ┌────┼────┬────────┐
  ▼    ▼    ▼        ▼
┌────┐┌────┐┌─────┐┌────┐
│주문││결제││배송  ││알림│  ← 각 서비스 내부에서도
│ BH ││ BH ││ BH  ││ BH │     DB, 외부 API별 벌크헤드
└────┘└────┘└─────┘└────┘
```

| 레벨 | 격리 대상 | 도구 |
|------|----------|------|
| **Gateway** | 서비스별 요청 수 | Spring Cloud Gateway Rate Limiter |
| **서비스** | 외부 호출별 동시성 | Resilience4j Bulkhead |
| **인프라** | DB, Redis 커넥션 | HikariCP, Lettuce 풀 분리 |
| **컨테이너** | CPU, 메모리 | K8s Resource Limits |

## 면접에서 이렇게 답하자

> "벌크헤드 패턴은 **선박의 격벽**처럼, 서비스의 자원(스레드, 커넥션)을 격리해서 한 곳의 장애가 전체로 전파되지 않도록 하는 패턴입니다.
>
> 예를 들어 배송 API가 느려져도, 배송 전용 스레드 풀만 소진되고 주문이나 결제는 정상 동작합니다.
> 격리 방식은 **세마포어**(가벼움, 내부 호출용)와 **스레드 풀**(완전 격리, 외부 API용)이 있고, Spring Boot에서는 **Resilience4j**로 구현합니다.
>
> 실무에서는 벌크헤드 단독이 아니라 **서킷브레이커, 타임리미터와 조합**해서 사용합니다. 벌크헤드가 동시성을 제한하고, 서킷브레이커가 실패율을 감지해 빠르게 차단하고, 타임리미터가 느린 응답을 끊습니다."
