# 메시지큐 아키텍처

> **핵심 질문**: 메시지큐는 왜 필요하고, 어떤 상황에서 어떤 메시지큐를 선택해야 하는가?

---

## 한 줄 요약

**"메시지큐는 서비스 간 직접 통화를 문자로 바꾸는 것이다. 상대가 바쁘든, 자리에 없든, 메시지는 전달된다."**

## 메시지큐가 필요한 이유

### 동기 호출의 한계 — 주문 시나리오

```
동기 방식: 주문 → 결제 → 재고 → 알림 (직렬 호출)

Client ──→ 주문 서비스 ──→ 결제 서비스 ──→ 재고 서비스 ──→ 알림 서비스
                                                              │
           ←──────────── 전체 응답 대기 (2초+) ────────────────┘
```

**문제 1: 응답 지연 누적**

```
주문 처리: 100ms
결제 호출: 500ms
재고 차감: 200ms
알림 발송: 800ms
─────────────────
총 응답시간: 1,600ms ← 사용자가 1.6초 기다림
```

**문제 2: 장애 전파 (Cascading Failure)**

```
알림 서비스 장애 발생!

Client → 주문 → 결제(성공) → 재고(성공) → 알림(timeout) → 주문 실패!
                                                           ↑
                                          알림이 안 되는 것뿐인데
                                          주문 자체가 실패?
```

**문제 3: 강결합**

```
주문 서비스가 모든 하위 서비스를 직접 알아야 한다:
- 결제 서비스 URL
- 재고 서비스 URL
- 알림 서비스 URL
- 포인트 서비스 URL ← 신규 추가 시 주문 서비스 코드 수정 필요
- 분석 서비스 URL  ← 또 수정
```

### 비동기 전환 — 메시지큐 도입

```
비동기 방식:

Client → 주문 서비스 → 메시지큐에 "주문 생성됨" 이벤트 발행 → 즉시 응답 (200ms)
                           │
                           ├──→ 결제 서비스 (구독)
                           ├──→ 재고 서비스 (구독)
                           ├──→ 알림 서비스 (구독)
                           └──→ 포인트 서비스 (구독) ← 리스너만 추가, 주문 코드 변경 없음
```

| 이점 | 설명 |
|------|------|
| **디커플링** | 주문 서비스가 하위 서비스를 몰라도 됨 |
| **장애 격리** | 알림 서비스 죽어도 주문은 성공. 복구 후 메시지 처리 |
| **버퍼링** | 순간 트래픽 폭증 시 메시지큐가 버퍼 역할 |
| **피크 트래픽 흡수** | 초당 10만 주문이 와도 Consumer가 자기 속도로 처리 |
| **응답 속도** | 핵심 처리만 하고 즉시 응답. 부가 작업은 비동기 |

```
피크 트래픽 흡수 시각화:

요청량   ██████████████                        ← 피크: 10만/s
         ██████████████
         ██████████████
처리량   ████████                               ← Consumer: 3만/s
         ████████
         ████████████████████████████████████   ← 피크 지나고도 꾸준히 처리
         ████████
         ████
                     시간 →

메시지큐가 없으면: 피크 시점에 서버 다운
메시지큐가 있으면: 큐에 쌓이고, Consumer가 자기 속도로 처리
```

## 메시지큐 핵심 개념

### Producer / Consumer / Broker

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Producer │────→│  Broker  │────→│ Consumer │
│ (발행자) │     │ (중개자) │     │ (소비자) │
└──────────┘     └──────────┘     └──────────┘

Producer: 메시지를 만들어서 보내는 쪽 (주문 서비스)
Broker:   메시지를 저장하고 전달하는 중간자 (Kafka, RabbitMQ, SQS)
Consumer: 메시지를 받아서 처리하는 쪽 (결제 서비스, 알림 서비스)
```

### Queue vs Topic (P2P vs Pub/Sub)

```
Point-to-Point (Queue):
  Producer → [Queue] → Consumer 1개만 소비

  메시지 A ──→ [  A  B  C  ] ──→ Consumer A가 가져감
                                  Consumer B는 다른 메시지를 가져감
  → 1개의 메시지 = 1개의 Consumer가 처리 (작업 분배)
  → 예: 주문 처리 워커, 이미지 리사이즈


Publish-Subscribe (Topic):
  Producer → [Topic] → 모든 구독자가 소비

  메시지 A ──→ [Topic] ──→ Consumer A (결제) ← 같은 메시지
                       ──→ Consumer B (알림) ← 같은 메시지
                       ──→ Consumer C (분석) ← 같은 메시지
  → 1개의 메시지 = N개의 Consumer가 각각 처리 (이벤트 전파)
  → 예: 주문 생성 이벤트 → 여러 서비스가 각자 반응
```

| 방식 | 메시지 소비 | 사용 사례 |
|------|------------|-----------|
| **Queue (P2P)** | 1개 Consumer만 | 작업 큐, 워커 분배 |
| **Topic (Pub/Sub)** | 모든 구독자 | 이벤트 전파, 알림 |

### Delivery Semantics

| 의미론 | 설명 | 유실 | 중복 | 실무 사용 |
|--------|------|------|------|-----------|
| **At-most-once** | 최대 1번 전달. 실패해도 재시도 안 함 | 가능 | 없음 | 로그, 메트릭 (유실 허용) |
| **At-least-once** | 최소 1번 전달. 실패 시 재시도 | 없음 | 가능 | **대부분의 실무** (Kafka 기본) |
| **Exactly-once** | 정확히 1번 전달 | 없음 | 없음 | 이론적 이상. 실현 비용 매우 높음 |

```
현실적인 선택:

At-least-once + Consumer 멱등성 = 사실상 Exactly-once

이유:
- At-most-once: 결제 메시지 유실? → 돈 안 빠지는데 물건 보냄 → 망함
- Exactly-once: 분산 환경에서 진짜 한 번만 보장하려면 성능 희생 큼
- At-least-once: 중복은 오지만, Consumer가 멱등하면 문제없음

→ "중복은 허용하되, 멱등하게 처리한다"가 업계 표준
```

### Dead Letter Queue (DLQ)

```
정상 흐름:
Message → Consumer → 처리 성공 → ACK

실패 흐름 (DLQ 없을 때):
Message → Consumer → 실패 → 재시도 → 실패 → 재시도 → 실패 → 재시도...
→ 무한 루프! 이 메시지 때문에 다른 메시지도 처리 못함 (Head-of-line blocking)

실패 흐름 (DLQ 있을 때):
Message → Consumer → 실패 → 재시도(1) → 실패 → 재시도(2) → 실패 → 재시도(3)
→ 3회 초과 → DLQ로 이동 → 원래 큐 정상 진행
                              │
                              → 운영자가 나중에 확인/재처리
```

**DLQ 운영 포인트**:

| 포인트 | 설명 |
|--------|------|
| 모니터링 필수 | DLQ에 메시지가 쌓이면 알림 발생시켜야 함 |
| 원인 분석 | 왜 실패했는지 메타데이터 (에러 메시지, 실패 횟수, 원본 큐) 저장 |
| 재처리 도구 | DLQ 메시지를 원래 큐로 돌려보내는 도구 필요 |
| 보관 기간 | 무기한 보관이 아니라 TTL 설정 (보통 7~14일) |
| 별도 Consumer | DLQ 전용 처리 로직이 있으면 좋음 (수동 개입 최소화) |

### Backpressure 처리

```
문제:
Producer: 초당 100,000건 발행
Consumer: 초당 10,000건 처리
→ 초당 90,000건씩 큐에 쌓임 → 메모리/디스크 초과 → Broker 장애

대응 전략:

1. Consumer 스케일 아웃
   Consumer 1대 → 10대로 늘려서 처리량 맞춤

2. Producer 속도 제한 (Rate Limiting)
   큐 크기가 임계치 초과 시 Producer에게 429 Too Many Requests

3. 큐 크기 제한 (Bounded Queue)
   RabbitMQ: max-length 설정 → 초과 시 oldest 메시지 drop or reject
   Kafka: retention.bytes로 파티션 크기 제한

4. 배압 전파 (Reactive Backpressure)
   Consumer가 Broker에게 "나 지금 바빠, 천천히 줘" 신호
   Kafka: max.poll.records로 한 번에 가져오는 양 조절
```

## Kafka vs RabbitMQ vs AWS SQS 깊이 비교

### 아키텍처 차이

```
Kafka (분산 로그):
┌─────────────────────────────────────┐
│ Topic: order-events                 │
│ ┌─────────────────────────────────┐ │
│ │ Partition 0: [0][1][2][3][4]... │ │  ← Append-only Log
│ │ Partition 1: [0][1][2][3][4]... │ │  ← Consumer가 offset으로 읽기
│ │ Partition 2: [0][1][2][3][4]... │ │  ← 읽어도 삭제 안 됨
│ └─────────────────────────────────┘ │
└─────────────────────────────────────┘
Consumer가 Pull: "나 offset 3부터 줘"
메시지 보관: retention 기간 동안 유지 (기본 7일)


RabbitMQ (전통적 메시지 브로커):
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Exchange │────→│  Queue   │────→│ Consumer │
│ (라우터) │     │ (저장소) │     │          │
└──────────┘     └──────────┘     └──────────┘
Broker가 Push: 메시지를 Consumer에게 밀어줌
소비 후: ACK 받으면 큐에서 삭제


AWS SQS (관리형 큐):
┌──────────┐     ┌─────────────────┐     ┌──────────┐
│ Producer │────→│  SQS Queue      │────→│ Consumer │
│          │     │  (AWS 관리)     │     │          │
└──────────┘     └─────────────────┘     └──────────┘
Consumer가 Pull: ReceiveMessage API 호출
Visibility Timeout: 처리 중 다른 Consumer가 못 봄
처리 완료: DeleteMessage 호출 → 큐에서 삭제
```

### 상세 비교표

| 항목 | Kafka | RabbitMQ | AWS SQS |
|------|-------|----------|---------|
| **모델** | 분산 로그 (Pull) | 메시지 브로커 (Push) | 관리형 큐 (Pull) |
| **처리량** | 100만+ msg/s | 수만 msg/s | 무제한 (자동 스케일) |
| **지연시간** | ms 단위 (배치 시 증가) | us~ms 단위 (가장 낮음) | ms~수십ms |
| **메시지 보관** | 설정 기간 동안 보관 (기본 7일) | 소비 후 삭제 | 최대 14일, 소비 후 삭제 |
| **재처리** | offset 되돌려서 자유롭게 | 불가 (이미 삭제됨) | 제한적 (Visibility Timeout 내) |
| **순서 보장** | 파티션 내 보장 | 기본 미보장 (Single Active Consumer로 가능) | Standard: 미보장 / FIFO: 보장 |
| **스케일링** | 파티션 추가 + Consumer 추가 | 큐 추가 + Consumer 추가 | 자동 (AWS 관리) |
| **운영 복잡도** | 높음 (Zookeeper/KRaft, Broker 관리) | 중간 (Erlang 클러스터) | 없음 (완전 관리형) |
| **프로토콜** | 자체 바이너리 프로토콜 | AMQP, MQTT, STOMP | HTTP/HTTPS (AWS SDK) |
| **Consumer Group** | 네이티브 지원 | Plugin 또는 수동 구현 | 자동 (동일 큐 구독) |
| **비용** | 인프라 직접 관리 or MSK ($$$) | 인프라 직접 관리 or AmazonMQ | 종량제 (메시지 수 기반, 저렴) |
| **메시지 크기** | 기본 1MB (설정 변경 가능) | 기본 상한 128MB (3.8+), 실무 수 KB~수 MB 권장 | 최대 256KB |
| **라우팅** | Topic + Partition Key | Exchange Type (Direct, Fanout, Topic, Headers) | 없음 (큐 단위) |

### 성능 특성 정리

```
처리량 (Throughput):
  Kafka  ████████████████████████████████  (100만+ msg/s)
  SQS    ████████████████████████████████  (무제한, 자동 스케일)
  Rabbit ████████████                      (수만 msg/s)

지연시간 (Latency):
  Rabbit ██                                (가장 낮음, us~ms)
  Kafka  ████                              (ms, 배치 설정에 따라)
  SQS    ████████                          (ms~수십ms)

운영 복잡도:
  SQS    ██                                (관리형, 신경 안 씀)
  Rabbit ████████                          (클러스터, Erlang)
  Kafka  ████████████████                  (파티션, ISR, Rebalancing...)
```

### 각각의 실무 사용 사례

**Kafka를 쓰는 경우**:
- 이벤트 스트리밍 (실시간 사용자 행동 추적, 클릭 스트림)
- 대용량 로그 수집 (ELK 파이프라인 앞단)
- MSA 이벤트 버스 (서비스 간 이벤트 전파)
- CQRS의 이벤트 소싱 (이벤트 재생으로 상태 재구축)
- CDC (Change Data Capture) — Debezium + Kafka Connect

**RabbitMQ를 쓰는 경우**:
- 작업 큐 (이미지 리사이즈, PDF 생성, 이메일 발송)
- RPC 패턴 (요청-응답 비동기 처리)
- 복잡한 라우팅 (헤더 기반 분기, 조건부 큐 배분)
- 우선순위 큐 (긴급 작업 먼저 처리)
- 소규모~중규모 서비스의 비동기 처리

**AWS SQS를 쓰는 경우**:
- Lambda 트리거 (SQS → Lambda, 서버리스 아키텍처)
- 간단한 비동기 작업 분배 (운영 부담 최소)
- AWS 생태계 내 서비스 연동 (SNS → SQS → Lambda)
- 스타트업 초기 (관리 인력 없을 때)
- 배치 처리 트리거 (S3 이벤트 → SQS → 처리)

## 메시지큐 선택 의사결정 트리

```
메시지큐가 필요하다!
│
├─ Q1: 이벤트를 나중에 재처리(replay)해야 하는가?
│  ├─ YES → Kafka
│  │         (offset 기반 재처리, 이벤트 소싱)
│  │
│  └─ NO → Q2로
│
├─ Q2: 초당 처리량이 10만 건 이상인가?
│  ├─ YES → Kafka
│  │         (대용량 스트리밍에 최적화)
│  │
│  └─ NO → Q3로
│
├─ Q3: 복잡한 라우팅이 필요한가?
│  │   (헤더 기반 분기, 조건부 큐, 우선순위)
│  ├─ YES → RabbitMQ
│  │         (Exchange + Binding 조합으로 유연한 라우팅)
│  │
│  └─ NO → Q4로
│
├─ Q4: 운영 인력/인프라 관리 여력이 있는가?
│  ├─ NO → AWS SQS (+ SNS for Pub/Sub)
│  │        (관리형, 종량제, 즉시 사용)
│  │
│  └─ YES → Q5로
│
├─ Q5: 여러 Consumer Group이 같은 메시지를 독립적으로 소비해야 하는가?
│  ├─ YES → Kafka
│  │         (Consumer Group 네이티브 지원)
│  │
│  └─ NO → Q6로
│
└─ Q6: 지연시간이 극도로 낮아야 하는가?
   ├─ YES → RabbitMQ
   │         (Push 기반, 마이크로초 수준 지연)
   │
   └─ NO → AWS SQS
            (가장 간단, 충분한 성능)
```

**현실적 조언**:

```
스타트업 초기       → SQS (관리 부담 0, 비용 저렴)
서비스 성장기       → RabbitMQ or Kafka (트래픽, 요구사항에 따라)
대규모 MSA         → Kafka (이벤트 버스 + 데이터 파이프라인)
AWS 올인           → SQS + SNS + EventBridge (네이티브 통합)
이벤트 소싱/CQRS    → Kafka (유일한 현실적 선택)
```

## 메시지큐 운영 포인트

### Consumer Lag 모니터링

```
Producer offset:  1,000,000  (Producer가 여기까지 썼다)
Consumer offset:    950,000  (Consumer가 여기까지 읽었다)
─────────────────────────
Consumer Lag:       50,000   (아직 처리 안 된 메시지)

Lag 추이:
  정상: Lag이 0 근처에서 유지
  경고: Lag이 꾸준히 증가 → Consumer 처리 속도 < 유입 속도
  위험: Lag이 폭증 + retention 기간 초과 → 메시지 유실!
```

**대응 전략**:

| Lag 상태 | 대응 |
|----------|------|
| Lag 증가 추세 | Consumer 인스턴스 추가 (파티션 수 이내) |
| Lag 폭증 | Consumer 처리 로직 병목 확인 (DB 쿼리? 외부 API?) |
| 특정 파티션만 Lag | 해당 파티션의 메시지 처리 실패 확인 |
| Lag = 0인데 느림 | Producer 쪽 문제 or 메시지 자체가 적음 |

모니터링 도구: **Kafka Lag Exporter** + Grafana, **Burrow**, CloudWatch (MSK)

### 멱등성 보장 (중복 메시지 처리)

At-least-once 환경에서 중복 메시지는 **반드시** 온다.

```java
@KafkaListener(topics = "order-events", groupId = "payment-processor")
public void handleOrderCreated(OrderCreatedEvent event, Acknowledgment ack) {
    // 1. 이미 처리한 이벤트인가?
    if (processedEventRepository.existsByEventId(event.eventId())) {
        log.info("이미 처리된 이벤트, skip: {}", event.eventId());
        ack.acknowledge();
        return;
    }

    // 2. 처리 + 기록을 같은 트랜잭션으로 (원자적)
    paymentService.processWithIdempotency(event);

    // 3. 수동 커밋
    ack.acknowledge();
}

@Transactional
public void processWithIdempotency(OrderCreatedEvent event) {
    // INSERT IGNORE: 이미 있으면 무시
    int inserted = processedEventRepository.insertIgnore(event.eventId());
    if (inserted == 0) return; // 이미 처리됨

    // 실제 결제 처리
    paymentRepository.save(Payment.from(event));
}
```

### 메시지 순서 보장 전략

```
Kafka:
  같은 Partition Key → 같은 Partition → 순서 보장

  kafkaTemplate.send("order-events", orderId.toString(), event);
  //                                  ^^^^^^^^ Partition Key

  orderId=100 → 항상 Partition 2 → 이 주문의 이벤트는 순서대로
  orderId=200 → 항상 Partition 0 → 이 주문의 이벤트도 순서대로
  → 서로 다른 주문 간 순서? 보장 안 됨 (보장할 필요도 없음)


RabbitMQ:
  Single Active Consumer 패턴: 큐당 1개 Consumer만 활성화
  → 순서 보장되지만 처리량 희생


SQS FIFO:
  MessageGroupId로 순서 보장
  같은 GroupId → 순서대로 처리
  다른 GroupId → 병렬 처리
```

### 장애 시 메시지 유실 방지

```
지점별 유실 가능성과 대응:

1. Producer → Broker 구간
   ┌──────────┐     ┌──────────┐
   │ Producer │──X──│  Broker  │  ← 네트워크 장애
   └──────────┘     └──────────┘
   대응: acks=all + retries + idempotent producer (enable.idempotence=true)

2. Broker 내부
   ┌──────────┐
   │  Broker  │  ← 디스크 장애, Broker 다운
   └──────────┘
   대응: replication.factor=3, min.insync.replicas=2

3. Broker → Consumer 구간
   ┌──────────┐     ┌──────────┐
   │  Broker  │──X──│ Consumer │  ← Consumer 처리 중 죽음
   └──────────┘     └──────────┘
   대응: 수동 커밋 (enable.auto.commit=false)
         처리 완료 후에만 ack.acknowledge()
```

**Kafka Producer 안전 설정**:

```yaml
spring:
  kafka:
    producer:
      acks: all                    # 모든 ISR 확인
      retries: 2147483647          # 무한 재시도 (delivery.timeout.ms로 제한)
      properties:
        enable.idempotence: true   # 중복 전송 방지
        max.in.flight.requests.per.connection: 5  # 멱등 모드에서 안전
        delivery.timeout.ms: 120000
```

### Retry + DLQ 전략

```java
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public DefaultErrorHandler errorHandler(KafkaTemplate<String, Object> template) {
        // DLQ로 보내는 Recoverer
        DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(template,
                (record, ex) -> new TopicPartition(
                    record.topic() + ".DLT",   // 원본 토픽명 + .DLT
                    record.partition()
                ));

        // Exponential Backoff: 1초 → 2초 → 4초, 최대 3회
        ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
        backOff.setMaxElapsedTime(15000L); // 최대 15초

        DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);

        // 재시도하면 안 되는 예외 (비즈니스 에러)
        handler.addNotRetryableExceptions(
            InvalidOrderException.class,
            DuplicatePaymentException.class
        );

        return handler;
    }
}
```

```
Retry 흐름:

Message → 처리 실패
  → 1초 후 재시도 (1회)
  → 2초 후 재시도 (2회)
  → 4초 후 재시도 (3회)
  → 실패 → order-events.DLT로 이동

DLT Consumer:
  → 알림 발송 (Slack, PagerDuty)
  → 실패 원인 로깅
  → 필요 시 수동 재처리 도구로 원래 토픽에 재발행
```

### Consumer Group 관리

```
Consumer Group "payment-processor":

  Topic: order-events (6 partitions)

  Consumer A ← Partition 0, 1     (2개 담당)
  Consumer B ← Partition 2, 3     (2개 담당)
  Consumer C ← Partition 4, 5     (2개 담당)

  Consumer D 추가 시 → Rebalance 발생!
  Consumer A ← Partition 0, 1
  Consumer B ← Partition 2, 3
  Consumer C ← Partition 4
  Consumer D ← Partition 5

  Consumer 6개 초과? → 놀고 있는 Consumer 발생 (파티션보다 많으니까)
```

**Rebalance 주의점**:

| 포인트 | 설명 |
|--------|------|
| Rebalance 중 처리 중단 | 파티션 재할당 동안 Consumer가 멈춤 |
| Sticky Assignor 사용 | 기존 할당 최대한 유지 (불필요한 이동 최소화) |
| session.timeout.ms | Consumer 헬스체크 주기 (기본 45초) |
| max.poll.interval.ms | 처리 지연 시 강제 Rebalance (기본 5분) |
| Static Group Membership | group.instance.id 설정으로 Rebalance 최소화 |

## 실무 아키텍처 패턴

### Event-Driven Architecture (EDA)

```
기존: 명령형 (Command-Driven)
  주문 서비스 → "결제해라" → 결제 서비스
  주문 서비스 → "재고 차감해라" → 재고 서비스
  → 주문 서비스가 모든 흐름을 알아야 함 (오케스트레이션)

EDA: 이벤트 기반
  주문 서비스 → "주문이 생성됐다" → [Kafka]
                                      ├→ 결제 서비스: "주문 생겼네? 결제하자"
                                      ├→ 재고 서비스: "주문 생겼네? 재고 차감하자"
                                      └→ 알림 서비스: "주문 생겼네? 알림 보내자"
  → 각 서비스가 자율적으로 반응 (코레오그래피)
```

**EDA의 현실적 고통**:

```
1. 디버깅이 어렵다
   "결제가 왜 안 됐지?" → 이벤트 발행됐나? → 컨슈머가 받았나? → 처리 중 에러났나?
   → 분산 추적(Distributed Tracing) 필수: Jaeger, Zipkin, AWS X-Ray

2. 이벤트 순서 보장이 어렵다
   "주문 생성" → "주문 취소"가 순서 뒤바뀌면?
   → Partition Key로 같은 주문은 같은 파티션에

3. 이벤트 스키마 변경이 어렵다
   OrderCreatedEvent에 필드 추가하면 기존 Consumer가 깨질 수 있음
   → Schema Registry (Avro, Protobuf) 사용
```

### CQRS와 메시지큐

```
CQRS (Command Query Responsibility Segregation):
  쓰기(Command)와 읽기(Query)를 분리

  ┌──────────────┐     ┌─────────┐     ┌──────────────┐
  │ Command API  │────→│  Kafka  │────→│  Query API   │
  │ (Write DB)   │     │         │     │  (Read DB)   │
  │ MySQL/PG     │     │ 이벤트  │     │ Elasticsearch│
  └──────────────┘     └─────────┘     │ Redis Cache  │
                                       └──────────────┘

흐름:
1. 주문 생성 → Write DB에 저장
2. OrderCreatedEvent → Kafka로 발행
3. Read 서비스가 이벤트 수신 → 검색용 인덱스(ES) 갱신
4. 조회 요청 → Read DB에서 빠르게 읽기

장점: 쓰기/읽기 독립 스케일링, 읽기에 최적화된 DB 사용
단점: 결과적 일관성 (Eventually Consistent), 구현 복잡도
```

### Outbox Pattern (DB + 메시지큐 일관성)

```
문제:
  @Transactional
  public void createOrder(OrderRequest request) {
      orderRepository.save(order);           // 1. DB 저장
      kafkaTemplate.send("order-events", event);  // 2. Kafka 발행
  }

  1 성공 → 2 실패: DB에는 있는데 이벤트 안 나감 (불일치!)
  1 실패 → 2 성공: DB에는 없는데 이벤트 나감 (불일치!)
  → DB 트랜잭션과 Kafka 발행은 원자적이지 않다


해결: Outbox Pattern

  ┌─────────────────────────────────────────────┐
  │ 하나의 DB 트랜잭션                           │
  │                                             │
  │ 1. INSERT INTO orders (...)                 │
  │ 2. INSERT INTO outbox_events (...)          │
  │                                             │
  │ → 둘 다 같은 트랜잭션이니 원자적!            │
  └─────────────────────────────────────────────┘
           │
           ▼
  Outbox Relay (Debezium / Polling)
           │
           ▼
     Kafka Topic
```

```java
// Outbox 테이블
@Entity
@Table(name = "outbox_events")
public class OutboxEvent {
    @Id
    private String id;              // UUID
    private String aggregateType;   // "Order"
    private String aggregateId;     // orderId
    private String eventType;       // "OrderCreated"
    private String payload;         // JSON
    private LocalDateTime createdAt;
    private boolean published;      // 발행 여부
}

// 주문 생성 — 같은 트랜잭션으로 Outbox에 저장
@Transactional
public Order createOrder(OrderRequest request) {
    Order order = orderRepository.save(Order.from(request));

    OutboxEvent event = OutboxEvent.builder()
        .id(UUID.randomUUID().toString())
        .aggregateType("Order")
        .aggregateId(order.getId().toString())
        .eventType("OrderCreated")
        .payload(objectMapper.writeValueAsString(OrderCreatedEvent.from(order)))
        .published(false)
        .build();

    outboxEventRepository.save(event);  // 같은 트랜잭션!
    return order;
}
```

```
Outbox Relay 방식 비교:

1. Polling (간단하지만 지연)
   @Scheduled(fixedRate = 1000)  // 1초마다
   public void publishOutboxEvents() {
       List<OutboxEvent> events = outboxRepository.findByPublishedFalse();
       for (OutboxEvent event : events) {
           kafkaTemplate.send(event.getAggregateType(), event.getPayload());
           event.markAsPublished();
       }
   }
   → 장점: 구현 간단
   → 단점: 폴링 간격만큼 지연, DB 부하

2. CDC (Debezium) — 권장
   Debezium이 DB의 binlog/WAL을 감시 → outbox 테이블 변경 감지 → Kafka 자동 발행
   → 장점: 거의 실시간, DB 부하 적음
   → 단점: Debezium 운영 필요
```

### Fan-out Pattern

```
1개의 이벤트를 N개의 서비스에 동시 전파

Producer → [Kafka Topic: order-events]
              │
              ├──→ Consumer Group A (결제) → 결제 처리
              ├──→ Consumer Group B (재고) → 재고 차감
              ├──→ Consumer Group C (알림) → 알림 발송
              ├──→ Consumer Group D (분석) → 데이터 수집
              └──→ Consumer Group E (추천) → 추천 모델 갱신

핵심: 각 Consumer Group은 독립적으로 모든 메시지를 받음
    → 서비스 추가 시 Consumer Group만 추가 (Producer 변경 없음)


AWS에서의 Fan-out: SNS + SQS

Producer → [SNS Topic]
              ├──→ SQS Queue A → Lambda A (결제)
              ├──→ SQS Queue B → Lambda B (알림)
              └──→ SQS Queue C → Lambda C (분석)

→ SNS가 Fan-out, SQS가 버퍼 역할
→ 서버리스 아키텍처에서 자주 사용
```

### Producer 설정 및 코드

```java
@Service
@RequiredArgsConstructor
public class OrderEventPublisher {

    private final KafkaTemplate<String, Object> kafkaTemplate;

    public void publishOrderCreated(Order order) {
        OrderCreatedEvent event = OrderCreatedEvent.builder()
            .eventId(UUID.randomUUID().toString())
            .orderId(order.getId())
            .userId(order.getUserId())
            .totalAmount(order.getTotalAmount())
            .items(order.getItems())
            .occurredAt(LocalDateTime.now())
            .build();

        // Key = orderId → 같은 주문의 이벤트는 같은 파티션 → 순서 보장
        kafkaTemplate.send("order-events", order.getId().toString(), event)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("이벤트 발행 실패: orderId={}", order.getId(), ex);
                    // 보상 로직 or Outbox 패턴으로 전환
                } else {
                    log.info("이벤트 발행 성공: topic={}, partition={}, offset={}",
                        result.getRecordMetadata().topic(),
                        result.getRecordMetadata().partition(),
                        result.getRecordMetadata().offset());
                }
            });
    }
}
```

### Consumer 설정 및 코드

```yaml
# application.yml
spring:
  kafka:
    consumer:
      group-id: payment-processor
      auto-offset-reset: earliest
      enable-auto-commit: false
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
```

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class PaymentEventConsumer {

    private final PaymentService paymentService;

    @KafkaListener(
        topics = "order-events",
        groupId = "payment-processor",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void handleOrderCreated(
            @Payload OrderCreatedEvent event,
            @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
            @Header(KafkaHeaders.OFFSET) long offset,
            Acknowledgment ack) {

        log.info("이벤트 수신: eventId={}, partition={}, offset={}",
            event.eventId(), partition, offset);

        try {
            paymentService.processPayment(event);
            ack.acknowledge();  // 처리 성공 시에만 커밋
        } catch (Exception e) {
            log.error("결제 처리 실패: eventId={}", event.eventId(), e);
            // ack 안 함 → 재시도 (DefaultErrorHandler가 DLQ까지 처리)
            throw e;
        }
    }
}
```

## 면접에서 이렇게 답하자

> "메시지큐는 **동기 호출의 세 가지 문제** — 응답 지연 누적, 장애 전파, 강결합 — 을 해결합니다.
> 비동기로 전환하면 디커플링, 장애 격리, 피크 트래픽 버퍼링을 얻습니다.
>
> 메시지큐 선택 기준은 명확합니다.
> **Kafka**: 대용량 이벤트 스트리밍, 재처리 필요, MSA 이벤트 버스. 로그 기반이라 메시지를 보관하고 offset으로 재생 가능합니다.
> **RabbitMQ**: 복잡한 라우팅, 작업 큐, 저지연. Push 기반이라 실시간 작업 분배에 적합합니다.
> **SQS**: 운영 부담 최소, AWS 서버리스 아키텍처. 관리형이라 인프라 걱정 없습니다.
>
> Delivery Semantics는 현실적으로 **At-least-once + Consumer 멱등성**이 표준입니다. Exactly-once는 비용이 너무 큽니다.
>
> 운영에서 가장 중요한 건 **Consumer Lag 모니터링**, **DLQ 전략**, **멱등성 보장**입니다.
> DB와 메시지큐의 일관성 문제는 **Outbox Pattern**으로 해결합니다. 같은 DB 트랜잭션에 이벤트를 저장하고, CDC(Debezium)나 폴링으로 Kafka에 발행합니다."
