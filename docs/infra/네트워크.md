# 네트워크

> **핵심 질문**: 백엔드 개발자가 알아야 할 네트워크 지식은?

---

## 한 줄 요약

**"백엔드 개발자에게 네트워크는 '요청이 어떻게 서버에 도착하고, 응답이 어떻게 돌아가는지'를 아는 것이다."**

---

## 요청의 여정

```
브라우저에서 https://api.example.com/orders 입력

1. DNS 조회: api.example.com → 52.78.123.45
2. TCP 연결: 3-Way Handshake
3. TLS 연결: SSL/TLS Handshake (HTTPS)
4. HTTP 요청: GET /orders
5. 서버 처리: 라우팅 → 컨트롤러 → 서비스 → DB
6. HTTP 응답: 200 OK + JSON
7. 연결 유지 또는 종료: Keep-Alive / Close
```

---

## DNS (Domain Name System)

```
브라우저 캐시 → OS 캐시 → Router → ISP DNS → Root DNS → .com DNS → example.com DNS
                                                                    → 52.78.123.45
```

**백엔드에서 DNS가 중요한 이유**:
- **로드밸런싱**: Route 53 가중치 라우팅 → [부하-분산-아키텍처](../traffic/부하-분산-아키텍처.md)
- **Failover**: 헬스체크 실패 시 다른 리전으로
- **TTL**: DNS 캐시 시간. TTL이 길면 변경 반영이 느림 (기본 300초)
- **DB 마이그레이션**: CNAME 변경으로 DB 전환 → [글로벌-서비스-DB-마이그레이션](../database/글로벌-서비스-DB-마이그레이션.md)

## TCP/IP

### 3-Way Handshake (연결)

```
Client → SYN → Server
Client ← SYN+ACK ← Server
Client → ACK → Server
→ 연결 수립 (1.5 RTT)
```

### 4-Way Handshake (종료)

```
Client → FIN → Server
Client ← ACK ← Server
Client ← FIN ← Server
Client → ACK → Server
→ 연결 종료
```

### 백엔드에서 TCP가 중요한 이유

```
커넥션 풀의 이유:
TCP 연결 = SYN → SYN+ACK → ACK (1.5 RTT)
+ TLS = ClientHello → ServerHello → ... (2 RTT 추가)
= 총 3.5 RTT (100ms 환경에서 350ms)

매 요청마다 이걸 하면? → 느림
→ 커넥션 풀로 연결을 재사용 (DB, Redis, HTTP)
```

### Keep-Alive

```
HTTP/1.1 Keep-Alive:
한 TCP 연결로 여러 HTTP 요청을 처리
→ 매번 연결하는 오버헤드 제거

HTTP/2 Multiplexing:
한 TCP 연결로 여러 HTTP 요청을 동시에 병렬 처리
→ HTTP 레벨 Head-of-Line Blocking 해결 (TCP 레벨 HOL은 여전히 존재 → HTTP/3가 해결)
```

## HTTP

### 상태 코드 — 백엔드가 정확히 써야 하는 것

| 코드 | 의미 | 사용 |
|------|------|------|
| **200** | OK | 조회 성공 |
| **201** | Created | 리소스 생성 성공 |
| **202** | Accepted | 비동기 접수 (쿠폰 대기열 등) |
| **204** | No Content | 삭제 성공 (본문 없음) |
| **301** | Moved Permanently | URL 영구 변경 (SEO) |
| **302** | Found | 임시 리다이렉트 |
| **400** | Bad Request | 입력값 오류 |
| **401** | Unauthorized | 인증 필요 (로그인 안 됨) |
| **403** | Forbidden | 권한 없음 (로그인했지만 접근 불가) |
| **404** | Not Found | 리소스 없음 |
| **409** | Conflict | 충돌 (중복 생성, 버전 충돌) |
| **429** | Too Many Requests | Rate Limiting |
| **500** | Internal Server Error | 서버 오류 |
| **502** | Bad Gateway | 뒷단 서버 응답 오류 |
| **503** | Service Unavailable | 서비스 점검/과부하 |
| **504** | Gateway Timeout | 뒷단 서버 타임아웃 |

### HTTP 헤더 — 실무에서 자주 쓰는 것

```
요청:
Authorization: Bearer {JWT}           # 인증
Content-Type: application/json         # 요청 본문 형식
Accept: application/json               # 원하는 응답 형식
Idempotency-Key: abc-123              # 멱등성 키
X-Request-ID: uuid                     # 분산 추적용

응답:
Cache-Control: max-age=300             # 캐시 300초
ETag: "abc123"                         # 캐시 검증
Retry-After: 60                        # Rate Limiting 시 대기 시간
X-RateLimit-Remaining: 95             # 남은 요청 수
```

### HTTP/2 vs HTTP/1.1

```
HTTP/1.1:
요청 1 → 응답 1 → 요청 2 → 응답 2 (순차)
6개 병렬 TCP 연결로 완화

HTTP/2:
하나의 TCP에서 요청 1,2,3 동시 → 응답 1,2,3 동시 (멀티플렉싱)
헤더 압축 (HPACK)
서버 푸시

HTTP/3 (QUIC):
UDP 기반 → TCP Handshake 불필요 → 더 빠른 연결
패킷 손실 시 다른 스트림에 영향 없음
```

## HTTPS / TLS

```
TLS Handshake:
Client → ClientHello (지원 암호 목록)
Server → ServerHello (선택한 암호 + 인증서)
Client → 인증서 검증 → 키 교환 (TLS 1.3: ECDHE만 지원, TLS 1.2: RSA/ECDHE)
→ 이후 합의된 대칭키로 암호화 통신

비용:
TLS 1.2: 2 RTT 추가
TLS 1.3: 1 RTT로 단축 (0-RTT도 가능, 단 replay attack 취약 → 멱등성 요청에만 사용 권장)
```

**실무**: ALB/Nginx에서 TLS 종단 (SSL Termination) → 내부 통신은 HTTP.

## 네트워크 문제 디버깅

### 흔한 네트워크 이슈

| 증상 | 가능한 원인 | 확인 방법 |
|------|-------------|-----------|
| Connection Timeout | 대상 서버 다운, 방화벽 차단 | telnet, curl -v |
| Read Timeout | 서버가 처리 중이지만 느림 | 서버 로그, Slow Query |
| Connection Reset | 서버가 강제 종료, LB 타임아웃 | 서버 로그, LB 설정 |
| DNS Resolution Failed | DNS 설정 오류, 네트워크 단절 | nslookup, dig |
| 502 Bad Gateway | 뒷단 서버 비정상 응답 | Nginx/ALB 로그 |
| 504 Gateway Timeout | 뒷단 서버 응답 시간 초과 | 타임아웃 설정 확인 |

### 디버깅 도구

```bash
# DNS 확인
nslookup api.example.com
dig api.example.com

# TCP 연결 확인
telnet api.example.com 443
nc -zv api.example.com 443

# HTTP 요청 상세
curl -v https://api.example.com/health

# 네트워크 경로
traceroute api.example.com

# 포트 사용 확인
netstat -tlnp | grep 8080
ss -tlnp | grep 8080
```

## 면접에서 이렇게 답하라

> "백엔드에서 네트워크의 핵심은 **요청의 여정을 이해하는 것**입니다.
> DNS로 IP를 찾고, TCP 3-Way Handshake로 연결하고, TLS로 암호화한 뒤, HTTP로 요청/응답합니다.
> 실무에서는 **커넥션 풀**로 TCP/TLS 오버헤드를 줄이고, **Keep-Alive**로 연결을 재사용합니다.
> HTTP 상태 코드는 정확히 사용해야 합니다. 401은 인증 실패, 403은 권한 부족, 429는 Rate Limiting.
> 네트워크 문제 디버깅은 curl -v로 시작해서 서버 로그, LB 로그 순으로 추적합니다.
> DNS TTL, LB Idle Timeout, 서버 Keep-Alive Timeout이 서로 맞지 않으면 간헐적 502/504가 발생하므로 일치시켜야 합니다."
