# Firehose vs S3 직접 적재 — 결과가 같은데 왜 Kinesis를 쓰나

> **핵심 질문**: 서버에서 Kinesis Data Firehose로 보내나 S3에 바로 PutObject 하나, 결국 S3에 데이터가 쌓이는 건 똑같은데 Firehose를 쓰는 이점이 뭐지?

---

## 한 줄 요약

**"결과물(S3의 데이터)은 같아 보여도, 직접 쏘면 '버퍼링·배칭·압축·포맷변환·파티셔닝·재시도'를 전부 네가 만들어야 한다. Firehose는 그 적재 파이프라인을 통째로 관리형으로 대신해 준다. 잘게·끊임없이 들어오는 스트림일수록 차이가 벌어진다."**

---

## 1. "S3에 바로 쓰면 되잖아"의 함정

맞다. 결과만 보면 둘 다 S3에 객체가 생긴다. 문제는 **"어떤 모양으로, 어떤 비용으로 쌓이느냐"** 다. 이벤트/로그처럼 **건당 작고 끊임없이 들어오는 데이터**를 `PutObject`로 직접 적재하면 다음을 전부 직접 떠안는다.

### 작은 파일 폭탄 (the small files problem)

레코드 하나당 `PutObject` 한 번을 호출하면, 1초에 수천 건 들어오는 트래픽에서 **객체가 수백만 개**로 불어난다. 이게 왜 문제인가:

| 측면 | 작은 파일 수백만 개일 때 | 큰 파일 수천 개일 때 |
|------|------------------------|---------------------|
| Athena/Spark 쿼리 | 파일마다 오픈/리스트 오버헤드 → 느려짐 | 파일당 처리량 커서 빠름 |
| S3 PUT 요청 비용 | 요청 수 = 객체 수 → 비용 누적 | 배칭으로 요청 수 급감 |
| 메타데이터/리스팅 | `ListObjects`가 무거워짐 | 가벼움 |

분석 엔진(Athena, EMR/Spark, Redshift Spectrum)은 **파일을 여는 행위 자체**에 고정 비용이 있다. 그래서 "같은 1GB"라도 1KB짜리 백만 개보다 128MB짜리 8개가 훨씬 빠르고 싸다. 직접 적재에선 이걸 막으려면 **앱에서 레코드를 모아 큰 파일로 합치는 배칭 로직**을 직접 짜야 한다.

### 직접 만들어야 하는 것들의 목록

직접 S3 적재를 "제대로" 하려면 사실상 다음을 다 구현하게 된다.

```
[ 서버 ] --(레코드 1건)--> 직접 구현해야 하는 것들:
  1. 버퍼링/배칭   — 메모리에 모았다가 일정 크기/시간마다 flush
  2. 압축          — gzip/snappy 등으로 묶기 (저장·스캔 비용↓)
  3. 포맷 변환     — JSON 로그 → Parquet/ORC (컬럼형, 분석 최적)
  4. 파티셔닝      — s3://bucket/year=/month=/day=/ 경로로 쪼개기
  5. 재시도/실패격리 — 실패한 레코드를 잃지 않고 따로 모으기
  6. 스파이크 완충  — 트래픽 폭증 시 S3로 밀어내는 속도 조절(백프레셔)
```

이걸 직접 만들면 **그게 곧 미니 Firehose를 재발명하는 일**이다. 게다가 이 배칭 버퍼는 보통 인메모리라, 그 서버 인스턴스가 죽으면 아직 flush 안 된 데이터가 날아간다 — 즉 직접 구현엔 **내구성(durability) 책임**까지 따라온다.

> **여기서 한 번 끊고**: 결과(S3에 데이터)가 같아 보이는 건 "데이터가 들어간다"는 한 측면만 봤기 때문이다. 실제로는 **모양(파일 크기·포맷·파티션)·비용·내구성·운영 부담**이 전부 다르다.

---

## 2. Firehose가 대신해 주는 것

Kinesis Data Firehose는 **"스트리밍 데이터를 목적지에 적재(load)하는 일"에 특화된 완전관리형 서비스**다. 위에서 직접 만들어야 했던 목록을 설정값으로 끝낸다.

### 버퍼링 — 설정 두 개로 끝

Firehose는 들어오는 레코드를 내부 버퍼에 모았다가, 둘 중 **먼저 도달하는 조건**에서 한 파일로 묶어 내보낸다.

- **Buffer size** (MB 단위): "이만큼 쌓이면 flush"
- **Buffer interval** (초 단위): "이 시간이 지나면 flush"

즉 트래픽이 많으면 크기 조건으로, 적으면 시간 조건으로 flush된다. 이 한 쌍이 위의 "작은 파일 폭탄"을 자동으로 막아 준다. (레코드 1건 최대 크기는 1,000 KB.)

### 완전관리형 — 서버·스케일·HA를 안 만진다

- 프로비저닝/패치할 인스턴스가 없다. 처리량에 맞춰 **자동 스케일**.
- 데이터는 적재되기 전까지 **3개 시설(AZ)에 동기 복제**로 보관 → 가용성/내구성을 Firehose가 책임진다. (직접 구현 시 인메모리 버퍼가 날아갈 위험과 대비됨)

### 포맷 변환 · 압축 · 암호화 — 빌트인

| 기능 | 직접 구현 | Firehose |
|------|----------|----------|
| JSON → Parquet/ORC | Spark/람다로 변환 잡 작성 | **설정으로 내장 변환** |
| 압축(gzip/snappy 등) | 코드로 처리 | 설정 |
| 저장 시 암호화(KMS) | SDK로 직접 | 설정 |

Parquet/ORC 같은 **컬럼형 포맷**으로 바꿔 두면 Athena가 필요한 컬럼만 읽어 **스캔량(=비용)이 크게 줄어든다**.

### 동적 파티셔닝 — 분석 비용을 직접 깎는 기능

Firehose는 **레코드 안의 데이터 키**(예: `customer_id`, `event_type`, 날짜)를 보고 S3 prefix를 자동으로 갈라 준다.

```
동적 파티셔닝 ON:
  s3://logs/year=2026/month=06/day=21/event=click/...
  s3://logs/year=2026/month=06/day=21/event=purchase/...
```

왜 중요한가: Athena/EMR/Redshift Spectrum/QuickSight는 **WHERE 조건이 파티션과 맞으면 그 prefix만 스캔**한다. 즉 "6월 21일 purchase만" 조회 시 나머지를 안 읽는다 → **스캔 비용↓**. 직접 하면 별도 파티셔닝 앱을 짜야 할 일을 설정으로 끝낸다.

> 운영 메모: 동적 파티션에는 **기본 활성 파티션 한도 500개**(요청 시 2,500까지)가 있고, 동적 파티셔닝을 켜면 end-to-end 지연이 버퍼링 힌트의 **약 1.5배** 정도까지 늘 수 있다.

### 그 외 — 변환 훅 · 실패 격리 · 다중 목적지

- **인라인 Lambda 변환**: 적재 직전 레코드를 가공(필드 마스킹, 정제 등).
- **실패 레코드 격리**: 변환/적재에 실패한 레코드를 잃지 않고 **S3의 에러 prefix**에 따로 떨군다. 직접 구현 시 가장 빼먹기 쉬운 부분이다.
- **다중 목적지**: 같은 데이터를 S3뿐 아니라 **Redshift(S3 경유 COPY), OpenSearch, Splunk, Apache Iceberg 테이블, HTTP 엔드포인트(서드파티)** 로도 보낼 수 있다. 목적지를 바꿔도 프로듀서 코드는 그대로.
  - 소스(입력)도 다양: 커스텀 앱(SDK), **Kinesis Data Streams**, **Amazon MSK**, 일부 AWS 서비스.

### 비용 감각

Firehose는 **수집한 데이터양(GB) 기준 종량제**다. (포맷 변환·동적 파티셔닝 등 부가 기능은 별도 과금.) 예컨대 S3로 적재하는 경우 GB당 대략 **$0.029 수준**(Direct PUT·첫 500 TB/월 구간)으로 시작한다. (리전·구간·부가 기능에 따라 단가는 달라진다.) — 핵심은 "직접 만들 엔지니어링·운영 시간"과 이 종량 요금을 비교해야 한다는 점이다.

## 3. 언제는 그냥 S3 직접이 맞나 (균형)

Firehose가 항상 정답은 아니다. **이미 큰 파일을 가끔 통째로 올리는** 워크로드라면 Firehose는 오히려 과하다.

| 상황 | 추천 | 이유 |
|------|------|------|
| 야간 ETL 결과 파일 1개를 하루 1번 업로드 | **S3 직접 PutObject** | 이미 크고 가끔 → 버퍼링·배칭이 필요 없음 |
| Spark 잡이 만든 Parquet 출력 저장 | **S3 직접** | 이미 큰 파일·올바른 포맷, Firehose가 더 할 게 없음 |
| 잘게·끊임없이 들어오는 이벤트/로그 스트림 | **Firehose** | 작은 파일 폭탄·배칭·파티셔닝을 대신 처리 |
| 실시간 유입을 분석 스토어로 모으기만 | **Firehose** | 적재 파이프라인을 통째로 위임 |

**판별 질문 하나**: *"내가 모으는 데이터가 잘게·끊임없이 들어오는가, 아니면 이미 큰 덩어리가 가끔 오는가?"* 전자면 Firehose, 후자면 직접이 보통 맞다.

---

## 4. 보너스 — Kinesis Data Streams vs Firehose (이름이 같아 헷갈리는 핵심)

둘 다 "Kinesis"라 헷갈리지만 **역할이 다르다.** 한쪽은 **실시간 스트림 버스**, 한쪽은 **관리형 적재기**다.

| | **Kinesis Data Streams (KDS)** | **Kinesis Data Firehose** |
|---|---|---|
| 한 줄 정체 | 실시간 스트림을 **저장·재생**하는 버스 | 목적지로 **적재**하는 관리형 파이프라인 |
| 소비자 | **직접 작성**(KCL / Lambda 등) | 없음 — Firehose가 알아서 목적지로 |
| 다중 소비자 | O (같은 데이터를 여러 앱이 각자 소비) | X (정해진 목적지로 적재) |
| 재처리/재생 | O — 보존 기간 내 다시 읽기 (**최대 365일**) | X (한 번 적재되면 끝) |
| 지연 | **서브초**(put-to-get), EFO로 **~70ms** | **근실시간**(버퍼링 때문에 초~분 단위) |
| 순서 보장 | **샤드 단위로 순서** 보장 | 보장 X |
| 운영 부담 | 샤드/용량 직접 관리(또는 온디맨드) | **완전관리형**, 자동 스케일 |

### 자주 쓰는 조합

실무에선 **둘을 같이** 쓴다. 실시간 처리도 하고 적재도 해야 할 때:

```
[프로듀서]
    │
    ├──> Kinesis Data Streams ──> 실시간 소비자(Lambda/KCL): 알림·집계·이상탐지
    │            │
    │            └──(KDS를 소스로)──> Firehose ──> S3 (분석용 적재)
```

KDS가 "여러 소비자가 실시간으로 같은 스트림을 각자 처리 + 필요하면 재생"을 맡고, 그 KDS를 **Firehose의 소스로 연결**해 S3 적재는 Firehose에 위임한다. 적재 코드를 직접 안 짜도 되는 이유가 여기 있다.

## 5. 한 장 의사결정 요약

```
질문: 이 데이터로 뭘 하려는가?

┌─ "그냥 S3/분석 스토어에 모으기만 하면 됨"
│     └─> Kinesis Data Firehose
│         (버퍼링·포맷변환·파티셔닝·실패격리를 위임)
│
├─ "여러 소비자가 실시간 처리 + 재처리/순서 보장 필요"
│     └─> Kinesis Data Streams
│         (+ S3 적재가 필요하면 KDS를 소스로 Firehose 붙이기)
│
└─ "이미 큰 배치 파일을 가끔 올림 (야간 ETL 등)"
      └─> S3 직접 PutObject
          (버퍼링·배칭이 불필요, Firehose는 과함)
```

## 면접에서 이렇게 답하라

> "결과가 S3에 쌓이는 건 같아 보여도, **직접 PutObject로 쏘면 적재 파이프라인을 전부 직접 만들게 됩니다.**
> 레코드마다 PUT을 하면 작은 파일이 수백만 개로 불어나 Athena·Spark 쿼리가 느려지고 PUT 요청 비용도 누적됩니다. 이걸 막으려면 **버퍼링·배칭·압축·포맷 변환(JSON→Parquet)·파티셔닝·재시도/실패 격리**를 직접 구현해야 하고, 인메모리 버퍼는 인스턴스가 죽으면 유실될 위험까지 있습니다.
> **Kinesis Data Firehose**는 이걸 완전관리형으로 대신합니다. Buffer size/interval로 자동 배칭하고, 컬럼형 포맷 변환·압축·KMS 암호화·동적 파티셔닝으로 **분석 스캔 비용을 줄여 주고**, 실패 레코드는 S3 에러 prefix로 격리하며, S3·Redshift·OpenSearch 등 여러 목적지를 지원합니다.
> 다만 **이미 큰 파일을 가끔 올리는 배치**라면 Firehose는 과합니다 — Firehose는 *잘게 끊임없이 들어오는 스트림*에서 빛납니다.
> Firehose와 헷갈리는 **Data Streams**는 적재기가 아니라 **실시간 스트림 버스**입니다. 다중 소비자·재처리(보존 최대 365일)·샤드 단위 순서 보장이 필요할 때 쓰고, 실무에선 **KDS로 실시간 처리 + 그 KDS를 소스로 Firehose가 S3 적재**하는 조합을 자주 씁니다."
