한 줄 요약

트래픽을 줄이고, 분산하고,
버틸 수 있게.
처음부터 다 하지 않는다 — 캐싱만 잘해도 대부분 해결된다.
##

3가지 축, 한눈에

①줄인다서버까지 요청이 안 오게 — Caching · CDN
②분산한다여러 서버가 나눠 받게 — Load Balancing · Read Replica · Sharding
③버틴다처리 용량을 늘려 — Scaling · Async · Rate Limiting
##

① 줄인다 — Caching

서버까지 요청이
안 오게 하는 것이 최선.
읽기 Hit Rate 90%면 읽기 DB 부하는 약 1/10로 줄어든다.
##

캐시는 여러 계층에

①Browser Cache클라이언트 · Cache-Control 헤더 (TTL 분~일)
②CDNEdge Server · 정적 리소스(이미지·JS·CSS) (분~시간)
③App CacheRedis/Local · 비즈니스 데이터·세션 (초~분)
④DB 레벨Materialized View (MySQL Query Cache 아님 — 8.0 제거)
##

Cache-Aside 패턴

가장 많이 씀
ProductService.kt
fun getProduct(id: Long): Product {
    redis.get("product:$id")?.let { return it }  // 1) Hit
    val product = db.findById(id)                  // 2) Miss → DB
    redis.set("product:$id", product, 5.minutes) // 3) 채움
    return product
}
앱이 직접 캐시를 관리한다. 가장 단순하고 일반적. 쓰기형은 Write-Through / Write-Behind.
!!

캐시의 진짜 어려움

⚠️
가장 어려운 문제는 Cache Invalidation — "어려운 2가지: 캐시 무효화와 이름 짓기" 그리고 MySQL Query Cache는 전역 뮤텍스 병목으로 5.7.20 deprecated → 8.0에서 완전 제거됐다. 쓰지 말 것.
→ DB 레벨 캐싱이 필요하면 Materialized View를 쓴다.
##

Scale-Up vs Scale-Out

Scale-Up (수직)
서버 스펙 ↑ · 코드 변경 없음
단, 물리 한계 + 단일 장애점 · 고스펙일수록 비용 비선형
Scale-Out (수평)
서버 수 ↑ · 이론상 무한 확장
단, 상태 관리 복잡 + 정합성 · 비용 선형
처음엔 Scale-Up, 한계 오면 Scale-Out. Scale-Out엔 Stateless가 필수 — 세션은 Redis, 파일은 S3.
##

② 분산 — L4 vs L7

L4 (Transport)
IP · Port 기준 · 빠름
단순 분산 · 예: AWS NLB
L7 (Application)
URL · Header · Cookie 기준
경로 기반 라우팅 · 예: ALB, Nginx
하위 계층일수록 빠르고 단순, 상위 계층일수록 똑똑한 라우팅이 가능하다.
##

분산 알고리즘

①Round Robin순서대로 배분 (서버 스펙 동일할 때)
②Least Conn연결 적은 서버로 (처리 시간이 다양할 때)
③IP Hash출발지 IP 해싱 → 같은 IP는 같은 서버 (LB 무상태, 세션 유지)
⚠️
IP Hash는 서버 수가 바뀌면 재분배되어 세션이 유실될 수 있다 + NAT/CGNAT 뒤 다수 클라이언트가 한 IP를 공유하면 쏠림 발생
##

Read Replica로 읽기 분산

읽기 트래픽이 80% 이상인
서비스가 대부분.
⚠️
Replication Lag — 쓰기 직후 Replica에서 읽으면 이전 데이터가 보일 수 있다 Write는 Primary, Read는 Replica로 분산. 강한 일관성이 필요하면 Primary 읽기.
##

③ 버틴다 — 비동기 처리

모든 요청을 즉시 처리할 필요는 없다
// 동기: 결제 → 재고 → 알림 → 응답 (3초)
Client → API → 결제 → 재고차감 → 알림 → Response

// 비동기: 핵심만 처리하고 응답 (0.5초)
Client → API → 결제 → Response
                 ↳ Message Queue → 재고차감 · 알림
응답 시간 ↓ · 각 처리가 독립적으로 실패·재시도 가능. 도구: Kafka · RabbitMQ · SQS.
##

Rate Limiting

초과 → 429
①Token Bucket일정 속도로 토큰 충전, 있을 때만 허용 (버스트 허용)
②Sliding Window시간 창 내 요청 수 제한 (정확하지만 메모리 사용)
③Fixed Window고정 시간 단위 카운트 (간단하나 경계에서 2배 허용 가능)
처리 용량 이상의 요청은 429 Too Many Requests로 거부해 시스템을 지킨다.
##

실무는 단계적으로

1캐싱 (Redis·CDN)DB 부하 급감 ← 여기서 대부분 해결
2Read Replica읽기 분산
3Auto Scaling트래픽 대응 탄력성
4비동기 (MQ)응답 시간 단축
5샤딩쓰기 분산 (최후 수단 — 극소수만 필요)
