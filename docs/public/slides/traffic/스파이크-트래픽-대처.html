##

스파이크란?

일반 대용량
점진적 증가 · 예측 가능
대응 시간 여유
스파이크
수초~수분 내 10~100배
대응 시간 거의 없음
예: 자정 선착순 쿠폰, 수능 성적 발표 — 순간 폭증에 살아남는 게 관건.
##

대처 전략, 한눈에

①사전 준비예측 가능 → 미리 띄운다 (Scheduled Scaling·워밍업)
②트래픽 제어들어오는 양 조절 → Rate Limit·대기열·Throttling
③비동기 버퍼링MQ로 피크 평탄화 (Peak Shaving)
④인프라 방어Circuit Breaker·Bulkhead·Timeout으로 장애 차단
!!

오토스케일의 함정

새 인스턴스는 즉시 뜨지 않는다
스파이크 → 감지(30초) → 기동(1~10분+) → 헬스체크(30초)
                              ↑
              이 수 분 동안 기존 서버가 먼저 죽는다
감지부터 트래픽 받기까지 수 분의 공백. 스파이크엔 너무 느리다.
##

Pre-Warming

미리 띄운다
Scheduled Scaling (AWS)
// 자정 쿠폰 이벤트 → 23:50에 미리 증설
{
  "StartTime": "2026-03-22T23:50:00Z",
  "DesiredCapacity": 20,  // 평소 4대 → 20대
  "MinSize": 20
}
정기 패턴은 Scheduled Scaling으로 선제 증설. JVM JIT 워밍업도 함께.
##

커넥션 풀 워밍업

application.yml — HikariCP
hikari:
  minimum-idle: 50          # 미리 50개 연결 확보
  maximum-pool-size: 100
  connection-timeout: 3000  # 못 받으면 빠르게 실패
⚠️
커넥션 생성은 비싸다 (TCP handshake + DB 인증)스파이크 때 생성이 몰리면 DB가 먼저 죽는다 → 미리 채워둔다.
##

Rate Limiting

초과는 거부
처리 가능 10,000/s · 유입 100,000/s
90,000개는 429로 거부
→ 서버는 살아있다
서버가 죽으면 100% 실패. 일부를 버려 전체를 살린다.
##

대기열 — Sorted Set

선착순 정석
Redis Sorted Set 대기열
# 1. 입장 — timestamp를 score로
ZADD waiting_queue {timestamp} {userId}

# 2. 순번 확인
ZRANK waiting_queue {userId}     → 32847번째

# 3. Worker가 앞에서부터 꺼내 처리
ZPOPMIN waiting_queue 100      → 100명씩
"기다리는 중"이 "접속 불가"보다 100배 낫다. 티켓팅·예약의 정석.
##

Throttling vs Rate Limit

Rate Limiting
초과 시 거부 (429)
API·봇 방어에 적합
Throttling
초과 시 지연·품질 저하
사용자 대면에 적합
Graceful Degradation — 추천·실시간 재고는 끄고, 상품 상세 + 캐시 리뷰만 살린다.
##

MQ로 Peak Shaving

순간을 평탄화
Producer 즉시 응답 · Consumer 일정 속도
// Producer — 즉시 202 Accepted
kafkaTemplate.send("coupon-request", event)

// Consumer — 일정 속도로 처리
@KafkaListener(topics=["coupon-request"], concurrency="5")
fun process(e: CouponEvent) { ... }
스파이크는 MQ가 흡수하고, Consumer가 일정 속도로 드레인한다.
!!

concurrency ≠ 처리율 제어

⚠️
Consumer 처리량의 상한은 토픽의 파티션 수다 파티션 5개면 concurrency 5 초과는 스레드가 놀 뿐. concurrency는 병렬도를 늘릴 뿐, 드레인 속도를 늦추는 레이트 리미터가 아니다.
→ 진짜 처리율 제한은 Consumer 안에 명시적 RateLimiter로.
##

Circuit Breaker

장애 전파 차단
상태 전이 · Resilience4j 기본값
CLOSED ─ 실패율 50%↑ (failureRateThreshold) ─→ OPEN
                                              ↓ 일정 시간
                                          HALF-OPEN
   성공 → CLOSED   /   실패 → OPEN
⚠️
실패율은 minimumNumberOfCalls(기본 100)를 채운 뒤에야 평가호출 적은 엔드포인트는 100% 실패해도 임계 호출 수 미달이면 OPEN 안 됨.
##

Bulkhead · Timeout

격벽으로 격리 · 모든 외부 호출에 타임아웃
// Bulkhead — 스레드풀 격리
결제 Pool(40) · 상품 Pool(40) · 리뷰 Pool(20)
→ 결제 지연돼도 상품·리뷰는 정상

# Timeout — 필수
read-timeout: 3s    // timeout: 0(무한) = 스레드 고갈 = 사망
하나가 죽어도 전체가 안 죽게 격리하고, 무한 대기를 막는다.
