한 줄 요약

전부 올리지 말고,
흘려보내라.
DB → 앱 → 파일/응답을 스트림으로 연결하면 메모리가 O(N) → O(1)로 바뀐다.
##

전형적인 OOM 코드

ExportService.java
List<Order> orders = orderRepository.findAll();  // 500만 건 → ~2GB

Workbook wb = new XSSFWorkbook();  // 전체를 메모리 DOM으로 → +2GB
for (Order o : orders) sheet.createRow(...);

wb.write(response.getOutputStream());  // 총 ~4GB → OOM
메모리는 DB 적재와 엑셀 생성 두 군데서 동시에 터진다.
List 2GB + Workbook 2GB · Heap 2GB
4GB
2GB 힙에 4GB를 올리려다 OutOfMemoryError. 데이터 건수에 비례해 무너진다.
##

해결 — 전 구간 스트리밍

배치 적재 → 스트리밍
// 기존: 전체를 메모리에 모았다가 한 번에
DB ──전체로드──▶ List ──전체생성──▶ Workbook ──출력──▶ 응답

// 개선: 1건 읽고 즉시 쓰고 버린다
DB ──커서──▶ 1건 읽기 ──즉시쓰기──▶ 파일/응답 스트림
        └─ 메모리: 처리 중인 1건(또는 청크)만 점유
읽는 즉시 쓰고 버린다 → 메모리가 데이터량과 무관해진다.
##

① DB 조회를 Stream으로

JPA Stream
OrderRepository / Service
@QueryHints(@QueryHint(name = HINT_FETCH_SIZE, value = "1000"))
Stream<Order> streamByCreatedAtBetween(...);

@Transactional(readOnly = true)  // 커서가 트랜잭션에 묶임 — 필수
stream.forEach(o -> { writeRow(o); em.detach(o); });
Stream<T> 반환 시 Hibernate ScrollableResults 사용. 반드시 트랜잭션 안에서.
##

저수준 — JDBC 커서 직접

정밀 제어
JdbcTemplate.query
PreparedStatement ps = con.prepareStatement(sql,
    ResultSet.TYPE_FORWARD_ONLY,   // 순방향 전용
    ResultSet.CONCUR_READ_ONLY);   // 읽기 전용
ps.setFetchSize(1000);  // MySQL은 양수 무시 → 아래 슬라이드 참고

while (rs.next()) writeRow(out, rs.getLong("id"), ...);
한 행씩 읽어 즉시 쓴다. TYPE_FORWARD_ONLY + CONCUR_READ_ONLY가 스트리밍 전제.
!!

커서 동작 — DB별 함정

MySQL · row-by-row
fetchSize = Integer.MIN_VALUE → 서버가 보내고 한 행씩 읽음
MySQL · 서버 커서
useCursorFetch=true + 양수 fetchSize → 진짜 서버 커서
MySQL 기본값은?
전체 메모리 적재 — Stream 써도 그냥 OOM
PostgreSQL
fetchSize > 0 + autoCommit=false → 자동 서버 커서
!!

영속성 컨텍스트 누수

⚠️
스트림으로 읽어도 detach 안 하면 1차 캐시에 계속 쌓여 결국 OOM 500만 건 처리 후 영속성 컨텍스트에 500만 엔티티가 남는다. 처리 직후 em.detach(o) 또는 StatelessSession 사용.
→ 핵심: 읽고 → 쓰고 → 즉시 detach. 윈도우 밖은 GC가 회수한다.
##

② 엑셀 — SXSSF 윈도우

~10MB
ExportService.java
SXSSFWorkbook wb = new SXSSFWorkbook(100);  // 메모리에 100행만 유지
wb.setCompressTempFiles(true);     // 임시 파일 압축
orders.forEach(o -> createRow(o));   // 윈도우 초과분은 디스크로 플러시
wb.write(out);
finally { wb.dispose(); }  // 임시 파일 정리 — 미호출 시 디스크 누수
XSSF가 전체를 DOM으로 들고 있다면, SXSSF는 최근 100행만 메모리에 두고 나머지는 임시 파일로.
##

엑셀 라이브러리 비교

100만 행 기준
①XSSFWorkbook~2GB · 전체 메모리 적재 (소규모 전용)
②SXSSFWorkbook~10MB · 슬라이딩 윈도우 (임시 파일 사용 · dispose 필수)
③EasyExcel~5MB · Alibaba · 가장 가볍고 API 간단 (1000건씩 flush)
④CSV 스트리밍거의 0 · 한 줄씩 쓰기 · UTF-8 BOM 필수 (서식 불가)
!!

스트리밍의 대가 — 커넥션

⚠️
스트리밍은 DB 커넥션을 오래 점유한다 500만 건 × 0.01ms = ~50초 점유. 풀이 10개인데 5명이 동시에 받으면 나머지 5개로 전체 서비스를 운영해야 한다.
→ 대응: Replica 분리 · 동시 다운로드 제한(Semaphore) · 비동기 전환
##

대용량의 정석 — 비동기 + S3

10만 건↑
AsyncExportService.java
@Async @Transactional(readOnly = true)
void generateExcelAsync(...) {
    exportExcelStreaming(out, orders);   // 백그라운드에서 생성
    s3Client.putObject(bucket, key, file);
    String url = s3.generatePresignedUrl(...);  // 1h 유효
    notify(requestId, url);  // 완료 알림 → S3가 직접 서빙(서버 부하 0)
}
요청은 202 Accepted로 즉시 응답. 파일은 Worker가 만들어 Pre-signed URL로 내려준다.
!!

@Async · Pre-signed 함정

💥
@Async self-invocation — 같은 클래스 내부 호출은 프록시를 안 타서 비동기가 안 된다 void @Async의 예외는 호출자에 전파 안 됨 → AsyncUncaughtExceptionHandler 필요.
🔒
Pre-signed URL은 유출 시 인증 우회 — 민감 데이터면 만료 5~15분 · 감사 로그 · HTTPS 강제
