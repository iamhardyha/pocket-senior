한 줄 요약

장애 대응은 기술이 아니라
프로세스다.
개인 역량이 아니라 팀 체계로 해결한다.
##

장애 대응 5단계

①감지 Detection모니터링·신고로 빨리 안다 (0~1분)
②공유 Communication채널 오픈·IC 지정 (1~5분)
③완화 Mitigation원인보다 복구 우선 (5~15분)
④복구 Recovery근본 원인·정상 확인 (15분~)
⑤회고 Post-mortem재발 방지 (1~3일 내)
##

① 감지 (Detection)

빨리 아는 것이
절반이다.
모니터링 알람과 사용자 신고로 발생 1분 안에 인지하는 것이 목표.
##

Golden Signals 4지표

Google SRE
LLatency응답 시간 · p99 > 3초
TTraffic요청량 · QPS 급등/급락
EErrors에러율 · 5xx > 1%
SSaturation자원 포화도 · CPU·Mem 한계 (서비스별 조정)
##

알람 임계 단계

지표가 악화되며 대응 강도가 올라간다
정상  Error 0.1%  · p99 200ms · CPU 40%
경고  Error 1%    · p99 1s    · CPU 70%   ← 슬랙 알림
위험  Error 5%    · p99 5s    · CPU 90%   ← 전화/PagerDuty
장애  Error 30%   · p99 30s   · CPU 100%  ← 전원 소집
급락도 장애다. 트래픽이 0이면 앞단 LB·DNS를 의심하라.
!!

알람은 Actionable해야 한다

⚠️
알람이 너무 많으면 무시하게 된다 — Alert Fatigue 울리면 누군가 반드시 뭔가를 해야 하는 상황만 남겨라. 진짜 중요한 것만.
스택 예시: Prometheus·Grafana → AlertManager → Slack/PagerDuty
##

② 공유 (Communication)

ICIncident Commander전체 지휘·의사결정
TLTech Lead원인 분석·수정 방향
CMCommunicator이해관계자·고객 공유
OPOperator실제 조치 실행(배포·설정)
5분 안에 공유하지 않으면 늦은 것. 상태 업데이트는 10분마다. 침묵이 최악이다.
##

③ 완화 (Mitigation)

원인을 찾는 동안
서비스는 계속 죽어있다.
완벽한 수정보다 빠른 완화가 100배 중요하다.
##

즉시 완화 조치

↺롤백최근 배포 후 장애 — 가장 흔한 원인 (1~5분)
⏻Feature Flag OFF특정 기능 문제 — 즉시
⛔Rate Limiting트래픽 급증 — + 서버 증설
⚡Circuit Breaker외부 API 장애 — + Fallback
모르겠으면? → 일단 최근 배포 롤백.
##

④ 복구 (Recovery)

복구 확인 체크리스트
□ Error Rate 정상 수준 회복?
□ Latency 정상 수준 회복?
□ 밀린 데이터 처리(MQ 적체 등) 완료?
□ 관련 팀에 복구 완료 공유?
□ 추가 모니터링 구간에 알람 설정?
데이터 정합성까지 확인해야 진짜 복구. 결제됐는데 주문이 없는 건은?
##

⑤ Blameless Post-mortem

✅
사람을 탓하지 않는다. 시스템을 고친다. 타임라인 · 근본 원인 · 재발 방지 액션 아이템(담당·기한)을 문서로 남긴다.
교훈은 개인이 아니라 시스템에 남긴다 — 같은 장애를 두 번 겪지 않도록.
##

장애 등급 (Severity)

1SEV-1서비스 전체 불가·매출 영향 — 전원 소집, 10분 내 완화
2SEV-2주요 기능 장애 — 온콜+관련팀, 30분 내 완화
3SEV-3부분 기능 저하 — 온콜이 업무 시간 내 처리
4SEV-4경미한 이슈 — 다음 스프린트에 수정
##

관통하는 한 가지 원칙

완화 vs 수정?
빠른 완화 > 완벽한 수정
혼자 vs 팀?
팀 체계 — 5분 내 공유
사람 vs 시스템?
Blameless — 시스템을 고친다
복구 기준은?
정합성까지 확인 완료
