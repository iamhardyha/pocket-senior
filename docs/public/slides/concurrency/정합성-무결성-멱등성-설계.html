정합성은 맞는가,
무결성은 올바른가,
멱등성은 여러 번 해도 같은가.
별개가 아니라 서로 보완 관계 — 셋 다 지켜야 신뢰할 수 있다.
##

A → B 10만원 이체로 보면

정정합성 ConsistencyA가 -10만이면 B는 반드시 +10만 · 총합이 안 맞으면 깨짐
무무결성 Integrity잔고는 음수 불가, 계좌는 존재해야 · 규칙 위반 데이터 금지
멱멱등성 Idempotency같은 요청 2번 보내도 1번만 실행 · 몇 번 해도 결과 동일
!!

깨지면 무슨 일?

잔고가 -500원
멱등성이 깨지면
결제가 2번 됨

신뢰할 수 있는 시스템
##

정합성 ① 단일 DB는 트랜잭션

OrderService.java
@Transactional  // 전부 성공하거나, 전부 실패
public void placeOrder(OrderRequest req) {
    inventoryService.decrease(req.productId(), req.qty());  // 1
    Order order = orderRepository.save(Order.from(req)); // 2
    paymentService.pay(order.getId(), req.amount());        // 3
}  // 3에서 실패하면 1·2도 롤백
한계: 트랜잭션은 단일 DB 안에서만 동작한다.
##

정합성 ② 서비스 너머는 결과적 일관성

Strong Consistency
모든 시점에 일치 → 성능 포기
Eventual Consistency
일시 불일치 OK, 결국 일치 → 성능 유지
MSA는 DB가 서비스마다 다르다. 분산 트랜잭션은 어렵다 → 실무는 대부분 Eventual.
##

패턴 — Outbox + CDC

주문 저장과 이벤트 발행을 한 트랜잭션으로
@Transactional
  orders  INSERT      // 1) 비즈니스 데이터
  outbox  INSERT      // 2) 발행할 이벤트 (같은 DB·같은 TX)

  └─ CDC(Debezium) / Polling ─→ Kafka ─→ 결제·재고·알림
저장과 발행이 원자적 → 메시지 유실 방지. Consumer 장애도 Kafka에 남아 복구 가능.
##

패턴 — Saga (보상 트랜잭션)

실패하면 거꾸로 되돌린다
// 성공
주문생성 → 결제 → 재고차감 → 배송요청
// 재고에서 실패 → 보상 트랜잭션 역순 실행
주문생성 → 결제 → 재고부족 → 결제환불 → 주문취소
Choreography
서비스가 이벤트로 다음 단계 트리거 · 소규모
Orchestration
중앙 오케스트레이터가 순서 제어 · 대규모
##

신뢰하되 검증하라

주문-결제 정합성 체크 (매일 배치)
SELECT o.id, o.status, p.status
FROM orders o LEFT JOIN payments p ON o.id = p.order_id
WHERE o.status = 'COMPLETED'
  AND (p.id IS NULL OR p.status != 'PAID');
결과가 있으면 정합성 깨진 것 → 알람 → 수동 보정. 비동기 설계의 안전망.
##

무결성 ① DB 제약 조건

가장 강한 방어선
¬∅NOT NULL필수값 보장 · 주문에 user_id 필수
≠UNIQUE중복 방지 · 이메일 유일성
✓CHECK값 범위 제한 · amount > 0
→FOREIGN KEY참조 무결성 · 존재하는 사용자만
##

무결성 ② 3단계로 막는다

①입력 Validation@NotNull · @Min · @Pattern — Controller에서 형식 검증
②비즈니스 규칙상태 전이·재고 확인 — Service/도메인에서 검증
③DB 제약NOT NULL·CHECK·FK — 최후의 보루
✅
원칙: 앞에서 막을수록 비용이 싸다. 입력 → 비즈니스 → DB.
##

DB로 못 막는 규칙은 코드로

Order.java — 상태 전이 규칙
public void cancel() {
    if (status == OrderStatus.SHIPPING)
        throw new IllegalStateException("배송 중 취소 불가");
    if (paidAt.isBefore(now().minus(24, HOURS)))
        throw new IllegalStateException("24시간 경과, 취소 불가");
    status = OrderStatus.CANCELLED;
}
CHECK로 표현 못하는 상태 머신·시간 규칙은 도메인 객체가 지킨다.
##

멱등성 ① f(f(x)) = f(x)

✓GET /orders/123멱등 · 몇 번 호출해도 같은 결과
✓DELETE /orders/123멱등 · 이미 삭제돼도 결과 동일
✕POST /orders비멱등 · 호출마다 새 주문 생성
⚠POST /payments비멱등 · 호출마다 결제 — 이게 위험
##

멱등성 ② Idempotency Key

PaymentController.java
@PostMapping("/payments")
public ... pay(@RequestHeader("Idempotency-Key") String key, ...) {
    var hit = idempotencyStore.get(key);    // 1) 이미 처리?
    if (hit.isPresent()) return hit.get();  // 저장 결과 반환
    var result = paymentService.pay(req); // 2) 처리
    idempotencyStore.save(key, result, Duration.ofHours(24));
}
같은 키로 재시도하면 저장된 결과를 그대로 반환 → 이중 결제 방지.
!!

함정 — 락 없으면 이중 실행

⚠️
앞 코드는 단순화한 happy path다. get → pay() → save 사이에 락이 없다 동시 재시도가 둘 다 "없음"으로 통과 → 이중 실행 레이스 발생.
✅
안전하게: SET-NX in-progress 락 또는 키에 DB Unique 제약
