# 쿠폰 중복처리
정합성

const question = "선착순 100장인데 101장이 나갔다. 왜?";
// 중복을 어떻게 막고, 정합성을 어떻게 지킬까?
>

한 줄 요약

정합성 = 확인과 실행을
원자적으로 만드는 것.
쿠폰에서 배운 패턴은 재고·포인트·좋아요 모든 차감/증가에 그대로 쓰인다.
##

왜 중복이 생기는가

쿠폰 잔여 1장 · 동시 요청
User A: ──READ(1)────────WRITE(0)──
User B: ──────READ(1)────────WRITE(0)──  // 같은 값을 읽음

// 결과: 1장 남았는데 둘 다 발급 → 2장 나감
Race Condition — "확인"과 "실행"이 원자적이지 않아서다.
##

해결 전략, 한눈에

①DB Unique 제약중복 INSERT를 DB가 차단 · 1인 1쿠폰 정석
②Pessimistic Lock먼저 잡은 사람만 · 정합성 최우선
③Optimistic Lock충돌 감지 후 재시도 · 충돌 적을 때
④Redis Atomic + Lua단일 스레드 원자 명령 · 고성능 선착순
⑤MQ 직렬화요청을 줄 세워 순차 처리 · 대규모 선착순
##

① DB Unique 제약

1인 1쿠폰
coupon_issue
CREATE TABLE coupon_issue (
    coupon_id BIGINT, user_id BIGINT,
    UNIQUE KEY uk_coupon_user (coupon_id, user_id)  -- 핵심
);
// 두 번째 발급 → DataIntegrityViolationException → 중복 거부
코드와 무관하게 DB가 보장한다. 단, "총 100장" 수량 제한은 별도로 막아야 한다.
##

② Pessimistic Lock

일단 잠근다
CouponRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT c FROM Coupon c WHERE c.id = :id")
Coupon findByIdForUpdate(@Param("id") Long id);
// SELECT ... FOR UPDATE → 커밋까지 다른 트랜잭션 대기
확실하다. 데이터가 절대 안 꼬인다. 단 직렬화 → 처리량 급감, Timeout·Deadlock 위험.
##

③ Optimistic Lock

충돌하면 재시도
@Version 으로 충돌 감지
@Version private Long version;   // 엔티티 버전 필드

-- 커밋 시 실행되는 SQL
UPDATE coupon SET remaining=99, version=2
 WHERE id=1 AND version=1;  // 0 rows → 예외 → 재시도
Lock 없이 빠르다. 단 충돌이 많으면 재시도 폭발 → 오히려 느려진다.
##

비관 vs 낙관, 선택 기준

충돌이 잦은 선착순?
Pessimistic — 강력 보장, 직렬화
충돌 드문 일반 주문?
Optimistic — 병렬, 높은 처리량
초당 10건 이하?
Optimistic 로 충분
초당 100건 이상?
Pessimistic 또는 Redis
!!

④ Redis DECR의 함정

단순 DECR/INCR 롤백 패턴
Long remaining = redis.decrement(key);   // 원자적
if (remaining < 0) redis.increment(key);  // 롤백
// 발급 기록 ...
⚠️
DECR ≥0 직후 기록/롤백 전에 죽으면 카운터와 실제 발급이 불일치 과도기적 음수도 다른 읽기에 노출된다. 단독 사용 금지.
##

④ Redis + Lua Script

완벽한 원자성
issue_coupon.lua
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -1 -- 이미 발급 (중복)
end
local r = tonumber(redis.call('GET', KEYS[1]))
if not r or r <= 0 then return 0 end  -- 소진
redis.call('DECR', KEYS[1]); redis.call('SADD', KEYS[2], ARGV[1])
return 1 -- 성공
Script가 하나의 명령처럼 실행 → 확인·차감·기록이 한 방에. 끼어들 틈이 없다.
##

⑤ MQ 직렬화

대규모 선착순
API는 즉시 접수, Consumer가 순차 처리
kafkaTemplate.send("coupon-issue", req);  // 202 Accepted

@KafkaListener(topics="coupon-issue", concurrency="1")
void process(Request req) { /* 1건씩 발급 */ }
⚠️
concurrency="1"은 인스턴스당 스레드 수일 뿐 전역 직렬화엔 단일 파티션 또는 키(couponId) 기반 파티셔닝이 필요하다.
##

실전 조합 — 3중 안전망

1Redis Lua원자적 수량 차감 + 중복 방지 (1차 방어)
2DB Unique KeyRedis 장애에도 막아주는 최종 안전망 (2차)
3Consumer 멱등성메시지 중복 처리 방지 (3차 방어)
단일 전략이 아니라 겹겹의 방어막으로 설계한다.
##

사용 시 중복도 막는다

상태 머신으로 전이 제어
ISSUED ──사용──> USED   // (O) 정상 사용
USED   ──사용──> USED   // (X) 중복 사용 불가
EXPIRED──사용──> USED   // (X) 만료 쿠폰 불가
발급뿐 아니라 사용도 FOR UPDATE + 상태 검증으로 한 번만 전이시킨다.
##

운영 — 정합성 검증 배치

매일 배치로 불일치 감지
SELECT coupon_id,
  CASE WHEN COUNT(ci.id) != (total - remaining)
       THEN 'MISMATCH' ELSE 'OK' END
FROM coupon c LEFT JOIN coupon_issue ci ...
