# 락과
동시성 제어

const question = "동시에 같은 데이터를 건드릴 때, 정합성을 어떻게 지킬까?";
// 어떤 락을, 언제 고를까?
>

한 줄 요약

락은 '누가 먼저'를 정하는 규칙이다.
목적에 안 맞는 락을 고르면 성능을 죽이거나 데이터를 깨뜨린다.
##

락 없이 동시 차감하면?

balance = 1000 에서 동시에 차감
A: READ 1000 → -500 → WRITE 500
B:      READ 1000 → -300 → WRITE 700

기대값 = 1000 - 500 - 300 = 200
실제값 = 700   // A의 차감이 통째로 사라짐
한쪽 변경이 덮어쓰여 사라진다 → Lost Update.
##

동시성 문제 4가지

①Lost Update한쪽 변경이 사라짐 · 잔고 차감 누락
②Dirty Read커밋 안 된 값을 읽음 · 롤백될 데이터를 노출
③Non-Repeatable같은 조회인데 값이 다름 · 조회 중 수정됨
④Phantom Read같은 조회인데 행 수가 다름 · 조회 중 INSERT
##

락 전체 지도

①Pessimistic비관적 락 · S Lock / X Lock · FOR UPDATE
②Optimistic낙관적 락 · @Version / CAS
③Named Lock이름 기반 논리 락 · GET_LOCK
④Applicationsynchronized · 단일 JVM 한정
⑤분산 락Redis(Redisson) · 여러 서버 동기화
##

① 비관적 락

충돌 가정
"충돌이 일어날 거라 보고,
미리 잠근다."
S Lock(읽기) — 여럿이 같이 읽되 쓰기는 막는다.
X Lock(쓰기 · FOR UPDATE) — 하나만 접근, 나머지는 대기.
##

비관적 락 — JPA

X Lock
AccountRepository.java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdForUpdate(@Param("id") Long id);
확실한 정합성. 대신 대기 시간이 생기고 Deadlock 위험이 따라온다.
!!

함정 — Deadlock

💥
A는 1→2 순서로, B는 2→1 순서로 잠그면 서로의 락을 영원히 기다린다 엇갈린 락 획득 순서가 교착의 본질.
→ 방어: 항상 같은 순서로 잠그기(ID 오름차순) · Lock Timeout · 짧은 트랜잭션
##

② 낙관적 락

@Version
UPDATE 시 version을 조건에 건다
// 읽을 때 본 version = 1
UPDATE product SET stock = 99, version = 2
WHERE id = 1 AND version = 1;
// 누가 먼저 바꿨으면 version≠1 → 0 rows
// → OptimisticLockingFailureException
"충돌 안 날 거라 보고, 충돌하면 다시 한다." Lock을 안 잡아 빠르다.
##

충돌하면 재시도

@Retryable
StockService.java
@Retryable(
    retryFor = OptimisticLockingFailureException.class,
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, multiplier = 2))
@Transactional
public void decreaseStock(Long id, int qty) { … }
100ms → 200ms → 400ms 백오프. 충돌이 잦으면 재시도 비용이 커진다.
##

Version 없는 CAS

실무 최다
한 방의 조건부 UPDATE로 끝낸다
int updated = jdbcTemplate.update(
  "UPDATE product SET stock = stock - ? " +
  "WHERE id = ? AND stock >= ?", qty, id, qty);

if (updated == 0) throw new InsufficientStockException();
⭐
version 컬럼 없이 이전 값과 비교 → 간단하고 확실. 실무에서 가장 많이 쓰는 패턴.
##

③ Named Lock

이름 기반
MySQL — Row가 아니라 '이름'을 잠근다
SELECT GET_LOCK('coupon_issue_100', 3);  -- 3초 대기
-- ... 크리티컬 섹션 ...
SELECT RELEASE_LOCK('coupon_issue_100');
⚠️
커넥션을 점유한다 → Connection Pool 고갈 위험 반드시 Lock 전용 DataSource를 분리할 것. 분산 락의 간이 버전.
!!

④ synchronized의 벽

💥
synchronized·ReentrantLock은 단일 JVM 안에서만 동작한다 Server 1과 Server 2는 락을 공유하지 못한다 → 동시에 같은 데이터 수정 → 정합성 붕괴.
→ 쓸 곳: 서버 1대 또는 로컬 캐시 동시 접근 제어뿐.
##

⑤ 분산 락 — Redisson

서버 간 동기화
CouponService.java
RLock lock = redisson.getLock("lock:coupon:" + id);
boolean ok = lock.tryLock(10, 3, SECONDS); // 대기10·임대3
try { /* 크리티컬 섹션 */ }
finally { if (lock.isHeldByCurrentThread()) lock.unlock(); }
여러 서버가 한 자원에 줄 서게 한다. AOP 어노테이션으로 깔끔하게 감싸는 게 정석.
!!

함정 — Lock·Tx 순서

⚠️
Tx 안에서 락을 잡으면 락 해제 → 커밋 순서가 되어, 그 틈에 다른 스레드가 커밋 안 된 데이터를 읽는다 반드시 Lock이 Transaction을 감싸야 한다 — 커밋 완료 후 락 해제.
올바른 중첩: 락 획득 → @Transactional → 커밋 → 락 해제
!!

함정 — Lease Time

💥
임대 시간(3초)보다 처리가 길면 락이 먼저 풀려 다른 스레드가 진입 → 정합성 붕괴 반대로 임대가 없으면 서버 사망 시 락이 영원히 안 풀린다.
✅
Redisson Watchdog: leaseTime 생략 시 살아있는 동안 자동 연장 tryLock(10, 3, S)는 비활성 · tryLock(10, S)는 활성.
##

락 선택 가이드

서버가 1대인가?
synchronized / ReentrantLock
충돌이 드문가?
Optimistic (@Version / CAS)
충돌이 잦은가?
Pessimistic (FOR UPDATE)
여러 서버·여러 DB?
분산 락 (Redisson)
