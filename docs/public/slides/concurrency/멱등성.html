한 줄 요약

한 번 하든 열 번 하든 결과는 같다.
못 지키면 돈이 두 번 빠지고, 알림이 열 번 간다.
##

요청은 반드시 중복된다

1네트워크 타임아웃응답을 못 받은 클라이언트가 재시도
2LB · MQ 재전달로드밸런서 retry · MQ at-least-once
3사용자 더블 클릭버튼 두 번 → 요청 두 번
4모바일 끊김네트워크 단절 → 자동 재전송
!!

멱등하지 않은 결제 API

응답 전송 중 네트워크가 끊기면…
Client → 결제 요청 → Server 처리 완료 → 응답 중 끊김
Client → 응답 못 받음 → "실패인가?" → 재시도
Server → 또 결제 실행 → 돈 2배 빠짐 💥
서버는 처리했지만 클라이언트는 성공을 모른다. 그래서 또 보낸다.
##

HTTP 메서드, 어디가 위험한가

OGET · HEAD · PUT · DELETE조회·전체교체·삭제는 몇 번이든 결과 동일
XPOST생성 → 매번 새 리소스 · 항상 비멱등
△PATCHstock += 1 비멱등 · status = CANCELLED 멱등
→ 멱등성 보장이 필요한 건 POST · 일부 PATCH다.
##

구현 패턴 5가지, 한눈에

①Idempotency Key클라이언트 고유 키로 처리 여부 확인 · 가장 범용
②DB Unique 제약유니크 컬럼이 중복을 막음 · 가장 간단
③상태 기반상태 머신으로 이미 전이됐으면 무시
④CASWHERE 조건 + 로그 체크로 DB 레벨 멱등
⑤MQ ConsumerEventId로 중복 메시지 걸러냄
##

① Idempotency Key

가장 범용
PaymentController.java
@PostMapping("/payments")
ResponseEntity pay(@RequestHeader("Idempotency-Key") String key) {
    if (store.find(key).isPresent())  // 이미 처리?
        return ok(store.get(key));    // 저장된 결과 반환
    // 처음 보는 키 → 실제 처리 후 결과 저장 (TTL 24h)
}
클라이언트가 고유 키를 보내고, 서버는 이 키로 처리했는지만 본다.
##

동시 중복은 Redis SET NX로

RedisIdempotencyStore.java
boolean tryLock(String key) {
    // SET NX — 없을 때만 저장 (Lock 역할), 30s 자동 해제
    return redis.setIfAbsent(
        "idem:lock:" + key, "PROCESSING", Duration.ofSeconds(30));
}
// 락 실패 → 409 Conflict "처리 중입니다"
Key 생성 주체: 클라이언트가 UUID 생성해야 재시도 시 같은 키를 쓴다.
##

② DB Unique 제약

가장 간단
payment 테이블 + 저장 로직
CREATE TABLE payment (
    idempotency_key VARCHAR(64) NOT NULL UNIQUE  -- 이것만으로 방지
);

try { paymentRepository.save(payment); }
catch (DataIntegrityViolationException e) {
    return repo.findByIdempotencyKey(key);  // 이미 처리됨
}
DB가 유니크 위반을 던지면, 기존 결과를 조회해 돌려준다.
##

③ 상태 기반 — 상태 머신

Order.java
public void pay() {
    if (status != PENDING) return;  // 이미 결제·취소 → 무시
    this.status = PAID;
}
public void cancel() {
    if (status == CANCELLED) return;  // 이미 취소 → 멱등
}
상태 전이가 명확하면, 같은 요청이 와도 이미 전이된 상태라 안전하다.
##

④ CAS — 차감을 멱등하게

포인트 차감: 로그에 없을 때만 실행
-- 비멱등: 2번 실행하면 200 차감
UPDATE wallet SET balance = balance - 100 WHERE user_id = 1;

-- 멱등: 트랜잭션 로그에 없을 때만 차감
UPDATE wallet SET balance = balance - 100
WHERE user_id = 1 AND NOT EXISTS (
  SELECT 1 FROM transaction_log WHERE transaction_id = 'txn-abc');
로그에 이미 있으면 UPDATE가 0행 → 두 번째 실행은 아무 일도 안 한다.
##

⑤ MQ Consumer 멱등성

at-least-once
PaymentConsumer.java
@KafkaListener(topics = "payment-events")
void handle(PaymentEvent e) {
    if (repo.existsByEventId(e.eventId())) return;  // 이미 처리
    paymentService.process(e);
    repo.save(new ProcessedEvent(e.eventId()));  // 기록
}
Kafka는 최소 1번 전달 = 중복 가능. EventId로 걸러낸다.
!!

처리와 기록이 따로 놀면

⚠️
체크 → 처리 → 기록이 원자적이지 않으면 구멍이 생긴다. 처리는 됐는데 기록 직전에 죽으면 → 재전달 시 또 처리된다.
INSERT IGNORE로 원자적 처리
@Transactional
void handleSafely(PaymentEvent e) {
    if (repo.insertIgnore(e.eventId()) == 0) return;  // 이미 처리
    paymentService.process(e);
}
##

어디에 적용할까

결제 · 주문 생성?
필수 — Idempotency Key + DB Unique
알림 발송 · 쿠폰 발급?
필수 — 발송 기록 · DB Unique
포인트 적립 · 차감?
필수 — Transaction ID + CAS
조회 · 좋아요 토글?
불필요 — 이미 자연스럽게 멱등
!!

흔한 실수 4가지

1TTL 없는 Key저장소 무한 증가 → 24~48h TTL
2실패도 캐싱재시도 불가 → 성공한 결과만 저장
3Key를 서버가 생성재시도 시 다른 키 → 클라이언트가 생성
4처리+기록 비원자기록 실패 시 중복 → 같은 트랜잭션
