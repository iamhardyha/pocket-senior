# 무결성

const question = "데이터가 비즈니스 규칙을 항상 만족하게 하려면?";
// 잘못된 데이터가 들어가면, 그 위 모든 로직이 무너진다
>

한 줄 요약

무결성은 방어선이다.
DB 제약 → 도메인 규칙 → 입력 검증, 3겹으로 쌓아야 한다.
>

왜 무결성인가

잘못된 데이터가 들어가는 순간,
그 위에 쌓이는 모든 로직이 무너진다.
버그는 고치면 끝이지만, 오염된 데이터는 시간이 갈수록 복구가 어려워진다.
##

무결성의 4가지 종류

①개체 무결성모든 행은 고유 식별 · PK NULL 불가 · 중복 불가
②참조 무결성참조 대상은 반드시 존재 · 주문의 user_id는 users에 존재
③도메인 무결성컬럼 값이 허용 범위 내 · 나이 > 0 · 정해진 상태만
④비즈니스 무결성비즈니스 규칙 만족 · 잔고 < 주문금액이면 주문 불가
##

3겹 방어선 — 안에서 밖으로

1층입력 검증 (Controller)Bean Validation · 형식·범위·필수값 · 비용 가장 낮음
2층비즈니스 규칙 (Domain)불변식 · 상태 머신 · 핵심 방어선
3층DB 제약 조건NOT NULL · UNIQUE · CHECK · FK · 최후의 방어선
##

1층 — 입력 검증

Bean Validation
CreateOrderRequest.java
@NotNull          Long productId;
@Min(1) @Max(9999) int quantity;
@NotBlank @Size(max=200) String address;
@Pattern(regexp="^\\d{5}$") String zipCode;
원칙: Controller에선 형식과 범위만. 비즈니스 로직은 넣지 않는다.
##

2층 — 도메인 불변식

핵심 방어선
Order.java
// 불변식: 주문 금액은 항상 양수
public void setTotalAmount(BigDecimal amount) {
    if (amount.compareTo(BigDecimal.ZERO) <= 0)
        throw new IllegalArgumentException("금액은 0보다 커야");
    this.totalAmount = amount;
}
객체가 스스로 규칙을 강제한다. 잘못된 상태로는 만들어질 수 없게.
##

상태 머신 — 허용된 전이만

OrderStatus 전이 규칙
PENDING ─→ PAID ─→ SHIPPING ─→ DELIVERED
                              ↑ 정상 흐름

// 허용되지 않는 전이
CANCELLED → PAID        (X)
DELIVERED → PENDING     (X)
전이 맵을 코드로 명시하고, 허용되지 않으면 예외를 던진다.
##

3층 — DB 제약 조건

최후의 방어선
orders 테이블
user_id BIGINT NOT NULL,
FOREIGN KEY (user_id) REFERENCES users(id),   -- 참조
CHECK (total_amount > 0),                    -- 도메인
order_number VARCHAR(32) UNIQUE,            -- 개체
UNIQUE KEY (order_id, product_id)            -- 복합
앞단을 모두 뚫어도 DB가 막는다. 데이터의 마지막 진실은 여기.
!!

DB 제약도 완전무결은 아니다

⚠️
MySQL CHECK 제약은 8.0.16+부터 실제 적용 — 그 이전엔 파싱만 되고 무시됐다. FK 검증도 foreign_key_checks=0 같은 설정으로 우회될 수 있다.
그래서 단일 방어선에 의존하지 않고 3겹으로 쌓는다.
##

무결성이 깨지는 흔한 4가지

①소프트 삭제 + FK행은 존재 → FK 통과 · 논리적으론 삭제된 사용자 참조
②Enum 불일치DB CHECK는 넓고 Java enum은 좁음 · 조회 시 역직렬화 실패
③숫자 정밀도float/double로 금액 계산 · 0.1+0.2 ≠ 0.3
④NULL의 함정= NULL은 0건 · IS NULL 써야 · 집계에서 제외됨
!!

금액은 절대 float/double 금지

0.1 + 0.2 의 함정
// 위험
double price = 0.1 + 0.2;   // 0.30000000000000004

// 안전
BigDecimal p = new BigDecimal("0.1").add(new BigDecimal("0.2"));
// 0.3 정확
원칙: 금액은 BigDecimal 또는 정수(원 단위). float/double은 절대 안 된다.
!!

NULL은 비교가 안 된다

WHERE coupon_id = NULL
SELECT * FROM orders WHERE coupon_id = NULL;   -- 0건!
SELECT * FROM orders WHERE coupon_id IS NULL;   -- 올바름

SELECT AVG(rating) FROM reviews;   -- NULL 행 제외 → 왜곡
원칙: 기본값이 있으면 NOT NULL + DEFAULT. 정말 "없음"일 때만 NULL.
