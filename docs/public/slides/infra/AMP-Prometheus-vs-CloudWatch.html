# AMP — AWS에서
Prometheus 굴리기

const question = "CloudWatch 두고 왜 또 Prometheus를?";
// AMP가 뭐고, ECS Fargate에선 어떻게 붙이나?
>

한 줄 요약

AMP는 Prometheus 서버 운영을 AWS가 대신 맡는 서버리스 백엔드다.
CloudWatch는 AWS 리소스를 손 안 대고 한곳에서 볼 때, AMP는 PromQL·Grafana·exporter 생태계를 그대로 이식할 때 강하다.
##

AMP가 정확히 뭔가

서버리스 · Prometheus 호환
①Prometheus 그대로동일 데이터 모델 + PromQL · 기존 설정·대시보드·알람 룰을 코드 변경 없이 이식
②서버리스스토리지·HA·스케일·보존을 AWS가 대신 — 굴릴 서버가 없다
③고가용성 내장워크스페이스 데이터를 같은 리전 3개 AZ에 복제
보존 기본 150일, 최대 3년(1095일). AMP는 '수집기'가 아니라 '백엔드' — 긁기는 ADOT가, AMP는 받아 저장·쿼리만.
##

관측성 3축에서 AMP의 자리

📈메트릭 Metrics시간에 따른 숫자 시계열(RPS·p99·CPU) → Prometheus / AMP · CloudWatch Metrics
📜로그 Logs사건의 텍스트 기록 → CloudWatch Logs · Loki · OpenSearch
🧭트레이스 Traces요청이 거쳐 간 경로(span 트리) → X-Ray · Tempo · Jaeger
AMP는 메트릭 한 축만 담당 — 로그·트레이스는 다루지 않는다. 겨루는 상대도 CloudWatch 전체가 아닌 '메트릭 부분'.
##

"CloudWatch 있는데 왜 또?"

①생태계 재사용exporter·Grafana 대시보드·알람 룰을 그대로 가져온다 (옮기려면 전부 다시 만든다)
②PromQL 표현력rate()·histogram_quantile()·라벨 기반 집계가 메트릭 분석에 특화
③이식성표준 포맷 → 온프렘·타 클라우드로 그대로 (CloudWatch는 AWS 락인)
④멀티클러스터 집계여러 EKS/ECS를 하나의 워크스페이스로 통합 대시보드
##

반대로 CloudWatch가 이기는 때

▸AWS 서비스 메트릭EC2·RDS·ALB·Lambda·SQS가 자동 수집 — 손 안 댐 (AMP는 별도 구성 필요)
▸운영 단순함메트릭·로그·알람·대시보드를 한 콘솔에서 올인원
▸학습 비용Prometheus 안 써 봤다 → 새 도구 학습 부담 X
🤝
현실 답은 자주 "둘 다" — AWS 인프라 메트릭은 CloudWatch, 앱/쿠버네티스 메트릭은 AMP+Grafana 하이브리드.
##

데이터 흐름 — Fargate에서 실제로

ECS Fargate Task · networkMode: awsvpc
[app 컨테이너]  Micrometer → GET /actuator/prometheus (:8080)
       │ scrape (localhost:8080)
[사이드카]      ADOT collector  (receiver → remotewrite + sigv4)
       │ remote_write (HTTPS · SigV4 서명)
   AMP 워크스페이스   (3-AZ 복제 · 기본 150일)
       │ PromQL (QueryMetrics API)
   Amazon Managed Grafana   (대시보드)
5단계 — ① 노출 → ② 스크레이프 → ③ SigV4 remote_write → ④ 3-AZ 저장 → ⑤ Grafana 조회.
!!

Fargate엔 왜 '사이드카'인가

⚠️
Fargate는 노드(호스트)를 네가 관리하지 않는다. EC2/EKS면 노드마다 DaemonSet으로 에이전트를 한 번 깔면 끝 — 그런데 Fargate는 호스트 접근이 안 돼 노드 단위 에이전트를 깔 자리가 없다. 완전관리형 collector는 EKS 전용 → ECS/Fargate는 ADOT를 직접 띄워야 한다.
→ 정답: ADOT 수집기를 앱과 같은 Task의 사이드카 컨테이너로. 같은 Task라 localhost로 /actuator/prometheus를 긁는다.
##

설정 ① 메트릭 노출

Spring Boot · Micrometer
application.yml  (+ micrometer-registry-prometheus 의존성)
management:
  endpoints.web.exposure.include: health,info,prometheus
  metrics:
    tags.application: order-service   # 공통 라벨
    distribution.percentiles-histogram:
      http.server.requests: true   # _bucket 생성 → p99 가능
레지스트리가 클래스패스에 있으면 /actuator/prometheus가 자동으로 켜진다. histogram을 켜야 histogram_quantile(p99)가 나온다.
##

설정 ② ADOT 수집기

긁기 → 보내기 → 서명
adot-config.yaml
receivers:
  prometheus:                  # /actuator/prometheus 스크레이프
    scrape_interval: 30s       # ★ 비용의 핵심 손잡이
    targets: ['localhost:8080']   # 같은 Task → localhost
exporters:
  prometheusremotewrite:       # AMP로 remote_write
    endpoint: ".../ws-xxxx/api/v1/remote_write"
    auth: { authenticator: sigv4auth }   # SigV4 서명
extensions: { sigv4auth: { service: aps } }   # aps = AMP
세 부품 — prometheus receiver(긁기) → remotewrite exporter(보내기) → sigv4auth(AWS 인증).
##

설정 ③ Task 사이드카 + IAM

task-definition.json — 같은 Task에 컨테이너 둘
{ "networkMode": "awsvpc",
  "taskRoleArn": "...order-task-role",   // ★ AMP 쓰기 권한
  "containerDefinitions": [
    { "name": "app", "portMappings": [{ containerPort: 8080 }] },
    { "name": "adot-collector",            // localhost:8080 긁음
      "image": "public.ecr.aws/.../aws-otel-collector" } ] }
🔑
Task Role에 AmazonPrometheusRemoteWriteAccess 누락 = 사이드카는 떠도 메트릭이 안 들어가고 403/AccessDenied — 현장 1순위 함정.
##

PromQL — 제일 많이 쓰는 패턴

라벨 기반 시계열 쿼리
# ① 초당 요청 수(RPS) — counter엔 항상 rate()
rate(http_server_requests_seconds_count[5m])

# ② p99 지연 — 히스토그램 분위수 (_bucket 필요)
histogram_quantile(0.99,
  sum by (le) (rate(http_server_requests_seconds_bucket[5m])))

# ③ 엔드포인트·상태코드별로 쪼개기
sum by (uri, status) (rate(..._count[5m]))
직관 셋 — counter는 항상 rate(), p99는 _bucket 시계열 필요(히스토그램 켜야), sum by (label)로 집계 축 자유.
##

AMP vs CloudWatch — 결정 기준

AWS 서비스 메트릭을 손 안 대고?
CloudWatch — 자동 수집
Prometheus·Grafana 자산 재사용?
AMP — 그대로 이식
멀티클라우드·멀티클러스터·이식성?
AMP — 표준 포맷
한 콘솔 올인원 · 운영 단순?
CloudWatch — 락인 대가
생태계·이식성이면 AMP, AWS 네이티브·단순함이면 CloudWatch — 현실은 하이브리드가 가장 흔하다.
##

비용 — 어디서 돈이 새나

종량제 · 4개 축
①수집 ingestion들어온 샘플 수 (≈$0.09/백만) — 보통 비용의 대부분
②저장 storage$0.03/GB · 보통 소액
③쿼리 QSP$0.10 / 10억 샘플 처리 — PromQL이 처리한 데이터포인트
④collectormanaged collector(EKS) 쓸 때만 · ADOT 자체엔 없음
인입(Data Transfer IN) 무료. 규모 예시(EKS 10노드·30초)는 월 ~893M 샘플 → 수집이 압도적으로 지배.
##

비용 최적화 — 손잡이 순서

①scrape_interval ↑30s → 60s면 수집 샘플 절반 · 가장 즉효
②relabel_config로 drop안 쓰는 메트릭을 적재 전 제거 → active series 감소
③알람은 AMP 네이티브 룰외부에서 PromQL 반복 호출은 중복 쿼리로 비용↑
🧮
별개 기능 — Prometheus Native Histogram(Micrometer percentiles-histogram과 무관)은 채워진 bucket을 0.25 샘플로 과금 → 히스토그램 워크로드 비용 최대 75%↓. 리텐션 단축은 효과 미미 — 저장이 원래 소액이라 결국 '수집'을 줄여야 한다.

면접에서 이렇게 답하라

AMP는 Prometheus 호환 서버리스 메트릭 백엔드 — 서버 운영(HA·스케일)은 AWS가, 모델·PromQL은 그대로. 3-AZ·기본 150일.
Fargate는 노드 에이전트를 못 깔아 ADOT를 같은 Task 사이드카로 띄워 localhost 스크레이프 → SigV4 remote_write.
비용은 수집이 지배 — scrape_interval↑ · relabel drop · 네이티브 알람 순으로 최적화.
