# JDK
마이그레이션

const question = "JDK 8 → 21, 무엇을 고려했나?";
// 단순 버전 업이 아니다.
>

한 줄 요약

버전 업이 아니라 호환성 점검이다.
삭제된 API · 모듈 시스템 · 의존성 · GC를 전부 확인해야 한다.
##

왜 JDK 21인가

8레거시 LTS아직 가장 많이 쓰임 · Oracle은 유료 연장
17현재 타겟 LTS많은 프로젝트의 현행 버전 · javax→jakarta 분기
21최신 LTS2031까지 지원 · Virtual Thread · Generational ZGC
##

고려할 것, 4가지 + 1

①삭제·변경 APIjavax→jakarta 전환이 가장 크다
②모듈 시스템JPMS · 리플렉션 라이브러리가 깨질 수 있다
③의존성 호환성Spring Boot 3.2+ · Hibernate 6.x 버전 정렬
④GC 변경기본이 G1 · ZGC도 프로덕션 가능
⑤Virtual Thread가장 큰 변화 · I/O 처리량 대폭 개선
##

① javax → jakarta

가장 큰 변화
Entity.java
import javax.persistence.Entity;    // ❌ 구 네임스페이스
import jakarta.persistence.Entity;  // ✅ 변경
JDK가 지운 게 아니다 — Jakarta EE 9의 네임스페이스 변경. Spring Boot 3+ / Hibernate 6+가 강제한다.
!!

사라진 것들

JAXBjavax.xml.bindJDK 11부터 제거 · jakarta.xml.bind 의존성 추가
JSNashorn 엔진JDK 15에서 삭제 · GraalJS로 대체
SecSecurity ManagerJEP 411 deprecated · 실제 삭제는 JDK 24
##

② 모듈 시스템 (JPMS)

JDK 9+
리플렉션으로 내부 API 접근 시 터진다
java.lang.reflect.InaccessibleObjectException:
  Unable to make field java.lang.String.value accessible

# 해결: JVM 옵션에 --add-opens 추가
--add-opens java.base/java.lang=ALL-UNNAMED
JDK 8은 classpath로 전부 접근, JDK 9+는 모듈 경계로 제한. 내부 API 찌르던 라이브러리가 깨진다.
##

③ 의존성 호환성

버전 정렬
SBSpring BootJDK 21은 3.2+ 필수 · 2.x는 JDK 17까지
HHibernate6.x 필요 · javax→jakarta 동반
QQueryDSL · JacksonQueryDSL 5.x(jakarta) · Jackson 2.15+
B빌드 도구Gradle 8.3+ · Maven 3.9+
##

④ GC, 기본값이 바뀐다

기본 GC 변화
JDK 8  기본: Parallel GC
JDK 9~21 기본: G1 GC  // JDK 9부터 변경

# 추천 시작점
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
기존 GC 튜닝 옵션이 무효화될 수 있다. 저지연이 중요하면 JDK 21의 Generational ZGC.
##

⑤ Virtual Thread

가장 큰 변화
Platform Thread → Virtual Thread
// Before: 무거움, 수천 개 한계
Executors.newFixedThreadPool(200);

// After: 가벼움, 수백만 개 가능
Executors.newVirtualThreadPerTaskExecutor();
경량 스레드로 I/O 바운드 처리량이 폭발적으로 늘어난다. CPU 바운드엔 효과 없음.
##

Spring Boot 3.2+ 설정 한 줄

application.yml
spring:
  threads:
    virtual:
      enabled: true
→ Tomcat이 Virtual Thread로 요청을 처리. 설정 한 줄로 동시 처리량이 크게 오른다.
!!

Virtual Thread 함정

⚠️
synchronized가 carrier thread를 묶는다 (pinning) → Lock 전환 권장. ThreadLocal 남용은 메모리 부담. DB 커넥션은 유한하니 풀 사이즈를 다시 점검.
스레드가 수백만 개여도 커넥션 풀이 병목이면 의미 없다.
##

전략 — 4단계로 나눠라

P1빌드 확인JDK 21 컴파일 · 에러 수정 · 의존성 업그레이드
P2테스트전체 스위트 실행 · 성능 벤치마크
P3스테이징GC·메모리·응답시간 모니터링 · 카나리 10%
P4운영 배포10→30→50→100% · JDK 8 빌드 롤백 준비
!!

한 번에 8 → 21은 위험

권장 경로 — 징검다리로 건넌다
JDK 8 → JDK 11  // 모듈 시스템 적응
      → JDK 17  // javax → jakarta
      → JDK 21  // Virtual Thread 등
또는 프레임워크와 함께: Spring Boot 2.x+8 → 3.x+17 → JDK 21(런타임만).
