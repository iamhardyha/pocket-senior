한 줄 요약

네트워크는 요청의 여정이다.
요청이 어떻게 서버에 도착하고 응답이 어떻게 돌아오는지 — 그 길 전체를 아는 것이 핵심이다.
##

요청의 여정 — 한 번의 호출

GET https://api.example.com/orders
1. DNS   도메인 → IP 주소
2. TCP   3-Way Handshake로 연결
3. TLS   암호화 채널 수립 (HTTPS)
4. HTTP  GET /orders → 서버 처리
5. 응답  200 OK + JSON → Keep-Alive
백엔드 네트워크 지식 = 이 다섯 단계를 끝까지 설명할 수 있는 것.
##

오늘 다룰 5가지

①DNS도메인 → IP · 로드밸런싱·Failover·TTL
②TCP / IP3-Way Handshake · 커넥션 풀의 이유
③HTTP상태 코드·헤더 · 1.1 vs 2 vs 3
④HTTPS / TLS암호화 핸드셰이크 · SSL Termination
⑤디버깅Timeout·502·504 · curl → 로그 추적
##

① DNS — 도메인을 IP로

캐시 계층
이름 해석은 캐시를 타고 내려간다
브라우저 캐시 → OS 캐시 → Router → ISP DNS
   → Root → .com → example.com DNS
   → 52.78.123.45
백엔드에 중요한 이유: 로드밸런싱(가중치 라우팅) · Failover(헬스체크) · DB 전환(CNAME).
!!

DNS TTL — 변경이 느린 이유

⚠️
TTL은 DNS 캐시 유지 시간 (기본 300초) TTL이 길면 IP를 바꿔도 반영이 느리다 → 마이그레이션·Failover 전엔 TTL을 미리 낮춰둔다.
실무 함정: DNS TTL · LB Idle Timeout · 서버 Keep-Alive가 안 맞으면 간헐적 502/504.
##

② TCP — 3-Way Handshake

1.5 RTT
연결 수립
Client → SYN →     Server
Client ← SYN+ACK ← Server
Client → ACK →     Server
// 연결 수립 (요청은 마지막 ACK에 실어 전송)
종료는 4-Way Handshake(FIN·ACK·FIN·ACK). 연결마다 왕복 비용이 든다.
TCP 1 RTT + TLS 1.2 2 RTT = 매 요청마다
~3 RTT
100ms 환경이면 연결에만 ~300ms. 그래서 커넥션 풀로 연결을 재사용한다 (DB·Redis·HTTP).
##

Keep-Alive & 멀티플렉싱

1.1Keep-Alive한 TCP로 여러 요청 순차 재사용 → 연결 오버헤드 제거
2Multiplexing한 TCP로 여러 요청 동시 처리 → HTTP HOL Blocking 해결
3QUICUDP 기반 → TCP-level HOL까지 해결 (HTTP/3)
##

③ 헷갈리는 상태 코드

401Unauthorized인증 필요 · 로그인 안 됨
403Forbidden권한 없음 · 로그인했지만 접근 불가
409Conflict중복 생성·버전 충돌
429Too Many RequestsRate Limiting · Retry-After
502Bad Gateway뒷단 서버 비정상 응답 (vs 504 타임아웃)
##

HTTP/1.1 → 2 → 3

1.1순차 처리요청1→응답1→요청2… · 병렬 TCP 6개로 완화
2멀티플렉싱한 TCP에서 동시 처리 · HPACK 헤더 압축·서버 푸시
3QUIC (UDP)핸드셰이크 불필요 · 패킷 손실이 다른 스트림에 무영향
##

실무에서 자주 쓰는 헤더

요청 / 응답 헤더
# 요청
Authorization: Bearer {JWT}      # 인증
Idempotency-Key: abc-123       # 멱등성
X-Request-ID: uuid              # 분산 추적
# 응답
Cache-Control: max-age=300      # 캐시
Retry-After: 60                 # 429 시 대기
ETag·Cache-Control은 캐시 검증, X-Request-ID는 분산 추적의 출발점.
##

④ TLS — 암호화 채널 수립

1.3 = 1 RTT
TLS Handshake
Client → ClientHello  지원 암호 목록
Server ← ServerHello  선택 암호 + 인증서
Client → 인증서 검증 → 키 교환
// 이후 대칭키로 암호화 통신
비용: TLS 1.2 = 2 RTT, 1.3 = 1 RTT (0-RTT는 replay 취약 → 멱등 요청에만).
!!

SSL Termination

💡
ALB / Nginx에서 TLS를 종단한다 외부는 HTTPS, 내부 통신은 HTTP → 암호화 비용을 게이트웨이에 모으고 애플리케이션 서버는 가볍게.
그래서 앱 서버는 평문 HTTP를 받지만, 바깥 경계는 항상 암호화되어 있다.
##

⑤ 네트워크 문제 디버깅

Connection Timeout?
서버 다운·방화벽 → telnet · curl -v
Read Timeout?
처리 중인데 느림 → 서버 로그·Slow Query
DNS 실패?
nslookup · dig로 해석 확인
502 / 504?
뒷단 비정상·타임아웃 → Nginx·ALB 로그
