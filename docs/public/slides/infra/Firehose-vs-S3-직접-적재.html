# Firehose vs
S3 직접 적재

const question = "결과가 같은데 왜 Kinesis를 쓰나?";
// S3에 데이터 쌓이는 건 똑같은데, Firehose의 이점은?
>

한 줄 요약

결과물은 같아 보여도, 직접 쏘면 적재 파이프라인을 전부 네가 만든다.
Firehose는 그 파이프라인을 통째로 관리형으로 대신한다. 잘게·끊임없이 들어올수록 차이가 벌어진다.
##

"S3에 바로 쓰면 되잖아"

둘 다 결국 S3에 객체가 생긴다 — 정말 똑같을까?
// A) 서버 → Kinesis Data Firehose → S3
producer.putRecord(event)   // 모아서 적재는 Firehose가

// B) 서버 → S3 직접 PutObject
s3.putObject(bucket, key, event)   // 한 건 = 한 객체
결과(S3에 데이터)만 봤을 뿐 — 실제로는 모양·비용·내구성·운영 부담이 전부 다르다.
건당 PutObject · 초당 수천 건 →
수백만 객체
분석 엔진은 파일 여는 행위 자체에 고정 비용 — 1KB 백만 개보다 128MB 8개가 훨씬 빠르고 싸다. PUT 요청 비용도 객체 수만큼 누적.
##

직접 적재 = 직접 만들 목록

1버퍼링·배칭모았다가 크기/시간마다 flush
2압축·포맷 변환gzip/snappy · JSON → Parquet/ORC
3파티셔닝year=/month=/day= 경로로 쪼개기
4재시도·백프레셔실패 격리 + 스파이크 완충
⚠️
이걸 다 만들면 미니 Firehose 재발명. 인메모리 버퍼는 인스턴스가 죽으면 flush 전 데이터 유실 — 내구성 책임까지 떠안는다.
##

② 버퍼링 — 설정 두 개로 끝

먼저 닿는 조건에 flush
Firehose buffering hints
Buffer size     = 128 MB    // 이만큼 쌓이면 flush
Buffer interval = 300 s    // 이 시간이 지나면 flush

// 트래픽 많으면 크기로, 적으면 시간으로 → 한 파일로 묶음
이 한 쌍이 작은 파일 폭탄을 자동으로 차단. (레코드 1건 최대 1,000 KB.)
##

② 완전관리형 + 빌트인 변환

서버·스케일·HA
프로비저닝 없음 · 자동 스케일 · 적재 전까지 3 AZ 동기 복제
포맷 변환
JSON → Parquet/ORC 설정으로 내장
압축
gzip/snappy 설정
저장 시 암호화
KMS 설정
컬럼형 포맷으로 두면 Athena가 필요한 컬럼만 읽어 스캔량(=비용)이 크게 준다.
##

② 동적 파티셔닝

분석 비용을 직접 깎는다
레코드 안의 키로 S3 prefix를 자동 분기
s3://logs/year=2026/month=06/day=21/event=click/...
s3://logs/year=2026/month=06/day=21/event=purchase/...
WHERE가 파티션과 맞으면 그 prefix만 스캔 — "6/21 purchase만" 조회 시 나머지를 안 읽는다 → 스캔 비용↓.
##

② 실패 격리 · 다중 목적지

↺실패 레코드 격리실패분을 S3 에러 prefix에 따로 — 직접 구현 시 가장 빼먹는 부분
⇉다중 목적지S3 · Redshift · OpenSearch · Splunk · Iceberg · HTTP (프로듀서 코드 그대로)
λ인라인 Lambda 변환적재 직전 필드 마스킹·정제
💲
수집 데이터양(GB) 종량제 — 액세스 로그 적재면 GB당 ~$0.029에서 시작 (조건부: 리전·구간·부가기능에 따라 변동. 핵심은 "직접 만들 엔지니어링·운영 시간"과의 비교.)
!!

그럼 직접 S3가 맞는 때

야간 ETL 결과 파일 1개/일
S3 직접 PutObject — 이미 크고 가끔, 버퍼링 불필요
Spark이 만든 Parquet 출력
S3 직접 — 이미 큰 파일·올바른 포맷
🧭
판별 질문 하나: "잘게·끊임없이 들어오나, 큰 덩어리가 가끔 오나?" 전자 → Firehose · 후자 → S3 직접이 보통 맞다.
##

③ Data Streams vs Firehose

이름만 같다
Kinesis Data Streams (KDS)
실시간 스트림 버스 · 소비자 직접 작성 · 다중 소비자 O · 재생 최대 365일 · 샤드 단위 순서 · 지연 서브초
Kinesis Data Firehose
관리형 적재기 · 소비자 없음 · 단일 목적지 · 재처리 X · 순서 보장 X · 지연 근실시간(초~분)
한쪽은 실시간 스트림 버스(저장·재생), 한쪽은 완전관리형 적재기.
##

자주 쓰는 조합

실시간 처리 + 적재를 둘 다 해야 할 때
[프로듀서]
   ├──> Kinesis Data Streams ──> 실시간 소비자(Lambda/KCL)
   │              │              // 알림·집계·이상탐지
   │              └──(KDS를 소스로)──> Firehose ──> S3
   │                                            // 분석용 적재
KDS가 다중 실시간 처리 + 재생을 맡고, S3 적재는 Firehose에 위임 — 적재 코드를 직접 안 짠다.

면접에서 이렇게 답하라

“직접 PutObject로 쏘면 적재 파이프라인을 전부 직접 만들게 됩니다.
Firehose는 버퍼링·포맷변환·동적 파티셔닝·실패 격리를 관리형으로 대신합니다.”
단, 큰 파일을 가끔 올리는 배치엔 과함 · Data Streams는 적재기가 아니라 실시간 스트림 버스(다중 소비자·재처리·순서).
