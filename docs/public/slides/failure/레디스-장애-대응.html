# Redis
장애 대응

if (redis.down) { "서비스도 같이 죽는가?" }
// Redis가 장애 나면 어떻게 대응하는가?
>

한 줄 요약

Redis가 죽었을 때 서비스가 같이 죽으면,
Redis에 너무 의존한 것이다.
##

Redis 장애, 4가지 얼굴

①완전 다운연결 불가 · 프로세스 크래시 / OOM Kill
②응답 지연타임아웃 급증 · 메모리 부족 / Slow Query / 네트워크
③부분 장애일부 노드만 불통 · Cluster 노드 장애 / 네트워크 파티션
④데이터 유실키가 사라짐 · eviction / 재시작
>

대응의 핵심

Redis의 역할에 따라
대응이 완전히 다르다.
캐시냐 · 세션이냐 · 분산 락이냐 · 카운터냐 — 역할별로 따로 본다.
##

역할별 장애 영향, 한눈에

①캐시DB로 직행 → Cache Stampede · DB 과부하
②세션전원 로그아웃 · CS 폭주
③분산 락동시성 제어 불가 → 쿠폰 중복 · 재고 마이너스
④카운터/큐수량 관리 불가 → 초과 발급 or 중단
##

① 캐시 — Stampede

캐시 HIT 99% 였는데, Redis 다운 →
10만 req → DB
막아주던 캐시가 사라지자 트래픽이 DB로 직행. DB가 같이 죽는다.
##

대응 — Local Cache 폴백

Local → Redis → DB
ProductService.java
Product cached = localCache.getIfPresent(key); // 1차: Local
if (cached != null) return cached;
try {
    cached = redis.get(key);            // 2차: Redis
} catch (RedisConnectionFailureException e) {
    log.warn("Redis 실패, DB 직접 조회");  // 장애 시 스킵
}
return repo.findById(id).orElseThrow(); // 3차: DB
Redis가 죽어도 서버 메모리 캐시가 먼저 막고, 끝내 안 되면 DB로.
##

대응 — Circuit Breaker

실패율 50% ↑ → OPEN
ProductService.java
@CircuitBreaker(name = "redis", fallbackMethod = "getFromDB")
Product getFromCache(Long id) {
    return redis.get("product:" + id);
}
Product getFromDB(Long id, Throwable t) {
    return repo.findById(id).orElseThrow();
}
⭐
Circuit OPEN이면 Redis 호출 자체를 안 함 — 죽은 Redis를 두드리느라 쌓이는 지연도 차단
!!

함정 — 복구 직후 Stampede

⚠️
Redis가 살아나도 캐시가 텅 비어 있으면 동시 요청이 다시 DB로 몰린다 → Mutex Lock으로 하나의 요청만 DB 조회 후 캐시 채우기 (setIfAbsent)
→ 단순 sleep 후 재조회는 함정: 락 보유 스레드가 100ms 넘기면 miss가 샌다. 운영은 제한 재시도 + 백오프.
##

② 세션 — 전원 로그아웃

장애 연쇄
Redis 다운 → 세션 조회 실패 → 인증 실패
           → 전원 로그아웃 → CS 폭주
세션을 Redis에만 두면 Redis 장애 = 인증 불가. 의존을 줄이는 게 핵심.
##

세션 방식별 장애 영향

Redis 세션
완전 의존 → 전원 로그아웃
JWT + Redis 블랙리스트
부분 의존 → 폐기 토큰만 영향
JWT only
영향 없음 (단, 토큰 폐기 불가)
불가피하게 Redis면?
세션 전용 Cluster 별도 분리
##

③ 분산 락 — DB 폴백

DB Unique = 최종 안전망
CouponService.java
try {
    issueWithRedisLock(couponId, userId);  // 1차: Redis 락
} catch (RedisConnectionFailureException e) {
    issueWithDBLock(couponId, userId);     // 2차: DB 비관 락
}
// + DB Unique(coupon_id, user_id) = 항상 작동하는 최종 방어
✅
Redis 락이 다 실패해도 DB Unique Key가 중복 발급을 끝까지 막는다
##

④ 카운터 — DB로 전환

선착순 수량 관리
CouponService.java
try {
    return issueViaRedis(couponId, userId);  // 빠름
} catch (RedisConnectionFailureException e) {
    log.warn("Redis 장애, DB 직접 처리");
    return issueViaDB(couponId, userId);     // 느리지만 안전
}
평소엔 Redis로 빠르게, 장애 땐 DB로 느려도 정확하게. 공통 패턴은 폴백.
##

인프라 — HA 구성 선택

단일 Redis
개발/소규모 → 전체 서비스 영향
Sentinel
Master 죽으면 Replica 승격 → Failover 중 수초 불가
Cluster
16384 슬롯 분산 → 장애 샤드만 영향
Failover 시간?
기본 30초↑ · 튜닝 시 5~15초 (그 사이 쓰기 불가 → 재시도)
