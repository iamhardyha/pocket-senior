한 줄 요약

DB가 죽으면 거의 다 죽는다.
그래서 죽지 않게 만들고, 죽으면 빠르게 전환하는 게 답이다.
##

DB는 단일 실패점(SPOF)

💥
애플리케이션은 여러 대로 늘려도, DB 한 대가 죽으면 전부 멈춘다 그래서 대응은 "복구"보다 "예방·이중화"가 먼저다. 대응 전략은 예방 → 자동 전환 → 복구 3단계.
장애는 갑자기 한 가지로만 오지 않는다 — 유형부터 구분하자.
##

DB 장애 유형 5가지

①완전 다운연결 불가 · 디스크·OOM·프로세스 크래시
②응답 지연Slow Query 폭증 · Lock 경합·풀스캔
③Replication 지연Replica가 뒤처짐 · Write 과다·네트워크
④디스크 풀쓰기 불가 · 로그·데이터 증가
⑤커넥션 고갈새 연결 불가 · 커넥션 누수·트래픽 급증
##

1단계 — 원인 빠르게 진단

5분 이내
MySQL · 진단 쿼리
SHOW PROCESSLIST;                 -- 실행 중인 쿼리
SHOW STATUS LIKE 'Threads_connected';  -- 커넥션 수
SELECT * FROM information_schema.innodb_lock_waits; -- Lock
df -h   # 디스크 사용량 (OS)
먼저 무엇이 막혀 있는지부터 본다. 추측보다 PROCESSLIST가 빠르다.
##

2단계 — 즉시 완화

⚡Slow QueryKILL {id}로 문제 쿼리 종료
⚡커넥션 고갈Sleep 커넥션 정리 · max_connections 임시 ↑
⚡Lock 경합장시간 Lock 보유 트랜잭션 Kill
⚡디스크 풀불필요 로그 삭제 · binlog purge
⚡Primary 다운Replica를 Primary로 승격 (Failover)
##

Failover — Primary 장애 시

죽은 Primary 대신
최신 Replica를 Primary로 승격한다.
핵심은 "가장 덜 뒤처진 Replica"를 고르는 것 — SHOW REPLICA STATUS로 lag 확인.
##

수동 Failover

5~30분
승격 절차 — Replica → 독립 Master
# 1. 가장 최신 Replica 선택 (lag 최소)
STOP REPLICA;        # 2. 복제 중단
RESET REPLICA ALL;   # 3. 독립 Master로 전환
# 4. 앱 DB 연결을 새 Master로 변경
# 5. 나머지 Replica를 새 Master에 연결
⚠️
수동은 느리고 데이터 유실 가능 사람이 매번 판단·실행 → 5~30분 소요. 새벽 장애에 취약하다.
##

자동 Failover

10~30초
Aurora · MHA
// AWS Aurora
장애 감지(10s) → Replica 승격(30s 이내) → DNS 엔드포인트 그대로
   → 앱은 자동 재연결 // 보통 30초 이내

// MySQL + MHA
MHA Manager 감지 → 최신 Replica 승격 + VIP 전환 // 10~30초
✅
엔드포인트가 그대로 유지되므로 앱은 코드 변경 없이 재연결만 하면 된다
##

Failover 방식 비교

수동
5~30분 · 유실 가능
운영 복잡도 낮음
MHA
10~30초 · 유실 최소화
운영 복잡도 중간
Aurora
30초 이내 · 커밋 데이터 보존
진행 중 트랜잭션은 롤백
한 줄 결론
자동 Failover를 구성하라 — 사람은 느리다
!!

DB는 멀쩡한데 장애?

💥
커넥션 풀이 전부 Slow Query에 점유되면 11번째 요청부터 커넥션 대기 → connection-timeout 초과 → 에러. 사용자에겐 "서비스 장애"로 보인다.
DB가 아니라 애플리케이션 ↔ DB 사이가 막힌 것.
##

커넥션 풀 튜닝

HikariCP
application.yml
hikari:
  maximum-pool-size: 30      # (core*2)+disk, 많다고 좋지 않다
  connection-timeout: 3000   # 3초 (기본 30초는 위험)
  max-lifetime: 1800000      # 30분마다 커넥션 갱신
  leak-detection-threshold: 5000  # Leak 의심 로그
⭐
쿼리 타임아웃도 필수: jakarta.persistence.query.timeout 한 쿼리가 풀 전체를 잡아먹는 연쇄 장애를 끊는다
!!

읽기/쓰기 분리 — Lag 함정

주문 직후 조회 시나리오
Primary: INSERT orders ...   // 즉시 반영
Replica: ... 3초 후 반영       // Lag

사용자: 주문완료 → 목록조회 → "주문이 없습니다" 💥
✅
대응: 쓰기 직후 읽기는 Primary에서 read-after-write가 꼭 필요하면 Reader가 아닌 Writer에서 읽는다. Aurora Reader도 lag은 0이 아니다.
##

데이터 복구 — PITR

"어제 오후 3시로 되돌려줘"
풀 백업 + Binlog로 Point-in-Time Recovery. Aurora는 Backtrack(수분) / 1초 단위 PITR을 제공한다.
⚠️
백업보다 중요한 건 복구 훈련 실제로 해보지 않은 복구 절차는 장애 때 작동하지 않는다.
