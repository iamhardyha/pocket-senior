##

지속적 과부하란?

스파이크 트래픽
순간 폭증 — 잠깐 버티면 끝
지속 과부하 (이 노트)
감당 한계를 계속 초과
증설로도 따라잡지 못하는 상황 → 단계별로 버티고, 버린다.
##

심각도 순서, 4단계

①자동 대응Auto Scaling이 알아서 · 인적 개입 불필요
②능동 대응엔지니어 개입 · 캐시·Rate Limit·쿼리 튜닝
③비상 대응Graceful Degradation · 비핵심 기능 OFF
④최후 수단대기열 모드 / 점검 페이지 전환
##

① 자동 대응

시스템이 알아서
auto-scaling
# CPU 70% 돌파 →
Auto Scaling 트리거 → 서버 증설 → 자동 해결
✅
전제: Scale-Out 가능한 Stateless 아키텍처상태를 서버에 두면 증설해도 늘어난 서버를 못 쓴다.
##

② 능동 대응

엔지니어 개입
즉시캐시 TTL ↑DB 부하 감소 · 가장 빠른 한 수
즉시Rate Limiting 강화과도한 요청 차단
수십분Read Replica 추가읽기 분산
수시간쿼리 최적화·불필요 호출 제거자원 효율 개선
트리거: Auto Scaling Max 도달 · DB CPU 80% · Redis 메모리 90%
##

③ Graceful Degradation

단계적 기능 축소
경고추천·랭킹 OFF리뷰는 캐시만 반환
위험검색 축소·저해상도비핵심 API 503 반환
긴급신규 주문만 허용조회는 캐시 · 정적 페이지(S3) 전환
핵심 기능만 살리고 나머지는 끈다.
##

런타임 스위치 — Feature Flag

ProductController.java
ProductResponse res = productService.getProduct(id);

// Level 1: 추천 OFF
if (featureFlag.isEnabled("degradation-level-1"))
    res.setRecommendations(Collections.emptyList());

// Level 2: 리뷰는 캐시만
if (featureFlag.isEnabled("degradation-level-2"))
    res.setReviews(cachedReviewService.get(id));
배포 없이 런타임에 즉시 켜고 끈다 — 그게 Feature Flag의 핵심.
!!

④ 최후 수단

💥
전체 서비스 유지 불가 → 대기열 모드 또는 점검 페이지 전체 사용자를 대기열에 진입시켜 유입 자체를 통제하거나, 정적 점검 페이지로 전환한다.
여기까지 오면 이미 늦은 것 — 3단계 안에서 막는 게 목표.
##

병목은 4곳에서 터진다

CPU연산·GC 과다비동기 처리(MQ) · JVM·GC 튜닝
MEM대용량 로딩·누수페이징·스트리밍 · 힙 덤프 분석
DBSlow Query·Lock·풀 고갈캐시·Replica · 최후엔 샤딩
NET대용량 응답·대역폭gzip·DTO · CDN 오프로드
##

메모리: 한번에 → 흘려보내기

OrderService.java
// Bad: 100만 건을 한번에 메모리에 → OOM
List<Order> all = orderRepository.findAll();

// Good: 스트리밍 + 처리한 건 즉시 해제
try (Stream<Order> s = orderRepository.streamAll()) {
    s.forEach(o -> { process(o); entityManager.detach(o); });
}
전체 로딩 대신 Stream·Cursor로 부분 처리하면 힙이 터지지 않는다.
##

DB 과부하: 대응 순서

1Slow Query Kill지금 죽이는 쿼리부터 즉시 차단
2캐시 강화·Read Replica읽기 부하를 DB 밖으로
3쿼리 튜닝·Scale-Up인덱스·실행계획 · 스펙 업
4샤딩최후 수단 — 복잡도가 폭증
싼 것·빠른 것부터. 샤딩은 정말 마지막에.
서버 4대 · 8,000 QPS 한계 · 월 15% 성장 →
3개월 후 95%
5,000 → 7,604 QPS. 6개월이면 한계 초과. 미리 보면 보인다.
##

70 · 80 · 90 원칙

70%증설 계획여유 있을 때 미리 설계
80%증설 실행실제로 서버·Replica 추가
90%이미 늦음비상 대응 영역으로 진입
평소 용량 계획이 3단계까지 가지 않게 막아준다.
