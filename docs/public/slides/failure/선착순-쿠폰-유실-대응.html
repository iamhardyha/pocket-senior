# 선착순 쿠폰
유실 대응

const question = "자정 폭주에 서버가 죽어 요청이 유실되면?";
// 순서 보장 + 유실 방지 + 장애 내성, 셋 다.
>

한 줄 요약

순서·유실·장애를 한 번에 풀려면 MQ 기반 대기열이다.
핵심은 API 서버는 접수만, 발급 판단은 Consumer에서 분리하는 것.
##

자정 선착순, 동시에 만족해야 할 4가지

①순서 보장먼저 온 사람이 받아야 · 공정성
②정확히 N장초과 발급 불가 · 수량 정합성
③중복 발급 불가1인 1장 · 멱등성
④유실 방지장애 나도 안 사라짐 · 내구성
자정 00:00:00 · 수십만 요청이 1초 내 폭주 →
정합성
서버 다운 → 요청 데이터 유실 → 수량은 깎였는데 발급 기록은 없음.
그리고 CS 폭주: "저 먼저 눌렀는데요?"
##

MQ 기반 선착순 — 데이터 흐름

접수와 발급을 분리한다
User ──폭주──> API (순번만 부여) ──> 202 "접수됨"
                          │
                  Kafka // 파티션 1개 = 순서 보장
                          │
                  Consumer // 중복→수량→발급, 순차
                          │
                  DB + Redis ──> 결과 알림
버퍼(Kafka)가 폭주를 흡수하고, 발급은 차분히 순차 처리한다.
##

오늘 다룰 4축

①순서 보장단일 파티션 + concurrency=1
②유실 방지acks=all · 복제 · 수동 커밋 + 멱등성
③중복·수량Redis Lua 원자 차감 + DB Unique
④장애 시나리오API·Kafka·Consumer·Redis 5가지 대응
##

① 순서 보장 — 단일 파티션

key 고정
Producer / Consumer
// key 고정 → 같은 파티션 → 도착 순서 유지
kafkaTemplate.send("coupon-queue", "coupon-100", req);

@KafkaListener(topics = "coupon-queue", concurrency = "1")
public void process(CouponRequest req) { issue(req); }
같은 key는 같은 파티션으로, concurrency=1이면 하나씩 순서대로.
##

API 서버의 역할은 오직 "접수"

CouponController.java
// 1. 소진 플래그면 즉시 거부 (빠른 필터)
if (redis.get("coupon:"+id+":soldout") != null) ...
// 2. 이미 접수했으면 중복 응답
if (redis.sIsMember("...:requested", userId)) ...
// 3. Kafka 적재 → 4. 접수 기록 → 5. 즉시 202
kafkaTemplate.send("coupon-queue", key, req);
return ResponseEntity.accepted();
발급 판단은 절대 여기서 안 한다 — 접수하고 즉시 응답.
##

② 유실 방지 — Kafka 내구성

1acks=allISR 모든 브로커 기록 확인 후 응답 · 유실 0
2replication.factor=33개 브로커 복제 · 1대 죽어도 OK
3min.insync.replicas=2ISR 2 미만이면 쓰기 거부 (유실보다 안전)
4결과API·Consumer 죽어도 메시지는 Kafka에 남음
##

③ Consumer — 멱등 + 원자 차감

CouponIssueService.java
// 1. 멱등성: 이미 발급됐으면 스킵
if (repo.existsByCouponIdAndUserId(id, userId)) return;

// 2. Lua Script로 원자적 차감 (남은 수량 반환, -1=소진)
Long remaining = redis.execute(decrIfPositiveScript, keys);
if (remaining == null || remaining < 0) { soldOut(); return; }

// 3. DB 발급 기록 (Unique 제약 = 최종 안전망)
repo.save(new CouponIssue(id, userId, key));
DB 행 하나가 남고(0) 소진(-1)을 구분 — 0도 발급 성공이다.
!!

함정 — 차감 롤백의 over-issuance

💥
DB 중복 예외가 나도 Redis 수량을 INCR로 되돌리면 안 된다 그 1개는 직전 정상 발급에서 이미 소비된 것. 복원하면 N개 초과 발급(over-issuance). 단순 DECR↔INCR 롤백도 Race Condition으로 over-increment 위험.
✅
차감·복원은 Lua Script로 원자적. 보정은 blind INCR이 아니라 DB 실제 발급 건수와 대조해서.
!!

장애 — Kafka & Consumer 다운

폴백 + 수동 커밋
// Kafka 완전 다운 → DB 직접 처리로 폴백
try { kafkaTemplate.send(...).get(3, SECONDS); }
catch (Exception e) { service.issueWithDBLock(id, userId); }

// Consumer: 처리 완료 후에만 수동 커밋
service.issue(req);
ack.acknowledge();  // 여기서 죽으면 → 재시작 시 재처리 → 멱등 안전
enable.auto.commit=false + 멱등성 = 중복 처리해도 안전.
!!

장애 — Redis 다운 & 사후 복구

🛡️
Redis가 죽어도 DB가 최종 안전망 Unique Key + 수량 체크로 초과·중복 방지. 성능 저하를 감수하고 DB 폴백, Redis Cluster로 고가용성.
🔎
"요청했는데 못 받음" → Kafka 로그 ↔ 발급 기록 대조 미발급 사용자 식별 → 수량 내면 수동 발급, 초과면 추가 수량/보상 쿠폰. 모든 요청이 Kafka에 남아 복구 가능.
##

진짜 1ms 단위 순서가 필요할까?

💡
네트워크 지연은 사용자마다 다르다(WiFi vs 5G). "공정한 선착순"의 정의 자체가 모호하다. Kafka 도착 순서 = 서버 도착 순서 ≈ 합리적 선착순. 밀리초 분쟁은 증명조차 불가.
사용자 경험 관점: "접수됨 → 결과 알려줌"이면 충분하다.
