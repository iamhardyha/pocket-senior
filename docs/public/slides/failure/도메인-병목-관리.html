한 줄 요약

가장 느린 서비스가 전체 속도를 결정한다.
동기로 묶지 말고, 끊어서 비동기로 연결하라.
##

주문 하나에 4개 도메인

동기 직렬 처리
1. 상품 서비스  재고 확인        // 50ms
2. 쿠폰 서비스  할인 계산        // 80ms
3. 결제 서비스  PG 결제 요청     // 500ms~3s ← 병목
4. 알림 서비스  주문 확인 발송   // 200ms
하나의 요구사항이 여러 서비스를 직렬로 거치면 시간이 더해진다.
50 + 80 + 500 + 200 =
830 ms
PG가 느리면 3,330ms까지. 병목 = 가장 느린 구간이 전체를 지배.
##

병목을 끊는 5가지 전략

①동기 최소화꼭 필요한 것만 기다리고 나머지는 비동기
②의존성 끊기이벤트 기반으로 강결합 → 약결합
③타임아웃 + 폴백느린 서비스를 기다리지 않는다
④캐싱의존 호출 자체를 줄인다
⑤BFF 패턴조합 로직을 별도 계층으로
##

① 사용자가 기다려야 하는가?

상품 조회 · 쿠폰 · 결제
동기 — 응답 결과에 직접 영향
알림 발송
비동기 — 나중에 가도 됨
포인트 적립
비동기 — 실패해도 재시도
정산 기록
비동기 — 내부 처리
판단 기준 하나: 사용자가 응답을 기다려야 하는가?
##

동기 안에서도 병렬로

130ms → 80ms
OrderService.java
// 직렬 50+80=130ms → 병렬 80ms (긴 쪽)
CompletableFuture<Product> pf =
    CompletableFuture.supplyAsync(() -> product.get(id), ioExecutor);
CompletableFuture<Coupon> cf =
    CompletableFuture.supplyAsync(() -> coupon.validate(id), ioExecutor);
⚠️
executor를 생략하면 공용 ForkJoinPool을 쓴다 — 블로킹 I/O엔 부적합풀 고갈 → 전체 영향. 벌크헤드 관점에서 전용 bounded executor로 격리하라.
##

② 동기 의존 → 이벤트 기반

강결합 vs 약결합
// 강결합: 알림 서비스가 죽으면 → 주문 전체 실패
주문 --HTTP--> 재고 --HTTP--> 결제 --HTTP--> 알림

// 약결합: 알림이 죽어도 → 주문은 성공
주문 --이벤트--> Kafka --> 재고 · 알림 · 정산
알림 서비스 복구 후 Kafka에서 메시지를 꺼내 처리하면 된다.
##

③ 타임아웃 + 폴백

Circuit Breaker
CouponService.java
@CircuitBreaker(name = "coupon", fallbackMethod = "fallback")
@TimeLimiter(name = "coupon")   // 2초 타임아웃
public CompletableFuture<Coupon> validate(Long id) { ... }

// 폴백: 쿠폰 서비스 장애 → 할인 없이 원가로 진행
public ... fallback(Long id, Throwable t) { return noDiscount(); }
느린 서비스를 기다리지 말고 폴백으로 진행해 전체가 멈추지 않게.
##

서비스별 타임아웃 · 폴백

1초상품 조회폴백 · 캐시된 상품 정보
2초쿠폰 검증폴백 · 할인 없이 진행
5초결제폴백 · 결제 실패 처리
∞알림비동기 · MQ 재시도
##

④ 캐싱으로 호출 자체를 제거

ProductService.java
@Cacheable(value = "product", key = "#id")
public Product getProduct(Long id) {
    return productClient.getProduct(id);  // 캐시 미스 시에만
}
// 캐시 HIT → 다른 서비스 호출 자체가 사라진다 (TTL 5분)
⚠️
가격처럼 정확해야 하는 데이터는 TTL을 짧게, 또는 캐시하지 않는다.
##

⑤ BFF — 호출 4번을 1번으로

Backend For Frontend
// Before: 클라이언트가 4번 호출
Client --> 상품 --> 재고 --> 리뷰 --> 추천

// After: BFF가 내부에서 병렬 호출 + 조합 + 폴백
Client --> BFF --> (상품 · 재고 · 리뷰 · 추천 병렬)
클라이언트는 1번만 호출. 모바일·웹별로 다른 BFF도 가능.
##

어디가 병목? 분산 추적

Jaeger · Zipkin · AWS X-Ray
[Order API] ───────────────────── 830ms
  ├─ [Product]  ── 50ms
  ├─ [Coupon]   ─── 80ms
  ├─ [Payment]  ──────────── 500ms  ← 병목!
  └─ [Notify]   ────── 200ms
각 호출 시간을 한눈에 — 어디가 병목인지 즉시 보인다.
##

병목을 알려주는 4가지 지표

①응답 시간서비스별 p50 · p95 · p99
②호출 실패율서비스 간 에러 비율
③대기열 적체량MQ backlog 누적
④풀 사용률스레드 풀 · 커넥션 풀 포화
!!

비동기가 만능은 아니다

💥
한 서비스의 느린 호출이 공용 스레드 풀을 다 잡으면 멀쩡한 호출까지 막힌다 → 서비스별로 전용 bounded executor로 격리(벌크헤드). 타임아웃 없는 비동기는 풀을 영원히 점유한다.
✅
비동기에도 타임아웃 · 격리 · 폴백은 반드시 함께.
