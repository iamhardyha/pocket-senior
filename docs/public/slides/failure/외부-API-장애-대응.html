# 외부 API
장애 대응

const question = "외부 API가 타임아웃 나면, 내 서비스는 어떻게 살아남나?";
// '만약에'가 아니라 '언제' 나느냐의 문제다.
>

한 줄 요약

외부 API는 반드시 죽는다.
핵심은 내 서비스가 같이 죽지 않는 것 — 격리하고, 빠르게 실패하라.
##

외부 1개가 내 전부를 죽인다

Cascading Failure
// 외부 PG사가 30초간 응답이 없다
Client → 내 API → 외부 PG (30초 대기...) → Timeout
                    ↑ 스레드 1개가 30초간 점유
200 동시요청 → 스레드 풀 고갈 → 내 서비스도 죽음
외부 API 1개의 지연이 스레드 풀을 말려 죽인다. 이것이 연쇄 장애.
##

5겹 방어, 한눈에

①Timeout무한 대기 차단 · 가장 기본이자 가장 중요
②Circuit Breaker반복 실패 감지 → 빠른 차단 (Fail Fast)
③Bulkhead서비스별 스레드 격리 · 전파 차단
④Retry일시 장애 재시도 · 멱등 + Backoff
⑤Fallback대체 응답 · 다중 벤더 · 캐시
##

① Timeout

가장 기본
RestTemplateBuilder
RestTemplate rt = new RestTemplateBuilder()
    .setConnectTimeout(Duration.ofSeconds(1))  // 연결: 1초
    .setReadTimeout(Duration.ofSeconds(3))     // 응답: 3초
    .build();
모든 외부 호출에 반드시 Timeout. 연결과 읽기를 따로 건다.
평균 200ms · p99 1.5s 일 때 →
3초
Timeout = p99 × 2 가 합리적. 기본값 30초는 위험, 무제한은 자살.
##

② Circuit Breaker

Resilience4j
PaymentService.java
@CircuitBreaker(name = "paymentService", fallbackMethod = "payFallback")
public PaymentResponse pay(PaymentRequest req) {
    return paymentClient.pay(req);   // OPEN이면 호출 안 함
}
// 최근 10건 중 50% 실패 → OPEN → 30초 차단
반복 실패를 감지하면 외부 호출을 끊고 즉시 Fallback.
##

CLOSED → OPEN → HALF-OPEN

state machine
CLOSED (정상)  ──실패율 50%↑──▶  OPEN (차단)
   ▲                              │ 즉시 Fallback · 스레드 보호
   │ 성공                         │ 30초 경과
   │                              ▼
   └──────  HALF-OPEN (시험 3건)  ──실패──▶ OPEN
죽은 외부를 계속 부르는 건 바보짓. 빠르게 실패(Fail Fast)하고 스레드를 지킨다.
##

③ Bulkhead — 격리

전파 차단
서비스별 스레드 풀 격리
// Before — 공유 풀
결제 지연 → 전체 스레드 200개 점유 → 상품 조회도 죽음

// After — 격리
결제 지연 → 결제 스레드 20개만 점유 → 상품 조회 정상
배의 격벽처럼, 한 칸이 침수돼도 배는 가라앉지 않는다.
##

④ Retry — 일시 장애만

Backoff + Jitter
Exponential Backoff
1차 실패 → 500ms 대기
2차 실패 → 1000ms 대기
3차 실패 → 2000ms 대기 → 포기
+ Jitter: 모든 클라가 같은 시점에 재시도하지 않게
간격을 점점 늘리고, 횟수를 3~5회로 제한한다.
!!

함정 — 멱등하지 않은 재시도

💥
멱등하지 않은 POST 결제를 재시도하면 결제 2번 실행 → 돈 2배 응답만 유실됐을 뿐 서버는 이미 처리했을 수 있다.
✅
GET은 안전, POST 결제는 Idempotency Key 필수 — 같은 키면 한 번만 처리
##

⑤ Fallback 전략

결제다른 PG로 전환이니시스 장애 → 토스페이먼츠
배송캐시된 결과마지막 조회 + "업데이트 지연" 안내
추천인기 상품실시간 대신 주간 인기 상품
SMSPush 전환SMS 장애 → FCM Push
!!

함정 — 타임아웃 폴백 = 이중 결제

⚠️
타임아웃을 그냥 보조 PG로 폴백하면 이중 결제 메인 PG가 실제로 카드를 승인했을 수 있다. 응답만 못 받았을 뿐.
→ 먼저 메인 PG에 거래 상태를 verify, "확실히 미승인"일 때만 폴백.
##

특정 시간대 타임아웃이라면?

패턴 분석 → 사전 대응
// 매일 14~15시 타임아웃 급증 → 외부 배치/피크 의심
@Scheduled(cron = "0 50 13 * * *")  // 문제 10분 전
void prepareForPeak() {
    warmUpCache();                              // 캐시 워밍업
    config.set("payment.timeout", "5s");     // 3→5초
    config.set("payment.failureRate", "70");  // 50→70%
}
시각·빈도 대시보드로 패턴을 찾고, 그 시간에 임계값을 동적 조정.
##

방어 조합 — 실행 순서

Resilience4j 권장 순서 (바깥 → 안쪽)
@Retry           // 4. 실패 시 재시도 (가장 바깥)
@CircuitBreaker  // 3. 실패율 높으면 차단
@TimeLimiter     // 2. 타임아웃
@Bulkhead        // 1. 스레드 격리 (가장 안쪽)
public CompletableFuture<PaymentResponse> pay(...) { ... }
바깥에서 안으로 Retry → CB → TimeLimiter → Bulkhead → 실제 호출.
