한 줄 요약

한 번에 바꾸지 마라.
단계적 전환이 전부다.
다운타임 없는 마이그레이션의 핵심은 여러 단계로 쪼개는 것.
##

왜 글로벌은 다른가

traffic across regions
# 리전마다 시간대가 다르다
Seoul      UTC+9     // 한국 새벽 = 미국 한낮
Virginia   UTC-5     // 미국 새벽 = 유럽 아침
Frankfurt  UTC+1     // 어딘가는 항상 피크
24시간 트래픽이 존재 → "새벽에 점검"이 불가능하다.
##

세 종류의 마이그레이션

①스키마 변경컬럼/테이블 구조 → Backward Compatible 전략
②엔진/버전 전환MySQL → Aurora 등 → Blue-Green 전략
③대용량 데이터 이전1억 건 이전 → CDC로 실시간 동기화
공통 전제: 구버전·신버전 코드가 동시에 살아 있어야 한다.
##

① 원칙 — Backward Compatible

구 코드와 신 코드가
동시에 동작할 수 있어야 한다.
배포는 순간이 아니라 롤링이다 — 둘이 공존하는 구간을 견뎌야 한다.
##

컬럼 추가 — 안전

NULL 허용으로 시작
add column safely
-- 1) NULL 허용으로 추가
ALTER TABLE orders ADD COLUMN discount DECIMAL(10,2) NULL;
-- 2) 코드 배포 (구·신 코드 모두 정상)
-- 3) 백필은 배치로 쪼개서 (한방 UPDATE = Lock)
UPDATE orders SET discount = 0 WHERE discount IS NULL;
MySQL 8.0.12+는 대부분의 ADD COLUMN을 INSTANT로 처리(제약 있음, 불가 시 INPLACE 폴백).
!!

컬럼 삭제 — 순서가 전부

⚠️
삭제 먼저 → 배포는 사고. 배포 완료 전 구 코드가 사라진 컬럼을 참조해 에러. 롤링 배포 중에는 구 코드 인스턴스가 아직 살아 있다.
→ 올바른 순서: ① 코드에서 사용 중지·배포 → ② 전 서버 신 코드 확인 → ③ 삭제
!!

이름 변경 — RENAME 금지

1새 컬럼 추가total_price 신규
2듀얼 라이트old·new 양쪽 모두 쓰기
3데이터 동기화기존 행을 new로 백필
4읽기 전환 → 구 컬럼 쓰기 중지 → 삭제단계별 안전 종료
RENAME 한 방은 구 코드와 즉시 깨진다 — 듀얼 라이트로 다리를 놓는다.
##

대용량 테이블 — Lock 없이

pt-online-schema-change (Percona)
1. 새 테이블 생성 (변경된 스키마)
2. 트리거로 원본 변경분 실시간 복제
3. 기존 데이터 배치 복사
4. 복사 완료 후 테이블 RENAME 스왑
5. 원본 삭제
대용량은 Online DDL 또는 pt-osc / gh-ost로 — Lock 없이 스키마를 바꾼다.
##

② 엔진 전환 — Blue-Green

MySQL → Aurora
# 준비: 신규 DB에 실시간 Replication
App ──→ MySQL(현재) ══(replication)══> Aurora(신규)

# 검증: Aurora 읽기 테스트 · 정합성 · 벤치마크
# 전환: Write/Read를 Aurora로 → 구 DB는 롤백용 대기
App ──→ Aurora(신규)        MySQL(구) // standby
복제로 따라잡고 → 검증 → 전환 → 구 DB는 롤백용으로 남긴다.
##

전환 시 다운타임 최소화

①DNS / CNAME가장 일반적 · CNAME 변경 → TTL(60s) 후 전환 · 수초~1분
②ProxySQL / HAProxy프록시 라우팅 변경 → 거의 무중단
③Feature Flag앱 레벨 DataSource 전환 · 즉시 롤백 · 코드 복잡↑
즉시성·롤백 편의는 ③ > ② > ①, 단순함은 ① > ② > ③.
!!

1억 건 — dump는 함정

💥
mysqldump → import는 수시간~수일. 그 사이 변경분이 통째로 유실된다. 멈춰 있는 스냅샷을 옮기는 동안 원본은 계속 바뀐다.
→ 살아 있는 DB는 "멈춰서 복사"가 불가능하다.
##

정석 — CDC로 따라잡기

Change Data Capture
snapshot + binlog tailing
1. 초기 스냅샷 복사 (백그라운드)
2. 복사 중 변경분은 CDC가 binlog로 실시간 추적
3. 스냅샷 완료 후 밀린 변경분 적용
4. 차이가 0에 수렴하면 전환
⭐
도구: AWS DMS · Debezium · gh-ost — 동기화하며 옮긴다
##

롤백 — 항상 준비

1즉시 전환트래픽을 DNS/Proxy로 구 DB로 되돌린다
2차이 반영전환 후 발생분을 구 DB에 반영
3원인 분석재시도 일정 수립
✅
핵심: 구 DB를 최소 2주 유지 — 늦게 발견되는 문제까지 롤백 가능
