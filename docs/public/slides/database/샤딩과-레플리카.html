# 샤딩과
레플리카

// 읽기 분산 vs 쓰기 분산. 무엇을 먼저?

한 줄 요약

레플리카는 읽기를,
샤딩은 쓰기를 분산한다.
레플리카를 먼저 하고, 그래도 안 되면 샤딩을 고려한다.
##

레플리카란?

Replication
Primary가 쓰고, 복사본이 읽는다
Primary (Master): 읽기 + 쓰기
Replica (Slave):  읽기 전용 (복사본)

Primary ──복제──> Replica 1
                 ──> Replica 2
Primary의 데이터를 여러 Replica로 복제 → 읽기 트래픽을 여러 대로 분산.
##

복제 방식 3가지

①동기 Sync커밋 전 Replica 확인 · Lag 없음·유실 없음 · 성능 저하 큼
②비동기 Async커밋 후 전달 · Lag 있음·유실 가능 · 대부분 이걸 쓴다
③반동기 Semi-Sync최소 1대 확인 후 커밋 · Lag·유실 최소화
실무 기본은 비동기. 동기는 정합성을 얻고 성능을 잃는다.
##

Read / Write 분리

RoutingDataSource
OrderService.java
@Transactional(readOnly = true)   // → Replica
public List<Order> getOrders() { ... }

@Transactional                    // → Primary
public void createOrder() { ... }
readOnly 플래그로 라우팅. 읽기는 Replica, 쓰기는 Primary로 자동 분기.
!!

레플리카의 한계

⚠️
레플리카는 읽기를 분산하지만 쓰기는 분산하지 못한다 모든 쓰기가 Primary 1대에 몰림 → 쓰기 QPS 한계 · Primary CPU 100% → Replica를 아무리 늘려도 소용없다.
→ 쓰기 분산이 필요하면? 샤딩으로 간다.
##

샤딩이란?

Sharding
데이터를 여러 DB에 나눠서 저장
// Before: 단일 DB
orders: 10억 건 → 1대가 감당

// After: 샤딩
Shard 1: user_id 1~1000만
Shard 2: user_id 1000만~2000만
Shard 3: user_id 2000만~3000만
데이터를 쪼개 여러 대가 쓰기를 나눠 감당한다.
##

샤딩 키 — 가장 중요한 결정

좋은 키 ✅
균등 분배 + 주요 쿼리 포함 + 불변
예: user_id → 사용자별 단일 샤드 접근
나쁜 키 ❌
created_at (최신 샤드 핫스팟)
status (편향) · 자주 바뀌는 값
키 하나가 전체 분산 효율과 운영 난이도를 좌우한다.
##

샤딩 전략 3가지

①Rangeuser_id 범위별 · 범위 쿼리 용이 · 핫스팟 가능
②Hashhash(key) % N · 균등 분배 · 범위 쿼리 불가
③Directory매핑 테이블로 관리 · 유연한 배치 · 매핑 의존
Hash 예: shard = hash(user_id) % 3 → 3개 샤드로 균등 분배.
!!

샤딩의 고통 5가지

①Cross-Shard JOIN 불가다른 DB라 SQL JOIN 안 됨 · 앱에서 합쳐야
②Cross-Shard 트랜잭션2샤드 걸친 원자적 처리 어려움
③Resharding샤드 추가 시 대부분 데이터 재배치
④글로벌 유니크 IDAuto Increment 충돌 · Snowflake·UUID
⑤운영 복잡도 폭증스키마·백업·모니터링 샤드 수만큼
##

Resharding을 덜 아프게

Consistent Hashing
샤드 3대 → 5대로 늘릴 때 이동량
// 일반 Hash: % N 이 바뀌면 거의 다 이동
3대 → 5대: 대부분의 키 이동 (60%+)

// Consistent Hashing: 해시 링 위에 배치
3대 → 5대: 인접 키만 이동 (약 20%)
키를 링 위에서 시계 방향 가장 가까운 노드에 배치 → 노드 추가 시 인접 키만 이동.
##

"정말 샤딩이 필요한가?"

①쿼리 최적화인덱스·실행 계획 · 대부분 여기서 해결
②캐싱 (Redis)읽기 부하 해결
③Read Replica읽기 분산
④파티셔닝·수직 분할단일 DB 내 분할 · 서비스별 DB 분리
⑤샤딩최후의 수단
!!

냉정한 현실

💡
한국에서 샤딩이 필요한 서비스는 네이버·카카오·쿠팡·배민 정도 대부분의 서비스는 Aurora + Read Replica로 충분하다. 샤딩은 운영 비용이 막대해 진짜 필요할 때만 해야 한다.
샤딩은 멋이 아니라 최후의 수단이다.
##

테이블 파티셔닝 (샤딩 대안)

단일 DB 내에서 논리적 분할
CREATE TABLE orders ( ... )
PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026)
);
-- 2024년 삭제: DROP PARTITION → 즉시
운영이 훨씬 단순하고 오래된 데이터 아카이빙에 강력. 단, 같은 DB라 쓰기 분산은 안 된다.
