# DynamoDB

한 줄 요약

AWS의 완전 관리형 NoSQL.
single-digit ms 응답 + 키 기반 조회가 메인이면 — 최적이다.
##

왜 RDB가 아닌가

트래픽이 커질수록
Scale-Up은 한계에 부딪힌다.
DynamoDB는 자동 Scale-Out + 서버리스. 트래픽이 폭증해도 항상 일정한 응답.
##

RDB vs DynamoDB

데이터 모델
RDB 고정 스키마 · Dynamo Key-Value/Document
쿼리
RDB SQL·JOIN 자유 · Dynamo PK/SK 조회만
확장
RDB Scale-Up · Dynamo 자동 Scale-Out
관리·비용
RDB 인스턴스 고정 · Dynamo 서버리스·요청당
##

핵심 개념, 한눈에

①키 구조 (PK / SK)데이터 분산 + 정렬·범위의 기본 축
②쿼리 방식GetItem · Query는 O.K, Scan은 금지
③GSIPK가 아닌 속성으로 조회 (남발 금지)
④접근 패턴 우선 설계쿼리를 먼저, 테이블은 그에 맞춰서
⑤Single Table Design한 테이블에 여러 엔티티 · JOIN 대체
##

① 키 구조 — PK / SK

분산 + 정렬
orders table · key schema
# Partition Key — 데이터 분산 기준 (필수)
PK = userId      // 사용자별로 파티션이 나뉜다

# Sort Key — PK 내부 정렬·범위 (선택)
SK = orderId     // 한 사용자의 주문이 정렬되어 저장
PK는 어디에 저장할지, SK는 그 안에서 어떻게 정렬·범위 조회할지를 정한다.
##

② 쿼리 — 되는 것

효율적
fast paths
GetItem  // PK + SK 단건 조회 → O(1)
Query    // PK 고정 + SK 범위 조회 → 효율적

// 예: user-1의 order-001 ~ order-003
WHERE userId = "user-1"
  AND orderId BETWEEN "order-001" AND "order-003"
키를 알고 들어가면 언제나 single-digit ms. 이게 DynamoDB의 본령.
!!

② 안 되는 것 — Scan / JOIN

⚠️
Scan은 전체 테이블을 순회한다 — 운영에서 절대 금지 JOIN 불가 · SUM·AVG 집계 불가 → 집계는 애플리케이션·별도 분석 스토어에서 처리.
→ DynamoDB는 키로 찌르는 DB. 훑는 순간 NoSQL의 장점이 사라진다.
##

③ GSI — 다른 축으로 조회

보조 인덱스
Global Secondary Index
// 메인: PK = userId, SK = orderId

// GSI-1 (상태별): status = "PAID"를 최신순으로
GSI-1  PK = status     SK = createdAt

// GSI-2 (상품별): 특정 상품의 주문 이력
GSI-2  PK = productId  SK = createdAt
⚠️
GSI마다 별도 비용 + 쓰기 시 GSI도 갱신 → 정말 필요한 접근 패턴에만. GSI 남발 금지.
##

④ 접근 패턴을 먼저

RDB: 데이터 구조 먼저.
DynamoDB: 쿼리(접근 패턴) 먼저.
"무엇을 조회할지"를 모두 적어두고 → 그 패턴을 전부 만족하는 PK/SK를 거꾸로 설계한다.
##

패턴을 키로 옮기면

예시
access patterns → key design
# 접근 패턴
# 1) 사용자별 주문 목록  2) 주문 단건
# 3) 특정 기간 주문      4) 상태별 주문

PK = USER#{userId}
SK = ORDER#{createdAt}#{orderId}
GSI-1  PK = STATUS#{status}  SK = {createdAt}
1~3번은 메인 키로, 4번은 GSI로 — 모든 패턴이 키 하나로 풀린다.
##

⑤ Single Table Design

JOIN 대체
한 테이블 · 여러 엔티티 (PK/SK 접두사로 구분)
USER#001      PROFILE             { name, email }
USER#001      ORDER#2026-03-01#A1 { amount, status }
USER#001      ORDER#2026-03-15#A2 { amount, status }
PRODUCT#P100  INFO                { name, price }
✅
한 번의 Query로 사용자 + 주문을 동시에 — RDB의 JOIN을 키 설계로 대체한다.
##

이럴 때 쓴다

✓세션 저장소Key-Value · 빠른 읽기/쓰기
✓게임 리더보드높은 쓰기 처리량
✓IoT 센서 데이터대량 쓰기 · 시계열
✓장바구니 · 이벤트 로그사용자별 독립 · Append + 범위 조회
!!

이럴 땐 RDB로

💥
복잡한 JOIN · Ad-hoc 쿼리 · 집계/분석 쿼리 패턴이 미리 정해지지 않거나, 관계가 복잡하거나, 강한 정합성이 필수라면 — RDB가 맞다.
DynamoDB 기본은 Eventual Consistency. 강한 일관성이 늘 필요하면 신중히.
$

면접에서 이렇게 답하라

“키 기반 조회 + 일관된 ms 응답이면 DynamoDB.
RDB와 반대로 접근 패턴을 먼저 정의하고 PK/SK를 설계하고,
Single Table로 JOIN 없이 한 Query로 가져온다.”
복잡한 JOIN·집계면 RDB, 대량 단순 읽기/쓰기면 Dynamo — 완전 관리형·자동 확장이 최대 장점.
