# N+1 문제와
조회 최적화

const question = "N+1 · fetch join · entity graph · DTO · QueryDSL";
// 각각 무엇이고, 언제 무엇을 고를까?
>

한 줄 요약

N+1은 ORM의 원죄다.
해결책은 여러 개 — 상황에 맞는 걸 고르는 것이 실력이다.
##

N+1 문제란?

OrderService.java
List<Order> orders = orderRepository.findAll();  // 쿼리 1개

for (Order o : orders) {
    o.getUser().getName();   // 주문마다 User 쿼리 +1
    o.getItems().size();   // 주문마다 Items 쿼리 +1
}
LAZY 로딩된 연관을 반복문에서 건드릴 때마다 추가 쿼리가 나간다.
주문 100개 조회 · 1 + 100 + 100 =
201 쿼리
1개면 될 쿼리가 201개로 폭발한다. 이것이 N+1.
##

해결책 5가지, 한눈에

①Fetch Join1쿼리 · 가장 직관적 (단, 컬렉션 페이징·카티전 함정)
②@EntityGraph어노테이션으로 간편 · LEFT OUTER 고정
③@BatchSize설정만으로 IN절 묶음 · 페이징+컬렉션 정석
④DTO Projection필요한 컬럼만 · 읽기 전용 최速
⑤QueryDSL타입 세이프 · 동적 쿼리
##

① Fetch Join

1 query
OrderRepository.java
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.user " +
       "JOIN FETCH o.items WHERE o.status = :s")
List<Order> findWithDetails(@Param("s") OrderStatus s);
연관을 한 방의 JOIN으로 가져온다. 기본 INNER, LEFT도 직접 선택 가능.
!!

함정 1 — 컬렉션 + 페이징

⚠️
컬렉션 Fetch Join에 페이징을 걸면 HHH000104: applying in memory 전체를 메모리에 올린 뒤 잘라낸다 → OOM 위험. 페이징은 메모리에서 일어난다.
→ 해결: ToOne만 Fetch Join + ToMany는 @BatchSize (③ 참고)
!!

함정 2 — 부모 중복과 DISTINCT

1:N Fetch Join은 부모 행이 자식 수만큼 중복된다
// distinct 없이: Order 1이 Item 개수만큼 중복 등장
@Query("SELECT o FROM Order o JOIN FETCH o.items")

// DISTINCT: 앱 레벨에서 중복 부모 제거 (SQL DISTINCT와 별개)
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
✅
Hibernate 6 / JPA 3.0+ 부터는 엔티티 조회 시 중복 제거가 기본 — distinct 없이도 OK (구버전은 distinct 필수)
!!

함정 3 — 다중 컬렉션

💥
컬렉션을 2개 이상 JOIN FETCH → MultipleBagFetchException Bag(List) 2개의 카티전 곱을 막으려 Hibernate가 던지는 방어 예외. Set·@OrderColumn으로 피해도 진짜 카티전 곱이 터진다.
→ 해결: 하나만 Fetch Join + 나머지는 @BatchSize
##

② @EntityGraph

LEFT OUTER 고정
OrderRepository.java
@EntityGraph(attributePaths = {"user", "items"})
@Query("SELECT o FROM Order o WHERE o.status = :s")
List<Order> findByStatus(@Param("s") OrderStatus s);
JPQL 없이 연관 로딩. 단 컬렉션을 포함하면 Fetch Join과 같은 함정(페이징·카티전·MultipleBag)을 그대로 겪는다.
##

③ @BatchSize

페이징 정석
application.yml / Entity
# 전역 설정
hibernate.default_batch_fetch_size: 100

// 자식 조회를 IN 절로 묶음 (부모 100개↓ → 1쿼리, 초과 시 1+α)
SELECT * FROM order_items WHERE order_id IN (1,2,…,100);
⭐
페이징 + 컬렉션의 정석: ToOne = Fetch Join, ToMany = @BatchSize
##

④ DTO Projection

읽기 전용 최速
OrderRepository.java
@Query("SELECT new com.app.OrderDto(" +
       "o.id, u.name, o.totalAmount) " +
       "FROM Order o JOIN o.user u")
List<OrderDto> findDtos();
필요한 컬럼만 + 영속성 컨텍스트 관리 없음 → 가장 가볍다. 원칙: 읽기는 DTO, 수정은 Entity.
##

⑤ QueryDSL

타입 세이프 동적
OrderQueryRepository.java
queryFactory.selectFrom(order)
    .where(
        statusEq(status),     // null이면 조건 무시
        amountGoe(minAmount)  // null이면 조건 무시
    )
    .fetch();
문자열 JPQL 조합의 지옥을 컴파일 타임 체크로 바꾼다. 동적 검색·필터에 최적.
##

상황별 선택 가이드

수정이 필요한가?
Entity — Fetch Join / EntityGraph
읽기 전용 + 동적 조건?
QueryDSL + DTO Projection
컬렉션 + 페이징?
ToOne Fetch Join + ToMany @BatchSize
단순 읽기?
DTO Projection — 가장 가벼움
