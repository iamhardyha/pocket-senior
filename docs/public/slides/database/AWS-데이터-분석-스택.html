# AWS 데이터
분석 스택

const stack = "Athena · Glue · Redshift · DynamoDB";
// 뭐가 뭐고, 언제 뭘 쓰고, 어떻게 엮나?
##

먼저 두 층으로 가른다

OLTP vs OLAP
운영(키로 읽고/쓰기)이냐 분석(대량 스캔·집계)이냐 — 이 층 구분이 결정의 9할.
DynamoDB
운영 OLTP · 키 조회 NoSQL · ms급
Athena
분석 OLAP · 서버리스 SQL로 S3 직접
Redshift
분석 OLAP · 컬럼형 MPP 웨어하우스
Glue
분석의 받침대 · 카탈로그 + ETL
##

언제 뭘 쓰나 — 결정 흐름

decision-flow · 위에서 아래로
Q1. 운영 트랜잭션? (앱이 키로 읽고/쓰는 실시간)
     예  → DynamoDB (또는 RDB)        ← OLTP, 끝
     아니오(분석·집계·리포트) ↓                ← OLAP

Q2. 이미 S3에 있고 · 가끔·애드혹·탐색적?
     예  → Athena (서버리스 · 스캔당 과금)
     아니오 ↓

Q3. 정형 BI 반복 · 복잡 조인 · 낮은 지연/동시성?
     예  → Redshift (상시 클러스터 / RPU)

* 어느 쪽이든 스키마·파티션·포맷은 Glue가 받친다.
##

Athena vs Redshift

분석 스택의 갈림길
Athena — 서버리스
S3 직접 쿼리 · 클러스터 없음
애드혹 · 간헐적 · 탐색적
스캔량 $5/TB · 안 쓰면 0
Redshift — 상시 웨어하우스
적재 후 고성능 쿼리
반복 BI · 복잡 조인 · 낮은 지연
노드/RPU + RMS · 띄우면 과금
판단의 핵심은 쿼리 빈도와 성능 요구 — 드물면 Athena, 반복·고성능·동시성이면 Redshift. 카탈로그를 공유하면 둘 다로 같은 S3를 쿼리.
##

운영 DB vs 분석 · Glue의 자리

DynamoDB는 분석 엔진이 아니다
키 조회 챔피언 · 풀스캔·집계엔 부적합
핫 파티션 throttle 위험
→ 집계는 Streams·CDC로 레이크에
Glue는 선택지가 아니라 받침대
Catalog · Crawler · ETL
Athena·Redshift가 한 카탈로그 공유
엔진이 아니라 공통 기반
원칙: 운영 테이블에서 큰 집계를 직접 돌리지 마라 — 변경을 비동기로 분석 스택에 흘려보낸다.
##

레퍼런스 A — 서버리스 레이크

가끔 들여다본다
운영팀 없이 · 애드혹 분석 · 유휴 비용 0
[ 앱·서비스 로그/이벤트 ]
      │  Firehose (버퍼링 → Parquet → 파티셔닝)
      ▼
[ S3 데이터 레이크 ]  year=/month=/day=/
      │  [ Glue ] Crawler 스키마 추론 · ETL raw→Parquet
      ▼
[ Glue Data Catalog ]  (테이블·파티션)
      ▼
[ Athena ] → 애드혹 SQL → [ QuickSight BI ]
전 구간 서버리스 → 띄워 둘 클러스터가 없어 유휴 비용 0. 빈도가 낮고 들쭉날쭉할수록 유리.
##

레퍼런스 B — 운영 → 분석

운영 DB 무영향
운영 테이블 성능을 절대 건드리지 않는다
[ DynamoDB ]  (운영 OLTP · 키 조회)
      │  Streams / CDC  (아이템 변경 캡처)
      ▼
[ Firehose / Lambda ] → [ S3 데이터 레이크 ]
      ▼
[ Glue 카탈로그 / ETL ]  (정규화·Parquet)
      ▼
[ Athena ] 애드혹 검증   ·   [ Redshift ] 정기 BI
변경을 비동기로 레이크에 흘려보내 운영 DynamoDB는 무영향. 탐색은 Athena로 싸게, 굳으면 Redshift로.
##

레퍼런스 C — 웨어하우스 + 레이크하우스

핫은 빠르게 · 콜드는 싸게 · 카탈로그는 하나로
          [ Glue Data Catalog ]  ← 공유 메타스토어 (단일 진실원)
            ╱                ╲
   [ Athena ]          [ Redshift (RA3) ]
   (S3 직접)          hot → RMS(SSD) · cold → S3
       │                    │
       ▼                    ▼
   [ S3 데이터 레이크 ] ◄──── 외부 테이블로 같은 데이터 조인
한 카탈로그를 Athena·Redshift가 공유 — 핫은 RA3/RMS로 빠르게, 콜드는 S3로 싸게. 이게 레이크하우스.
##

과금 축은 서비스마다 다르다

AAthena / Spectrum스캔한 데이터량 $5/TB · 손잡이 = 파티션 + 컬럼형
GGlueDPU-hour $0.44 · DPU 수 · 잡 시간 · 크롤러 빈도
RRedshift노드 시간 / RPU + RMS 스토리지 · 가동 시간
DDynamoDB요청 / 용량 · 읽기·쓰기 패턴 · 키 분산
분석 비용 = 스캔량 싸움. 파티셔닝 + 컬럼형으로 읽는 바이트를 줄이면 쿼리당 30~90% 절감.

면접에서 이렇게 답하라

“먼저 운영(OLTP) · 분석(OLAP)으로 층을 가른다.
키 조회는 DynamoDB, 분석은 S3 레이크 위에서 —
가끔이면 Athena · 반복·고성능이면 Redshift,
Glue는 둘이 공유하는 카탈로그·ETL.”
비용은 Athena=스캔량 · Redshift=가동 시간 · Glue=변환 시간 · DynamoDB=요청. 핵심은 파티셔닝+컬럼형으로 스캔량 줄이기.
