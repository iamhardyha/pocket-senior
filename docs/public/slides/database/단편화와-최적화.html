# 단편화와
최적화

DELETE → 디스크 == "그대로?";
// 지웠는데 왜 용량이 안 줄어들까?
>

한 줄 요약

DB는 DELETE해도
공간을 돌려주지 않는다.
빈 자리가 쌓이면 I/O가 늘고 성능이 떨어진다 — OPTIMIZE TABLE로 재정리하라.
##

DELETE는 지우지 않는다

InnoDB 내부 동작
DELETE FROM orders WHERE id = 100;

// 1. 행에 "삭제됨" 플래그(delete mark)만 설정
// 2. MVCC 위해 Undo Log에 이전 버전 보관
// 3. Purge Thread가 나중에 물리 제거 → 재사용 공간으로
물리적으로 디스크에서 지우지 않고, "재사용 가능" 공간으로만 바꾼다. OS엔 반환 안 됨.
##

왜 바로 안 지울까?

①MVCC다른 트랜잭션이 삭제 전 데이터를 읽고 있을 수 있음
②성능물리 삭제 + 페이지 재배치는 비용이 큼
③재사용빈 공간에 새 데이터 → 새 페이지 할당보다 빠름
④크래시 복구Redo/Undo 로그 기반 복구 위해 즉시 제거 불가
##

단편화 (Fragmentation)

16KB 페이지
테이블스페이스 — 16KB 페이지 단위
██ = 데이터    ░░ = 빈 공간 (삭제/UPDATE로 발생)

Page1 [██████]  Page2 [██░░██]  Page3 [░░░░░░]  Page4 [██░░██]
       꽉 참            중간 삭제        전부 삭제됨      행 크기 변동
INSERT→UPDATE→DELETE 반복으로 물리적 저장 순서가 뒤죽박죽되는 현상.
##

오늘의 지도

①성능 저하 3가지I/O 증가 · Buffer Pool 효율 ↓ · 공간 낭비
②확인 방법information_schema · SHOW TABLE STATUS
③OPTIMIZE TABLE데이터를 PK 순서로 재배치 = 테이블 재생성
④운영 도구pt-osc · gh-ost — 무중단 최적화
⑤예방 설계파티셔닝 · AUTO_INCREMENT PK
##

① 순차 I/O → 랜덤 I/O

클러스터드 인덱스 (PK 순서로 물리 저장)
[정상] 연속 저장
Page1 → Page2 → Page3 → Page4   순차 읽기 → 빠름

[단편화] 흩어진 저장
Page1 → Page5 → Page12 → Page3   랜덤 읽기 → 느림
단편화가 심하면 PK 순서와 물리 위치가 어긋나 순차 I/O가 랜덤 I/O로 바뀐다.
랜덤 vs 순차 읽기 격차
200배 / 6배
HDD는 200배, SSD에서도 6배 차이. SSD라고 단편화를 무시할 수 없다.
##

② Buffer Pool 효율 저하

Buffer Pool 1GB 기준 — 빈 공간이 캐시를 잡아먹는다
[정상]    페이지 충전율 90% → 유효 데이터 ~900MB
[단편화]  페이지 충전율 50% → 유효 데이터 ~500MB

→ 같은 데이터에 페이지를 더 많이 읽음 → 캐시 히트율 하락
반쯤 빈 페이지가 메모리에 올라오면 실제 캐시할 수 있는 데이터가 줄어든다.
##

③ 디스크 공간 낭비

💥
실제 데이터는 1GB인데 500MB가 빈 공간으로 낭비 data_free는 OS에 반환되지 않으므로 디스크에선 1.5GB를 차지한다. (fragmentation 50%)
→ I/O·메모리·디스크, 세 군데서 동시에 손해를 본다.
##

단편화 확인하기

information_schema.tables
SELECT table_name,
       ROUND(data_free / 1024 / 1024, 2) AS free_mb,
       ROUND(data_free / data_length * 100, 2) AS frag_ratio
FROM information_schema.tables WHERE table_schema = 'mydb';
📐
기준: frag_ratio 20% 이상이면 최적화 고려 data_free는 innodb_file_per_table=ON(5.6.6+ 기본)일 때만 정확
##

OPTIMIZE TABLE — 재생성

내부적으로 = ALTER TABLE ... FORCE
OPTIMIZE TABLE orders;

// 1. 새 테이블스페이스 생성
// 2. 데이터를 PK 순서대로 복사 → 3. 인덱스 재구축
// 4. 기존 삭제 → 5. 새 것으로 교체
빈 공간을 압축해 페이지를 다시 채운다 — 3페이지(빈틈 多) → 2페이지로 응축.
!!

OPTIMIZE의 함정

⚠️
기본 ONLINE(INPLACE)이라 DML은 대부분 허용되지만 시작/종료에 짧은 메타데이터 락 임시로 테이블 크기만큼 추가 디스크 필요 · 크기 비례 소요 시간 · Replica에서도 실행 → 복제 지연
→ 소형 테이블(< 1GB)엔 OK, 대형 테이블엔 전용 도구를 쓴다.
##

무중단 최적화 도구

①pt-online-schema-changePercona · 트리거로 변경분 반영 · chunk + max-lag 자동 조절
②gh-ostGitHub · 트리거 대신 binlog를 읽어 반영 → 성능 부담 ↓
⭐
둘 다 임시 테이블 복사 → RENAME로 원자적 교체 · 복제 지연 시 자동 일시정지
##

도구 선택 가이드

작은 테이블 (< 1GB)?
OPTIMIZE TABLE — 내장, 가장 쉬움
중대형 (< 100GB)?
pt-osc — 거의 무중단, 복제 안전
모든 크기 · 트리거 부담 X?
gh-ost — binlog 기반
애초에 안 쌓이게?
파티셔닝 + DROP PARTITION
##

애초에 덜 쌓이게

①파티셔닝날짜 파티션 → DROP PARTITION으로 즉시 공간 회수
②AUTO_INCREMENT PK순차 삽입 → 페이지 분할 최소화 (UUID는 단편화 유발)
③배치 삭제대량 삭제 대신 LIMIT 1000씩 나눠 삭제
④Soft Deletedeleted_at 컬럼 사용 (조회 WHERE 조건 필요)
!!

UUID가 단편화를 부른다

삽입 위치가 단편화를 결정한다
AUTO_INCREMENT: 항상 맨 뒤 삽입 → 페이지 분할 없음
UUID (v4)     : 랜덤 위치 삽입 → 페이지 분할 빈번 → 단편화

// UUID 필수면: UUIDv7(시간순) 또는 UUID_TO_BIN(UUID(), 1)
UUIDv7은 앞부분이 타임스탬프라 생성 자체가 시간순 → 순차 삽입에 가깝다.
