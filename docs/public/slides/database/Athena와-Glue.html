# Athena + Glue
S3를 SQL로

const question = "S3에 쌓인 데이터를 서버 없이 SQL로?";
// Athena와 Glue는 어떻게 맞물려 돌아갈까?
>

한 줄 요약

Athena는 엔진일 뿐,
스키마는 Glue Catalog가 쥔다.
데이터(S3) · 메타데이터(Catalog) · 엔진(Athena)이 분리 — 원본은 그대로 두고 읽을 때 스키마를 입힌다(schema-on-read).
##

Glue = 세 가지의 묶음

①Data Catalog중앙 메타스토어 — 테이블·스키마·파티션 정의 (Athena가 의존)
②CrawlerS3 스캔 → 스키마 자동 추론 → 카탈로그에 테이블·파티션 등록
③ETL JobSpark로 변환·정제 — JSON → Parquet · 파티셔닝
Catalog는 정의를 보관, Crawler는 정의를 채움, ETL은 데이터를 가공.
##

원천 → 쿼리까지

data flow — 데이터·메타데이터·엔진의 분리
S3 raw  // JSON / CSV / 로그
   │  ① Crawler: 스캔 → 스키마 추론
   ▼
Glue Data Catalog  // 메타스토어
   │  ② ETL: JSON → Parquet · 파티셔닝
   ▼
S3 curated  // dt=2026-06-30/ ...
   │  ③ Crawler 재실행 / 파티션 등록
   ▼
Athena  // 카탈로그 보고 S3 스캔
②(Parquet 변환)는 선택이지만 비용·성능 때문에 사실상 필수.
##

CREATE EXTERNAL TABLE

DDL은 무료
스키마가 안정적이면 Crawler 없이 DDL로 직접 정의
CREATE EXTERNAL TABLE events (
  user_id    string,
  event_type string,
  amount     double,
  ts         timestamp
)
PARTITIONED BY (dt string)   -- 파티션 컬럼 분리 선언
STORED AS PARQUET             -- 컬럼형 포맷
LOCATION 's3://my-lake/curated/events/';
EXTERNAL → DROP TABLE 해도 S3 데이터는 보존, 카탈로그 정의만 삭제.
##

스캔을 줄이는 두 레버

파티션 프루닝 + 컬럼형 = 읽는 바이트↓
SELECT event_type, COUNT(*), SUM(amount)
FROM events
WHERE dt = '2026-06-30'   -- 이 폴더만 읽음 (pruning)
  AND event_type = 'purchase'
GROUP BY event_type;
⭐
WHERE에 파티션 컬럼을 걸어야 효과 — 빼면 전체 스캔. Parquet은 고른 컬럼만 읽어 또 줄인다.
요금 = 결과 행 수가 아니라 스캔한 바이트
$5 / TB
압축 · 파티셔닝 · 컬럼형 세 레버로 스캔량을 줄이면 보통 30~90% 절감. DDL·실패 쿼리는 무료.
💡
Glue: Catalog 첫 100만 객체 무료 · ETL $0.44/DPU-hour (1 DPU = 4 vCPU+16 GB) · Crawler 최소 10분예) 6 DPU × 0.25h × $0.44 ≈ $0.66
##

Athena 심화 세 가지

①Capacity Reservations스캔량 대신 시간당 컴퓨팅 용량으로 과금 · per-query와 혼용 가능
②Federated QueryLambda 커넥터로 S3 밖(RDS·DynamoDB)을 한 SQL로 조인
③CTAS쿼리 결과를 Parquet·파티션 테이블로 물질화 · 가벼운 변환은 ETL 없이

“Glue Catalog가 S3 파일에 스키마를 입혀 테이블로 보이게 하고,

비용은 곧 스캔한 바이트($5/TB) — Parquet·파티셔닝으로 줄이는 게 핵심.”
schema-on-read 구조 — Catalog 없으면 Athena는 스키마를 모른다.
