# Aurora
업그레이드

const question = "RDS → Aurora, 뭐가 좋아지고 뭘 고민할까?";
// 공유 스토리지가 바꾸는 것들
>

한 줄 요약

Aurora는 관리형 고가용성을 산다.
MySQL·PostgreSQL 호환은 그대로. 대신 비용·벤더 락인을 지불한다.
##

왜 RDS에서 갈아탈까?

⏳
일반 RDS는 binlog 복제 — 피크엔 Lag 5~10초, Failover 1~2분. Replica는 각자 EBS 스토리지를 따로 들고, 추가할 때마다 데이터를 통째로 복사한다.
읽기 부하가 커질수록 복제 지연과 운영 부담이 한계로 다가온다.
##

핵심은 공유 스토리지

aurora-storage.txt
# 일반 RDS: 인스턴스마다 독립 스토리지
Primary ──(binlog)──> Replica   // 복제 = 데이터 전송

# Aurora: 하나의 분산 스토리지를 공유
Primary ─┐
Replica ─┼─> 공유 스토리지 (3 AZ · 6 복제본)
Replica ─┘   // 복제 = 같은 디스크 읽기
Replica는 데이터를 복사하지 않고 같은 스토리지를 읽는다. 여기서 모든 장점이 파생된다.
Replication Lag · 5~10초 →
20ms 이하
Failover도 1~2분 → 30초 이내. 사실상 즉시 동기화·즉시 승계.
##

RDS vs Aurora

◆스토리지EBS 단일 AZ → 분산 3 AZ · 6 복제본
◆복제binlog 기반(느림) → 스토리지 공유(빠름)
◆Failover1~2분 → 30초 이내
◆스토리지 확장수동 설정 → 10GB~128TB 자동
◆백업스냅샷 → 연속 백업 + PITR(1초 단위)
##

전환 전 고민 4가지

①비용인스턴스 +20% · 스토리지·I/O 별도 (소규모는 손해)
②호환성99%지만 100%는 아니다 (검증 필수)
③벤더 락인전용 기능 쓸수록 이탈 어려움 (표준 SQL 고수)
④커넥션 관리Failover 시 커넥션 끊김 (풀 검증·재연결)
##

① 비용 — 언제 역전되나

cost-compare.txt (r6g.xlarge 기준)
RDS    $0.48/hr ≈ 월 $350
Aurora $0.58/hr ≈ 월 $420  // +20%, 스토리지·I/O 별도

소규모 → Aurora가 더 비쌈
대규모 → Replica·DBA 운영비 절감으로 역전
판단 기준: 고가용성이 필요한가? DBA가 없는가? → Yes면 값을 한다.
!!

② 호환성 — 99%의 함정

⚠️
MySQL 5.7/8.0과 거의 같지만 차이가 있다. 시스템 변수 차이 · MyISAM 미지원(InnoDB만) · 일부 플러그인 미지원 · mysql_native_password deprecated.
검증: 테스트 스위트 전체 실행 · 실행계획 비교 · 부하 테스트.
!!

③ 벤더 락인

🔒
전용 기능을 쓸수록 이탈이 어려워진다. Serverless · Global Database · Backtrack · Parallel Query — 강력하지만 Aurora 종속.
✅
완화: 앱은 표준 SQL만 사용하고, 전용 기능은 인프라 레벨에서만 활용 — 코드에 Aurora 의존성 넣지 않기.
##

④ 커넥션 관리

Failover 대비
application.yml
spring:
  datasource:
    hikari:
      connection-test-query: SELECT 1   # 유효성 검사
      validation-timeout: 3000
      max-lifetime: 1800000         # 30분마다 갱신
Aurora는 Failover 시 커넥션이 끊긴다. 풀 검증 + 짧은 timeout으로 빠르게 재연결.
##

나아진 점 — 운영 편의성

Replica 추가
수십분 복사 → 클릭 한 번
Failover
수동·1~2분 → 자동 30초
백업 복구
스냅샷 → Backtrack / PITR
스토리지
수동 확장 → 자동 (관리 불필요)
##

장애 내성 — 6 복제본

storage-durability.txt (3 AZ · 6 복제본)
2개 유실  → 읽기/쓰기 정상
3개 유실  → 읽기 정상, 쓰기 불가
// 손상된 복제본은 자동 재생성
스토리지가 스스로 치유한다. AZ 하나가 통째로 죽어도 데이터는 살아남는다.
##

전환은 Zero Downtime

🟢
Replication 기반 Blue-Green으로 무중단 전환. 기존 DB → Aurora로 복제를 걸어 따라잡힌 뒤, 엔드포인트만 전환(컷오버)한다.
컷오버 전 실행계획·부하 테스트를 끝내 두는 것이 핵심.
