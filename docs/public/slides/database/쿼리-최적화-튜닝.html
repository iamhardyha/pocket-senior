# 쿼리
최적화·튜닝

const question = "쿼리를 어떻게 분석하고 최적화하는가?";
// EXPLAIN으로 시작해서 인덱스로 끝난다.
>

한 줄 요약

EXPLAIN을 읽으면 절반이 풀린다.
나머지 절반은 인덱스를 이해하는 것이다.
##

튜닝은 정해진 순서다

①느린 쿼리 발견Slow Query Log로 문제 쿼리부터 찾는다
②EXPLAIN 분석실행 계획으로 왜 느린지 본다
③원인 파악풀스캔? 인덱스 미스? 정렬·임시 테이블?
④인덱스·쿼리 수정근거 있는 한 가지만 바꾼다
⑤개선 효과 확인다시 EXPLAIN으로 검증한다
##

EXPLAIN, 4개 컬럼만 봐라

type접근 방식좋음 const·eq_ref·ref / 나쁨 ALL(풀스캔)
key사용된 인덱스인덱스 이름이 떠야 정상 / NULL = 미사용
rows예상 검사 행적을수록 좋다 / 전체 행 수면 위험
Extra추가 정보좋음 Using index / 나쁨 filesort·temporary
##

type 성능 순서

좋은 순
왼쪽일수록 빠르다
const > eq_ref > ref > range > index > ALL

const  : PK/Unique로 1행 조회 (최고)
eq_ref : JOIN에서 PK/Unique 매칭
ref    : 일반 인덱스로 여러 행
range  : 인덱스 범위 스캔 (BETWEEN, >, <)
ALL    : 테이블 풀 스캔 (최악)
ALL이 보이면 일단 의심하라 — 풀스캔 신호다.
!!

Extra — 보이면 멈칫

⚠️
Using filesort · Using temporary 인덱스로 정렬을 못 해 별도 정렬 / GROUP BY·DISTINCT로 임시 테이블 → 둘 다 느림.
✅
Using index = 커버링 인덱스 테이블을 안 보고 인덱스만으로 끝낸다 — 가장 빠른 신호.
##

인덱스를 못 타는 순간

WHERE 절
-- 인덱스 사용 O
WHERE user_id = 100                  -- 등호
WHERE created_at > '2026-01-01'     -- 범위

-- 인덱스 사용 X
WHERE YEAR(created_at) = 2026       -- 컬럼에 함수
WHERE name LIKE '%검색어'          -- 앞쪽 와일드카드
WHERE amount * 1.1 > 10000        -- 컬럼에 연산
규칙: 인덱스 컬럼은 가공하지 마라. 함수·연산·앞 와일드카드는 인덱스를 죽인다.
##

복합 인덱스는 선두부터

INDEX (user_id, status, created_at)
-- 사용 O : 선두부터 연속으로 쓰면 탄다
WHERE user_id = ?
WHERE user_id = ? AND status = ?

-- 사용 X : 선두 컬럼(user_id) 누락
WHERE status = ?
WHERE created_at > ?
선두 컬럼이 빠지면 뒤 컬럼은 의미가 없다. 왼쪽부터 연속으로 써야 탄다.
##

인덱스 컬럼 순서 원칙

①동등 조건 (=)맨 앞에 — 가장 좁히는 조건
②범위 조건 (> < BETWEEN)그 뒤에 — 범위부턴 정렬이 깨진다
③정렬 (ORDER BY)맨 마지막에 — filesort 회피
예) WHERE user_id=? AND status=? ORDER BY created_at → (user_id, status, created_at)
##

커버링 인덱스

테이블 접근 0
인덱스에 SELECT 컬럼까지 모두 포함
CREATE INDEX idx_cov ON orders
  (user_id, status, created_at, total_amount);

SELECT status, created_at, total_amount  -- 인덱스에 다 있음
FROM orders WHERE user_id = 100;
-- Extra: Using index  → 테이블을 안 본다
필요한 컬럼이 전부 인덱스 안에 있으면 디스크 테이블 조회가 사라진다.
##

흔한 성능 함정 5선

①큰 OFFSET 페이징앞 행을 다 읽고 버림 → 커서 기반으로
②COUNT(*) 전체풀스캔 위험 → 근사치·"더보기" 패턴
③서브쿼리 IN→ JOIN / EXISTS로 재작성
④SELECT *전송량↑·커버링 불가 → 필요 컬럼만
⑤OR 조건인덱스 비효율 → UNION ALL 분리
!!

① OFFSET → 커서 페이징

100만 번째 페이지
-- 느림: 100만 행을 읽고 버린 뒤 10개 반환
SELECT * FROM orders ORDER BY id DESC
  LIMIT 10 OFFSET 1000000;

-- 빠름: 인덱스로 곧장 점프 (No-Offset)
SELECT * FROM orders WHERE id < 12345
  ORDER BY id DESC LIMIT 10;
마지막 본 id를 커서로 들고 다니면, OFFSET이 아무리 커도 비용이 일정하다.
!!

③ 서브쿼리 IN → JOIN·EXISTS

VIP 회원의 주문
-- 느림: 서브쿼리 IN
SELECT * FROM orders
WHERE user_id IN (SELECT id FROM users WHERE grade='VIP');

-- 빠름: JOIN / 큰 테이블이면 EXISTS
SELECT o.* FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.grade = 'VIP';
옵티마이저가 못 풀어주는 서브쿼리를 명시적 JOIN/EXISTS로 풀어준다.
##

실행 계획으로 잡는다

Before — 인덱스 없음
orders type=ALL · rows=1,000,000
→ 풀스캔
After — created_at 인덱스
orders type=range · rows=50,000
→ 범위 스캔
측정 → 인덱스 추가 → 다시 EXPLAIN. type이 ALL에서 range로 떨어지는지 눈으로 확인한다.
