한 줄 요약

Redshift = 컬럼형 + MPP.
빠름의 핵심은 읽을 양 줄이기(컬럼형·정렬키) · 노드 간 이동 없애기(분산키). 반복 BI엔 Redshift, 가끔 애드혹엔 Athena.
##

MPP 아키텍처

리더 · 컴퓨트 · 슬라이스
①리더 노드 (1개)클라이언트와 통신·쿼리 파싱·실행계획→컴퓨트 배포·부분결과 최종 집계. 데이터는 안 들고 있음
②컴퓨트 노드 (N개)각자 독립 CPU·메모리·디스크로 실제 데이터 저장·연산
③슬라이스(slice)컴퓨트를 더 쪼갠 병렬 처리 최소 단위. 노드 N × 슬라이스 = N갈래 병렬
##

왜 빠른가 — 컬럼형 × MPP

행 기반 (OLTP)
한 행의 모든 컬럼이 붙어 있음 → amount만 필요해도 전체 블록을 읽음
컬럼형 (Redshift)
같은 컬럼끼리 붙어 있음 → 필요한 컬럼 블록만 + 높은 압축률
읽을 컬럼만(컬럼형) × 압축으로 더 적게 × 여러 슬라이스가 동시에(MPP) = 대규모 집계 가속.
⚠️
단건 조회·잦은 UPDATE/DELETE엔 오히려 불리 분석용 웨어하우스지 운영 DB가 아니다.
##

재분배 — 조인의 가장 큰 비용

조인할 두 행이 다른 노드에 있으면 네트워크로 옮겨야 한다 = 재분배(redistribution).
💸
EXPLAIN에 비용이 보인다 — DS_DIST_NONE(이동 없음)이 최선, DS_DIST_BOTH(양쪽 재분배)가 최악. 큰 쿼리에선 재분배가 비용의 가장 큰 덩어리.
→ DISTKEY로 조인 키를 맞추면 매칭 행이 같은 슬라이스에 모여(코로케이션) 재분배가 사라진다 → DS_DIST_NONE.
##

분산 스타일 4가지

DISTSTYLE
KKEYDISTKEY 컬럼 해시로 배치 · 같은 키=같은 슬라이스. 자주 조인하는 큰 팩트 테이블
AALL전체를 모든 노드에 복제. 작고(<5백만 행) 안 바뀌는 디멘션 · 이동 0, 저장 N배
EEVEN라운드로빈 균등 분산(키 무관). 조인 적고 스캔 위주, 마땅한 키 없을 때
∗AUTO (기본)크기·패턴 보고 자동 선택·변경. 잘 모르면 AUTO로 두고 모니터링
분산키 선택: ① 자주 조인하는 키 ② 카디널리티 높아 고르게 분산(쏠리면 스큐).
##

정렬키 — zone map으로 스캔 스킵

SORTKEY
block skipping
-- SORTKEY(created_at) 로 정렬 저장 (블록마다 min/max = zone map)
블록1: 2026-01-01 ~ 01-10
블록2: 2026-01-11 ~ 01-20
블록3: 2026-01-21 ~ 01-31

WHERE created_at >= '2026-01-21'
  → zone map 보고 블록1·2 통째로 스킵, 블록3부터만 읽음
범위·WHERE에 자주 쓰는 컬럼(특히 날짜)을 SORTKEY로 → 스캔량 급감. 기본 Compound(첫 키로 거르기 강함).
##

실전 DDL — 스타 스키마 + COPY

fact / dimension 설계
CREATE TABLE fact_orders (
    order_id BIGINT, user_id BIGINT,
    amount DECIMAL(12,2), created_at TIMESTAMP
)
DISTSTYLE KEY DISTKEY (user_id)        -- 조인 키 분산 → 재분배 제거
COMPOUND SORTKEY (created_at);          -- 날짜 범위 가속 (zone map)

CREATE TABLE dim_user ( ... ) DISTSTYLE ALL;  -- 전 노드 복제 → 이동 0
⭐
적재는 INSERT가 아니라 COPY — S3 파일을 여러 슬라이스가 병렬로 빨아들인다. 파일을 슬라이스 수의 배수로 쪼개면 병렬이 고르게 된다.
##

컴퓨트 / 스토리지 분리

RA3 · Serverless · Spectrum
▣RA3 + RMS핫=로컬 SSD, 콜드=S3 기반 RMS 자동 계층화. 컴퓨트와 스토리지(GB-month) 분리 과금
∞Serverless클러스터 없이 RPU 단위 · base RPU 잡고 오토스케일, 쓴 만큼 과금
⌕Spectrum외부 테이블로 S3를 적재 없이 직접 SQL · Glue Catalog 공유(Athena와 동일)
⚠️
Spectrum(S3 외부테이블) 스캔은 $5/TB 별도. 무과금은 Serverless(RPU 포함)뿐 — 프로비저닝 RA3는 S3 쿼리에 $5/TB가 따로 붙는다.
##

Athena vs Redshift — 언제 무엇을

Athena
애드혹·탐색·간헐 쿼리
스캔 $5/TB · 안 쓰면 0
S3 그대로 · 운영 부담 0
Redshift
반복 BI·복잡 조인·낮은 지연·많은 동시 사용자
노드/RPU + RMS
적재해 분산·정렬 튜닝
가끔·탐색이면 Athena, 반복·고정 BI면 Redshift. 둘은 Glue Catalog 공유 → Spectrum으로 핫+콜드 한 쿼리(레이크하우스).
