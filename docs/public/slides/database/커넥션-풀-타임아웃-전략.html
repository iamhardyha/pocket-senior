# 커넥션 풀
타임아웃 전략

const question = "maxIdleTime · maxLifetime, 왜 무한대로 두면 안 되나?";
// 커넥션은 한 번 만들면 계속 쓰면 되는 거 아닌가?
>

한 줄 요약

커넥션은 살아 있는 자원이다.
오래 유지할수록 네트워크 장비·DB·OS 레벨에서 몰래 썩는다.
##

커넥션은 객체가 아니다

TCP 소켓 + DB 세션 + 중간 장비 상태
셋 중 하나라도 상태가 변하면 — 겉으론 살아있지만 실제론 죽은 half-open 상태가 된다.
##

두 개의 타임아웃

⏱maxIdleTime유휴 커넥션을 풀에 유지하는 최대 시간 (마지막 사용 후 경과)
⏳maxLifetime커넥션이 생성 시점부터 존재할 수 있는 최대 시간
idle는 "안 쓴 지 얼마나", lifetime은 "태어난 지 얼마나". 기준이 다르다.
##

무한대가 위험한 5가지 이유

①중간 장비 종료방화벽·NAT·LB가 유휴 세션을 몰래 끊는다
②DB 측 정리MySQL wait_timeout · PG idle_session_timeout
③세션 오염·누수세션 변수·Prepared Statement가 계속 쌓인다
④페일오버 미반영DNS 바뀌어도 죽은 Primary에 계속 연결
⑤OS 자원 고갈FD·소켓 메모리·DB 세션이 반환 안 됨
!!

① 방화벽이 몰래 끊는다

App ──TCP──▶ [방화벽/NAT/LB] ──TCP──▶ DB
1. 커넥션 30분 미사용
2. 방화벽이 NAT 테이블에서 세션 제거 (FIN/RST 안 보냄)
3. App은 살아있다고 믿음 (keepalive 미설정 시)
4. 요청 → 패킷 DROP → 재전송 → 수십 초 타임아웃
종료가 한쪽에만 통보되거나 양쪽 다 통보 안 된다. 이것이 가장 흔한 실무 장애.
##

환경별 유휴 타임아웃

350sAWS NLB고정 · 사용자 변경 불가 · NAT Gateway도 동일
60sAWS ALB기본 60초 · 1~4000초 범위 설정 가능
30m+일반 방화벽30분~1시간 · 장비·정책에 따라 상이
5dLinux conntrack기본 5일 · 클라우드는 VPC가 먼저 적용
##

② DB가 먼저 끊는다

서버 측 정리
DB 서버의 유휴 커넥션 정리 설정
-- MySQL: 기본 8시간 (28800초)
SHOW VARIABLES LIKE 'wait_timeout';  // JDBC는 이 값 적용

-- PostgreSQL (postgresql.conf)
idle_session_timeout = '30min'  // PG 14+ 일반 유휴 정리
💥
야간 무트래픽 → 8시간 유휴 → DB가 종료 → 출근 첫 요청에 "Communications link failure"
!!

③ 세션이 오염된다

커넥션은 DB 서버에 상태(State)를 가진다
// 어떤 요청이 세션 변수를 바꾸고 반납
connection.execute("SET SESSION transaction_isolation" +
                   " = 'READ-UNCOMMITTED'");

// 다음 요청이 같은 커넥션을 받으면?
→ READ-UNCOMMITTED가 그대로 적용된다!
세션 메모리·임시 테이블·Statement 캐시도 쌓인다. 주기적 재생성이 세션을 클린하게 초기화한다.
!!

④ 페일오버가 안 먹힌다

Aurora / RDS Multi-AZ 페일오버
1. Primary 10.0.1.100 장애
2. DNS가 Replica 10.0.1.200으로 변경
3. 기존 커넥션은 이미 10.0.1.100에 TCP 직결
4. 계속 죽은 Primary에 연결 시도 → DNS 무시
maxLifetime이 있으면 오래된 커넥션이 폐기·재생성되며 새 DNS가 반영된다.
##

HikariCP 권장 설정

Spring Boot 기본 풀
application.yml
spring.datasource.hikari:
  max-lifetime: 1800000    # 30분 — wait_timeout보다 짧게
  idle-timeout: 600000     # 10분 — 방화벽 idle보다 짧게
  keepalive-time: 300000   # 5분 — TCP 생존 확인 (4.0+)
  connection-timeout: 3000 # 3초 — 획득 대기
HikariCP는 maxLifetime을 반드시 설정하라고 강조한다. 0(무한대)은 명시적 비권장.
기억할 두 개의 부등식
maxLifetime < DB wait_timeout
idleTimeout < 방화벽 idle timeout
내가 먼저 폐기해야 — DB·방화벽이 끊은 죽은 커넥션을 받지 않는다.
##

왜 한꺼번에 안 만료될까

Variance
💡
HikariCP는 maxLifetime에서 최대 ~2.5%를 랜덤 차감한다 예) 30분 → 최대 ~45초 편차. 모든 커넥션이 동시에 만료돼 재생성이 몰리는 것(thundering herd)을 방지한다.
커넥션 재생성이 시간대별로 자연스럽게 분산된다.
##

유효성 검사, 어떻게?

Borrow 시 검증 (testOnBorrow)
죽은 커넥션 100% 차단 · 매 요청 쿼리 → 느림
유휴 중 주기 검사 (keepaliveTime)
성능 영향 적음 · 검사 사이 죽을 수 있음
시간 기반 폐기 (maxLifetime)
오버헤드 0 · 가장 깔끔 · 유효한 것도 버림
실무 최적 조합
maxLifetime + keepaliveTime
!!

사례 — 월요일 아침마다 장애

⚠️
주말 트래픽 감소 → 수시간 유휴 → NLB가 350초 후 세션 제거 App은 모름 → 월요일 첫 요청에 dead connection 할당 → 타임아웃 장애.
✅
