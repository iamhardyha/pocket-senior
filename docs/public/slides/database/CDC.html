# CDC
Change Data Capture

const question = "CDC가 무엇인가?";
// DB 변경을 어떻게 실시간으로 다른 시스템에 전파하나?
>

한 줄 요약

DB 트랜잭션 로그를 읽어
변경을 실시간 전파한다.
애플리케이션 코드 변경 없이, DB 수준에서 변경 이벤트를 스트리밍한다.
##

왜 필요한가?

DB가 바뀌면 캐시·검색·다른 서비스도
같이 바뀌어야 한다.
매번 코드에 동기화 로직을 끼우면 결합도가 폭발한다. CDC는 DB 변경을 이벤트로 떼어낸다.
##

오늘 다룰 것

①구현 방식 3가지Polling · Trigger · Log-based(주류)
②Debezium 아키텍처binlog/WAL → Connector → Kafka
③CDC vs Dual Write정합성을 로그가 보장
④실무 활용Outbox · CQRS · ETL · 캐시 무효화
⑤주의사항멱등성 · 스키마 · WAL 디스크 풀
##

① 구현 방식 3가지

1Pollingupdated_at 주기 조회 · 지연·DELETE 누락·DB 부하
2Trigger트리거로 Shadow 테이블 기록 · 원본 TX 성능 저하
3Log-based (주류)트랜잭션 로그 직접 읽기 · 부하↓·순서 보장
대표 구현체는 Debezium — Log-based 오픈소스.
!!

왜 Polling·Trigger를 안 쓰나

⚠️
Polling은 폴링 간격 내 다중 변경이 하나로 합쳐져 중간 상태가 유실되고, 물리 DELETE를 못 잡는다. Trigger는 원본 트랜잭션에 끼어들어 성능을 깎고, DB 엔진 내부 실행이라 디버깅이 어렵다.
→ 그래서 트랜잭션 로그를 직접 읽는 Log-based가 답.
##

로그는 어디서 읽나

MyMySQLbinlog (ROW format) · binlog_format=ROW · GTID 권장
PgPostgreSQLWAL → logical decoding · wal_level=logical · replication slot
💡
WAL은 물리적 로그다. CDC가 읽는 건 pgoutput 같은 output plugin이 논리적 변경 이벤트로 디코딩한 결과.
##

② Debezium 아키텍처

Kafka Connect
Source DB → Connector → Kafka
Source DB  ──binlog/WAL──▶  Debezium Connector  ──이벤트──▶  Kafka Topic
                          (Kafka Connect 위에서 동작)

##

실행 모드 3가지

①Kafka Connect가장 일반적 · Connect 위에서 동작
②Debezium ServerKafka 없이 standalone · Pulsar·Kinesis·Pub/Sub·Redis
③Embedded Engine애플리케이션 내 라이브러리로 임베딩
##

③ CDC vs Dual Write

CDC
트랜잭션 로그 기반
순서·완전성 보장 · 비동기 · 부하↓
Dual Write
앱이 두 곳에 직접 쓰기
2번째 쓰기 실패 시 불일치
Dual Write는 원자성 보장에 분산 트랜잭션이 필요하다. CDC는 이미 커밋된 로그를 읽으니 그 문제가 없다.
!!

CDC는 강한 일관성이 아니다

⚠️
CDC는 eventual consistency다. 로그 순서대로 전파하니 순서는 보장하지만, 전파 시점에 지연(lag)이 존재한다 → 강한 일관성(strong consistency)은 아니다.
"실시간"은 맞지만 즉시(synchronous)는 아니다. 면접에서 자주 파고드는 지점.
##

④ 실무 활용 사례

①Outbox + CDCMSA 간 데이터 동기화 · 가장 권장
②CQRSCommand DB → Read DB 실시간 동기화
③실시간 ETL데이터 웨어하우스/레이크로 변경 스트리밍
④캐시 무효화DB 변경 → Redis 캐시 자동 갱신
⑤감사 로그Audit Trail 자동 생성
!!

주의 1 — at-least-once

💥
Kafka Connect는 at-least-once — Connector 장애 시 중복 이벤트가 발생한다. Consumer 측 멱등성 처리 필수. Tombstone(DELETE 후 key만·value=null) 이벤트는 log compaction과 연계해 처리.
→ 같은 이벤트가 두 번 와도 결과가 같도록 소비자를 설계한다.
!!

주의 2 — PG 디스크 풀 장애

⚠️
PostgreSQL은 replication slot이 소비될 때까지 WAL을 보관한다. CDC가 장시간 중단되면 WAL이 무한 축적 → 디스크 풀 장애. (MySQL은 binlog 만료 전 소비 필요.)
→ PG 13+의 max_slot_wal_keep_size 설정 필수.
##

주의 3 — 스키마와 순서

스키마 변경(DDL)
파이프라인에 영향
Consumer 호환성 관리 필요
순서 보장
같은 PK는 같은 파티션으로
Debezium은 PK 기반 파티셔닝
