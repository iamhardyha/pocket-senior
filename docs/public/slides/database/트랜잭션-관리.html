# 트랜잭션
관리

const question = "여러 DB에 걸친 트랜잭션, 어떻게 처리하나?";
// 단일 DB는 @Transactional. 그 너머는?
>

한 줄 요약

단일 DB는 @Transactional로 충분하다.
여러 DB에 걸치면 분산 트랜잭션이 필요한데, 대부분은 피하는 게 답이다.
##

ACID, 깨지면 무슨 일?

AAtomicity 원자성전부 성공 or 전부 실패 · 깨지면 송금 중 한쪽만 차감
CConsistency 일관성제약 조건 항상 만족 · 깨지면 잔고가 음수
IIsolation 격리성동시 트랜잭션 간섭 없음 · 깨지면 Dirty Read · Lost Update
DDurability 지속성커밋 후 영구 보존 · 깨지면 커밋했는데 사라짐
##

오늘 다룰 5가지

①@Transactional프록시가 begin·commit·rollback을 대신 감싼다
②전파 Propagation기존 TX에 참여? 새로? · REQUIRED / REQUIRES_NEW
③격리 수준 Isolation동시성 이상 vs 성능의 트레이드오프
④흔한 함정같은 클래스 내부 호출은 프록시를 안 거친다
⑤분산 트랜잭션2PC는 피하고 · Saga · Outbox
##

① @Transactional 동작 원리

프록시 AOP
프록시가 메서드를 감싼다
1. 프록시가 요청을 가로챔
2. TransactionManager.begin()
3. 실제 메서드 실행
4a. 정상 → commit()
4b. 예외 → rollback()
개발자는 비즈니스 로직만 쓰고, 트랜잭션 경계는 프록시가 책임진다.
##

② 전파 — 5가지 한눈에

①REQUIRED (기본)있으면 참여, 없으면 생성 · 대부분의 경우
②REQUIRES_NEW항상 새 TX (기존 보류) · 독립적 로그·감사
③NESTED기존 TX 내 Savepoint · 부분 롤백
④NOT_SUPPORTEDTX 없이 실행 (기존 보류) · 조회 전용
⑤MANDATORY기존 TX 없으면 예외 · TX 필수 보장
##

REQUIRES_NEW 실전

롤백과 무관한 로그
AuditService.java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void log(Order order) {
    // 별도 TX → 외부 TX가 롤백돼도 이 로그는 유지
    auditLogRepository.save(new AuditLog(order));
}
주문은 롤백돼도 감사 로그는 남아야 할 때. 새 트랜잭션으로 떼어낸다.
##

③ 격리 수준 — 막는 이상

L0READ_UNCOMMITTEDDirty·NonRepeatable·Phantom 전부 허용 · 성능 최고
L1READ_COMMITTEDDirty만 차단 · PostgreSQL 기본
L2REPEATABLE_READDirty·NonRepeatable 차단 · MySQL 기본
L3SERIALIZABLE전부 차단 · 성능 최저
!!

MySQL의 Phantom Read

💡
SQL 표준상 REPEATABLE_READ는 Phantom을 허용하지만, MySQL InnoDB는 Next-Key Lock(Gap + Record)으로 사실상 방지한다. 그래서 MySQL에서는 △ — 표준이 아니라 구현이 막아준다.
실무: 대부분 DB 기본값 그대로. 격리 수준을 바꿀 일은 극히 드물다.
!!

④ 함정 — 같은 클래스 내부 호출

OrderService.java
public void outer() {
    inner();   // @Transactional이 안 먹힘!
}

@Transactional
public void inner() {
    // 프록시를 안 거쳐서 TX 없이 실행됨
}
원인: Spring AOP는 프록시 기반. 같은 객체 내 호출은 프록시를 거치지 않는다.
##

해결 — 프록시를 거치게

✅
별도 클래스로 분리한다 (권장) @Transactional 메서드를 다른 빈으로 옮기면 호출 시 프록시를 거친다.
self-injection도 가능하지만 순환참조·가독성 문제로 비추. 분리가 깔끔하다.
##

readOnly = true 한 줄의 값

1Dirty Checking 생략Hibernate가 변경 감지를 안 함 → 성능 향상
2Replica 라우팅읽기를 Read 전용 노드로 보낼 수 있음
3의도 명확화"이 메서드는 수정하지 않는다"는 선언
##

⑤ 여러 DB, @Transactional로 묶을까?

서로 다른 데이터 소스
서비스 A (DB-A): 주문 저장
서비스 B (DB-B): 재고 차감

@Transactional으로 묶을 수 있나?  → 불가능.
// 두 DB는 서로 다른 TransactionManager를 쓴다
하나의 로컬 트랜잭션으로는 두 DB의 원자성을 보장할 수 없다.
!!

2PC — 이론적 정답, 실무적 비추

⚠️
Prepare → Commit 2단계로 강한 일관성을 주지만, Lock 보유 2배 · Coordinator 단일 장애점 · 파티션 시 해결 불가 대부분의 클라우드 DB가 XA 트랜잭션을 미지원하거나 비권장한다.
→ 그래서 실무는 강한 일관성을 포기하고 Saga로 간다.
##

Saga — 실무의 답

로컬 TX + 보상
각 서비스가 로컬 트랜잭션으로 처리
// 성공
주문 생성(DB-A) → 재고 차감(DB-B) → 결제(PG) → 완료

// 재고에서 실패 → 보상 트랜잭션
주문 생성(DB-A) → 재고 차감 실패 → 주문 취소(보상)
실패하면 되돌리는 보상 트랜잭션으로 정합성을 맞춘다. 이벤트 기반(Choreography) · 중앙 제어(Orchestration).
##

Outbox — 메시지 유실 방지

권장
OrderService.java
@Transactional  // 같은 DB, 같은 트랜잭션
public void createOrder(OrderRequest req) {
    orderRepository.save(...);        // 1. 주문 저장
    outboxRepository.save(...);       // 2. 이벤트도 같은 TX
}  // 커밋 → 주문 + 이벤트가 원자적으로 저장
별도 프로세스(CDC·Polling)가 outbox를 읽어 Kafka에 발행. 저장과 발행의 원자성을 한 TX로 보장한다.
##

분산 방법, 어떻게 고를까

강한 일관성이 절대 필요?
2PC (XA) — 단, 성능·운영 비용 큼 · 비추
소규모 · 단순 흐름?
Saga Choreography — 이벤트 기반
대규모 · 복잡한 흐름?
Saga Orchestration — 중앙 제어
이벤트 유실을 막아야?
Outbox + CDC — 권장 기본값
