# 이벤트 리스너
vs 비동기

const question = "이벤트 리스너를 쓰는 이유 · 비동기를 쓰는 이유 · 둘의 차이";
// 그냥 비동기로 처리해도 되지 않을까?
>

한 줄 요약

이벤트는 설계,
비동기는 실행의 문제다.
이벤트 = 누가 처리할지 모르게 알리기 · 비동기 = 빨리 끝내려고 따로 실행하기.
##

질문이 다르다

비동기 (@Async)
"이 작업을 별도 스레드에서 해줘"
→ 실행 방식의 문제
이벤트 리스너
"이런 일이 일어났어"
→ 설계 구조의 문제
두 축은 직교한다 — 섞어서도, 따로도 쓴다.
##

비동기 — 별도 스레드에서

NotificationService.java
@Async
public void sendOrderConfirmation(Order order) {
    // 별도 스레드 → 주문 응답은 이미 반환됨
    emailClient.send(order.getUserEmail(), "주문 확인");
}
호출자가 대상을 직접 호출한다 — 응답 시간에서만 제외할 뿐, 결합은 그대로다.
!!

비동기만으로는 — OCP 위반

요구사항 추가될 때마다 주문 서비스가 비대해진다
public void createOrder(OrderRequest req) {
    orderRepository.save(...);
    notificationService.sendOrderConfirmation(order); // +알림
    pointService.earnPoints(order);                  // +포인트
    analyticsService.trackOrder(order);              // +분석
    // 주문이 모든 것을 알아야 한다 → OCP 위반
}
처리 대상이 늘 때마다 주문 서비스 코드를 수정해야 한다.
##

이벤트 — 사실만 발행

OrderService.java
@Transactional
public void createOrder(OrderRequest req) {
    Order order = orderRepository.save(...);
    // "주문이 생성됐다"는 사실만 알림
    eventPublisher.publishEvent(new OrderCreatedEvent(order));
    // 누가 듣는지 주문 서비스는 모른다
}
ApplicationEventPublisher로 발행 — 발행자는 구독자를 모른다 (Pub-Sub).
##

리스너만 추가하면 끝

독립 리스너들 — 주문 서비스 코드 변경 없음
@EventListener
public void onNotify(OrderCreatedEvent e) { /* 알림 */ }

@EventListener
public void onPoint(OrderCreatedEvent e) { /* 포인트 */ }

@EventListener
public void onTrack(OrderCreatedEvent e) { /* 분석 */ }
✅
새 기능은 리스너만 추가 — 주문 서비스를 안 건드린다 개방-폐쇄 원칙(OCP)을 코드로 지킨다.
!!

기본 @EventListener의 함정

기본 @EventListener — 같은 스레드 · 같은 트랜잭션
@EventListener
public void handle(OrderCreatedEvent e) {
    // 주문 트랜잭션 안에서 동기로 실행된다
    // 여기서 예외 발생 → 주문도 함께 롤백!
}
동기 · 동일 트랜잭션 · 동일 스레드가 기본값. 리스너 실패가 원본까지 끌고 내려간다.
##

커밋 후로 미루기

@TransactionalEventListener — phase로 실행 시점 지정
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(OrderCreatedEvent e) {
    // 주문 트랜잭션 커밋 후 실행
    // 여기서 예외 → 주문은 이미 커밋됨 (안전)
    notificationClient.send(e.getOrder());
}
phase: AFTER_COMMIT(성공 후) · AFTER_ROLLBACK(실패 후). 원본 트랜잭션과 분리된다.
##

실무 황금 조합

안전 + 빠름
@Async + @TransactionalEventListener(AFTER_COMMIT)
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(OrderCreatedEvent e) {
    // 1. 커밋 후  2. 별도 스레드  3. 응답과 무관
}
⭐
커밋 후 · 별도 스레드 — 안전하고 성능도 좋은 기본기 실무에서 가장 많이 쓰는 조합.
!!

커밋 후 죽으면? — 유실

⚠️
DB는 커밋됐는데 핸들러/프로세스가 죽으면 비동기 side-effect가 그대로 사라진다 인프로세스 이벤트는 영속성·재시도가 없다 → 알림 안 감, 포인트 누락. 아무도 모른다.
→ 중요하거나 서비스 간 전파가 필요하면 Outbox 패턴(내구성 보장)으로.
##

Spring Event vs Kafka

Spring ApplicationEvent
단일 JVM · 비영속(메모리)
장애 시 유실 · 재시도 직접 구현
Kafka Event
서비스 간 분산 · 영속(디스크)
메시지 보존 · offset으로 자동 재시도
모놀리스 내 도메인 분리는 Spring Event, MSA 간 통신은 Kafka. 잘 써두면 MSA 전환이 쉽다.
##

판단 기준

처리 대상이 늘어날 수 있나?
YES → 이벤트 · NO → @Async로 충분
원래 트랜잭션과 독립이어야?
YES → @TransactionalEventListener
응답 시간에서 제외해야?
YES → @Async 추가
유실되면 안 되는 일?
YES → Outbox · 서비스 간이면 Kafka
