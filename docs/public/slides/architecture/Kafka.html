한 줄 요약

Kafka는 분산 이벤트 스트리밍 플랫폼.
메시지를 '보내고 잊는' 게 아니라 '기록하고 읽는' 구조다.
##

핵심 구성 요소

PProducer / Topic메시지를 Topic(논리 카테고리)에 발행
▦PartitionTopic의 물리 분할 · 순서 보장의 단위
CConsumer Group파티션을 그룹 내 Consumer에 분배
↳Offset각 Consumer가 어디까지 읽었는지 기록
##

파티션과 순서 보장

Topic: order-events
Partition 0: [msg1, msg4, msg7, …]  ← Consumer A
Partition 1: [msg2, msg5, msg8, …]  ← Consumer B
Partition 2: [msg3, msg6, msg9, …]  ← Consumer C
파티션 내부에서만 순서 보장 · 파티션 간에는 순서 보장 안 됨.
##

메시지 키 → 같은 파티션

KafkaProducer.java
kafkaTemplate.send("order-events", userId.toString(), event);
//                              ^^^^^^ key
같은 key → 항상 같은 파티션(murmur2 해시로 결정). 키 값과 파티션 번호가 일치하는 건 아니다. 같은 사용자 이벤트는 순서 보장.
파티션 수 = 최대 병렬 Consumer 수
3 파티션
Consumer 3개면 1:1 최적 · 5개면 2개는 논다 · 1개면 혼자 다 처리(느림).
##

Consumer Group · 리밸런스

group: order-processor
Consumer A ← Partition 0
Consumer B ← Partition 1
Consumer C ← Partition 2

// Consumer가 추가/이탈하면 파티션 재배분(리밸런스)
그룹에 Consumer가 들고 나면 파티션을 자동 재할당한다. 스케일 아웃의 단위는 곧 파티션 수.
##

offset 수동 커밋

유실 방지
OrderConsumer.java
@KafkaListener(topics = "order-events")
public void consume(OrderEvent e, Acknowledgment ack) {
    orderService.process(e);   // 처리
    ack.acknowledge();        // 완료 후에만 커밋
}
enable-auto-commit: false · 실패하면 커밋 안 함 → 다음 poll에서 재처리.
!!

재처리의 전제 — 멱등성

⚠️
수동 커밋은 "최소 한 번(at-least-once)" — 같은 메시지가 다시 올 수 있다 실패한 메시지는 재전달된다. Consumer가 멱등하지 않으면 중복 처리·중복 결제로 이어진다.
→ 해결: eventId 같은 멱등성 키로 중복을 걸러낸다.
##

acks — 유실 vs 성능

0응답 안 기다림최고 성능 · 유실 가능
1Leader만 확인균형 · Leader 장애 시 유실
all모든 ISR 확인느림 · 유실 없음 (권장)
##

acks=all 의 짝꿍

application.yml
# Producer: 모든 ISR 확인 후 응답 → 유실 방지
acks: all
# Broker/Topic: 최소 몇 개의 동기 복제본이 살아있어야 쓰기 허용
min.insync.replicas: 2
⭐
acks=all 단독으로는 부족 — min.insync.replicas≥2가 함께 있어야 복제본 1개로 줄었을 때 쓰기를 막아 유실을 진짜로 차단한다
##

재시도 → Dead Letter Topic

ErrorHandler 설정
new DefaultErrorHandler(
    recoverer,                       // 실패 시 DLT로 발행
    new FixedBackOff(1000L, 3)  // 1초 간격, 재시도 3회
);
최초 1회 + 재시도 3회 = 총 4회 시도 후 실패 → order-events.DLT로 격리.
쓴 offset 1,000,000 − 읽은 offset 950,000 =
50,000 Lag
Lag이 계속 증가 → 처리 속도 < 유입 속도. Consumer 추가 또는 로직 최적화. (Burrow · Lag Exporter)
##

전달 보장, 어디까지?

기본 (수동 커밋)
at-least-once — 멱등성으로 중복 흡수
트랜잭션 + read_committed
exactly-once — Kafka 내부 한정·비용 큼
현실적인 선택
at-least-once + 멱등 Consumer
