# 벌크헤드

>

한 줄 요약

자원을 격리한다.
한 곳이 터져도 전체가 죽지 않게 — 선박의 격벽에서 온 패턴.
##

왜 "벌크헤드"인가

선박의 격벽(Bulkhead)
// 격벽 없는 배
[ 하나의 공간 ] → 한 곳 침수 → 배 전체 침몰

// 격벽 있는 배
[구역1][구역2][💧구역3][구역4] → 한 구역만 침수, 나머지 무사
소프트웨어도 같다 — 스레드·커넥션·메모리를 서비스별로 격리한다.
!!

격리 없는 시스템의 비극

공유 스레드 풀 200개 — 배송 API가 먹통이 되면
1. 배송 호출 스레드들이 타임아웃(30s)까지 대기
2. 스레드 200개 전부 배송 대기에 묶임
3. 주문·결제·알림도 스레드를 못 받음
4. 전체 다운 ← 배송 하나 때문에 전부 죽음
이것이 연쇄 장애(Cascading Failure). 벌크헤드가 막는다.
##

격리 방식 3가지

①스레드 풀 격리서비스별 전용 풀 · 완전 격리, 단 메모리·컨텍스트 스위칭 비용
②세마포어 격리동시 호출 수만 제한 · 가볍지만 호출자 스레드 점유
③커넥션 풀 격리DB·외부별 풀 분리 · HikariCP pool-name 분리
##

스레드 풀 vs 세마포어

세마포어
카운터만 — 가벼움 · 호출자 스레드 공유 · 내부 호출용
스레드 풀
전용 스레드 — 완전 격리 · 독립 타임아웃 · 느린 외부 API용
반환 타입도 다르다 — 세마포어=동기, 스레드 풀=CompletionStage(비동기).
##

Resilience4j · 세마포어

가벼운 격리
application.yml
resilience4j:
  bulkhead:
    instances:
      deliveryService:
        maxConcurrentCalls: 20     # 동시 최대 20개
        maxWaitDuration: 100ms    # 초과 시 Fast Fail
슬롯이 차면 대기 100ms 후 거부. 빠르게 실패시켜 호출자를 풀어준다.
##

@Bulkhead + 폴백

OrderService.java
@Bulkhead(name = "deliveryService", fallbackMethod = "fallback")
public DeliveryResult requestDelivery(Long orderId) {
    return deliveryClient.request(orderId);
}
// 벌크헤드 포화 시 호출되는 폴백
private DeliveryResult fallback(Long id, BulkheadFullException e) {
    return DeliveryResult.queued(id); // 큐로 전환 후 재시도
}
포화 시 던지는 예외가 BulkheadFullException — 폴백 시그니처에 명시.
##

Resilience4j · 스레드 풀

완전 격리
application.yml — thread-pool-bulkhead
resilience4j:
  thread-pool-bulkhead:
    instances:
      deliveryService:
        maxThreadPoolSize: 20     # 전용 스레드 20개
        coreThreadPoolSize: 10
        queueCapacity: 50         # 대기 큐 50개
type = Bulkhead.Type.THREADPOOL + 반환은 CompletionStage여야 한다.
##

벌크헤드는 혼자 쓰지 않는다

🧩
벌크헤드만 쓰면 느린 호출이 제한된 슬롯을 계속 점유한다. 서킷브레이커(실패율 감지·차단) + 타임리미터(느린 응답 끊기)와 함께 써야 완성된다.
동시성은 Bulkhead, 실패율은 CircuitBreaker, 지연은 TimeLimiter가 맡는다.
##

Aspect 실행 순서

CB가 바깥
Resilience4j 기본 Aspect 우선순위
CircuitBreaker → TimeLimiter → Bulkhead → 실제 호출

// 서킷이 OPEN이면 Bulkhead 체크도 없이 즉시 거부
@CircuitBreaker(name = "deliveryService")  // 가장 바깥
@TimeLimiter(name = "deliveryService")
@Bulkhead(name = "deliveryService")       // 가장 안쪽
CB가 바깥, Bulkhead가 안쪽. 차단이 우선이라 불필요한 슬롯 소모를 막는다.
##

어느 단계에서 무슨 예외가?

⛔서킷 OPENCircuitBreaker 즉시 차단 · CallNotPermittedException
⏱3초 초과TimeLimiter가 취소 · TimeoutException
🚧동시 21번째Bulkhead가 거부 · BulkheadFullException
⚠️
@TimeLimiter는 CompletionStage 반환에서만 동작동기 메서드에 붙이면 타임아웃이 작동하지 않는다.
RPS 100 × 응답 0.2s = (Little's Law)
20 × 1.5 = 30
필요 동시성 = RPS × 평균 응답시간. 여유 1.5배 버퍼 → maxConcurrentCalls: 30.
!!

너무 작아도, 너무 커도 문제

너무 작으면
정상 트래픽도 거부 — 불필요한 폴백
너무 크면
격리 효과 소멸 — 장애 시 자원 고갈
기준: 피크 트래픽(평소 2~3배) · 다운스트림 용량 이내로 잡는다.
