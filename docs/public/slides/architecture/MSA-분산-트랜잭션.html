# MSA
분산 트랜잭션

const question = "여러 서비스에 걸친 트랜잭션을 어떻게 관리하는가?";
// @Transactional은 죽었다. 그럼 무엇으로 대체하나?
>

한 줄 요약

MSA에서 @Transactional은 죽었다.
Saga + Outbox로 결과적 일관성을, 보상 트랜잭션으로 실패를 되돌려라.
##

왜 기존 트랜잭션이 깨지나

DB가 나뉘면 ACID가 무너진다.
@Transactional은 단일 DB 커넥션 위에서만 동작한다. DB가 3개면 트랜잭션도 3개 — 하나로 못 묶는다.
!!

결제는 됐는데 재고 차감 실패

💸
카드에서 돈은 빠졌는데 재고는 그대로 → 돈만 빠진 유령 주문 MSA에서 "전부 성공 or 전부 실패"는 불가능하다. 가능한 건 "전부 성공 or 실패한 것을 되돌리기".
→ 이것이 결과적 일관성(Eventual Consistency)이다.
##

2PC / XA — 왜 안 쓰나

✗성능·가용성Prepare에서 전 DB에 Lock 잡고 대기 · 가장 느린 서비스에 맞춰짐
✗Coordinator SPOF조율자가 죽으면 Lock 잡힌 채 전체 멈춤
✗in-doubt 상태Prepare OK 후 네트워크 끊기면 커밋도 롤백도 못 하는 불확정
✗자율성 상실독립 배포 불가 · NoSQL·외부 API는 XA 미지원
MSA를 하겠다면서 2PC를 쓰는 건 모순이다.
##

대안 — Saga 패턴

긴 트랜잭션을 로컬 트랜잭션의 연쇄로 쪼갠다.
각 단계는 자기 DB에 커밋하고 다음 단계로 이벤트를 넘긴다. 실패하면 보상 트랜잭션으로 앞 단계를 되돌린다.
##

Saga ① Choreography

중앙 조율자 없음
각 서비스가 이벤트를 발행·구독하며 자율 반응.
바통을 알아서 넘긴다. 느슨한 결합·SPOF 없음 ↔ 플로우 추적 어려움, 이벤트 스키마 암묵 결합, 순환 의존 위험.
##

Saga ② Orchestration

중앙 지휘자
중앙 Orchestrator가 순서대로 지시.
상태 머신으로 추적·디버깅 쉬움 ↔ Orchestrator가 SPOF·God Object 위험. 실무에선 이쪽이 더 많이 쓰인다.
##

Choreography vs Orchestration

서비스 3~4개 · 단순 직선형?
Choreography — 추가 쉬움
5개 이상 · 분기·복잡한 보상?
Orchestration — 추적 용이
단일 장애점
Choreo 없음 / Orch = Orchestrator
디버깅
Orch이 압도적으로 쉬움
##

보상 트랜잭션 ≠ 롤백

이미 커밋된 것을 되돌리는 새로운 트랜잭션.
주문 생성→주문 취소, 결제 승인→환불 API, 재고 차감→수량 +1. 단 "물리적 원복"이 아닌 "논리적 취소"일 수 있다(환불은 3~5영업일).
##

Outbox — 왜 필요한가

OrderService.java — 위험한 코드
@Transactional
public void createOrder(...) {
    orderRepository.save(order);             // 1. DB 커밋
    kafkaTemplate.send("order.created", e); // 2. Kafka 발행
}
DB 트랜잭션과 메시지 발행은 서로 다른 시스템 — 하나로 못 묶는다. 한쪽만 성공하면 유실 또는 유령 결제.
##

Outbox 테이블

원자성 확보
Kafka에 직접 보내지 말고
같은 DB의 Outbox 테이블에 저장.
비즈니스 데이터 + 이벤트를 한 트랜잭션으로 커밋 → 원자성 보장. 별도 프로세스가 Outbox를 읽어 Kafka로 중계한다.
!!

Outbox는 at-least-once

⚠️
send 성공 후 published 마킹 커밋이 실패하면 다음 폴링에서 재발행된다. 정확히 한 번(exactly-once)이 아니다 — Outbox는 본질적으로 at-least-once.
✅
그래서 소비자 멱등성 처리가 필수 — 같은 이벤트를 두 번 받아도 결과가 같아야 한다.
##

Outbox → Kafka 중계

방법 1 — Polling Publisher
주기적 SELECT로 발행 · 구현 간단 ↔ 지연·DB 부하·스케일 한계
방법 2 — CDC (Debezium)
binlog 감지로 거의 실시간 발행 · 프로덕션 권장
Debezium Outbox Event Router: 변경 감지 → topic 필드로 라우팅 → payload를 본문, aggregate_id를 키로 순서 보장.
##

멱등성 — 마지막 안전벨트

PaymentService.java — 환불 멱등 처리
@Transactional
public void refund(String sagaId, Long paymentId) {
    if (refundRepository.existsBySagaId(sagaId)) {
        return;   // 이미 처리됨 → 중복 무시
    }
    // ... 환불 처리 + PG API 호출
}
네트워크 장애로 중복 메시지는 반드시 온다. eventId/sagaId로 중복을 걸러라.
