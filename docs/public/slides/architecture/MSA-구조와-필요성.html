한 줄 요약

MSA는 조직의 문제를 푸는 도구다.
독립 배포·확장 요구가 Monolith의 결합도를 못 견딜 때 — 그리고 분산 복잡성이라는 대가를 반드시 치른다.
##

왜 Monolith가 아픈가

!배포 지옥1줄 수정에도 전체 빌드·회귀 테스트 · 팀 간 배포 충돌
!장애 전파알림 모듈 메모리 누수 → 주문·결제까지 전체 다운
!스케일링 비효율조회만 폭증해도 전체 인스턴스를 동일하게 복제
한 모듈의 문제가 중요도와 무관하게 전체로 퍼진다.
##

Conway's Law

조직 = 아키텍처
“시스템은 그것을 만든 조직의 커뮤니케이션 구조를 닮는다.”
MSA는 기술적 선택이 아니라 조직적 선택이다. 팀 5명이 도입하면 오버헤드만 늘고, 팀 30명이 서로의 코드를 밟을 때 비로소 고민할 문제다.
##

나눠야 할 때 vs 아닐 때

MSA 적합
팀 20명+ · 명확한 도메인 경계 · 모듈별 트래픽 격차 큼 · 잦은 독립 배포
Monolith 유지
팀 10명 이하 · 도메인 탐색 중 · 균일한 트래픽 · 전체 배포로 충분
⛔
스타트업 초기·작은 팀·불명확한 경계에서는 하지 마라잘못 그은 서비스 경계는 Monolith보다 고치기 훨씬 어렵다.
##

먼저 Modular Monolith

내부 모듈 경계부터 깔끔히,
필요할 때 서비스로 분리.
모듈 간 직접 의존 금지 — Facade·이벤트로만 통신하고, ArchUnit으로 경계를 테스트에서 강제한다. 실무에서 가장 현실적인 전략.
##

설계 원칙 4가지

①단일 비즈니스 능력기능(CRUD)이 아니라 도메인(주문/결제/배송) 단위로
②Database per ServiceDB를 나눠 결합도·SPOF 제거
③Bounded Context같은 용어가 다른 의미를 갖는 경계로 서비스 분리
④API First구현 전 계약(OpenAPI)부터 → 팀 병렬 개발
##

Bounded Context로 경계 잡기

상품 Context
Product = 이름·설명·카테고리·재고
주문 Context
Product = 주문항목·수량·가격
결제 Context
Product = 결제금액·환불정책
배송 Context
Product = 무게·크기·배송지
같은 "Product"가 Context마다 다른 의미·속성을 가진다 → 각 Context가 하나의 서비스.
##

Database per Service

결합도 제거
✅
스키마 변경이 해당 서비스에만 영향 · 서비스별 최적 DB 엔진 · 독립 스케일링공유 DB는 결합도 폭발 + SPOF + DB 병목으로 독립성이 무너진다.
⚖️
대가: 서비스 간 JOIN 불가 · 분산 트랜잭션→ API Composition / CQRS로 조회, Saga로 트랜잭션 해결.
##

동기 vs 비동기

동기 (REST · gRPC)
즉시 응답이 필요할 때 — 주문 → 결제. 호출자가 응답을 기다림.
비동기 (Kafka)
나중에 처리해도 될 때 — 주문 완료 → 알림·재고·포인트(Fan-out).
실무는 둘을 혼합한다. 동기 호출엔 반드시 Timeout + Circuit Breaker로 장애 전파를 막는다.
##

REST vs gRPC

REST (JSON/HTTP)
외부·클라이언트 대면 — curl로 바로 테스트, 브라우저 완벽 지원, 느슨한 계약.
gRPC (Protobuf/HTTP2)
서비스 간 내부 통신 — 바이너리 직렬화로 더 빠름, 양방향 스트리밍, 엄격한 타입 계약.
정답은 상황에 따라 — 외부 API는 REST, 고빈도 내부 통신·실시간엔 gRPC.
##

API Gateway

단일 진입점
GatewayConfig.java — Spring Cloud Gateway
builder.routes()
  .route("order-service", r -> r
    .path("/api/v1/orders/**")
    .filters(f -> f
      .circuitBreaker(c -> c.setName("orderCb"))
      .retry(rc -> rc.setRetries(3)))
    .uri("lb://order-service"))
라우팅·인증/인가·Rate Limiting·로드밸런싱·Correlation ID를 한 곳에서 처리.
##

Service Discovery

Client-Side (Eureka)
인스턴스 목록을 받아 클라이언트가 직접 로드밸런싱.
Server-Side (K8s Service)
"payment-service:8080"만 알면 K8s DNS가 해결.
💡
Kubernetes를 쓰면 Eureka가 필요 없다K8s Service + CoreDNS가 대체 — Spring Cloud Netflix 스택은 레거시화되는 추세.
##

Circuit Breaker 상태 전이

●CLOSED (정상)모든 요청 통과 · 실패율 > 임계값이면 → OPEN
●OPEN (차단)즉시 fallback 반환 · 대기 시간 경과 후 → HALF_OPEN
●HALF_OPEN (시험)일부만 통과 · 성공 → CLOSED, 실패 → OPEN
없으면 결제 서비스 다운 시 스레드 풀 고갈 → 연쇄 장애.
!!

나눈 뒤의 대가

⚠분산 트랜잭션ACID 불가 → Saga Pattern · 복잡도 매우 높음
⚠데이터 조인 불가서비스 간 JOIN 안 됨 → CQRS · API Composition
⚠데이터 일관성Eventually Consistent → 보상 트랜잭션
⚠추적·디버깅어디서 났는지 → Distributed Tracing
비용은 나누는 것이 아니라 나눈 뒤의 분산 복잡성을 관리하는 것.
##

Strangler Fig Pattern

점진적 전환
1Proxy 도입Gateway를 앞단에 — 100% Monolith로 시작
2독립적인 것부터알림 등 의존성·임팩트 낮은 모듈 먼저 분리
3점진적 분리Monolith가 점점 작아지도록 한 조각씩
🔥
빅뱅 전환은 롤백 불가 도박 — 이중 비용·숨은 버그·개발 중단한 조각씩 감싸 대체하는 Strangler Fig가 가장 안전하다.
