##

동기 호출의 3대 한계

①응답 지연 누적결제·재고·알림 직렬 호출 → 1.6초+ 대기
②장애 전파알림만 죽었는데 주문 전체가 실패 (Cascading)
③강결합하위 서비스 추가마다 주문 코드 수정
##

비동기로 얻는 것

OrderService.java
orderService.create(req);
eventPublisher.publish("order-events", event); // 발행 후 즉시 응답
// 결제·재고·알림·포인트는 각자 구독 → 주문 코드 변경 0
디커플링 · 장애 격리 · 버퍼링 · 즉시 응답 — 한 번에 얻는다.
피크 10만/s 유입 · Consumer 3만/s 처리
버퍼링
큐가 없으면 피크에 서버 다운. 큐가 있으면 쌓아두고 자기 속도로 처리.
##

3대 구성요소

Producer → Broker → Consumer
Producer  // 메시지를 만들어 보낸다 (주문 서비스)
Broker    // 저장하고 전달하는 중간자 (Kafka·RabbitMQ·SQS)
Consumer  // 받아서 처리한다 (결제·알림 서비스)
발행자와 소비자는 서로를 모른다. Broker만 알면 된다.
##

Queue vs Topic

Queue · P2P
1 메시지 = 1 Consumer
작업 분배 — 워커, 리사이즈
Topic · Pub/Sub
1 메시지 = N Consumer
이벤트 전파 — 여러 서비스가 각자 반응
분배가 필요하면 Queue, 전파가 필요하면 Topic.
##

Delivery Semantics

AAt-most-once최대 1번 · 유실 가능 · 로그·메트릭
BAt-least-once최소 1번 · 중복 가능 · 대부분의 실무(Kafka 기본)
CExactly-once정확히 1번 · Kafka 0.11+ EOS · 단 Kafka 경계 안
!!

현실적 선택

At-least-once + Consumer 멱등성
= 사실상 Exactly-once
✅
EOS는 Kafka 경계 안에서만 보장 — DB write·외부 API 같은 side-effect는 자동 확장 안 됨→ end-to-end에선 "중복은 허용하되 멱등하게 처리"가 업계 표준.
##

Kafka · RabbitMQ · SQS

KKafka분산 로그·Pull · 100만+ msg/s · offset 재처리 · 보관(기본 7일)
RRabbitMQ메시지 브로커·Push · 최저 지연 · 복잡한 라우팅 · 소비 후 삭제
SAWS SQS관리형 큐·Pull · 운영부담 0 · 종량제 · 최대 14일 보관
##

무엇을 고를까?

재처리(replay)·이벤트 소싱?
Kafka — offset 되감기
복잡한 라우팅·저지연?
RabbitMQ — Exchange·Push
운영 인력 없음·서버리스?
SQS (+ SNS) — 관리형
대규모 MSA 이벤트 버스?
Kafka — Consumer Group
##

처리 후에만 커밋

PaymentEventConsumer.java
@KafkaListener(topics = "order-events")
public void handle(OrderEvent e, Acknowledgment ack) {
    paymentService.process(e);
    ack.acknowledge();  // 성공 시에만 offset 커밋
}
enable.auto.commit=false — 처리 중 죽으면 재전달되어 유실을 막는다.
##

Backpressure 대응

①Consumer 스케일 아웃처리량을 유입에 맞춤 (파티션 수 이내)
②Producer Rate Limit임계 초과 시 429 Too Many Requests
③Bounded QueueRabbitMQ max-length · Kafka retention.bytes
④배압 전파Kafka max.poll.records로 한 번에 가져오는 양 조절
!!

재시도는 횟수가 아니다

KafkaConsumerConfig.java
ExponentialBackOff backOff = new ExponentialBackOff(1000L, 2.0);
backOff.setMaxElapsedTime(15000L); // 누적 15초까지 (1+2+4+8)
⚠️
기본 ExponentialBackOff는 횟수가 아니라 maxElapsedTime(누적 경과시간)으로 끊긴다→ 정확히 N회로 끊으려면 ExponentialBackOffWithMaxRetries(n) 또는 FixedBackOff(간격, 횟수).
##

Dead Letter Queue

실패 → 재시도(1·2·3) → 초과 → order-events.DLT
// DLQ 없으면: 한 메시지가 무한 재시도 → 뒤 메시지도 막힘
//             (Head-of-line blocking)
// DLQ 있으면: 3회 초과분만 격리 → 원래 큐는 정상 진행
운영 포인트: 쌓이면 알림 · 실패 메타데이터 보관 · 재처리 도구 · TTL(7~14일).
##

순서 보장 & DB 정합성

메시지 순서?
같은 Partition Key → 같은 파티션
orderId로 보내면 그 주문은 순서대로
DB 저장 + 발행 원자성?
Outbox Pattern
한 트랜잭션에 outbox 저장 → CDC로 발행
DB 트랜잭션과 Kafka 발행은 원자적이지 않다 → Outbox로 일관성 확보.
