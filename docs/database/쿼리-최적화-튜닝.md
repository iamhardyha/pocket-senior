# 쿼리 최적화, 쿼리 튜닝

> **핵심 질문**: 쿼리를 어떻게 분석하고 최적화하는가?

---

## 한 줄 요약

**"실행 계획(EXPLAIN)을 읽을 줄 알면 절반은 해결된다. 인덱스를 이해하면 나머지도 해결된다."**

---

## 쿼리 튜닝 프로세스

```
1. 느린 쿼리 발견 (Slow Query Log)
2. EXPLAIN으로 실행 계획 분석
3. 원인 파악 (풀스캔? 인덱스 미스? 정렬?)
4. 인덱스 추가 또는 쿼리 수정
5. 개선 효과 확인
```

---

## EXPLAIN 읽는 법

```sql
EXPLAIN SELECT * FROM orders
WHERE user_id = 100 AND status = 'PAID'
ORDER BY created_at DESC
LIMIT 10;
```

| 컬럼 | 의미 | 좋은 값 | 나쁜 값 |
|------|------|---------|---------|
| **type** | 접근 방식 | const, eq_ref, ref | ALL (풀스캔) |
| **key** | 사용된 인덱스 | 인덱스 이름 | NULL (인덱스 미사용) |
| **rows** | 예상 검사 행 수 | 적을수록 좋음 | 전체 행 수 |
| **Extra** | 추가 정보 | Using index | Using filesort, Using temporary |

### type 성능 순서 (좋은 순)

```
system > const > eq_ref > ref > range > index > ALL

const:    PK/Unique로 1행 조회 (최고)
eq_ref:   JOIN에서 PK/Unique 매칭
ref:      일반 인덱스로 여러 행
range:    인덱스 범위 스캔 (BETWEEN, >, <)
index:    인덱스 풀 스캔 (테이블보다는 나음)
ALL:      테이블 풀 스캔 (최악)
```

### Extra 주의 항목

```
Using filesort:    인덱스로 정렬 못해서 별도 정렬 → 느림
Using temporary:   임시 테이블 사용 (GROUP BY, DISTINCT) → 느림
Using index:       커버링 인덱스 → 빠름 (Good)
Using where:       인덱스로 필터링 후 추가 WHERE 적용
```

## 인덱스 핵심

### 인덱스가 걸리는/안 걸리는 경우

```sql
-- 인덱스 사용 O
WHERE user_id = 100                    -- 등호
WHERE created_at > '2026-01-01'        -- 범위
WHERE user_id = 100 AND status = 'PAID' -- 복합 인덱스 선두 컬럼 포함

-- 인덱스 사용 X
WHERE YEAR(created_at) = 2026          -- 함수 적용
WHERE name LIKE '%검색어'              -- 앞쪽 와일드카드
WHERE amount * 1.1 > 10000             -- 연산
WHERE user_id IS NOT NULL              -- 경우에 따라
WHERE status != 'CANCELLED'            -- 부정 조건 (대부분)
```

### 복합 인덱스와 순서

```sql
CREATE INDEX idx_orders ON orders (user_id, status, created_at);

-- 인덱스 사용 O (선두부터 사용)
WHERE user_id = 100
WHERE user_id = 100 AND status = 'PAID'
WHERE user_id = 100 AND status = 'PAID' AND created_at > '2026-01-01'

-- 인덱스 사용 X (선두 컬럼 누락)
WHERE status = 'PAID'                          -- user_id 없음
WHERE status = 'PAID' AND created_at > '...'   -- user_id 없음
WHERE created_at > '2026-01-01'                -- user_id, status 없음
```

**복합 인덱스 순서 결정 원칙**:
1. **동등 조건(=)** 컬럼을 앞에
2. **범위 조건(>, <, BETWEEN)** 컬럼을 뒤에
3. **정렬(ORDER BY)** 컬럼을 마지막에
4. **카디널리티** 높은 컬럼을 앞에 (변별력 높은 것)

```sql
-- 예: WHERE user_id = ? AND status = ? ORDER BY created_at DESC
-- 최적 인덱스: (user_id, status, created_at)
--              ^^^^^^^^  ^^^^^^  ^^^^^^^^^^
--              동등       동등     정렬
```

### 커버링 인덱스

```sql
-- 인덱스만으로 쿼리를 완성 (테이블 접근 불필요)
CREATE INDEX idx_covering ON orders (user_id, status, created_at, total_amount);

SELECT status, created_at, total_amount  -- 인덱스에 포함된 컬럼만
FROM orders
WHERE user_id = 100;
-- Extra: Using index (커버링 인덱스!)
-- 테이블을 안 봐도 됨 → 매우 빠름
```

## 흔한 성능 문제와 해결

### 1. OFFSET 페이징 (큰 페이지)

```sql
-- 느림: 100만번째부터 10개
SELECT * FROM orders ORDER BY id DESC LIMIT 10 OFFSET 1000000;
-- 100만 행을 읽고 버린 후 10개 반환

-- 빠름: 커서 기반 페이징 (No-Offset)
SELECT * FROM orders WHERE id < 12345 ORDER BY id DESC LIMIT 10;
-- 인덱스로 바로 12345 위치로 점프
```

```java
// Cursor-based Pagination
@Query("SELECT o FROM Order o WHERE o.id < :lastId ORDER BY o.id DESC")
List<Order> findNextPage(@Param("lastId") Long lastId, Pageable pageable);
```

### 2. COUNT(*) 전체 개수

```sql
-- 느림: 전체 건수
SELECT COUNT(*) FROM orders WHERE status = 'PAID';
-- 전체 테이블 스캔 가능

-- 대안 1: 대략적인 수
SELECT TABLE_ROWS FROM information_schema.tables WHERE table_name = 'orders';

-- 대안 2: "더보기" 패턴 (정확한 count 불필요)
-- LIMIT + 1로 다음 페이지 유무만 확인
SELECT * FROM orders WHERE status = 'PAID' LIMIT 11;  -- 11개 나오면 "다음 있음"
```

### 3. JOIN 최적화

```sql
-- 느림: 서브쿼리
SELECT * FROM orders
WHERE user_id IN (SELECT id FROM users WHERE grade = 'VIP');

-- 빠름: JOIN
SELECT o.* FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.grade = 'VIP';

-- 더 빠름: EXISTS (큰 테이블일 때)
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM users u WHERE u.id = o.user_id AND u.grade = 'VIP');
```

### 4. 불필요한 SELECT *

```sql
-- 느림: 30개 컬럼 전부
SELECT * FROM orders WHERE user_id = 100;

-- 빠름: 필요한 것만
SELECT id, status, total_amount, created_at FROM orders WHERE user_id = 100;
-- 네트워크 전송량 감소 + 커버링 인덱스 가능
```

### 5. OR 조건

```sql
-- 인덱스 비효율
SELECT * FROM orders WHERE user_id = 100 OR status = 'PENDING';
-- 두 조건이 다른 인덱스 → 풀스캔 가능

-- 개선: UNION ALL
SELECT * FROM orders WHERE user_id = 100
UNION ALL
SELECT * FROM orders WHERE status = 'PENDING' AND user_id != 100;
```

## 실행 계획 기반 튜닝 예시

```sql
-- Before
EXPLAIN SELECT * FROM orders o
JOIN order_items oi ON o.id = oi.order_id
WHERE o.created_at BETWEEN '2026-01-01' AND '2026-03-31'
  AND oi.product_id = 500
ORDER BY o.created_at DESC;

-- 실행 계획:
-- orders: type=ALL, rows=1000000  ← 풀스캔!
-- order_items: type=ref, rows=5

-- 원인: created_at에 인덱스 없음, 또는 범위가 너무 넓음
```

```sql
-- After: 인덱스 추가
CREATE INDEX idx_orders_created ON orders (created_at);
CREATE INDEX idx_items_product_order ON order_items (product_id, order_id);

-- 실행 계획:
-- orders: type=range, rows=50000  ← 범위 스캔으로 개선
-- order_items: type=eq_ref, rows=1

-- 추가 개선: 조건 순서 변경 (product_id가 더 변별력 높으면)
-- order_items부터 조회 → orders JOIN
```

## 모니터링

```sql
-- Slow Query Log 활성화
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 1초 이상 쿼리 기록

-- 현재 실행 중인 느린 쿼리
SELECT * FROM information_schema.processlist
WHERE command != 'Sleep' AND time > 3;
```

## 면접에서 이렇게 답하라

> "쿼리 튜닝은 **EXPLAIN으로 시작해서 인덱스로 끝**납니다.
> 먼저 Slow Query Log에서 문제 쿼리를 찾고, EXPLAIN으로 type이 ALL(풀스캔)인지, Using filesort가 있는지 확인합니다.
> 복합 인덱스는 **동등 조건 → 범위 조건 → 정렬** 순서로 설계하고, 가능하면 커버링 인덱스를 활용합니다.
> 대량 페이징은 OFFSET 대신 **커서 기반 페이징**으로, SELECT *는 필요한 컬럼만 조회하도록 개선합니다.
> 인덱스 컬럼에 함수를 적용하거나 앞쪽 와일드카드를 쓰면 인덱스를 타지 않으므로 주의합니다."
