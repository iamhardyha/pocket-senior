# N+1 문제와 조회 최적화

> **핵심 질문**: N+1, fetch join, entity graph, dto projection, QueryDSL은 각각 무엇이고 어떻게 사용하는가?

---

## 한 줄 요약

**"N+1은 ORM의 원죄다. 해결책은 여러 개인데, 상황에 맞는 걸 골라야 한다."**

---

## N+1 문제란?

```java
// Entity
@Entity
public class Order {
    @Id private Long id;
    @ManyToOne(fetch = FetchType.LAZY)
    private User user;
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}
```

```java
// 주문 목록 조회
List<Order> orders = orderRepository.findAll();  // 쿼리 1개

for (Order order : orders) {
    order.getUser().getName();    // 주문마다 User 쿼리 1개
    order.getItems().size();     // 주문마다 Items 쿼리 1개
}
```

```sql
-- 실제 실행되는 쿼리
SELECT * FROM orders;                          -- 1번
SELECT * FROM users WHERE id = 1;              -- N번 (주문 수만큼)
SELECT * FROM order_items WHERE order_id = 1;  -- N번
SELECT * FROM users WHERE id = 2;
SELECT * FROM order_items WHERE order_id = 2;
...
-- 주문 100개면: 1 + 100 + 100 = 201개 쿼리
```

**1개 쿼리로 될 것이 N+1개가 실행된다.** 이것이 N+1 문제.

---

## 해결 방법 총정리

| 방법 | 쿼리 수 | 장점 | 단점 |
|------|---------|------|------|
| Fetch Join | 1개 | 가장 직관적 | 페이징 불가 (컬렉션), 카테시안 곱 |
| @EntityGraph | 1개 | 어노테이션으로 간편 | 복잡한 조건 불가 |
| @BatchSize | 1+α | 설정만으로 해결 | 최적은 아님 |
| DTO Projection | 1개 | 가장 빠름, 필요한 컬럼만 | Entity 아님, 영속성 없음 |
| QueryDSL | 1개 | 타입 안전, 동적 쿼리 | 설정 복잡, 빌드 필요 |

## 1. Fetch Join

```java
// JPQL Fetch Join
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.user " +
       "JOIN FETCH o.items " +
       "WHERE o.status = :status")
List<Order> findByStatusWithDetails(@Param("status") OrderStatus status);
```

```sql
-- 실행되는 쿼리: 1개
SELECT o.*, u.*, oi.*
FROM orders o
INNER JOIN users u ON o.user_id = u.id
INNER JOIN order_items oi ON oi.order_id = o.id
WHERE o.status = 'PAID';
```

### 주의: 컬렉션 Fetch Join + 페이징

```java
// 위험: 컬렉션 Fetch Join + 페이징
@Query("SELECT o FROM Order o JOIN FETCH o.items")
Page<Order> findAllWithItems(Pageable pageable);
// → 경고: HHH000104: applying in memory!
// → 전체 데이터를 메모리에 올린 후 페이징 → OOM 위험
```

**컬렉션 Fetch Join에서 페이징이 안 되는 이유**:
```
Order 1 → Item A, Item B  (2행)
Order 2 → Item C           (1행)

JOIN 결과:
Order 1 | Item A
Order 1 | Item B  ← 같은 Order인데 2행
Order 2 | Item C

LIMIT 2하면? Order 1의 Item B가 잘림 → 데이터 불완전
→ Hibernate가 전체를 메모리에 올려서 처리 (위험)
```

**해결: @BatchSize 또는 별도 쿼리**

### 주의: 다중 컬렉션 Fetch Join

```java
// 금지: 2개 이상 컬렉션 Fetch Join
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.items " +      // 컬렉션 1
       "JOIN FETCH o.coupons")      // 컬렉션 2
// → MultipleBagFetchException (카테시안 곱 폭발)

// 해결: 하나만 Fetch Join + 나머지 BatchSize
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id IN :ids")
List<Order> findWithItems(@Param("ids") List<Long> ids);
// coupons는 @BatchSize로 처리
```

## 2. @EntityGraph

```java
// 어노테이션으로 Fetch 전략 지정
@EntityGraph(attributePaths = {"user", "items"})
@Query("SELECT o FROM Order o WHERE o.status = :status")
List<Order> findByStatus(@Param("status") OrderStatus status);

// Entity에 미리 정의
@Entity
@NamedEntityGraph(
    name = "Order.withUserAndItems",
    attributeNodes = {
        @NamedAttributeNode("user"),
        @NamedAttributeNode("items")
    }
)
public class Order { ... }

// 사용
@EntityGraph("Order.withUserAndItems")
List<Order> findByStatus(OrderStatus status);
```

**Fetch Join과 차이**: EntityGraph는 LEFT OUTER JOIN, Fetch Join은 INNER JOIN이 기본.

| | Fetch Join | EntityGraph |
|---|-----------|-------------|
| JOIN 타입 | INNER (기본) | LEFT OUTER (기본) |
| 조건 추가 | JPQL에서 자유롭게 | 어노테이션으로 제한적 |
| 동적 제어 | 쿼리마다 다르게 | 같은 그래프 재사용 |

## 3. @BatchSize

```java
@Entity
public class Order {
    @OneToMany(mappedBy = "order")
    @BatchSize(size = 100)  // 100개씩 IN 절로 묶어서 조회
    private List<OrderItem> items;
}

// 또는 전역 설정
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

```sql
-- BatchSize 없이: N개 쿼리
SELECT * FROM order_items WHERE order_id = 1;
SELECT * FROM order_items WHERE order_id = 2;
SELECT * FROM order_items WHERE order_id = 3;
...

-- BatchSize=100: 1개 쿼리
SELECT * FROM order_items WHERE order_id IN (1,2,3,...,100);
```

**장점**: 코드 변경 없이 설정만으로 N+1 완화.
**적합**: 페이징 + 컬렉션 조회 (Fetch Join이 안 되는 경우).

```java
// 페이징 + 컬렉션 최적화: Fetch Join(ToOne) + BatchSize(ToMany)
@Query("SELECT o FROM Order o JOIN FETCH o.user")  // ToOne은 Fetch Join
Page<Order> findAllWithUser(Pageable pageable);
// items는 @BatchSize로 → 페이징 안전 + N+1 해결
```

## 4. DTO Projection — 가장 빠른 조회

Entity를 통째로 가져올 필요 없다. **필요한 컬럼만** 가져온다.

### 인터페이스 기반 Projection

```java
// Projection 인터페이스
public interface OrderSummary {
    Long getId();
    String getUserName();       // user.name에서 가져옴
    BigDecimal getTotalAmount();
    LocalDateTime getCreatedAt();
}

// Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<OrderSummary> findByStatus(OrderStatus status);
    // → SELECT o.id, u.name, o.total_amount, o.created_at
    //   FROM orders o JOIN users u ON o.user_id = u.id
    //   WHERE o.status = ?
}
```

### 클래스 기반 Projection (JPQL)

```java
// DTO
public record OrderDto(
    Long id,
    String userName,
    BigDecimal totalAmount,
    int itemCount
) {}

// JPQL
@Query("SELECT new com.example.dto.OrderDto(" +
       "o.id, u.name, o.totalAmount, SIZE(o.items)) " +
       "FROM Order o JOIN o.user u " +
       "WHERE o.status = :status")
List<OrderDto> findOrderDtos(@Param("status") OrderStatus status);
```

### Entity vs DTO 성능 차이

```
100만 건 주문 조회:

Entity 조회:  SELECT * FROM orders → 30개 컬럼 전부 로딩
              + 영속성 컨텍스트 관리 (1차 캐시, 변경 감지)
              → 메모리 多, 속도 느림

DTO Projection: SELECT id, user_name, total_amount → 3개 컬럼만
                + 영속성 컨텍스트 관리 없음
                → 메모리 少, 속도 빠름

성능 차이: DTO가 2~5배 빠름 (데이터 양에 따라)
```

**원칙**: 읽기 전용 조회는 DTO Projection. 수정이 필요하면 Entity.

## 5. QueryDSL — 타입 안전 동적 쿼리

### 설정

```groovy
// build.gradle
dependencies {
    implementation 'com.querydsl:querydsl-jpa:5.0.0:jakarta'
    annotationProcessor 'com.querydsl:querydsl-apt:5.0.0:jakarta'
    annotationProcessor 'jakarta.persistence:jakarta.persistence-api'
}
```

### 기본 사용

```java
@Repository
@RequiredArgsConstructor
public class OrderQueryRepository {
    private final JPAQueryFactory queryFactory;

    public List<OrderDto> searchOrders(OrderSearchCondition condition) {
        QOrder order = QOrder.order;
        QUser user = QUser.user;
        QOrderItem item = QOrderItem.orderItem;

        return queryFactory
            .select(Projections.constructor(OrderDto.class,
                order.id,
                user.name,
                order.totalAmount,
                order.createdAt
            ))
            .from(order)
            .join(order.user, user)
            .where(
                statusEq(condition.status()),
                amountGoe(condition.minAmount()),
                createdAfter(condition.startDate())
            )
            .orderBy(order.createdAt.desc())
            .offset(condition.offset())
            .limit(condition.limit())
            .fetch();
    }

    // 동적 조건: null이면 조건 무시
    private BooleanExpression statusEq(OrderStatus status) {
        return status != null ? QOrder.order.status.eq(status) : null;
    }

    private BooleanExpression amountGoe(BigDecimal minAmount) {
        return minAmount != null ? QOrder.order.totalAmount.goe(minAmount) : null;
    }

    private BooleanExpression createdAfter(LocalDateTime startDate) {
        return startDate != null ? QOrder.order.createdAt.after(startDate) : null;
    }
}
```

### QueryDSL의 진짜 가치: 동적 쿼리

```java
// JPQL로 동적 쿼리: 문자열 조합 → 지옥
StringBuilder jpql = new StringBuilder("SELECT o FROM Order o WHERE 1=1");
if (status != null) jpql.append(" AND o.status = :status");
if (minAmount != null) jpql.append(" AND o.totalAmount >= :minAmount");
// → 타입 안전 X, 오타 런타임 에러, 유지보수 최악

// QueryDSL: 컴파일 타임 체크 → 천국
queryFactory.selectFrom(order)
    .where(
        statusEq(status),        // null이면 무시
        amountGoe(minAmount)     // null이면 무시
    )
    .fetch();
```

## 상황별 선택 가이드

```
조회 목적이 뭔가?

수정이 필요한가? ──YES──→ Entity (Fetch Join / EntityGraph)
       │
       NO (읽기 전용)
       │
동적 조건이 있나? ──YES──→ QueryDSL + DTO Projection
       │
       NO (고정 조건)
       │
컬렉션이 포함되나? ──YES──→ 페이징 필요? ──YES──→ ToOne Fetch Join + @BatchSize
       │                         │
       │                        NO ──→ Fetch Join
       │
       NO ──→ DTO Projection (가장 가벼움)
```

| 상황 | 추천 | 이유 |
|------|------|------|
| 관리자 목록 (검색+필터+페이징) | QueryDSL + DTO | 동적 조건, 읽기 전용 |
| 상품 상세 (연관 데이터 포함) | Fetch Join | Entity 수정 가능, 연관 로딩 |
| 주문 목록 + 상품 목록 (페이징) | Fetch Join(ToOne) + BatchSize(ToMany) | 페이징 안전 |
| 대시보드 통계 | Native Query or QueryDSL + DTO | 집계 함수, 성능 |
| API 응답 (필요 필드만) | DTO Projection | 네트워크 비용 절감 |

## 면접에서 이렇게 답하라

> "N+1은 Lazy Loading에서 연관 엔티티 접근 시 발생하는 문제로, 해결 방법을 상황에 맞게 선택합니다.
> **읽기 전용 조회**는 DTO Projection으로 필요한 컬럼만 가져와 성능을 극대화합니다.
> **엔티티가 필요한 경우** ToOne 관계는 Fetch Join, ToMany 관계는 @BatchSize로 처리하면 페이징도 안전합니다.
> **동적 쿼리**가 필요하면 QueryDSL을 사용해 타입 안전하게 조건을 조합합니다.
> 다중 컬렉션 Fetch Join은 카테시안 곱이 발생하므로 하나만 Fetch Join하고 나머지는 BatchSize로 처리합니다.
> 핵심 원칙은 '읽기는 DTO, 수정은 Entity'입니다."
