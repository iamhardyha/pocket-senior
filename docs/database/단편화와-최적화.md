# 데이터베이스 단편화와 최적화

> **핵심 질문**: DELETE를 해도 디스크 용량이 안 줄어드는 이유는? 단편화가 성능에 미치는 영향과 최적화 방법은?

---

## 한 줄 요약

**"DB는 DELETE해도 공간을 돌려주지 않는다. 빈 자리가 쌓이면 I/O가 늘고 성능이 떨어진다. OPTIMIZE TABLE로 재정리하라."**

---

## DELETE해도 디스크가 안 줄어드는 이유

MySQL(InnoDB)은 행을 삭제할 때 **물리적으로 디스크에서 지우지 않는다**.

```
DELETE FROM orders WHERE id = 100;
```

실제 동작:

```
1. 해당 행에 "삭제됨" 플래그(delete mark)를 설정
2. MVCC를 위해 Undo Log에 이전 버전 보관
3. Purge Thread가 B-tree 페이지에서 삭제된 행을 물리적으로 제거
   → 해당 페이지 내에서 "재사용 가능" 공간으로 전환
   → 페이지가 완전히 비면 인접 페이지와 병합(merge) 가능
4. 하지만 페이지가 부분적으로만 비었으면 빈 공간은 남고,
   OS에 디스크 공간을 반환하지도 않음
```

왜 바로 안 지울까?

| 이유 | 설명 |
|------|------|
| **MVCC** | 다른 트랜잭션이 삭제 전 데이터를 읽고 있을 수 있음 |
| **성능** | 물리적 삭제 + 페이지 재배치는 비용이 큼 |
| **재사용** | 빈 공간에 새 데이터를 넣는 게 새 페이지 할당보다 빠름 |
| **크래시 복구** | Redo/Undo 로그 기반 복구를 위해 즉시 제거 불가 |

## 단편화(Fragmentation)란

데이터가 반복적으로 INSERT → UPDATE → DELETE 되면서 **물리적 저장 순서가 뒤죽박죽**되는 현상.

### InnoDB 페이지 구조와 단편화

InnoDB는 데이터를 **16KB 페이지** 단위로 관리한다.

```
┌─────────────────────────────────────────┐
│               테이블스페이스              │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │
│  │Page 1│ │Page 2│ │Page 3│ │Page 4│   │
│  │██████│ │██░░██│ │░░░░░░│ │██████│   │
│  │██████│ │██░░░░│ │░░░░░░│ │██░░██│   │
│  └──────┘ └──────┘ └──────┘ └──────┘   │
│  ██ = 데이터    ░░ = 빈 공간 (삭제됨)    │
└─────────────────────────────────────────┘
```

- **Page 2**: 중간 행이 삭제되어 빈 공간 발생
- **Page 3**: 모든 행이 삭제되었지만 페이지 자체는 남아있음
- **Page 4**: UPDATE로 행 크기가 바뀌면서 빈 공간 발생

---

## 단편화가 성능을 떨어뜨리는 3가지 이유

### 1. 디스크 I/O 증가

```
[정상 상태 — 연속 저장]
Page 1 → Page 2 → Page 3 → Page 4
순차 읽기(Sequential I/O): 한 번에 쭉 읽음 → 빠름

[단편화 상태 — 흩어진 저장]
Page 1 → Page 5 → Page 12 → Page 3
랜덤 읽기(Random I/O): 디스크 헤드가 이리저리 → 느림
```

InnoDB는 **클러스터드 인덱스**(PK 순서로 물리 저장)를 사용한다. 단편화가 심해지면 PK 순서와 물리적 저장 위치가 어긋나면서 **순차 I/O가 랜덤 I/O로 바뀐다**.

| I/O 유형 | HDD 성능 | SSD 성능 |
|----------|----------|----------|
| 순차 읽기 | ~200 MB/s | ~3,000 MB/s |
| 랜덤 읽기 | ~1 MB/s | ~500 MB/s |

HDD에서는 200배, SSD에서도 6배 차이. **SSD라고 단편화를 무시할 수 없다**.

### 2. 쿼리 성능 저하

```sql
-- Range Scan: PK 범위로 조회
SELECT * FROM orders WHERE id BETWEEN 1000 AND 2000;
```

```
[정상] 10개 페이지 읽기 → Buffer Pool에 10페이지 캐시
[단편화] 50개 페이지 읽기 → Buffer Pool에 50페이지 캐시
         ↑ 같은 데이터인데 페이지가 반쯤 비어있으니 더 많은 페이지를 읽어야 함
```

**Buffer Pool 효율도 떨어진다**. 같은 크기의 Buffer Pool에 빈 공간이 섞인 페이지가 올라오면, 실제 캐시할 수 있는 데이터 양이 줄어든다.

```
Buffer Pool 1GB 기준:
[정상] 페이지 충전율 90% → 유효 데이터 ~900MB
[단편화] 페이지 충전율 50% → 유효 데이터 ~500MB
→ 캐시 히트율 하락 → 디스크 I/O 추가 발생
```

### 3. 디스크 공간 낭비

```sql
-- 실제 데이터 vs 디스크 사용량 확인
SELECT
    table_name,
    ROUND(data_length / 1024 / 1024, 2) AS data_mb,
    ROUND(data_free / 1024 / 1024, 2) AS free_mb,
    ROUND(data_free / data_length * 100, 2) AS fragmentation_pct
FROM information_schema.tables
WHERE table_schema = 'mydb' AND table_name = 'orders';
```

```
+------------+---------+---------+--------------------+
| table_name | data_mb | free_mb | fragmentation_pct  |
+------------+---------+---------+--------------------+
| orders     | 1024.00 | 512.00  | 50.00              |
+------------+---------+---------+--------------------+
```

위 예시에서 실제 데이터는 1GB인데, **500MB가 빈 공간**으로 낭비되고 있다.
이 빈 공간은 OS에 반환되지 않으므로 디스크에서는 1.5GB를 차지한다.

## 단편화 확인 방법 (MySQL)

### 방법 1: information_schema

```sql
SELECT
    table_name,
    engine,
    ROUND(data_length / 1024 / 1024, 2) AS data_mb,
    ROUND(index_length / 1024 / 1024, 2) AS index_mb,
    ROUND(data_free / 1024 / 1024, 2) AS free_mb,
    ROUND(data_free / (data_length + index_length) * 100, 2) AS frag_ratio
FROM information_schema.tables
WHERE table_schema = 'mydb'
ORDER BY data_free DESC;
```

**기준**: `frag_ratio`가 **20% 이상**이면 최적화를 고려.

> `data_free`는 `innodb_file_per_table=ON`(MySQL 5.6.6+ 기본값)일 때만 해당 테이블의 빈 공간을 정확히 나타낸다. 공유 테이블스페이스를 사용하면 전체 공유 공간의 빈 영역이 표시되므로 주의.

### 방법 2: SHOW TABLE STATUS

```sql
SHOW TABLE STATUS LIKE 'orders'\G
```

```
           Name: orders
         Engine: InnoDB
        Version: 10
     Row_format: Dynamic
           Rows: 5000000
 Avg_row_length: 200
    Data_length: 1073741824    ← 실제 데이터
   Index_length: 268435456
      Data_free: 536870912     ← 빈 공간 (512MB!)
```

### 방법 3: innodb_metrics (MySQL 8.0+)

```sql
-- 페이지 단위로 확인
SELECT name, count, comment
FROM information_schema.innodb_metrics
WHERE name LIKE '%page%free%';
```

## 최적화(OPTIMIZE TABLE)

### 원리

```
OPTIMIZE TABLE orders;
```

내부적으로 이런 일이 일어난다:

```
1. 새 테이블스페이스 생성
2. 기존 데이터를 PK 순서대로 새 테이블스페이스에 복사
3. 인덱스 재구축
4. 기존 테이블스페이스 삭제
5. 새 테이블스페이스로 교체
```

```
[Before — 단편화]                    [After — 최적화 완료]
┌──────┐ ┌──────┐ ┌──────┐          ┌──────┐ ┌──────┐
│██░░██│ │░░░░░░│ │██░░██│    →     │██████│ │██████│
│██░░░░│ │░░░░░░│ │██░░░░│          │██████│ │████░░│
└──────┘ └──────┘ └──────┘          └──────┘ └──────┘
 3 페이지 (빈 공간 다수)               2 페이지 (빈 공간 최소)
```

### 실행 방법

```sql
-- 단일 테이블
OPTIMIZE TABLE orders;

-- 여러 테이블
OPTIMIZE TABLE orders, users, order_items;
```

InnoDB에서 `OPTIMIZE TABLE`은 내부적으로 **ALTER TABLE ... FORCE**와 동일하게 동작한다.

```sql
-- 사실상 같은 동작
ALTER TABLE orders FORCE;
ALTER TABLE orders ENGINE=InnoDB;
```

### 주의사항

| 항목 | 설명 |
|------|------|
| **테이블 락** | 전체 재구축 동안 **DML(쓰기) 차단**. 대형 테이블은 수십 분간 쓰기 불가 |
| **디스크 공간** | 임시로 테이블 크기만큼 **추가 디스크 필요** (원본 + 복사본) |
| **소요 시간** | 테이블 크기에 비례. 10GB 테이블 → 수분~수십 분 |
| **복제 지연** | Replica에서도 실행되므로 **복제 지연(Replication Lag)** 발생 가능 |
| **Lock wait** | 장시간 실행 시 다른 DDL이 대기할 수 있음 |

## 운영 환경에서의 최적화 전략

### 1. OPTIMIZE TABLE의 한계

`OPTIMIZE TABLE`과 `ALTER TABLE ... FORCE`는 항상 **ALGORITHM=COPY**(전체 테이블 재구축)로 동작한다.
`ALGORITHM=INPLACE`를 지정하면 에러가 발생한다.

```sql
-- 이렇게 하면 에러!
ALTER TABLE orders FORCE, ALGORITHM=INPLACE, LOCK=NONE;
-- ERROR 1846 (0A000): ALGORITHM=INPLACE is not supported.

-- 실제로는 항상 COPY로 동작
OPTIMIZE TABLE orders;  -- = ALTER TABLE orders FORCE (ALGORITHM=COPY)
```

즉, `OPTIMIZE TABLE`만으로는 **서비스 중단 없는 최적화가 불가능**하다.
대형 테이블에서는 반드시 아래 도구를 사용해야 한다.

### 2. pt-online-schema-change (Percona Toolkit)

대용량 테이블에서 **서비스 중단 없이** 최적화하는 업계 표준 도구.

```bash
pt-online-schema-change \
  --alter "ENGINE=InnoDB" \
  --execute \
  --chunk-size=1000 \
  --max-lag=1s \
  --check-replication-filters \
  D=mydb,t=orders
```

동작 원리:

```
1. _orders_new 임시 테이블 생성 (동일 스키마)
2. orders에 트리거 설치 (INSERT/UPDATE/DELETE → _orders_new에도 반영)
3. 기존 데이터를 chunk 단위로 _orders_new에 복사
4. 복제 지연(lag)이 1초 넘으면 자동 일시 정지
5. 복사 완료 후 RENAME TABLE로 원자적 교체
6. 트리거 및 기존 테이블 삭제
```

### 3. gh-ost (GitHub Online Schema Change)

트리거 대신 **binlog를 읽어서** 변경분을 반영. 트리거의 성능 부담이 없다.

```bash
gh-ost \
  --alter "ENGINE=InnoDB" \
  --database=mydb \
  --table=orders \
  --execute \
  --max-lag-millis=1500
```

### 도구 선택 기준

| 기준 | OPTIMIZE TABLE | pt-osc | gh-ost |
|------|---------------|--------|--------|
| **서비스 중단** | 전체 DML 차단 | 거의 없음 | 거의 없음 |
| **적합 크기** | < 1GB | < 100GB | 모든 크기 |
| **복제 안전** | 주의 필요 | 자동 조절 | 자동 조절 |
| **설치** | 내장 | Percona 설치 | GitHub 설치 |
| **난이도** | 쉬움 | 보통 | 보통 |

## 단편화 예방

최적화는 비용이 크므로, 애초에 단편화를 줄이는 설계가 중요하다.

| 전략 | 설명 |
|------|------|
| **Soft Delete** | 물리 삭제 대신 `deleted_at` 컬럼 사용. 단, 조회 시 WHERE 조건 추가 필요 |
| **파티셔닝** | 날짜 기반 파티션 → 오래된 파티션만 `DROP PARTITION` (즉시 공간 회수) |
| **배치 삭제** | 한 번에 대량 삭제 대신 `LIMIT 1000`씩 나눠서 삭제 |
| **AUTO_INCREMENT PK** | 순차 삽입으로 페이지 분할 최소화 (UUID PK는 단편화 유발) |
| **적절한 행 크기** | VARCHAR 대신 CHAR, TEXT를 별도 테이블로 분리 |

### UUID vs AUTO_INCREMENT

```
AUTO_INCREMENT: 항상 맨 뒤에 삽입 → 페이지 분할 없음
UUID (v4):      랜덤 위치에 삽입 → 페이지 분할 빈번 → 단편화 심각
```

UUID를 꼭 써야 한다면:

```sql
-- 방법 1: MySQL의 UUID() = UUIDv1 (시간 기반)
-- UUID_TO_BIN()의 두 번째 인자 1 = 시간 바이트를 앞으로 재배치
-- MySQL 8.0+에서 사용 가능
INSERT INTO orders (id) VALUES (UUID_TO_BIN(UUID(), 1));

-- 방법 2: 애플리케이션에서 UUIDv7 생성 (RFC 9562)
-- UUIDv7은 앞부분이 타임스탬프 → 생성 자체가 시간순
-- MySQL에 네이티브 UUID_v7() 함수는 없음 → 앱에서 생성 후 저장
```

### 파티셔닝으로 공간 즉시 회수

```sql
-- 월별 파티셔닝
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT,
    created_at DATETIME NOT NULL,
    ...
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
    PARTITION p202601 VALUES LESS THAN (202602),
    PARTITION p202602 VALUES LESS THAN (202603),
    PARTITION p202603 VALUES LESS THAN (202604)
);

-- 오래된 데이터 삭제: 즉시 디스크 공간 회수!
ALTER TABLE orders DROP PARTITION p202601;
-- vs DELETE WHERE created_at < '2026-02-01' → 단편화 발생
```

## 면접에서 이렇게 답하자

> "MySQL InnoDB에서 DELETE는 물리적 삭제가 아니라 **삭제 표시**만 합니다. MVCC와 성능을 위한 설계입니다.
> 이로 인해 반복적인 DML 작업 후 **단편화**가 발생하고, 순차 I/O가 랜덤 I/O로 바뀌면서 쿼리 성능이 저하됩니다. Buffer Pool 효율도 떨어지고 디스크 공간도 낭비됩니다.
>
> 단편화 해결은 `OPTIMIZE TABLE`로 데이터를 재배치합니다. 내부적으로 테이블을 재생성하는 것과 같습니다.
> 운영 환경에서는 서비스 영향을 최소화하기 위해 **pt-online-schema-change**나 **gh-ost** 같은 도구를 사용합니다.
>
> 근본적으로는 **파티셔닝**으로 DROP PARTITION을 통해 공간을 즉시 회수하거나, AUTO_INCREMENT PK를 사용해 페이지 분할을 최소화하는 등 설계 단계에서 단편화를 예방하는 것이 중요합니다."
