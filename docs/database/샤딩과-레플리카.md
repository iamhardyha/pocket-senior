# 샤딩과 레플리카

> **핵심 질문**: DB 샤딩과 레플리카는 무엇이고, 어떻게 설계하는가?

---

## 한 줄 요약

**"레플리카는 읽기를 분산하고, 샤딩은 쓰기를 분산한다. 레플리카를 먼저 하고, 그래도 안 되면 샤딩을 고려한다."**

---

## 레플리카 (Replication)

### 개념

```
Primary (Master): 읽기 + 쓰기
Replica (Slave):  읽기 전용 (Primary의 복사본)

Primary ──복제──→ Replica 1
                → Replica 2
```

### 종류

| 방식 | 동기화 | Lag | 데이터 유실 |
|------|--------|-----|-------------|
| 동기 (Sync) | 커밋 전에 Replica 확인 | 없음 | 없음 |
| 비동기 (Async) | 커밋 후 Replica에 전달 | 있음 | 가능 |
| 반동기 (Semi-Sync) | 최소 1대 확인 후 커밋 | 최소화 | 최소화 |

**대부분 비동기**를 쓴다. 동기는 성능 저하가 크다.

### Read/Write 분리

```java
// Spring: AbstractRoutingDataSource
@Transactional(readOnly = true)   // → Replica
public List<Order> getOrders() { ... }

@Transactional                    // → Primary
public void createOrder() { ... }
```

### 한계

```
레플리카는 읽기를 분산하지만, 쓰기는 분산하지 못한다.

Primary 1대에 모든 쓰기가 몰림:
- 쓰기 QPS 한계 도달
- Primary CPU 100%
- 레플리카 아무리 늘려도 소용없음

→ 쓰기 분산이 필요하면? → 샤딩
```

## 샤딩 (Sharding)

### 개념

데이터를 여러 DB에 **나눠서** 저장한다.

```
Before (단일 DB):
orders 테이블: 10억 건 → 1대에서 감당

After (샤딩):
Shard 1: user_id 1~1000만 (orders)
Shard 2: user_id 1000만~2000만 (orders)
Shard 3: user_id 2000만~3000만 (orders)
→ 3대가 나눠서 감당
```

### 샤딩 키 선택 — 가장 중요한 결정

```
좋은 샤딩 키:
- 데이터가 균등하게 분배됨
- 주요 쿼리에 포함됨
- 변경되지 않음

예: user_id
→ 사용자별 데이터가 한 샤드에 모임
→ 사용자별 조회 시 단일 샤드만 접근 → 빠름

나쁜 샤딩 키:
- created_at (최신 샤드에만 쓰기 몰림)
- status (편향 분배)
- 자주 변경되는 값
```

### 샤딩 전략

| 전략 | 방법 | 장점 | 단점 |
|------|------|------|------|
| **Range** | user_id 범위별 | 범위 쿼리 용이 | 핫스팟 가능 |
| **Hash** | hash(user_id) % N | 균등 분배 | 범위 쿼리 불가 |
| **Directory** | 매핑 테이블로 관리 | 유연한 배치 | 매핑 테이블 의존 |

```
Hash Sharding:
shard_number = hash(user_id) % 3

user_id=1  → hash(1) % 3 = 1 → Shard 1
user_id=2  → hash(2) % 3 = 2 → Shard 2
user_id=3  → hash(3) % 3 = 0 → Shard 0
```

### 샤딩의 고통

```
1. Cross-Shard JOIN 불가
   → user_id=1의 주문 + user_id=2의 주문을 JOIN?
   → 서로 다른 DB라 SQL로 JOIN 불가
   → 애플리케이션에서 합쳐야 함

2. Cross-Shard 트랜잭션 불가
   → 2개 샤드에 걸친 원자적 처리 어려움

3. Shard 추가 시 데이터 재배치 (Resharding)
   → 3대 → 5대로 늘리면?
   → hash(user_id) % 5로 변경 → 대부분의 데이터 이동
   → Consistent Hashing으로 완화 가능

4. 글로벌 유니크 ID
   → Auto Increment가 샤드마다 충돌
   → Snowflake ID, UUID 등 분산 ID 생성 필요

5. 운영 복잡도 폭증
   → 스키마 변경을 모든 샤드에 적용
   → 백업/복구도 샤드 수만큼
   → 모니터링도 샤드별로
```

### Consistent Hashing

샤드 추가/제거 시 데이터 이동을 최소화.

```
일반 Hash:
3대 → 5대: 대부분의 키가 다른 샤드로 이동 (60%+)

Consistent Hashing:
3대 → 5대: 이웃 샤드 데이터 일부만 이동 (약 20%)

원리: 해시 링(ring) 위에 노드와 키를 배치
키는 시계 방향으로 가장 가까운 노드에 배치
노드 추가 시 인접 키만 이동
```

## 샤딩 전에 고려할 것

```
"정말 샤딩이 필요한가?"

순서:
1. 쿼리 최적화 (인덱스, 실행 계획)     → 대부분 여기서 해결
2. 캐싱 (Redis)                        → 읽기 부하 해결
3. Read Replica                         → 읽기 분산
4. 테이블 파티셔닝 (단일 DB 내)         → 관리 단순
5. 수직 분할 (서비스별 DB 분리)         → MSA
6. 샤딩                                 → 최후의 수단
```

**냉정한 현실**: 한국에서 샤딩이 필요한 서비스는 네이버, 카카오, 쿠팡, 배민 정도. 대부분은 Aurora + Read Replica로 충분하다.

## 테이블 파티셔닝 (샤딩 대안)

단일 DB 내에서 테이블을 논리적으로 분할.

```sql
-- Range Partitioning
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    created_at DATE,
    ...
) PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026),
    PARTITION p2026 VALUES LESS THAN (2027)
);

-- 2024년 데이터 삭제: 테이블 DROP처럼 빠름
ALTER TABLE orders DROP PARTITION p2024;
```

- 샤딩보다 훨씬 간단
- 운영 복잡도 낮음
- 오래된 데이터 아카이빙에 효과적
- 하지만 쓰기 분산은 안 됨 (동일 DB)

## 면접에서 이렇게 답하라

> "레플리카와 샤딩은 목적이 다릅니다.
> **레플리카**는 읽기를 분산하고, **샤딩**은 쓰기를 분산합니다.
> 먼저 쿼리 최적화 → 캐싱 → Read Replica 순서로 시도하고, 그래도 안 되면 샤딩을 고려합니다.
> 샤딩 키는 **균등 분배 + 주요 쿼리 포함 + 불변** 조건을 만족해야 합니다.
> 가장 큰 고통은 Cross-Shard JOIN 불가, Resharding 비용, 운영 복잡도입니다.
> 대부분의 서비스는 Aurora + Read Replica + 파티셔닝으로 충분하고, 샤딩은 진짜 필요할 때만 해야 합니다."
