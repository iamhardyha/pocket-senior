# AWS 데이터 분석 스택 한눈에 — 뭘 언제 쓰고 어떻게 엮나

> **핵심 질문**: Athena·Glue·Redshift·DynamoDB가 다 "데이터" 서비스인데 뭐가 뭐고, 어떤 니즈일 때 무엇을 고르고 어떻게 조합해 아키텍처를 짜야 하지?

---

## 한 줄 요약

**"먼저 '운영(OLTP)이냐 분석(OLAP)이냐'로 층을 가른다. 키로 읽고 쓰는 운영은 DynamoDB, 분석은 S3 데이터 레이크 위에서 — 가끔·애드혹은 Athena, 반복 BI·복잡 조인은 Redshift. Glue는 그 둘이 공유하는 카탈로그+ETL이다. 그리고 분석 비용은 결국 '스캔량을 줄이는' 싸움이다."**

> 이 노트는 **결정맵 + 레퍼런스 아키텍처**다. 각 서비스의 내부 동작은 시리즈 다음 편들이 깊게 다룬다 — [Athena와 Glue](./Athena와-Glue.md)(2편), [Redshift](./Redshift-데이터-웨어하우스.md)(3편), [DynamoDB](./DynamoDB.md)(4편). 여기선 **"층 구분 → 언제 뭘 → 어떻게 조합"** 에만 집중한다.

---

## 1. 가장 먼저 할 일: 두 개의 층을 가른다 (OLTP vs OLAP)

네 서비스를 외우려 들면 헷갈린다. **워크로드를 두 층으로 가르는 것**이 결정의 9할이다.

- **운영(OLTP, Online Transaction Processing)**: 앱 백엔드가 실시간으로 한 건씩 읽고 쓴다. "주문번호로 주문 조회", "장바구니에 담기". 키 기반·짧은 트랜잭션·낮은 지연. → **DynamoDB**(또는 RDB).
- **분석(OLAP, Online Analytical Processing)**: 많은 행을 훑어 집계·조인·리포트한다. "지난달 카테고리별 매출 추이", "이탈 사용자 코호트". 대량 스캔·복잡 쿼리. → **Athena / Redshift**(데이터 레이크/웨어하우스 위에서).

**왜 층을 먼저 가르나** — 운영 DB에서 큰 분석 집계를 돌리면, 그 무거운 스캔이 운영 트랜잭션의 CPU·I/O·커넥션을 잡아먹어 **실서비스 지연·장애**로 번진다. 그래서 분석은 운영에서 **물리적으로 분리된 스택**으로 빼는 게 원칙이다. (이 분리를 잇는 다리가 뒤에 나오는 CDC/Streams다.)

### 네 서비스 한 줄 정체 + 층 정리

| 서비스 | 한 줄 정체 | 층 | 대표 워크로드 | 과금 축 |
|---|---|---|---|---|
| **DynamoDB** | 완전관리형 서버리스 **NoSQL** (키 조회, 단일 자릿수 ms) | 운영(OLTP) | 키로 읽고 쓰는 앱 백엔드 | 요청/용량 (on-demand · provisioned RCU/WCU) |
| **Athena** | **서버리스** SQL로 **S3를 직접** 쿼리 | 분석(OLAP) | 가끔·애드혹 탐색, 로그 분석 | **스캔량** ($5/TB) |
| **Redshift** | 컬럼형 **MPP 데이터 웨어하우스** | 분석(OLAP) | 반복 BI·대시보드·복잡 조인 | 노드/RPU + 스토리지(RMS) |
| **Glue** | **서버리스 데이터 통합** (카탈로그 + 크롤러 + ETL) | 분석의 **받침대** | 스키마 카탈로그·포맷 변환·파티셔닝 | DPU-hour ($0.44/Spark) + 카탈로그 |

> 핵심 직관: **DynamoDB는 다른 셋과 층이 다르다.** Athena·Redshift·Glue는 "분석 3종"이고, 그 안에서 Glue는 엔진이 아니라 **둘이 공유하는 기반**이다.

## 2. 언제 뭘 쓰나 — 결정 기준

### 결정 흐름 (위에서 아래로)

```
Q1. 운영 트랜잭션인가? (앱이 키로 읽고/쓰는 실시간 경로)
      예  → DynamoDB (또는 RDB)              ← 운영(OLTP) 층, 끝
      아니오(분석·집계·리포트) ↓              ← 분석(OLAP) 층
Q2. 데이터가 이미 S3에 있고, 쿼리가 가끔·애드혹·탐색적인가?
      예  → Athena (서버리스, 띄울 클러스터 없음, 스캔당 과금)
      아니오 ↓
Q3. 정해진 BI·대시보드를 반복 실행? 복잡한 다중 조인?
    낮은 지연·동시 사용자 多가 필요?
      예  → Redshift (적재된 웨어하우스, 상시 클러스터/RPU)

* 어느 쪽을 고르든 스키마·파티션·포맷 변환은 Glue가 받친다.
```

### Athena vs Redshift — 분석 스택 안에서의 갈림길

둘 다 SQL로 분석하지만 **모델이 정반대**다. Athena는 "데이터는 S3에 그대로 두고 쿼리할 때만 읽는다", Redshift는 "데이터를 웨어하우스에 적재해 두고 그 위에서 고성능으로 돈다".

| 기준 | **Athena** | **Redshift** |
|---|---|---|
| 운영 모델 | 서버리스 — 프로비저닝/관리할 클러스터 없음 | 상시 클러스터(또는 Serverless RPU) — 띄워 둔다 |
| 데이터 위치 | **S3를 직접** 쿼리 (데이터 레이크 in-place) | 웨어하우스에 **적재**(load) 후 쿼리 (+ Spectrum으로 S3도) |
| 쿼리 성격 | **애드혹·간헐적·탐색적**, 데이터 양·주기 들쭉날쭉 | **반복 BI·정형 리포트**, 복잡 조인, 낮은 지연 |
| 성능 지렛대 | 파일 포맷·파티션(읽는 쪽에서) | 분산키(DISTKEY)·정렬키(SORTKEY)로 데이터 배치 최적화 |
| 과금 | **스캔한 데이터량** ($5/TB) — 안 쓰면 0 | 노드 시간 / RPU + 스토리지(RMS) — 띄워 두면 과금 |
| "이럴 때" | "한 달에 몇 번 로그 좀 들여다본다" | "매일 아침 임원 대시보드가 수십 개 쿼리를 친다" |

**판단의 핵심**: *쿼리 빈도와 성능 요구*다. 드물고 들쭉날쭉하면 띄워 두는 비용이 아까우니 Athena(쓴 만큼). 반복·고성능·동시성이 필요하면 클러스터를 상시 운영하는 Redshift가 결국 싸고 빠르다. — 둘은 배타적이지 않다. **카탈로그를 공유하면 같은 S3 데이터를 둘 다로 쿼리**할 수 있다(섹션 3-C).

> 참고: Athena의 SQL 엔진은 Trino/Presto 계열이다. 현재 기본은 엔진 버전 3(Trino 기반)이며, 이전 v1/v2는 Presto 계열이다. Redshift Spectrum도 "클러스터에서 S3를 직접 쿼리"라 Athena와 닮았지만, **Redshift 클러스터 비용이 별도로 든다**(스캔은 동일하게 $5/TB). RA3 노드 타입은 Spectrum을 따로 둘 필요 없이 S3를 직접 본다.

### DynamoDB vs 분석 스택 — "운영 DB에서 큰 집계 돌리지 마라"

DynamoDB는 **키 기반 조회의 챔피언**이지, 분석 엔진이 아니다. 같은 파티션 키에 트래픽이 몰리면 테이블에 여유 용량이 있어도 **핫 파티션**으로 throttle이 나고(단일 파티션 상한 3000 RCU / 1000 WCU), 풀스캔·집계는 비싸고 느리다. 그래서:

- **키로 한 건 읽고 쓰는 운영 경로** → DynamoDB. (예: `userId`로 프로필 조회)
- **"전체를 훑는" 집계·조인·리포트** → 운영 테이블에서 직접 돌리지 말고, **변경을 분석 스택으로 흘려보낸 뒤** Athena/Redshift에서. 다리는 **DynamoDB Streams** 또는 [CDC](./CDC.md)다(섹션 3-B).

### Glue는 어디에? — 선택지가 아니라 받침대

Glue는 "Athena냐 Redshift냐"와 같은 층의 선택지가 **아니다**. 분석 3종의 **공통 기반**이다. 세 부품으로 본다.

- **Data Catalog** — 중앙 메타데이터 저장소. "이 S3 경로엔 이런 스키마의 테이블이 있다"를 기록. **Athena·Redshift Spectrum·EMR이 이 하나를 공유**한다 → 같은 데이터를 여러 엔진이 같은 정의로 본다.
- **Crawler** — S3를 스캔해 스키마를 자동 추론하고 카탈로그에 테이블/파티션을 만들어 준다.
- **ETL Job** — Spark/Python으로 변환·적재. 대표 작업이 **JSON 로그 → Parquet(컬럼형) 변환 + 파티셔닝**인데, 이게 곧 뒤에서 말할 "스캔량 줄이기"의 실행부다.

> 한 줄: **Athena/Redshift가 "무엇을 어떻게 쿼리하나"라면, Glue는 "그 데이터가 어떤 모양·어떤 스키마로 거기 있나"를 책임진다.** 적재 자체(스트림→S3)는 [Firehose vs S3 직접 적재](../infra/Firehose-vs-S3-직접-적재.md)가 다룬다.

## 3. 니즈별 레퍼런스 아키텍처

니즈를 받으면 아래 셋 중 하나(또는 조합)로 그린다.

### (A) 서버리스 데이터 레이크 분석 — "운영팀 없이, 가끔 들여다본다"

**니즈**: 앱/서비스 로그·이벤트를 모아 두고, 필요할 때 SQL로 탐색. 상시 분석 인프라를 운영할 사람이 없다.

```
[앱·서비스 로그/이벤트]
       │   Firehose 적재 (버퍼링 → Parquet 변환 → 파티셔닝)
       ▼
[ S3 데이터 레이크 ]   s3://.../year=2026/month=06/day=30/...  (Parquet)
       │
       ├── [Glue Crawler] ──► 스키마 추론 ──┐
       │                                     ▼
       │                          [ Glue Data Catalog ]  (테이블·파티션)
       │                                     │
       └── [Glue ETL Job] (raw→Parquet)      ▼
                                        [ Athena ] ── 애드혹 SQL
                                             │
                                             ▼
                                   [ QuickSight 등 BI ] 대시보드
```

**왜 이 모양**: 전 구간 서버리스 → 띄워 둘 클러스터가 없어 **유휴 비용 0**, 쿼리할 때 스캔량만 낸다. 빈도가 낮고 들쭉날쭉할수록 이 조합이 유리하다.

### (B) 운영 → 분석 파이프라인 — "운영 DB를 건드리지 않고 분석한다"

**니즈**: DynamoDB로 서비스를 운영 중인데, 그 데이터로 집계·리포트를 뽑고 싶다. **운영 테이블 성능은 절대 건드리면 안 된다.**

```
[ DynamoDB ] (운영 OLTP · 키 조회)
     │   DynamoDB Streams / CDC  (아이템 변경 캡처)
     ▼
[ Firehose / Lambda ] ──► [ S3 데이터 레이크 ]
                                │
                          [ Glue 카탈로그 / ETL ]   (정규화·Parquet·파티셔닝)
                                │
                  ┌─────────────┴─────────────┐
                  ▼                           ▼
            [ Athena ]                  [ Redshift ]
        (애드혹 탐색·검증)          (정기 BI·복잡 조인·대시보드)
```

**왜 이 모양**: 변경 이벤트를 **비동기로** 레이크에 흘려보내므로, 분석 쿼리가 아무리 무거워도 운영 DynamoDB는 영향을 받지 않는다. 탐색 단계는 Athena로 싸게, 정형화된 반복 리포트로 굳으면 Redshift로 올린다. 변경 캡처의 원리는 [CDC](./CDC.md), Streams는 [DynamoDB](./DynamoDB.md) 편 참고.

### (C) 엔터프라이즈 웨어하우스 + 레이크하우스 — "핫은 빠르게, 콜드는 싸게, 카탈로그는 하나로"

**니즈**: 대규모·고성능 BI가 상시 돌고, 동시에 수년치 콜드 데이터도 가끔 조인해야 한다. 데이터 정의는 **한 곳에서** 관리하고 싶다.

```
              [ Glue Data Catalog ]  ← 공유 메타스토어 (단일 진실원)
                ╱                  ╲
        [ Athena ]            [ Redshift (RA3) ]
       (S3 직접 쿼리)              │  hot  → RMS(SSD, 빠름)
              │                    │  cold → S3 (RA3가 직접/Spectrum)
              ▼                    ▼
        [ S3 데이터 레이크 ] ◄──────┘  외부 테이블로 같은 데이터 조인
```

**왜 이 모양**:
- **RA3 + Redshift Managed Storage(RMS)** 가 핫 데이터는 SSD, 콜드는 S3로 자동 계층화한다 — 컴퓨트와 스토리지가 **분리 과금**이라 둘을 독립적으로 키운다.
- 자주 쓰는 데이터는 웨어하우스 안에서 고성능으로, 오래된 데이터는 S3에 두고 **Spectrum/RA3로 직접 조인** → 다 적재하지 않아 스토리지 비용↓.
- **Glue Catalog 하나를 Athena와 Redshift가 공유** → "같은 데이터, 두 엔진". 정의가 한 곳이라 드리프트가 없다. 이게 *레이크하우스*의 실체다.

## 4. 비용 사고방식 한 컷

서비스마다 **과금 축이 다르다.** 이걸 모르면 "왜 청구서가 이렇지"가 된다.

| 서비스 | 과금 축 | 비용을 좌우하는 손잡이 |
|---|---|---|
| **Athena / Spectrum** | **스캔한 데이터량** — $5/TB (쿼리당 최소 10MB, 실패 쿼리 무료) | **얼마나 적게 읽느냐** = 파티션 + 컬럼형 |
| **Glue** | ETL: **DPU-hour** ($0.44/Spark DPU, 1 DPU=4 vCPU+16GB, 최소 1분) · 카탈로그: 첫 100만 객체+100만 접근 무료 후 월정액 | DPU 수·잡 시간, 크롤러 빈도 |
| **Redshift** | 노드 시간(또는 Serverless **RPU**) + 스토리지(**RMS**, GB-month) | 적재량·동시성·상시 가동 시간 |
| **DynamoDB** | 요청/용량 — on-demand(요청당) 또는 provisioned(설정 RCU/WCU) | 읽기/쓰기 패턴, 키 분산(핫 파티션 회피) |

### 분석 비용의 핵심: "스캔량을 줄여라"

Athena·Spectrum이 **스캔한 만큼** 돈을 받으므로, 분석 비용 최적화는 곧 **읽는 바이트를 줄이는 일**이다. 두 손잡이가 압도적이다.

- **파티셔닝**: `year=/month=/day=/` 경로로 쪼개면, `WHERE day='2026-06-30'` 쿼리가 **그 파티션만** 읽는다(파티션 프루닝). 전체 스캔 회피.
- **컬럼형 포맷(Parquet/ORC)**: 쿼리에 필요한 **컬럼만** 읽는다. 50개 컬럼 중 3개만 SELECT하면 나머지를 안 읽는다. + 압축까지.

이 둘을 적용하면 스캔량이 크게 줄어 비용도 같이 떨어진다(AWS는 압축+파티션+컬럼형으로 **쿼리당 30~90%** 절감 가능하다고 안내). 그리고 이 "raw JSON → Parquet 변환 + 파티셔닝"을 실제로 수행하는 게 바로 **Glue ETL**이다. 즉 **Glue에 약간 쓰고 Athena에서 크게 아끼는** 구조다.

> 비용 직관 한 줄: **Redshift는 "띄워 둔 시간"에, Athena는 "읽은 바이트"에, Glue는 "변환 시간"에, DynamoDB는 "요청·용량"에 돈을 낸다.** 워크로드가 어느 축에 무거운지를 보고 고른다.

---

## 5. 면접에서 이렇게 답하자

> "AWS 데이터 서비스는 먼저 **운영(OLTP) 층과 분석(OLAP) 층으로 가릅니다.** 키로 읽고 쓰는 실시간 운영은 **DynamoDB**, 대량 스캔·집계 분석은 S3 데이터 레이크 위에서 합니다. 분석 안에서는 **가끔·애드혹이면 서버리스라 유휴 비용이 없는 Athena**, **반복 BI·복잡 조인·고성능이면 상시 웨어하우스 Redshift**를 씁니다. **Glue**는 둘 중 하나가 아니라, Athena·Redshift가 **공유하는 카탈로그와 ETL**로 둘을 받칩니다.
>
> 실무 원칙은 **운영 DB에서 큰 집계를 직접 돌리지 않는 것** 입니다 — DynamoDB Streams나 CDC로 변경을 레이크에 흘려보내 분석 스택에서 처리합니다. 비용은 결국 **Athena는 스캔량, Redshift는 가동 시간, Glue는 변환 시간, DynamoDB는 요청량**에 달려 있고, 분석 비용 최적화의 핵심은 **파티셔닝 + 컬럼형 포맷으로 스캔량을 줄이는 것**입니다."

**꼬리질문 대비**
- *"Athena랑 Redshift Spectrum 차이?"* → 둘 다 S3를 SQL로 직접 쿼리(스캔 $5/TB)지만, Spectrum은 **Redshift 클러스터 비용이 별도**다. 이미 Redshift를 운영하며 콜드 데이터를 조인할 때 Spectrum, 클러스터 없이 순수 S3 분석이면 Athena.
- *"DynamoDB로 분석하면 안 되나?"* → 키 조회엔 최강이지만 풀스캔·집계엔 부적합하고, 핫 파티션 throttle 위험이 있다. 분석은 레이크로 빼라.
- *"레이크하우스가 뭐냐?"* → **Glue Catalog 하나를 공유**해 같은 S3 데이터를 Athena·Redshift 등 여러 엔진이 같은 정의로 쿼리하는 구조. 핫은 웨어하우스(RA3/RMS)에, 콜드는 S3에 두고 한 카탈로그로 잇는다.
