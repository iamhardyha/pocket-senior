# DynamoDB — 싱글테이블·키설계·핫파티션

> **핵심 질문**: DynamoDB는 무엇이고, 액세스 패턴에서 키를 어떻게 뽑아내며, 싱글테이블·보조 인덱스·핫파티션을 실무에서 어떻게 다루는가?

---

## 한 줄 요약

**"DynamoDB는 '쿼리를 먼저 정하고 그에 맞춰 키를 설계하는' DB다. RDB가 데이터를 정규화한 뒤 쿼리를 짠다면, DynamoDB는 액세스 패턴을 먼저 적고 PK/SK를 거기에 끼워 맞춘다. 잘 설계하면 어떤 규모에서도 한 자릿수 ms, 잘못 설계하면 핫파티션으로 throttle 난다."**

---

## 1. DynamoDB가 뭐고, 언제 쓰나

DynamoDB는 AWS의 **완전 관리형 서버리스 NoSQL**이다. 서버·샤드·레플리카를 직접 관리하지 않아도 내부적으로 데이터를 여러 파티션에 자동 분산하고, 키 기반 조회에서 데이터 양과 무관하게 **한 자릿수 ms**(single-digit millisecond) 응답을 목표로 한다.

핵심 전제는 하나다 — **"무엇을 어떻게 조회할지를 미리 안다."** 이걸 알면 DynamoDB는 거의 무한히 스케일하고, 모르면(애드혹 분석·자유로운 JOIN) RDB나 분석 엔진이 맞다.

### RDB vs DynamoDB

| 항목 | RDB (MySQL/PostgreSQL) | DynamoDB |
|------|----------------------|----------|
| 데이터 모델 | 테이블·행·열, 고정 스키마 | Key-Value / Document, 아이템별 유연 스키마 |
| 설계 순서 | 정규화 먼저 → 쿼리는 나중 | **액세스 패턴 먼저 → 키 설계** |
| 쿼리 | SQL (JOIN·집계·애드혹 자유) | PK/SK 기반 조회 (JOIN·서버측 집계 없음) |
| 확장 | 기본 Scale-Up, 샤딩은 수작업 | Scale-Out 자동 (파티션 자동 분할) |
| 성능 특성 | 쿼리·인덱스·데이터량에 따라 변동 | 키 조회는 데이터량과 무관하게 일정 |
| 트랜잭션 | 완전한 ACID, 임의 범위 | ACID지만 **한 트랜잭션당 최대 100개 아이템** |
| 일관성 | 기본 강한 일관성 | 읽기당 선택: 강한 일관성 vs 결과적 일관성 |
| 운영 | 서버·백업·패치 관리 | 완전 관리형(서버리스) |
| 비용 | 인스턴스 기반(켜두면 과금) | 요청·저장량 기반(쓴 만큼) |

> **왜 JOIN이 없나** — JOIN은 두 테이블을 런타임에 합치는 연산이라, 어느 노드에 어떤 데이터가 있는지 미리 모이게 두지 않으면 분산 환경에서 비싸진다. DynamoDB는 "조인할 데이터를 처음부터 같은 파티션에 같이 저장"(아이템 컬렉션)해서 조인 자체를 없앤다. 이게 싱글테이블 디자인의 뿌리다.

### 잘 맞는 곳 / 안 맞는 곳

| 잘 맞음 | 이유 | 안 맞음 | 이유 |
|---|---|---|---|
| 세션·장바구니·사용자 프로필 | 키 단건 read/write | 복잡한 다중 JOIN | 조인 불가 |
| 게임 리더보드·카운터 | 높은 쓰기 처리량 | 애드혹/탐색 쿼리 | 패턴 사전 고정 필요 |
| IoT·이벤트 로그 | 대량 append, 범위 조회 | SUM/GROUP BY 분석 | 서버측 집계 없음 |
| 서버리스 백엔드 원천 | Lambda·API GW와 궁합 | 관계가 자주 바뀌는 도메인 | 모델 변경 비용 큼 |

## 2. 키 모델 — 파티션 키와 정렬 키

DynamoDB의 프라이머리 키는 두 형태다.

1. **파티션 키(PK)만** — 순수 Key-Value. PK 하나로 아이템이 유일.
2. **파티션 키 + 정렬 키(SK) 복합** — PK가 같은 아이템들이 SK로 정렬되어 한 곳에 모인다(= **아이템 컬렉션**).

```
PK(Partition Key)  ──hash(PK)──▶  어느 물리 파티션에 저장할지 결정
SK(Sort Key)       ──────────▶  같은 PK 안에서 정렬 순서 + 범위 쿼리
```

- **PK의 역할**: 값을 해시해 데이터를 물리 파티션에 흩뿌린다. → PK가 **고카디널리티**(값 종류가 많고 트래픽이 고르게 퍼짐)여야 핫파티션을 피한다.
- **SK의 역할**: 같은 PK 안에서 사전식(lexicographic) 정렬. 그래서 SK를 `2026-06-30#...`처럼 **정렬 가능한 문자열**로 설계하면 범위·정렬 쿼리가 공짜로 따라온다.

### 읽기 연산 3종 — 비용이 완전히 다르다

| 연산 | 동작 | 비용 |
|---|---|---|
| `GetItem` | PK(+SK)로 **단건** 조회 | 가장 쌈, O(1) |
| `Query` | **하나의 PK** 안에서 SK 조건으로 다건 | 효율적 — 읽은 만큼만 과금 |
| `Scan` | **테이블 전체** 순회 후 필터 | 운영에서 금지 — 테이블 전체를 읽고 과금 |

핵심: `Query`는 **반드시 PK를 등호로 고정**해야 한다. PK 없이 "조건에 맞는 아이템 다 줘"는 Scan뿐이다. 그래서 "PK 없이 조회하고 싶은 패턴"이 생기면 → 그게 바로 **보조 인덱스(GSI/LSI)** 가 필요한 신호다.

### 복합키 쿼리 패턴 (SK 조건)

```
PK = "USER#123" 고정하고 SK에:
  = "ORDER#A1"                         → 정확 매칭
  begins_with(SK, "ORDER#")            → 이 유저의 주문 전부
  begins_with(SK, "ORDER#2026-06")     → 6월 주문만 (계층적 SK의 힘)
  between "ORDER#2026-01" and "..."    → 기간 범위
  > "ORDER#2026-06-15"                 → 그 이후
  + ScanIndexForward=false             → 역순(최신순) 정렬
```

> **실무 팁** — SK를 `ORDER#{날짜}#{주문id}`처럼 **굵은 분류 → 세밀한 분류** 순으로 계층화하면, `begins_with`로 "연도 → 월 → 일" 어느 깊이로든 잘라 조회할 수 있다. 날짜는 정렬되도록 ISO 8601(`2026-06-30`)을 쓴다. `6/30` 같은 포맷은 사전식 정렬이 깨진다.

## 3. 싱글테이블 디자인 — 액세스 패턴 먼저

DynamoDB 모델링은 RDB와 순서가 **정반대**다.

```
RDB         : 엔티티 도출 → 정규화 → 테이블 N개 → 쿼리는 JOIN으로 그때그때
DynamoDB    : 액세스 패턴 목록 작성 → 그걸 다 만족하는 키 설계 → 테이블 최소화(보통 1개)
```

### Step 1. 액세스 패턴을 먼저 다 적는다

예: 커머스 도메인.

1. 유저 프로필 조회
2. 유저의 주문 목록(최신순)
3. 주문 단건 조회
4. 특정 상태(PAID/PENDING)의 주문을 최신순으로 (운영자 화면)
5. 상품 정보 + 그 상품 리뷰 목록

### Step 2. 한 테이블에 여러 엔티티를 — PK/SK 오버로딩

PK/SK 속성명을 **엔티티 중립적**(`PK`, `SK`)으로 두고, 값에 `USER#`, `ORDER#`, `PRODUCT#` 같은 **타입 접두사**를 붙여 한 테이블에 섞는다. 이걸 **키 오버로딩**이라 한다.

```
PK              | SK                       | 속성...
----------------|--------------------------|---------------------------------
USER#123        | PROFILE                  | name=김개발, email=...
USER#123        | ORDER#2026-06-01#A1      | amount=50000, status=PAID
USER#123        | ORDER#2026-06-15#A2      | amount=30000, status=PENDING
PRODUCT#P100    | META                     | name=노트북, price=...
PRODUCT#P100    | REVIEW#2026-06-20#R1     | rating=5, text=...
```

이렇게 두면:

- 패턴 ①②: `Query(PK="USER#123")` 한 방에 **프로필 + 주문 전부**가 한 아이템 컬렉션으로 떨어진다. → **JOIN 없이 한 번의 왕복**.
- 패턴 ②만: `Query(PK="USER#123", SK begins_with "ORDER#", ScanIndexForward=false)`.
- 패턴 ③: `GetItem(PK="USER#123", SK="ORDER#2026-06-01#A1")`.
- 패턴 ⑤: `Query(PK="PRODUCT#P100")` → 상품 메타 + 리뷰 함께.

남은 건 패턴 ④("상태별 주문 조회") — PK가 `USER#`라 상태로는 못 찾는다. 여기서 **GSI**가 등장한다(다음 섹션).

> **함정** — 싱글테이블은 강력하지만 인지 비용이 높다. 한 테이블에 이질적인 아이템이 섞이고 속성명이 추상적이라, 팀이 작거나 패턴이 단순하면 **엔티티별 테이블 몇 개**로 나누는 게 유지보수에 낫다. "테이블 1개"는 목표가 아니라 **왕복 횟수·비용 최소화의 수단**이다. 맹신 금지.

## 4. 보조 인덱스 — GSI vs LSI

PK 외의 속성으로 조회하려면 보조 인덱스를 만든다. 둘의 차이가 면접 단골이다.

| 항목 | **GSI** (Global Secondary Index) | **LSI** (Local Secondary Index) |
|---|---|---|
| 키 | PK·SK **둘 다 새로 지정 가능** | **PK는 베이스와 동일**, SK만 다름 |
| 생성 시점 | **언제든 추가/삭제** | **테이블 생성 시에만**, 이후 추가 불가 |
| 일관성 | **결과적 일관성만** (강한 일관성 불가) | 강한 일관성 선택 가능 |
| 용량(프로비전드) | 테이블과 **독립**된 RCU/WCU | 베이스 테이블 용량을 공유 |
| 제약 | 테이블당 기본 20개(조정 가능) | 테이블당 5개, **파티션 키 값당 10GB 제한** |
| 비유 | "키를 갈아끼운 가상 테이블" | "같은 파티션에 정렬축 하나 더" |

### GSI를 어떻게 쓰나 (패턴 ④ 해결)

상태별 조회를 위해 GSI를 얹는다. 별도 속성 `GSI1PK`, `GSI1SK`를 두고, 인덱싱하고 싶은 아이템에만 채운다.

```
베이스                                        GSI1 (GSI1PK / GSI1SK)
PK=USER#123  SK=ORDER#...#A1  status=PAID   →  GSI1PK=STATUS#PAID    GSI1SK=2026-06-01#A1
PK=USER#123  SK=ORDER#...#A2  status=PENDING→  GSI1PK=STATUS#PENDING GSI1SK=2026-06-15#A2

운영자: Query(GSI1, GSI1PK="STATUS#PENDING", 최신순) → 결제 대기 주문 전부
```

GSI의 세 가지 핵심 동작:

1. **결과적 일관성** — 베이스에 쓰면 GSI 반영까지 보통 ms 단위 지연. "방금 쓴 걸 GSI로 즉시 읽기"는 보장되지 않는다.
2. **독립 용량** — 프로비전드 모드에서 GSI는 자기 RCU/WCU를 따로 가진다. **GSI 쓰기 용량이 부족하면 백프레셔로 베이스 테이블 쓰기까지 throttle된다** — GSI를 좁게(프로젝션 최소) 잡아야 하는 이유.
3. **프로젝션(projection)** — GSI에 복사할 속성을 고른다(`KEYS_ONLY` / `INCLUDE` / `ALL`). `ALL`은 편하지만 저장·쓰기 비용이 곱으로 든다. 필요한 속성만 프로젝션하는 게 정석.

### 고급 패턴 4종

```
1) Sparse index — GSI 키 속성을 "해당하는 아이템에만" 채운다.
   예: 미처리 주문에만 GSI1PK="UNPROCESSED" 세팅, 처리되면 그 속성 삭제.
   → GSI엔 미처리 건만 존재 = 작고 빠른 "작업 큐" 인덱스.

2) Index overloading — 하나의 GSI(GSI1PK/GSI1SK)를 여러 엔티티 타입이 공유.
   USER는 GSI1PK=EMAIL#..., ORDER는 GSI1PK=STATUS#... → 인덱스 1개로 여러 조회.

3) Adjacency list (M:N) — 관계를 아이템으로 표현 + GSI 키 반전(inverted index).
   PK=USER#1  SK=GROUP#A  /  PK=USER#2  SK=GROUP#A
   GSI: GSI1PK=SK, GSI1SK=PK  → Query(GSI1PK="GROUP#A")로 "그룹 A의 멤버 전부".
   베이스로는 "유저의 그룹", 반전 GSI로는 "그룹의 유저" — 양방향 M:N 한 테이블에서.

4) 큰 아이템은 분리 — 아이템은 400KB 한계. 큰 blob은 S3에 두고
   테이블엔 메타+S3 키만 저장(읽기 비용·아이템 크기 절감).
```

## 5. 용량 모드 — On-demand vs Provisioned

처리량 비용을 어떻게 낼지 두 모드 중 고른다.

| | **On-demand** (기본·대개 권장) | **Provisioned** |
|---|---|---|
| 과금 | **요청한 만큼** (RRU/WRU) | 미리 설정한 **RCU/WCU**(소비량 아닌 설정량) |
| 용량 계획 | 불필요 | 필요(피크 예측) |
| 스파이크 | 급증을 **즉시 수용** | 설정+오토스케일 한도까지 |
| 적합 | 트래픽 불규칙·신규·서버리스 | 예측 가능·꾸준한 워크로드, 비용 최적화 |
| 비용 성격 | 변동(쓴 만큼) | 예측 가능(켜둔 만큼) |

> **권장 흐름** — 패턴을 모르는 초기엔 **On-demand**로 시작(throttle 걱정 없음·계획 불필요). CloudWatch로 실제 소비를 본 뒤, 트래픽이 꾸준하고 비용이 커지면 **Provisioned + Auto Scaling**으로 전환해 단가를 낮춘다.

### RCU / WCU 정의

용량 단위는 **아이템 크기**와 **일관성**으로 환산된다:

```
1 RCU = 4KB까지 "강한 일관성 읽기" 1회/초
      = 4KB까지 "결과적 일관성 읽기" 2회/초   (그래서 결과적 일관성이 읽기 비용 절반)
1 WCU = 1KB까지 쓰기 1회/초

On-demand는 같은 환산을 RRU(읽기요청단위)/WRU(쓰기요청단위)로 부른다.

예: 8KB 아이템을 강한 일관성으로 초당 10번 읽기
   → 아이템당 2 RCU(8KB=4KB×2) × 10 = 20 RCU 필요
```

**트랜잭션은 용량을 2배 소비한다(트랜잭션 읽기 2 RCU, 쓰기 2 WCU).** 내부적으로 prepare·commit 두 단계로 처리되기 때문이다.

## 6. 핫 파티션과 처리량 한계 (실무 핵심)

DynamoDB가 느려지거나 throttle 나는 사고의 8할은 **핫 파티션**이다.

### 물리 한계

```
단일 파티션이 감당하는 최대치:  약 3,000 RCU / 1,000 WCU
(그리고 한 파티션의 물리 크기는 약 10GB)
```

테이블 전체 용량이 충분해도, **트래픽이 한 PK(=한 파티션)에 쏠리면** 그 파티션의 한계(1,000 WCU 등)에 먼저 부딪혀 `ProvisionedThroughputExceededException`(throttle)이 난다. "용량은 남는데 왜 throttle?"의 정답이 이것.

### Adaptive capacity — AWS가 먼저 막아준다

DynamoDB는 핫 파티션을 감지하면 **처리량을 그 파티션으로 자동 부스트**한다(adaptive capacity). 모든 테이블에 **자동·무료**로 켜져 있고, 가벼운 불균형은 대부분 알아서 흡수한다.

단, 한계가 있다 — 부스트는 **테이블 총 용량과 파티션 물리 한도(3,000/1,000) 안에서만** 가능하다. 즉 **단일 키에 대한 절대 한도(파티션당 1,000 WCU)는 못 넘는다**. 그래서 진짜 해결책은 인프라가 아니라 **데이터 모델링**이다.

### 핫 파티션 증상 → 원인 → 해결

| 증상 | 원인 | 해결 |
|---|---|---|
| 일부 키만 throttle, 전체 용량은 남음 | 저카디널리티 PK / 특정 키 쏠림 | **PK 카디널리티 ↑**, write sharding |
| 인기 게시물·핫 상품 집중 read | 단일 인기 아이템 폭주 | **앞단 캐시(DAX/ElastiCache)**, 읽기 분산 |
| 쓰기 throttle인데 베이스는 한가 | **GSI 용량 부족 → 백프레셔** | GSI 용량↑ / 프로젝션 축소 / sparse화 |
| 시계열 쓰기가 오늘 파티션에 몰림 | SK는 분산돼도 PK가 날짜 1개 | PK에 샤드 접미사 추가 |

### Write sharding — 쏠린 키를 강제로 흩뿌리기

핫한 논리 키를 `키#0..#N-1`로 쪼개 여러 파티션에 분산한다.

```
나쁨:  PK = "STATUS#OPEN"          → 모든 OPEN 주문이 한 파티션, 쓰기 폭주
좋음:  PK = "STATUS#OPEN#{0..9}"   → 쓰기 시 (id 해시 % 10)로 샤드 선택 → 10배 분산

읽을 때(scatter-gather):
  for s in 0..9:  Query(PK="STATUS#OPEN#"+s)
  → 10개 결과를 앱에서 merge (정렬 필요하면 merge-sort)
```

트레이드오프가 명확하다 — **쓰기는 N배 분산되지만, 전체 읽기는 샤드 N개를 모두 쿼리해 합쳐야 한다(읽기 fan-out↑).** 샤드 수 N은 "쓰기 분산 이득 vs 읽기 fan-out 비용"의 균형점으로 잡는다. 무조건 크게 잡으면 읽기가 비싸진다.

## 7. Streams와 Transactions

### DynamoDB Streams — 변경 캡처 → 이벤트

테이블의 **아이템 레벨 변경(생성·수정·삭제)** 을 시간순 로그로 캡처해, **Lambda를 자동 트리거**할 수 있다. 사실상 DynamoDB판 [CDC](./CDC.md)다.

```
DynamoDB write ──▶ Stream(변경 레코드) ──▶ Lambda
                                            ├─ 검색엔진(OpenSearch) 동기화
                                            ├─ 집계 테이블 갱신 (서버측 집계 부재 보완)
                                            ├─ 알림/후속 워크플로우
                                            └─ S3 데이터레이크로 적재 (분석 파이프라인 진입)
```

스트림 레코드는 변경 전/후 이미지(`OLD_IMAGE`/`NEW_IMAGE`)를 담을 수 있어, "무엇이 어떻게 바뀌었나"를 그대로 후속 시스템에 흘려보낸다. 이게 DynamoDB를 **이벤트 기반 아키텍처의 원천**으로 만든다.

### Transactions — 키-값 위의 ACID

`TransactWriteItems` / `TransactGetItems`로 **여러 아이템을 all-or-nothing**으로 묶는다(조건 위반 시 전체 롤백). "주문 생성 + 재고 차감 + 잔액 차감을 한 묶음으로" 같은 케이스에 쓴다.

제약: 한 트랜잭션에 **최대 100개 아이템**, 트랜잭션 연산은 **용량을 2배 소비**한다. 그래서 트랜잭션은 "꼭 원자성이 필요한 곳"에만 쓰고, 단건으로 충분한 곳엔 일반 write를 쓴다.

## 8. 실전: 한눈에 보는 설계 흐름

```
① 액세스 패턴 나열        ② 키 설계(싱글테이블)        ③ 못 푸는 패턴 → GSI
   - 유저 주문 목록          PK=USER#id                   상태별 조회:
   - 주문 단건              SK=ORDER#날짜#주문id            GSI1PK=STATUS#상태
   - 상태별 주문(운영)       (오버로딩으로 PROFILE도 동거)    GSI1SK=날짜#주문id
                                                          (sparse: 관심 상태만 세팅)

④ 핫키 점검              ⑤ 용량 모드                  ⑥ 변경 후속 처리
   인기 상태/날짜 쏠림?       초기 On-demand →             Streams → Lambda →
   → write sharding         꾸준하면 Provisioned          집계테이블·검색·S3
   PK에 #0..#N 접미사        +Auto Scaling
```

설계는 위에서 아래로 한 번에 끝나지 않는다 — ①을 빠뜨리면 ②가 틀어지고, ④를 빠뜨리면 운영에서 throttle로 되돌아온다. **"키는 코드보다 바꾸기 어렵다"** 는 게 DynamoDB 설계의 무게중심이다.

## 9. 자주 하는 실수 (면접에서 꼬리질문 나오는 지점)

- **Scan을 운영 쿼리로 쓴다** — 테이블 전체를 읽고 과금. 패턴이 Scan을 부르면 인덱스 설계가 잘못된 것.
- **PK 카디널리티가 낮다** — `STATUS`, `날짜`, `테넌트id` 단독 PK는 핫파티션 직행. 합성키/샤딩 필요.
- **GSI 남발 + `ALL` 프로젝션** — 쓰기마다 모든 GSI 갱신 + 저장 곱. 백프레셔로 베이스까지 throttle.
- **"방금 쓴 걸 GSI로 즉시 읽으려 함"** — GSI는 결과적 일관성. 직후 읽기는 베이스 테이블로.
- **나중에 LSI 추가하려 함** — LSI는 **테이블 생성 시에만**. 못 바꾼다(이후엔 GSI로).
- **트랜잭션 남용** — 용량 2배·아이템 100개 한계. 원자성이 진짜 필요한 곳에만.

## 면접에서 이렇게 답하자

> "DynamoDB는 **키 기반 조회가 메인이고 규모와 무관하게 한 자릿수 ms가 필요할 때** 씁니다. 설계는 RDB와 반대로 **액세스 패턴을 먼저 다 적고, 그걸 만족하는 PK/SK를 설계**합니다. PK는 어느 파티션에 갈지 정하니 **고카디널리티**여야 하고, SK는 정렬·범위 쿼리축이 됩니다.
>
> 여러 엔티티는 **싱글테이블에 키 오버로딩**으로 담아 한 번의 Query로 관련 데이터를 가져와 JOIN을 없앱니다. PK로 못 푸는 조회는 **GSI**로 푸는데, GSI는 결과적 일관성에 독립 용량이라 프로젝션을 좁게 잡고 sparse하게 씁니다. LSI는 PK가 같고 테이블 생성 시에만 만들 수 있어 보통 GSI를 선호합니다.
>
> 운영 사고의 대부분은 **핫 파티션**입니다. 단일 파티션 한계(약 3,000 RCU/1,000 WCU)를 한 키에 몰면 전체 용량이 남아도 throttle 납니다. Adaptive capacity가 어느 정도 자동으로 막아주지만 근본 해결은 모델링 — **write sharding으로 키를 흩뿌리고, 읽기 fan-out과의 트레이드오프를 잡습니다.** 변경 후속 처리는 **Streams로 Lambda를 트리거**해 검색 동기화·집계·데이터레이크 적재로 잇고, 원자성이 필요한 곳만 **Transactions**를 씁니다."
