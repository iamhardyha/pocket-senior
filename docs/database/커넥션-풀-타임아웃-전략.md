# 커넥션 풀 타임아웃 전략

> **핵심 질문**: 커넥션 풀의 maxIdleTime과 maxLifetime을 왜 무한대로 설정하면 안 되는가? 커넥션은 계속 유지하는 게 좋지 않은가?

---

## 한 줄 요약

**"커넥션은 TCP 위의 '살아 있는 자원'이다. 오래 유지할수록 네트워크 장비, DB, OS 레벨에서 몰래 썩는다."**

---

## 핵심 개념 정리

```
maxIdleTime  = 사용되지 않는 커넥션을 풀에 유지하는 최대 시간
maxLifetime  = 커넥션이 생성된 시점부터 존재할 수 있는 최대 시간
```

```
[커넥션 생명주기]

생성 ──▶ 사용 ──▶ 반납 ──▶ 유휴(Idle) ──▶ 사용 ──▶ 반납 ──▶ 유휴 ...
│                                                              │
│◀──────────────── maxLifetime ──────────────────────────────▶│
                          │◀── maxIdleTime ──▶│
                          (마지막 사용 후 경과)
```

---

## 왜 무한대가 위험한가?

직관적으로는 "커넥션을 한 번 만들어서 계속 쓰면 되지 않나?"라고 생각할 수 있다. 하지만 커넥션은 단순한 논리적 객체가 아니라 **TCP 소켓 + DB 세션 + 중간 네트워크 장비 상태**의 조합이다. 이 중 하나라도 문제가 생기면 커넥션은 "겉으로는 살아 있지만 실제로는 죽은" 상태가 된다.

### 1. 중간 장비의 유휴 커넥션 강제 종료 (가장 흔한 실무 장애)

```
App Server ──TCP──▶ [방화벽/NAT/LB] ──TCP──▶ DB Server

방화벽의 idle timeout = 30분 (기본값)
```

대부분의 방화벽, NAT 장비, 클라우드 로드밸런서는 **유휴 TCP 연결을 일정 시간 후 자동으로 끊는다.** 문제는 이 종료가 **한쪽에만 통보**되거나 **양쪽 모두 통보되지 않는다**는 것이다.

```
[장애 시나리오]

1. 커넥션이 30분 동안 미사용
2. 방화벽이 NAT 테이블에서 해당 세션 제거 (양쪽에 FIN/RST 안 보냄)
3. App은 커넥션이 살아있다고 믿음 (TCP keepalive 미설정 시)
4. 요청이 들어와서 해당 커넥션 사용 시도
5. 패킷이 방화벽에서 DROP → 응답 없음 → TCP retransmission 시작
6. 수십 초~수 분간 타임아웃 대기 후 실패
7. 사용자에게는 "서비스 응답 없음"
```

| 환경 | 기본 유휴 타임아웃 | 비고 |
|------|-------------------|------|
| AWS NLB | 350초 | 고정, 사용자 변경 불가 |
| AWS ALB | 60초 | 1~4000초 범위에서 설정 가능 |
| AWS NAT Gateway | 350초 | 고정 |
| Linux conntrack (`nf_conntrack_tcp_timeout_established`) | 5일 (432000초) | 클라우드에서는 VPC 레이어가 먼저 적용 |
| 일반 방화벽 | 30분~1시간 | 장비/정책에 따라 상이 |

**maxIdleTime을 무한대로 놓으면:** 방화벽이 이미 끊은 커넥션을 풀이 계속 보유하고 있다가, 요청이 들어올 때 죽은 커넥션을 할당하여 장애 발생.

### 2. DB의 커넥션 강제 정리

MySQL과 PostgreSQL 모두 오래된 커넥션을 서버 측에서 정리하는 메커니즘이 있다.

```sql
-- MySQL: 기본 8시간
SHOW VARIABLES LIKE 'wait_timeout';        -- 28800 (초) — JDBC 커넥션은 이 값이 적용됨
SHOW VARIABLES LIKE 'interactive_timeout'; -- 28800 (초) — mysql CLI 등 CLIENT_INTERACTIVE 플래그 사용 시에만 적용

-- PostgreSQL
-- postgresql.conf
idle_session_timeout = '30min'                      -- PG 14+ : 일반 유휴 커넥션 정리
idle_in_transaction_session_timeout = '10min'        -- BEGIN 후 트랜잭션 내 유휴 상태에만 적용 (일반 유휴와 다름)
```

```
[장애 시나리오]

1. MySQL wait_timeout = 8시간
2. 야간에 트래픽이 없어 커넥션이 8시간 이상 유휴
3. MySQL이 서버 측에서 커넥션 종료
4. App의 커넥션 풀은 여전히 해당 커넥션을 "유효"로 보유
5. 출근 시간 첫 요청 → "Connection is closed" 또는 "Communications link failure"
```

**maxLifetime을 무한대로 놓으면:** DB가 끊은 줄 모르고 죽은 커넥션을 반복적으로 할당.

### 3. 메모리 누수와 세션 오염

DB 커넥션은 **서버 측에 상태(State)를 가진다:**

```
커넥션 하나가 DB 서버에서 차지하는 자원:
├── 세션 메모리 (sort_buffer, join_buffer, read_buffer 등)
├── 임시 테이블
├── 준비된 Statement 캐시
├── 트랜잭션 로그 참조 (long-running tx 문제)
└── 세션 변수 (SET 명령으로 변경된 값)
```

```java
// 위험한 시나리오: 세션 변수 오염
connection.execute("SET SESSION transaction_isolation = 'READ-UNCOMMITTED'");
// ... 처리 후 반납

// 나중에 다른 요청이 같은 커넥션을 받으면?
// → READ-UNCOMMITTED가 그대로 적용됨!
```

커넥션을 주기적으로 재생성하면 **DB 세션이 클린 상태로 초기화**된다. maxLifetime은 이 오염을 자동으로 정리하는 안전장치다.

### 4. DB 인프라 변경 시 반영 불가

```
[시나리오: DB 페일오버]

1. Primary DB (10.0.1.100) → 장애 발생
2. DNS가 Replica (10.0.1.200)로 변경
3. 기존 커넥션은 이미 10.0.1.100에 TCP로 직접 연결되어 있음
4. DNS 변경이 반영 안 됨 — 계속 죽은 Primary에 연결 시도
```

maxLifetime이 있으면 **오래된 커넥션이 자연스럽게 폐기되고 새 커넥션이 생성**되면서 새 DNS가 반영된다. Aurora, RDS Multi-AZ 페일오버에서 특히 중요하다.

### 5. OS 레벨 자원 고갈

```bash
# 커넥션 하나당 OS 자원
- 파일 디스크립터: 1개
- TCP 소켓 메모리: ~4-16KB (send/recv buffer)
- DB 서버 프로세스/스레드: 1개

# 유휴 커넥션 100개 × 서버 10대 = DB에 1,000개 세션
# maxIdleTime이 없으면 트래픽이 줄어도 절대 반환되지 않음
```

## HikariCP 권장 설정

HikariCP는 Spring Boot의 기본 커넥션 풀이며, 이 문제에 대해 명확한 가이드라인을 제시한다.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10           # 최대 커넥션 수
      minimum-idle: 5                 # 최소 유휴 커넥션
      max-lifetime: 1800000           # 30분 (ms) — DB의 wait_timeout보다 짧게
      idle-timeout: 600000            # 10분 (ms) — 방화벽 idle timeout보다 짧게
      connection-timeout: 3000        # 3초 — 커넥션 획득 대기 시간
      validation-timeout: 3000        # 3초 — 유효성 검사 타임아웃
      keepalive-time: 300000          # 5분 — TCP keepalive (HikariCP 4.0+)
      connection-test-query: SELECT 1 # 유효성 검사 쿼리 (JDBC4 드라이버면 불필요)
```

### 핵심 원칙

```
maxLifetime < DB의 wait_timeout

예) MySQL wait_timeout = 28800초 (8시간)
    → HikariCP maxLifetime = 1800초 (30분) 이하 권장

idleTimeout < 방화벽/NAT의 idle timeout

예) AWS NLB idle timeout = 350초
    → HikariCP idleTimeout = 300초 이하 권장
```

### HikariCP의 maxLifetime 동작 방식

```
[커넥션 10개의 maxLifetime = 30분일 때]

HikariCP는 maxLifetime에서 최대 500ms를 랜덤으로 차감 (Jitter)
→ 모든 커넥션이 동시에 만료되는 것을 방지

conn1: 1800000ms - 127ms = 1799873ms 후 만료
conn2: 1800000ms - 489ms = 1799511ms 후 만료
conn3: 1800000ms - 312ms = 1799688ms 후 만료
...

→ 편차가 500ms 이내로 작지만, 커넥션 재생성이 미세하게 분산됨
```

이것이 HikariCP가 maxLifetime을 **반드시 설정하라**고 강조하는 이유다. 0(무한대)은 명시적으로 비권장한다.

## 커넥션 유효성 검사 전략

"그러면 사용 전에 커넥션이 살아있는지 확인하면 되지 않나?"라는 의문이 나올 수 있다.

| 전략 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **Borrow 시 검증** (DBCP2: `testOnBorrow`) | 풀에서 꺼낼 때마다 `SELECT 1` | 죽은 커넥션 100% 방지 | 매 요청마다 추가 쿼리 → 성능 저하 |
| **유휴 중 주기 검사** (DBCP2: `testWhileIdle`, HikariCP: `keepaliveTime`) | 유휴 중 주기적으로 검사 | 성능 영향 적음 | 검사 사이 죽을 수 있음 |
| **시간 기반 폐기** (`maxLifetime`) | 일정 시간 후 자동 폐기 | 가장 깔끔, 오버헤드 없음 | 아직 유효한 커넥션도 버릴 수 있음 |

**실무에서는 maxLifetime + keepaliveTime 조합이 최적이다.** HikariCP에는 `testOnBorrow` 개념이 별도로 없으며, JDBC4 드라이버의 `isValid()` 메서드로 커넥션 획득 시 자동 검증한다.

```
[추천 조합]
maxLifetime = 30분       ← 세션 오염, DNS 변경 대응
idleTimeout = 10분       ← 유휴 커넥션 반환
keepaliveTime = 5분      ← 유휴 중 생존 확인 (SELECT 1)
```

## 실무 장애 사례

### 사례 1: AWS 환경에서 월요일 아침마다 장애

```
원인: 주말 트래픽 감소 → 커넥션 수시간 유휴
     → NLB가 350초 후 TCP 세션 제거
     → App은 모름 → 월요일 첫 요청에 dead connection 할당

해결: idleTimeout = 300초 (NLB 350초보다 짧게)
     keepaliveTime = 120초 (주기적 생존 확인)
```

### 사례 2: Aurora 페일오버 후 5분간 오류

```
원인: maxLifetime = 무한대
     → 기존 커넥션이 이전 Primary의 IP에 TCP로 직접 연결
     → DNS 변경이 반영 안 됨 → 새 Primary로 전환 불가
     (Aurora Cluster Endpoint DNS TTL은 5초로 짧지만,
      JVM은 기본적으로 DNS를 영구 캐싱하여 TTL을 무시함)

해결:
  1. maxLifetime = 900초 (15분)
     → 최대 15분 내에 모든 커넥션이 새 엔드포인트로 재연결
  2. JVM DNS 캐싱 비활성화 (함께 설정 필수)
     -Dsun.net.inetaddr.ttl=0
     또는 java.security의 networkaddress.cache.ttl=0
```

### 사례 3: 메모리 누수 — DB 서버 OOM

```
원인: 오래된 커넥션이 Prepared Statement 캐시를 무한히 누적
     App 서버 20대 × 커넥션 20개 = 400개 세션
     각 세션이 수백 개의 Prepared Statement 보유

해결:
  1. 근본 해결: JDBC 드라이버 레벨에서 캐시 크기 제한
     - MySQL: prepStmtCacheSize=250 (기본)
     - PostgreSQL: preparedStatementCacheQueries=256 (기본)
  2. 안전장치: maxLifetime = 1800초 (30분)
     → 주기적으로 세션이 초기화되어 메모리 반환
```

## 면접에서 이렇게 답하자

### Q. 커넥션 풀의 maxIdleTime과 maxLifetime을 왜 무한대로 두면 안 되나요?

> "커넥션은 TCP 소켓 + DB 세션 + 중간 네트워크 장비 상태의 조합입니다. 이 세 가지 중 하나라도 상태가 변하면 커넥션은 겉으로는 살아있지만 실제로는 사용 불가능한 'half-open' 상태가 됩니다.
>
> 대표적으로 방화벽이나 NAT 장비가 유휴 TCP 세션을 양쪽 통보 없이 제거하면, App은 죽은 커넥션을 할당해서 수십 초간 타임아웃을 대기합니다. MySQL의 wait_timeout이 만료되어도 같은 문제가 생깁니다.
>
> 또한 maxLifetime이 없으면 Aurora 페일오버 시 DNS 변경이 반영되지 않고, DB 세션의 메모리 누적이나 세션 변수 오염 같은 문제도 자연 정리되지 않습니다.
>
> HikariCP에서는 maxLifetime을 DB의 wait_timeout보다 짧게, idleTimeout을 방화벽의 idle timeout보다 짧게 설정하는 것이 원칙입니다."

### Q. 그러면 매번 커넥션을 새로 만드는 게 나은 건가요?

> "아닙니다. 커넥션 생성은 TCP 3-way handshake + TLS 협상 + DB 인증까지 수 ms~수십 ms가 걸리는 비싼 작업입니다. 커넥션 풀의 목적은 이 비용을 줄이면서도 적절한 주기로 갱신하여 건강한 커넥션만 유지하는 것입니다. maxLifetime 30분이면 30분간 수천 건의 요청을 재사용하면서, 30분마다 한 번씩만 재생성 비용을 지불합니다."
