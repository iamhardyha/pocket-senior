# Redshift — 데이터 웨어하우스

> **핵심 질문**: Redshift는 왜 빠르고, 분산키·정렬키는 뭐고, Athena 두고 언제 Redshift를 쓰지?

---

## 한 줄 요약

**"Redshift는 컬럼형 저장 + MPP(여러 노드 병렬)로 대규모 집계·조인을 빠르게 처리하는 데이터 웨어하우스다. 빠름의 핵심은 '읽을 데이터를 어떻게 줄이고(컬럼형·정렬키), 노드 간 데이터 이동을 어떻게 없애느냐(분산키)'에 달려 있다. 반복되는 BI·복잡 조인·많은 동시 사용자엔 Redshift, 가끔 던지는 애드혹 SQL엔 Athena."**

---

## 1. 왜 빠른가 — 컬럼형 + MPP

Redshift가 빠른 이유는 두 축이다: **읽을 양을 줄이는 컬럼형 저장**, 그리고 **그 일을 여러 노드가 동시에 하는 MPP(Massively Parallel Processing)**.

### 클러스터 구조: 리더 노드 + 컴퓨트 노드 + 슬라이스

```
                  클라이언트 (BI 툴 / psql)
                          │  (리더 노드와만 통신)
                          ▼
        ┌──────────────────────────────────────┐
        │           리더 노드 (1개)              │
        │  · 쿼리 파싱 · 실행 계획 수립           │
        │  · 코드 컴파일 → 컴퓨트 노드로 배포      │
        │  · 각 노드 부분 결과 → 최종 집계         │
        └──────────────────────────────────────┘
              │            │            │
     ┌────────┘     ┌──────┘     └────────┐
     ▼              ▼                      ▼
 ┌─────────┐   ┌─────────┐           ┌─────────┐
 │컴퓨트노드1│   │컴퓨트노드2│   ...     │컴퓨트노드N│   각자 CPU·메모리·디스크
 │ slice 0 │   │ slice 2 │           │ slice 4 │
 │ slice 1 │   │ slice 3 │           │ slice 5 │   슬라이스 = 병렬 처리 단위
 └─────────┘   └─────────┘           └─────────┘
```

- **리더 노드(1개)**: 클라이언트와 통신하는 유일한 창구. 쿼리를 파싱해 실행 계획을 세우고, **컴파일된 코드 조각을 컴퓨트 노드에 배포**한 뒤, 각 노드가 돌려준 부분 결과를 **최종 집계**한다. 데이터를 직접 들고 있지 않다.
- **컴퓨트 노드(N개)**: 각자 독립된 CPU·메모리·디스크를 가지고 실제 데이터를 저장·연산한다.
- **슬라이스(slice)**: 각 컴퓨트 노드는 다시 여러 슬라이스로 쪼개진다. **슬라이스가 진짜 병렬 처리의 최소 단위** — 노드 메모리·디스크를 슬라이스 수만큼 나눠 갖고, 각 슬라이스가 자기 몫의 데이터를 동시에 처리한다. 즉 "노드 4개 × 슬라이스 2개 = 8갈래 병렬"이 된다.

### 컬럼형 저장이 집계에 강한 이유

행 기반(OLTP, MySQL 등) vs 컬럼형(Redshift)을 `SELECT SUM(amount) FROM orders WHERE created_at >= ...` 같은 집계로 비교하면 차이가 분명하다.

| | 행 기반 저장 | 컬럼형 저장 (Redshift) |
|---|---|---|
| 디스크 배치 | 한 행의 모든 컬럼이 붙어 있음 | **같은 컬럼 값끼리** 붙어 있음 |
| 위 집계가 읽는 양 | `amount`만 필요해도 **모든 컬럼이 섞인 블록**을 읽음 | `amount`·`created_at` **컬럼 블록만** 읽음 → I/O 대폭 감소 |
| 압축 | 한 블록에 타입이 뒤섞여 압축 효율 낮음 | 같은 타입·비슷한 값이 모여 **압축률 높음** → 읽을 바이트 자체가 적음 |

여기에 **MPP**가 곱해진다. 읽어야 할 컬럼 블록을 N개 노드 × 슬라이스가 동시에 스캔·부분합산하고, 리더 노드가 마지막에 합친다. **"읽을 컬럼만(컬럼형) × 압축으로 더 적게 × 여러 슬라이스가 동시에(MPP)"** — 이 셋이 곱해져 대규모 집계가 빨라진다. OLTP DB가 단건 행 조회에 강한 것과 정반대 방향의 최적화다.

> 함정: Redshift는 **단건 행 조회·잦은 소량 UPDATE/DELETE에는 오히려 불리**하다. 컬럼형은 한 행을 고치려고 여러 컬럼 블록을 건드려야 하고, MPP·압축의 이점도 안 나온다. "분석용 웨어하우스지 운영 DB가 아니다"가 핵심.

## 2. 분산 스타일(DISTSTYLE) — 노드 간 데이터 이동을 줄인다

데이터가 여러 노드에 흩어져 있으면, **조인할 두 행이 다른 노드에 있을 때 한쪽을 네트워크로 옮겨야** 한다. 이 이동을 **재분배(redistribution)**라 부르고, 큰 쿼리에선 비용의 가장 큰 덩어리가 된다. DISTSTYLE은 "행을 슬라이스에 어떻게 흩뿌릴지"를 정해 이 이동을 줄이는 설계다.

### 재분배가 비싼 이유

A 테이블과 B 테이블을 `user_id`로 조인한다고 하자. 두 테이블이 서로 다른 기준으로 흩어져 있으면, Redshift는 조인 전에:

- **재분배(redistribute)**: 한쪽(또는 양쪽) 테이블 행을 조인 키 기준으로 **노드 간에 다시 뿌린다** → 대량 네트워크 전송, 임시 디스크 사용.
- **브로드캐스트(broadcast)**: 작은 쪽 테이블 전체를 **모든 노드에 복사**한다 → 작으면 싸지만 크면 폭발.

실행 계획(`EXPLAIN`)에서 이 비용이 그대로 보인다:

| EXPLAIN 표시 | 의미 | 비용 |
|---|---|---|
| `DS_DIST_NONE` | 이동 없음 — 조인 행이 이미 같은 슬라이스에 모임 | 최선 |
| `DS_DIST_INNER` / `DS_DIST_OUTER` | 한쪽 테이블을 조인 키로 재분배 | 중 |
| `DS_BCAST_INNER` | 안쪽 테이블 전체를 전 노드에 브로드캐스트 | 작으면 OK |
| `DS_DIST_BOTH` | 양쪽 다 재분배 | 최악 |

**분산키(DISTKEY)로 조인 키를 맞춰두면** 매칭될 행이 처음부터 같은 슬라이스에 모여 있어(**collocation, 코로케이션**) 재분배가 사라진다(`DS_DIST_NONE`). 이것이 분산 설계의 핵심 목표다.

### 네 가지 스타일

| DISTSTYLE | 동작 | 언제 |
|---|---|---|
| **KEY** | `DISTKEY` 컬럼 값을 **해시**해서 슬라이스 배치. 같은 키 값은 같은 슬라이스로 | 조인·`GROUP BY`가 잦은 **큰 팩트 테이블**. 조인 키를 DISTKEY로 → 코로케이션으로 재분배 제거 |
| **ALL** | 테이블 **전체를 모든 노드에 복제** | **작고(가이드: 5백만 행 미만) 잘 안 바뀌는 디멘션** 테이블. 어느 노드에서 조인하든 옆에 있으니 이동 0. 단 저장공간 N배·쓰기 비용↑ |
| **EVEN** | 라운드로빈으로 고르게 분산 (조인 키 무관) | 조인을 거의 안 하고 **스캔 위주**인 큰 테이블, 마땅한 분산키가 없을 때 |
| **AUTO**(기본) | Redshift가 테이블 크기·사용 패턴을 보고 **스타일을 자동 선택·변경** (작을 땐 ALL → 커지면 KEY/EVEN 등) | 명시적 튜닝 전 기본값. 잘 모르겠으면 AUTO로 두고 모니터링 |

> 분산키 선택 원칙: ① **자주 조인하는 키**일 것 ② **카디널리티가 높아 고르게 분산**될 것(특정 값에 행이 쏠리면 그 슬라이스만 과부하 = 데이터 스큐). 예를 들어 `user_id`처럼 값 종류가 많고 조인에 쓰이는 컬럼이 좋은 후보다. `gender`처럼 값이 몇 개뿐인 컬럼은 스큐를 만든다. (→ [샤딩과-레플리카](./샤딩과-레플리카.md)의 샤드 키 선택과 같은 고민이다.)

## 3. 정렬키(SORTKEY) — 디스크 정렬 + zone map으로 스캔을 건너뛴다

분산키가 "**어느 노드에** 둘까"라면, 정렬키는 한 슬라이스 안에서 "**어떤 순서로** 디스크에 쌓을까"다.

Redshift는 데이터를 1MB 블록 단위로 저장하면서, 각 블록의 **최소·최댓값을 메타데이터(zone map)**로 들고 있다. 정렬키 컬럼으로 정렬해 두면:

```
SORTKEY(created_at) 로 정렬 저장 →
  블록1: created_at 2026-01-01 ~ 2026-01-10   (zone map: min/max 기록)
  블록2: created_at 2026-01-11 ~ 2026-01-20
  블록3: created_at 2026-01-21 ~ 2026-01-31
  ...

쿼리: WHERE created_at >= '2026-01-21'
  → zone map 보고 블록1·블록2는 통째로 스킵, 블록3부터만 읽음 (block skipping)
```

정렬이 안 돼 있으면 해당 값이 모든 블록에 흩어져 있어 전부 읽어야 한다. **`WHERE`·범위 조건에 자주 쓰는 컬럼(특히 날짜)**을 정렬키로 주면 스캔량이 크게 준다.

### Compound vs Interleaved

- **Compound(복합, 기본)**: 정렬키 컬럼들을 **나열한 순서대로** 우선순위를 둬 정렬한다(`SORTKEY(a, b)`면 a로 먼저, 같으면 b로). **첫 번째 키로 거르는 쿼리에 가장 강하다.** 대부분의 경우 정답. 단 두 번째 키만으로 필터하면 효과가 약하다.
- **Interleaved(인터리브)**: 여러 정렬키에 **동등한 가중치**를 줘, 어떤 컬럼으로 필터해도 비슷하게 효과를 본다. 대신 적재·`VACUUM` 등 **유지보수 비용이 더 크다**. 어느 컬럼으로 필터될지 예측이 안 되는 특수한 경우에만 고려.

실무 결론: **고민되면 날짜 같은 주력 필터 컬럼으로 compound SORTKEY** 한 줄. interleaved는 명확한 이유가 있을 때만.

## 4. 실전 DDL — 스타 스키마 설계 + S3 적재

전형적인 팩트/디멘션(star schema) 설계 예다.

```sql
-- 팩트 테이블: 크고, user_id로 자주 조인, created_at으로 범위 필터
CREATE TABLE fact_orders (
    order_id     BIGINT       NOT NULL,
    user_id      BIGINT       NOT NULL,
    product_id   INTEGER      NOT NULL,
    amount       DECIMAL(12,2),
    created_at   TIMESTAMP    NOT NULL
)
DISTSTYLE KEY
DISTKEY (user_id)               -- 조인 키로 분산 → 같은 user 행이 같은 슬라이스에 (재분배 제거)
COMPOUND SORTKEY (created_at);  -- 날짜 범위 필터 가속 (zone map 블록 스킵)

-- 디멘션 테이블: 작고(<5백만 행) 잘 안 바뀜 → 전 노드 복제
CREATE TABLE dim_user (
    user_id   BIGINT      NOT NULL,
    name      VARCHAR(100),
    grade     VARCHAR(20),
    region    VARCHAR(50)
)
DISTSTYLE ALL;                  -- 모든 노드에 복제 → 조인 시 네트워크 이동 0
```

조인 `fact_orders f JOIN dim_user d ON f.user_id = d.user_id`는:
- `fact_orders`가 `user_id`로 분산되어 있고,
- `dim_user`는 모든 노드에 복제되어 있으니,
- 어느 슬라이스든 짝이 바로 옆에 있다 → **재분배·브로드캐스트 없이** 조인된다.

### S3에서 대량 적재: COPY

Redshift에 데이터를 채우는 표준은 `INSERT`가 아니라 **`COPY`** — S3의 파일을 **여러 슬라이스가 병렬로** 빨아들인다(단건 INSERT는 컬럼형에 최악).

```sql
COPY fact_orders
FROM 's3://my-datalake/orders/dt=2026-06-30/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS PARQUET;             -- CSV/JSON/Parquet 등 지원. Parquet은 이미 컬럼형이라 궁합 좋음
```

> 함정: `COPY`는 가능한 한 **파일을 슬라이스 수의 배수로** 나눠 주면 병렬 적재가 고르게 된다. 한 개 거대 파일이면 병렬성을 못 살린다.

## 5. RA3 + RMS / Serverless / Spectrum — 컴퓨트와 스토리지를 분리한다

과거 노드 타입(dc2 등)은 컴퓨트와 스토리지가 한 몸이라, 저장공간이 부족하면 필요 없는 컴퓨트까지 늘려야 했다. **RA3**가 이걸 끊었다.

### RA3 + Redshift Managed Storage(RMS)

- Nitro 기반 RA3 노드는 **자주 쓰는 핫 데이터는 로컬 SSD**, **콜드 데이터는 자동으로 S3 기반 RMS**로 계층화한다.
- 그래서 **컴퓨트 비용과 스토리지 비용(RMS, GB-month)이 분리 과금**된다. 데이터가 많아도 컴퓨트는 워크로드에 맞게, 스토리지는 쓴 만큼만 낸다.

### Redshift Serverless

- 클러스터를 직접 띄우지 않고, 용량을 **RPU(Redshift Processing Unit)** 단위로 쓴다. **base RPU**(최소 용량)를 잡아두면 부하에 따라 오토스케일한다.
- 쓴 만큼(RPU-시간) 과금된다. 띄엄띄엄 쓰는 워크로드·운영 부담 최소화에 적합.
- **S3 데이터레이크 쿼리도 RPU에 포함** — Serverless에선 Spectrum 별도 과금이 없다.

### Spectrum — 클러스터에서 S3를 직접 SQL

- **외부 테이블**을 만들어 S3 파일을 적재 없이 바로 쿼리한다. 스캔 **$5/TB**(쿼리당 최소 10MB), **Glue Data Catalog를 메타스토어로 공유**한다 — Athena가 보는 그 카탈로그와 동일하다(→ [Athena와-Glue](./Athena와-Glue.md)).
- Athena와 엔진은 다르지만 "S3를 직접 쿼리" 개념은 같다. 차이는 **Redshift 클러스터 비용은 별도로 든다**는 점.
- **RA3(프로비저닝)에서 S3 외부테이블 쿼리는 Spectrum으로 처리되고 스캔이 $5/TB로 별도 과금된다. Spectrum 비용이 없는 건 Serverless(RPU 포함)뿐이며, 2026-05 신규 RG 인스턴스는 데이터레이크 쿼리엔진을 내장한다.**

```sql
-- Glue Data Catalog를 외부 스키마로 마운트
CREATE EXTERNAL SCHEMA spectrum
FROM DATA CATALOG
DATABASE 'analytics_db'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftSpectrumRole';

-- 클러스터 내부 핫 테이블 + S3 콜드 데이터를 한 쿼리로 조인
SELECT f.user_id, SUM(s.amount)
FROM   fact_orders f                       -- 클러스터(핫)
JOIN   spectrum.orders_archive s           -- S3(콜드)
       USING (user_id)
GROUP  BY f.user_id;
```

### 동시성 스케일링(Concurrency Scaling)

- 동시 쿼리가 몰리면 **임시 클러스터를 자동으로 추가**해 대기 없이 처리하고, 끝나면 회수한다.
- 메인 클러스터가 **24시간 가동될 때마다 무료 1시간 크레딧**이 쌓이고, 그 이상만 초당 과금. 많은 대시보드 사용자의 동시 접속(BI)에 유용하다.

## 6. Athena vs Redshift — 언제 무엇을

둘 다 SQL이고 둘 다 S3를 본다. 그래서 헷갈리는데, **"가끔·탐색이냐 vs 반복·고정 BI냐"**로 가르면 깔끔하다.

| | **Athena** | **Redshift** |
|---|---|---|
| 형태 | 완전 서버리스 (인프라 0) | 클러스터(RA3) 또는 Serverless |
| 과금 | **스캔 $5/TB** (쿼리당) — 안 쓰면 0 | 노드 시간 / RPU + RMS 스토리지 |
| 강한 곳 | **애드혹·탐색·간헐적** 쿼리, 로그 분석, "한 번 보고 끝" | **반복 BI 대시보드, 복잡한 다중 조인, 낮은 지연, 많은 동시 사용자** |
| 데이터 위치 | S3에 그대로 (적재 불필요) | 적재(COPY)해서 분산·정렬 최적화 / Spectrum으로 S3도 |
| 성능 튜닝 | 파티션·컬럼형 포맷 정도 | **분산키·정렬키·압축·결과 캐시·materialized view** 등 깊은 튜닝 |
| 한 줄 | "가끔 던지는 SQL, 운영 부담 0" | "매일 도는 BI, 갈수록 빨라지게 튜닝" |

판단 기준:
- **"하루에 몇 번, 어쩌다, 데이터는 S3에 이미 있음"** → Athena. 클러스터 띄울 이유가 없다.
- **"같은 대시보드가 매일·동시 사용자 많음·조인 복잡·지연 민감"** → Redshift. 적재해서 분산/정렬로 튜닝하면 반복 쿼리가 점점 싸고 빨라진다.

### 레이크하우스 — 둘을 잇기

양자택일이 아니다. **Glue Data Catalog를 공유 메타스토어**로 두면:

```
        ┌────────────── Glue Data Catalog (공유 스키마) ──────────────┐
        │                                                            │
   ┌────────┐                  S3 데이터 레이크                  ┌──────────────┐
   │ Athena │ ── 애드혹 SQL ──▶  (raw / 콜드 / 아카이브)  ◀── Spectrum ── │   Redshift   │
   └────────┘                       ▲                            │ (핫 = RMS/SSD)│
                                    │ COPY로 핫 데이터 적재         └──────────────┘
```

- 콜드·원천 데이터는 **S3에 두고** Athena/Spectrum으로 필요할 때 쿼리,
- 자주·빠르게 봐야 하는 핫 데이터는 **Redshift에 적재**해 튜닝,
- **Redshift Spectrum이 다리** 역할로 클러스터 내부 핫 테이블과 S3 콜드 데이터를 **한 쿼리로 조인**한다. (앞단의 적재 흐름은 → [Firehose-vs-S3-직접-적재](../infra/Firehose-vs-S3-직접-적재.md), 전체 그림은 → [AWS-데이터-분석-스택](./AWS-데이터-분석-스택.md))

## 7. 비용 정리

| 항목 | 과금 |
|---|---|
| 컴퓨트 | 프로비저닝 노드 시간(RA3) **또는** Serverless RPU-시간 |
| 스토리지 | RA3는 **RMS를 GB-month 별도** 과금 (컴퓨트와 분리) |
| Spectrum | S3 스캔 **$5/TB**(10MB 최소). **Serverless는 RPU에 포함, 프로비저닝 RA3는 $5/TB 별도 과금** |
| 동시성 스케일링 | 24h 가동당 **무료 1시간**, 초과분 초당 과금 |
| 백업(스냅샷) | **프로비저닝 스토리지 용량까지 100% 무료** |
| 데이터 전송 | **S3 ↔ Redshift 전송 무료** |

비용 절감 포인트:
- **스캔량 자체를 줄이는 최적화(컬럼형 포맷·파티셔닝·압축)**는 **Spectrum/S3 쿼리($5/TB)에 직접 효과** — 스캔 바이트가 곧 돈이다.
- 클러스터 내부 테이블은 스캔당 과금이 아니라 노드/RPU 시간이지만, **분산키·정렬키로 읽을 블록을 줄이면 같은 노드로 더 많은 쿼리를 소화** → 사실상 비용/성능이 좋아진다.

## 면접에서 이렇게 답하자

> "Redshift는 **컬럼형 저장 + MPP**로 대규모 집계·조인을 빠르게 처리하는 데이터 웨어하우스입니다. 쿼리는 **리더 노드**가 계획을 세워 **컴퓨트 노드의 슬라이스들에 병렬로 분배**하고 결과를 모읍니다.
>
> 빠름의 핵심은 두 가지입니다. **정렬키(SORTKEY)**는 데이터를 디스크에 정렬해 zone map으로 안 읽어도 되는 블록을 건너뛰게 하고 — 즉 **읽을 양을 줄이고**, **분산키(DISTKEY)**는 조인 키를 같은 슬라이스에 모아(코로케이션) **노드 간 재분배라는 비싼 네트워크 이동을 없앱니다**. 작은 디멘션은 `DISTSTYLE ALL`로 전 노드 복제해 이동을 0으로 만들고, 잘 모르면 `AUTO`로 둡니다.
>
> Athena와의 선택은 **'가끔 던지는 애드혹이냐, 반복되는 BI냐'**로 가릅니다. S3에 있는 데이터를 어쩌다 한 번 보면 서버리스인 Athena가 적합하고, 매일 도는 대시보드·복잡한 조인·낮은 지연·많은 동시 사용자라면 적재해서 분산/정렬로 튜닝하는 Redshift가 맞습니다. 둘은 **Glue Catalog를 공유**하므로, RA3의 핫 데이터와 S3의 콜드 데이터를 **Spectrum**으로 한 쿼리에 조인하는 레이크하우스 구성도 가능합니다."
