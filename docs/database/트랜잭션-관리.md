# 트랜잭션 + 여러 데이터 소스에서의 트랜잭션 관리

> **핵심 질문**: 트랜잭션을 어떻게 관리하고, 여러 DB에 걸친 트랜잭션은 어떻게 처리하는가?

---

## 한 줄 요약

**"단일 DB는 @Transactional로 충분하다. 여러 DB에 걸치면 분산 트랜잭션이 필요한데, 대부분은 피하는 게 답이다."**

---

## ACID 복습

| 속성 | 의미 | 깨지면? |
|------|------|---------|
| **Atomicity** (원자성) | 전부 성공 or 전부 실패 | 일부만 반영 (송금 중 한쪽만 차감) |
| **Consistency** (일관성) | 제약 조건 항상 만족 | 잔고가 음수 |
| **Isolation** (격리성) | 동시 트랜잭션 간 간섭 없음 | Dirty Read, Lost Update |
| **Durability** (지속성) | 커밋 후 영구 보존 | 커밋했는데 사라짐 |

## 단일 DB 트랜잭션

### @Transactional 동작 원리

```java
@Transactional
public void placeOrder(OrderRequest request) {
    Order order = orderRepository.save(Order.from(request));
    inventoryService.decrease(request.productId(), request.quantity());
    // 예외 발생 시 → 전부 롤백
}
```

```
실제 동작:
1. 프록시가 요청을 가로챔
2. TransactionManager.begin()
3. 실제 메서드 실행
4-a. 정상 → TransactionManager.commit()
4-b. 예외 → TransactionManager.rollback()
```

### 트랜잭션 전파 (Propagation)

```java
@Transactional
public void outer() {
    // 이미 트랜잭션 시작됨
    inner();  // inner()의 전파 설정에 따라 동작이 달라짐
}

@Transactional(propagation = Propagation.REQUIRED)  // 기본값
public void inner() {
    // outer의 트랜잭션에 참여 → 같이 커밋/롤백
}
```

| 전파 | 동작 | 사용 시점 |
|------|------|-----------|
| **REQUIRED** (기본) | 기존 TX 있으면 참여, 없으면 생성 | 대부분의 경우 |
| **REQUIRES_NEW** | 항상 새 TX 생성 (기존 TX 보류) | 독립적 로그 저장, 감사 |
| **NESTED** | 기존 TX 내 Savepoint 생성 | 부분 롤백 필요 시 |
| **NOT_SUPPORTED** | TX 없이 실행 (기존 TX 보류) | 조회 전용 |
| **MANDATORY** | 기존 TX 없으면 예외 | TX 필수 보장 |

### REQUIRES_NEW 실전 사용

```java
@Transactional
public void placeOrder(OrderRequest request) {
    Order order = orderRepository.save(Order.from(request));

    try {
        inventoryService.decrease(request.productId(), request.quantity());
    } catch (Exception e) {
        // 재고 차감 실패 → 주문도 롤백됨 (같은 TX)
        throw e;
    }

    // 주문 이력 로그: 주문 롤백과 무관하게 저장해야 함
    auditService.log(order);  // REQUIRES_NEW
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void log(Order order) {
    // 별도 TX → 외부 TX가 롤백되어도 이 로그는 유지
    auditLogRepository.save(new AuditLog(order));
}
```

### 격리 수준 (Isolation Level)

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void process() { ... }
```

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read | 성능 |
|-----------|-----------|--------------------|--------------|----|
| READ_UNCOMMITTED | O | O | O | 최고 |
| **READ_COMMITTED** | X | O | O | 좋음 (대부분 기본) |
| REPEATABLE_READ | X | X | △ | 보통 (MySQL 기본) |
| SERIALIZABLE | X | X | X | 최저 |

**참고**: SQL 표준상 REPEATABLE_READ는 Phantom Read를 허용하지만, **MySQL InnoDB는 Next-Key Lock(Gap Lock + Record Lock)으로 Phantom Read를 사실상 방지**한다. 따라서 MySQL에서는 △(구현으로 방지됨).

**실무 선택**: 대부분 DB 기본값 그대로 사용 (MySQL: REPEATABLE_READ, PostgreSQL: READ_COMMITTED). 변경이 필요한 경우는 극히 드물다.

### 흔한 실수: 같은 클래스 내부 호출

```java
@Service
public class OrderService {

    public void outer() {
        inner();  // @Transactional이 안 먹힘!
    }

    @Transactional
    public void inner() {
        // 프록시를 거치지 않아서 트랜잭션 없이 실행됨
    }
}
```

**원인**: Spring AOP는 프록시 기반. 같은 객체 내 호출은 프록시를 거치지 않는다.

**해결**:
```java
// 방법 1: 별도 클래스로 분리 (권장)
@Service
public class OrderTransactionService {
    @Transactional
    public void inner() { ... }
}

// 방법 2: self-injection (비추)
@Autowired private OrderService self;
public void outer() { self.inner(); }
```

### readOnly 최적화

```java
@Transactional(readOnly = true)
public List<Order> getOrders() {
    return orderRepository.findAll();
}
```

- Hibernate: Dirty Checking 생략 → 성능 향상
- DB: Replica로 라우팅 가능 (Read/Write 분리)
- 의도 명확화: "이 메서드는 수정하지 않는다"

## 여러 데이터 소스에서의 트랜잭션

### 문제 상황

```
서비스 A (DB-A): 주문 저장
서비스 B (DB-B): 재고 차감

@Transactional으로 묶을 수 있나? → 불가능.
두 DB는 서로 다른 Transaction Manager를 사용한다.
```

### 방법 1: 2PC (Two-Phase Commit) — 이론적 정답, 실무적 비추

```
Phase 1 (Prepare):
  Coordinator → DB-A: "커밋할 준비 됐나?" → "Yes"
  Coordinator → DB-B: "커밋할 준비 됐나?" → "Yes"

Phase 2 (Commit):
  Coordinator → DB-A: "커밋해라" → OK
  Coordinator → DB-B: "커밋해라" → OK
```

```java
// Spring JTA (Java Transaction API)
@Bean
public PlatformTransactionManager transactionManager() {
    return new JtaTransactionManager();  // Atomikos, Bitronix 등
}
```

**왜 비추인가**:
- 성능 저하 (Lock 보유 시간 2배)
- Coordinator 단일 장애점
- 네트워크 파티션 시 해결 불가
- 대부분의 클라우드 DB가 XA 트랜잭션 미지원 또는 비권장

### 방법 2: Saga 패턴 — 실무에서의 답

2PC 대신 **각 서비스가 로컬 트랜잭션**으로 처리하고, 실패 시 **보상 트랜잭션** 실행.

```
성공:
주문 생성(DB-A) → 재고 차감(DB-B) → 결제(PG) → 완료

재고에서 실패:
주문 생성(DB-A) → 재고 차감 실패(DB-B) → 주문 취소(DB-A, 보상)
```

#### Choreography Saga (이벤트 기반)

```
주문 서비스 → "주문 생성됨" 이벤트 발행
                  ↓
재고 서비스 → 재고 차감 → "재고 차감됨" 이벤트 발행
                  ↓
결제 서비스 → 결제 처리 → "결제 완료됨" 이벤트 발행
                  ↓
주문 서비스 → 주문 상태 → COMPLETED

재고 차감 실패 시:
재고 서비스 → "재고 부족" 이벤트 발행
                  ↓
주문 서비스 → 주문 취소 (보상)
```

#### Orchestration Saga (중앙 제어)

```java
// Saga Orchestrator
public class OrderSaga {
    public void execute(OrderRequest request) {
        SagaBuilder.create()
            .step("주문 생성")
                .action(() -> orderService.create(request))
                .compensation(() -> orderService.cancel(request.orderId()))
            .step("재고 차감")
                .action(() -> inventoryService.decrease(request))
                .compensation(() -> inventoryService.restore(request))
            .step("결제")
                .action(() -> paymentService.pay(request))
                .compensation(() -> paymentService.refund(request))
            .build()
            .execute();
    }
}
```

### 방법 3: Outbox 패턴 — 메시지 유실 방지

```java
@Transactional  // 같은 DB, 같은 트랜잭션
public void createOrder(OrderRequest request) {
    // 1. 주문 저장
    Order order = orderRepository.save(Order.from(request));

    // 2. 이벤트를 outbox 테이블에 저장 (같은 TX)
    outboxRepository.save(new OutboxEvent(
        "OrderCreated",
        objectMapper.writeValueAsString(order)
    ));
    // 커밋 → 주문 + 이벤트가 원자적으로 저장됨
}

// 별도 프로세스 (CDC 또는 Polling)가 outbox 테이블을 읽어 Kafka에 발행
```

```sql
CREATE TABLE outbox (
    id             BIGINT PRIMARY KEY AUTO_INCREMENT,
    aggregate_type VARCHAR(50) NOT NULL,   -- 'Order'
    aggregate_id   BIGINT NOT NULL,        -- order.id
    event_type     VARCHAR(50) NOT NULL,   -- 'OrderCreated'
    payload        JSON NOT NULL,
    created_at     TIMESTAMP DEFAULT NOW(),
    published      BOOLEAN DEFAULT FALSE   -- 발행 완료 여부
);
```

### 방법 비교

| 방법 | 정합성 | 성능 | 복잡도 | 추천도 |
|------|--------|------|--------|--------|
| 2PC (XA) | 강함 | 낮음 | 높음 | 비추 |
| Saga (Choreography) | 결과적 | 높음 | 중간 | 소규모 |
| Saga (Orchestration) | 결과적 | 높음 | 높음 | 대규모 |
| Outbox + CDC | 결과적 | 높음 | 중간 | 권장 |

## 면접에서 이렇게 답하라

> "단일 DB 트랜잭션은 Spring @Transactional로 충분합니다. 전파는 기본 REQUIRED를 쓰고, 독립적 로그가 필요하면 REQUIRES_NEW를 씁니다.
> 여러 DB/서비스에 걸치면 **2PC는 피하고 Saga 패턴**을 씁니다. 각 서비스가 로컬 트랜잭션으로 처리하고, 실패 시 보상 트랜잭션을 실행합니다.
> 이벤트 유실 방지를 위해 **Outbox 패턴**으로 데이터 저장과 이벤트 발행을 같은 트랜잭션에 묶습니다.
> 핵심은 **강한 일관성(2PC)을 포기하고, 결과적 일관성(Eventual Consistency)을 받아들이는 것**입니다. 대부분의 비즈니스에서 이것으로 충분합니다."
