# 여러 도메인이 걸칠 때 병목현상 관리

> **핵심 질문**: 하나의 요구사항이 여러 도메인(서비스)에 걸쳐야 하는 상황에서 병목현상을 어떻게 관리할 것인가?

---

## 한 줄 요약

**"가장 느린 서비스가 전체 속도를 결정한다. 동기로 묶지 말고, 끊어서 비동기로 연결하라."**

---

## 문제 상황

### 주문 생성 — 4개 도메인이 얽힘

```
주문 API 요청
   │
   ├── 1. 상품 서비스: 상품 정보 조회 + 재고 확인 (50ms)
   ├── 2. 쿠폰 서비스: 쿠폰 유효성 + 할인 계산 (80ms)
   ├── 3. 결제 서비스: PG사 결제 요청 (500ms~3s) ← 병목
   └── 4. 알림 서비스: 주문 확인 SMS/Push (200ms)

동기 직렬 처리 시: 50 + 80 + 500 + 200 = 830ms (최소)
PG사 느리면: 50 + 80 + 3000 + 200 = 3330ms
```

**병목 = 가장 느린 구간이 전체를 지배한다.**

---

## 전략 1: 동기 호출 최소화 — 꼭 필요한 것만 기다린다

### Before: 전부 동기

```
Client → 상품 조회 → 쿠폰 검증 → 결제 → 알림 → Response
         (동기)       (동기)      (동기)  (동기)
         ─────────── 총 830ms+ ───────────
```

### After: 필수만 동기, 나머지 비동기

```
Client → 상품 조회 + 쿠폰 검증(병렬) → 결제 → Response (202 Accepted)
         ──────── 130ms ──────────    500ms
         ──────────── 총 630ms ────────────
                                        │
                                   비동기 (MQ)
                                        │
                              ├── 알림 서비스 (나중에)
                              ├── 포인트 적립 (나중에)
                              └── 정산 데이터 (나중에)
```

**판단 기준: 사용자가 응답을 기다려야 하는가?**

| 작업 | 사용자 대기 필요? | 처리 방식 |
|------|-------------------|-----------|
| 상품 정보 조회 | O — 금액에 영향 | 동기 |
| 쿠폰 유효성 검증 | O — 할인에 영향 | 동기 |
| 결제 | O — 결과를 알아야 함 | 동기 |
| 알림 발송 | X — 나중에 가도 됨 | 비동기 |
| 포인트 적립 | X — 실패해도 재시도 가능 | 비동기 |
| 정산 기록 | X — 내부 처리 | 비동기 |

### 동기 호출 내에서도 병렬화

```java
// Before: 직렬 (130ms)
Product product = productService.getProduct(productId);      // 50ms
CouponResult coupon = couponService.validate(couponId);      // 80ms

// After: 병렬 (80ms — 둘 중 긴 쪽)
CompletableFuture<Product> productFuture =
    CompletableFuture.supplyAsync(() -> productService.getProduct(productId));
CompletableFuture<CouponResult> couponFuture =
    CompletableFuture.supplyAsync(() -> couponService.validate(couponId));

Product product = productFuture.get(3, TimeUnit.SECONDS);
CouponResult coupon = couponFuture.get(3, TimeUnit.SECONDS);
```

## 전략 2: 서비스 간 의존성 끊기 — 이벤트 기반

### 동기 의존 (강결합)

```
주문 서비스 ──HTTP 호출──→ 재고 서비스
              │
              └──HTTP 호출──→ 결제 서비스
                  │
                  └──HTTP 호출──→ 알림 서비스

문제: 알림 서비스가 죽으면 → 주문 전체가 실패
```

### 이벤트 기반 (약결합)

```
주문 서비스 ──"주문 생성됨" 이벤트──→ Kafka
                                       │
                                 ├──→ 재고 서비스 (재고 차감)
                                 ├──→ 알림 서비스 (알림 발송)
                                 └──→ 정산 서비스 (정산 기록)

알림 서비스가 죽어도 → 주문은 성공
알림 서비스 복구 후 → Kafka에서 메시지 꺼내서 처리
```

→ 자세한 내용: [이벤트-리스너-vs-비동기](../architecture/이벤트-리스너-vs-비동기.md), [Kafka](../architecture/Kafka.md)

## 전략 3: 타임아웃 + 폴백 — 느린 서비스를 기다리지 않는다

```java
// 쿠폰 서비스가 느릴 때 — 할인 없이 진행
@CircuitBreaker(name = "couponService", fallbackMethod = "couponFallback")
@TimeLimiter(name = "couponService")  // 2초 타임아웃
public CompletableFuture<CouponResult> validateCoupon(Long couponId) {
    return CompletableFuture.supplyAsync(() ->
        couponClient.validate(couponId)
    );
}

// 폴백: 할인 없이 원가 적용
public CompletableFuture<CouponResult> couponFallback(Long couponId, Throwable t) {
    log.warn("쿠폰 서비스 장애, 할인 없이 진행: {}", t.getMessage());
    return CompletableFuture.completedFuture(CouponResult.noDiscount());
}
```

**서비스별 타임아웃 설정**:

| 서비스 | 타임아웃 | 폴백 |
|--------|----------|------|
| 상품 조회 | 1초 | 캐시된 상품 정보 |
| 쿠폰 검증 | 2초 | 할인 없이 진행 |
| 결제 | 5초 | 결제 실패 처리 |
| 알림 | 비동기 | MQ 재시도 |

## 전략 4: 캐싱으로 의존 호출 줄이기

매번 다른 서비스를 호출할 필요가 없다.

```
Before:
주문할 때마다 → 상품 서비스 호출 (상품명, 가격)
               → 사용자 서비스 호출 (배송지)

After:
상품 정보 → Redis 캐시 (5분 TTL)
사용자 배송지 → Redis 캐시 (10분 TTL)
→ 캐시 HIT면 서비스 호출 자체가 없음
```

```java
@Cacheable(value = "product", key = "#productId", unless = "#result == null")
public Product getProduct(Long productId) {
    return productClient.getProduct(productId);  // 캐시 미스 시에만 호출
}
```

**주의**: 가격처럼 정확해야 하는 데이터는 캐시 TTL을 짧게, 또는 캐시하지 않는다.

## 전략 5: BFF (Backend For Frontend) 패턴

여러 도메인을 조합하는 로직을 별도 계층으로 분리.

```
Before:
Client → 상품 API → 재고 API → 리뷰 API → 추천 API
         (4번 호출)

After:
Client → BFF (1번 호출) → 상품 API
                         → 재고 API    (병렬 호출)
                         → 리뷰 API
                         → 추천 API
```

- 클라이언트는 1번만 호출
- BFF가 내부적으로 병렬 호출 + 조합 + 폴백 처리
- 모바일/웹별로 다른 BFF 가능

## 병목 진단 방법

### 분산 추적 (Distributed Tracing)

```
[Order API] ──────────────────────────────── 830ms
  ├── [Product Service] ────── 50ms
  ├── [Coupon Service] ─────────── 80ms
  ├── [Payment Service] ─────────────────────────── 500ms  ← 병목!
  └── [Notification Service] ──────────── 200ms
```

- **Jaeger**, **Zipkin**, 또는 **AWS X-Ray**로 시각화
- 각 서비스 호출의 소요 시간을 한눈에 파악
- 어디가 병목인지 즉시 알 수 있다

### 핵심 지표

```
- 각 서비스 호출의 p50, p95, p99 응답 시간
- 서비스 간 호출 실패율
- 대기열(MQ) 적체량
- 스레드 풀/커넥션 풀 사용률
```

## 면접에서 이렇게 답하라

> "여러 도메인이 걸치는 상황에서 병목 관리의 핵심은 **동기 호출을 최소화**하는 것입니다.
> 사용자 응답에 필수적인 호출만 동기로 하고, 나머지는 이벤트 기반 비동기로 처리합니다.
> 동기 호출 내에서도 **CompletableFuture로 병렬화**하고, 각 서비스에 **타임아웃과 Circuit Breaker**를 설정합니다.
> 느린 서비스에 대한 **폴백**을 준비해서 전체가 멈추지 않게 하고, **캐싱**으로 서비스 호출 자체를 줄입니다.
> 어디가 병목인지 파악하기 위해 **분산 추적(Jaeger)**을 반드시 도입합니다.
> 핵심 원칙은 '가장 느린 서비스가 전체를 지배하지 않도록 끊는 것'입니다."
