# 트래픽 과다 시 대응

> **핵심 질문**: 트래픽 사용량이 너무 많아진다면 어떻게 대응하는가?

---

## 한 줄 요약

**"전부 받으려다 전부 죽는다. 우선순위를 정하고, 버릴 것을 버려라."**

[스파이크-트래픽-대처](../traffic/스파이크-트래픽-대처.md)가 "순간 폭증"이라면, 여기서는 **지속적으로 감당 한계를 초과**하는 상황을 다룬다.

---

## 단계별 대응 — 심각도 순서

### Level 1: 자동 대응 (시스템이 알아서)

```
CPU 70% → Auto Scaling 트리거 → 서버 증설
         → 자동으로 해결
```

- Auto Scaling의 적정 동작 범위
- 별도 인적 개입 불필요
- **전제**: Scale-Out 가능한 Stateless 아키텍처

### Level 2: 능동적 대응 (엔지니어 개입)

```
Auto Scaling Max 도달 / DB CPU 80% / Redis 메모리 90%
→ 자동으로는 더 이상 안 됨
→ 엔지니어가 조치해야 함
```

| 조치 | 효과 | 소요 |
|------|------|------|
| 캐시 TTL 늘리기 | DB 부하 감소 | 즉시 |
| Rate Limiting 강화 | 과도한 요청 차단 | 즉시 |
| 불필요한 API 호출 제거 | 전체 부하 감소 | 수시간 |
| DB Read Replica 추가 | 읽기 분산 | 수십분 |
| 쿼리 최적화 | 자원 효율 개선 | 수시간~일 |

### Level 3: 비상 대응 (서비스 일부 포기)

```
시스템 한계 초과 → 핵심 기능만 살리고 나머지는 끈다
```

**Graceful Degradation (단계적 기능 축소)**:

```
단계 1 (경고):
  - 추천 시스템 OFF
  - 실시간 랭킹 OFF
  - 리뷰 목록 캐시만 반환

단계 2 (위험):
  - 검색 기능 축소 (상세 필터 OFF)
  - 이미지 저해상도 전환
  - 비핵심 API 503 반환

단계 3 (긴급):
  - 신규 주문만 허용 (조회는 캐시 반환)
  - 비로그인 사용자 접근 차단
  - 정적 페이지로 전환 (S3)
```

```java
// Feature Flag 기반 Degradation
@RestController
public class ProductController {

    @GetMapping("/products/{id}")
    public ProductResponse getProduct(@PathVariable Long id) {
        ProductResponse response = productService.getProduct(id);

        // Level 1: 추천 OFF
        if (featureFlag.isEnabled("degradation-level-1")) {
            response.setRecommendations(Collections.emptyList());
        }

        // Level 2: 리뷰 캐시만
        if (featureFlag.isEnabled("degradation-level-2")) {
            response.setReviews(cachedReviewService.get(id));
        }

        return response;
    }
}
```

### Level 4: 최후 수단

```
전체 서비스 유지 불가
→ 대기열 모드 전환 (전체 사용자 대기열 진입)
→ 또는 점검 페이지 전환
```

## 병목 지점별 대응

### CPU 과부하

```
원인: 연산 과다, 비효율적 코드, GC 과다
진단: top, htop, jstat -gc

대응:
1. Auto Scaling (서버 추가)
2. 무거운 연산 비동기 처리 (MQ로)
3. 코드 최적화 (N+1, 불필요한 직렬화)
4. JVM 튜닝 (GC 알고리즘, 힙 사이즈)
```

### 메모리 과부하

```
원인: 대용량 데이터 로딩, 메모리 누수, 세션 과다
진단: jmap -heap, GC 로그, 힙 덤프 분석

대응:
1. 페이징 처리 (전체 로딩 → 부분 로딩)
2. 스트리밍 처리 (Stream API, Cursor)
3. 메모리 누수 수정 (힙 덤프 분석)
4. 서버 스펙 업 (임시)
```

```java
// Bad: 100만 건을 한번에 메모리에
List<Order> all = orderRepository.findAll();  // OOM

// Good: 스트리밍
@Transactional(readOnly = true)
public void processOrders() {
    try (Stream<Order> stream = orderRepository.streamAll()) {
        stream.forEach(order -> {
            process(order);
            entityManager.detach(order);  // 메모리 해제
        });
    }
}
```

### DB 과부하

```
원인: Slow Query, Lock 경합, 커넥션 풀 고갈, Write 폭증
진단: SHOW PROCESSLIST, Slow Query Log, Performance Schema

대응 순서:
1. Slow Query Kill (즉시)
2. 캐시 강화 (Redis TTL 늘리기)
3. Read Replica로 읽기 분산
4. 쿼리 튜닝 (인덱스 추가, 실행 계획 분석)
5. DB 스펙 업 (Scale-Up)
6. 샤딩 (최후 수단)
```

### 네트워크 과부하

```
원인: 대용량 응답, 불필요한 데이터 전송, 대역폭 한계
진단: 네트워크 모니터링 (CloudWatch, Grafana)

대응:
1. 응답 압축 (gzip, brotli)
2. 필요한 필드만 반환 (DTO Projection)
3. 페이지네이션 강제
4. CDN으로 정적 리소스 오프로드
5. 대역폭 확장
```

## 트래픽 증가 예측과 용량 계획

### Capacity Planning

```
현재 상태:
- 서버 4대, 각 서버 최대 2,000 QPS = 총 8,000 QPS
- 현재 트래픽: 5,000 QPS (62.5% 사용)
- 월 성장률: 15%

예측:
- 3개월 후: 5,000 × 1.15³ = 7,604 QPS (95% 사용) ← 위험
- 6개월 후: 5,000 × 1.15⁶ = 11,569 QPS ← 한계 초과

필요 조치:
- 3개월 내 서버 2대 추가 (총 12,000 QPS)
- DB Read Replica 1대 추가
- 캐시 적중률 90% → 95%로 개선 (DB 부하 50% 감소)
```

**원칙**: 용량의 **70%**에 도달하면 증설 계획. **80%**에서 실행. **90%**는 이미 늦음.

---

## 면접에서 이렇게 답하라

> "트래픽 과다 시 **4단계**로 대응합니다.
> 1단계는 **Auto Scaling**으로 자동 대응하고,
> 2단계는 **캐시 강화, Rate Limiting, 쿼리 최적화**로 엔지니어가 능동적으로 대응합니다.
> 3단계는 **Graceful Degradation**으로 비핵심 기능을 끄고 핵심만 살립니다. Feature Flag로 런타임에 즉시 전환합니다.
> 4단계는 **대기열 모드나 점검 페이지**로 전체 트래픽을 제어합니다.
> 평소에는 **Capacity Planning**으로 용량 70% 도달 시 증설을 계획해서 3단계까지 가지 않게 합니다."
