# 선착순 쿠폰 — 자정 트래픽 장애로 유실 시 대응

> **핵심 질문**: 쿠폰 선착순일 때 순서를 꼭 보장해야 하는데, 자정에 트래픽 장애가 발생해 유실되거나 못 받을 수 있다면?

---

## 한 줄 요약

**"순서 보장 + 유실 방지 + 장애 내성. 셋 다 만족하려면 MQ 기반 대기열이 답이다."**

---

## 문제의 본질

자정 선착순 쿠폰의 난이도:

```
00:00:00 ──→ 수만~수십만 요청이 1초 내에 폭주
           ├── 순서 보장 필수 (먼저 온 사람이 받아야)
           ├── 정확히 N장만 발급 (초과 발급 불가)
           ├── 중복 발급 불가 (1인 1장)
           └── 장애나도 유실되면 안 됨 (이미 받을 자격이 있는 사용자)
```

**최악의 시나리오**:
1. 자정에 트래픽 폭증 → 서버 다운
2. 서버 복구 후 → 이미 요청했던 사용자 데이터 유실
3. 쿠폰 수량은 차감됐는데 발급 기록은 없음 → 정합성 깨짐
4. CS 폭주: "저 먼저 눌렀는데 왜 안 받았나요?"

## 아키텍처: MQ 기반 선착순 시스템

```
                    ┌───────────────────┐
[사용자] ──────────→│   API Server      │──→ 202 "접수됨"
  00:00:00 폭주     │  (순번만 부여)     │
                    └────────┬──────────┘
                             │
                    Kafka (coupon-queue)
                    파티션 1개 = 순서 보장
                             │
                    ┌────────▼──────────┐
                    │   Consumer        │
                    │  (순차 처리)       │
                    │  1. 중복 체크      │
                    │  2. 수량 체크      │
                    │  3. 발급 기록      │
                    └────────┬──────────┘
                             │
                    DB + Redis 업데이트
                             │
                    사용자에게 결과 알림
                    (WebSocket / Push / Polling)
```

### 순서 보장

```java
// Producer: Kafka 파티션 1개에 순서대로 적재
// key를 고정하면 같은 파티션으로 → 순서 보장
kafkaTemplate.send("coupon-queue", "coupon-100", new CouponRequest(userId, couponId, requestedAt));

// Consumer: concurrency=1 → 하나씩 순서대로 처리
@KafkaListener(topics = "coupon-queue", concurrency = "1")
public void process(CouponRequest request) {
    couponIssueService.issue(request);
}
```

### 유실 방지 — Kafka의 내구성

```
Kafka 설정:
- acks=all → 모든 브로커에 기록 확인 후 응답
- replication.factor=3 → 3개 브로커에 복제
- min.insync.replicas=2 → 최소 2개에 기록되어야 성공

→ 브로커 1대 죽어도 메시지 유실 없음
→ API 서버 죽어도 Kafka에 메시지가 남아있음
→ Consumer 죽어도 복구 후 이어서 처리
```

## 단계별 흐름 상세

### 1단계: 요청 접수 (API Server)

```java
@PostMapping("/coupon/{couponId}/issue")
public ResponseEntity<AcceptResponse> issueCoupon(
    @PathVariable Long couponId,
    @RequestHeader("Idempotency-Key") String idempotencyKey
) {
    // 1. 빠른 필터링 (Redis)
    // 이미 소진된 쿠폰이면 즉시 거부
    if (redis.get("coupon:" + couponId + ":soldout") != null) {
        return ResponseEntity.ok(AcceptResponse.soldOut());
    }

    // 2. 이미 접수된 요청이면 중복 응답
    if (redis.sIsMember("coupon:" + couponId + ":requested", getCurrentUserId())) {
        return ResponseEntity.ok(AcceptResponse.alreadyRequested());
    }

    // 3. Kafka에 적재
    CouponRequest request = new CouponRequest(
        getCurrentUserId(), couponId, idempotencyKey, Instant.now()
    );
    kafkaTemplate.send("coupon-queue", "coupon-" + couponId, request);

    // 4. 접수 기록
    redis.sAdd("coupon:" + couponId + ":requested", getCurrentUserId());

    // 5. 즉시 응답
    return ResponseEntity.accepted().body(AcceptResponse.accepted());
}
```

**API 서버의 역할은 오직 "접수"**. 발급 판단은 Consumer에서.

### 2단계: 순차 처리 (Consumer)

```java
@KafkaListener(topics = "coupon-queue", concurrency = "1")
@Transactional
public void processCouponIssue(CouponRequest request) {
    Long couponId = request.couponId();
    Long userId = request.userId();

    // 1. 멱등성 체크 (같은 요청 재처리 방지)
    if (couponIssueRepository.existsByCouponIdAndUserId(couponId, userId)) {
        log.info("이미 발급됨: couponId={}, userId={}", couponId, userId);
        return;
    }

    // 2. 수량 체크 (Redis 원자적 차감)
    Long remaining = redis.decrement("coupon:" + couponId + ":remaining");
    if (remaining < 0) {
        redis.increment("coupon:" + couponId + ":remaining");  // 롤백
        redis.set("coupon:" + couponId + ":soldout", "true");  // 소진 플래그
        notifyUser(userId, couponId, IssueResult.SOLD_OUT);
        return;
    }

    // 3. DB 발급 기록
    try {
        couponIssueRepository.save(new CouponIssue(couponId, userId, request.idempotencyKey()));
        notifyUser(userId, couponId, IssueResult.SUCCESS);
    } catch (DataIntegrityViolationException e) {
        // DB Unique 제약 위반 (최종 안전망)
        redis.increment("coupon:" + couponId + ":remaining");
        log.warn("중복 발급 시도: {}", e.getMessage());
    }
}
```

### 3단계: 결과 알림

```java
// 방법 1: SSE (Server-Sent Events)
@GetMapping(value = "/coupon/{couponId}/result", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter getResult(@PathVariable Long couponId) {
    SseEmitter emitter = new SseEmitter(60_000L);  // 60초 타임아웃
    resultNotifier.register(getCurrentUserId(), couponId, emitter);
    return emitter;
}

// 방법 2: Polling
@GetMapping("/coupon/{couponId}/status")
public ResponseEntity<IssueStatus> getStatus(@PathVariable Long couponId) {
    // Redis에서 결과 조회
    String result = redis.get("coupon-result:" + couponId + ":" + getCurrentUserId());
    if (result == null) {
        return ResponseEntity.ok(IssueStatus.PROCESSING);
    }
    return ResponseEntity.ok(IssueStatus.from(result));
}
```

## 장애 시나리오별 대응

### 시나리오 1: API 서버 장애 (자정에 서버 다운)

```
문제: 사용자 요청을 받지 못함
영향: Kafka에 메시지 자체가 안 들어감

대응:
1. Auto Scaling + Pre-warming으로 서버 미리 확보
2. L7 LB가 살아있는 서버로 라우팅
3. Rate Limiting으로 서버 보호
4. 최악의 경우: 이벤트 시간 연장 공지
```

### 시나리오 2: Kafka 장애

```
문제: 메시지 적재 불가
영향: 접수는 됐다고 응답했는데 실제로 안 들어감

대응:
1. Kafka Cluster (3 브로커 이상)로 단일 장애 방지
2. Producer 재시도 (retries=3, retry.backoff.ms=100)
3. Kafka 완전 다운 시 → DB 직접 처리로 폴백

폴백:
try {
    kafkaTemplate.send(...).get(3, TimeUnit.SECONDS);
} catch (Exception e) {
    // Kafka 장애 → DB 직접 처리 (Pessimistic Lock)
    couponIssueService.issueWithDBLock(couponId, userId);
}
```

### 시나리오 3: Consumer 장애 (처리 중 다운)

```
문제: Kafka에서 메시지를 꺼냈는데 처리 중 Consumer 죽음
영향: 메시지 유실 가능?

대응: Kafka offset 관리
- enable.auto.commit=false (자동 커밋 OFF)
- 처리 완료 후 수동 커밋
- Consumer 재시작 시 마지막 커밋 지점부터 재처리
- Consumer 멱등성 보장 (중복 처리 안전)

@KafkaListener(topics = "coupon-queue")
public void process(CouponRequest request, Acknowledgment ack) {
    couponIssueService.issue(request);  // 처리
    ack.acknowledge();  // 처리 완료 후에만 커밋
    // 여기서 죽으면? → 커밋 안 됨 → 재시작 시 다시 처리 → 멱등성으로 안전
}
```

### 시나리오 4: Redis 장애

```
문제: 수량 관리, 중복 체크 불가
영향: 초과 발급 or 서비스 중단

대응:
1. DB가 최종 안전망 (Unique Key + 수량 체크)
2. Redis → DB 폴백 (성능 저하 감수)
3. Redis Cluster로 고가용성 확보
```

### 시나리오 5: "자정에 유실" — 요청했는데 못 받은 사용자

```
복구 프로세스:
1. Kafka 메시지 로그 확인 (누가 언제 요청했는지 전부 기록됨)
2. 실제 발급 기록과 대조
3. 요청은 있으나 발급이 안 된 사용자 식별
4. 수량 내라면 → 수동 발급
5. 수량 초과라면 → 추가 수량 확보 후 발급 or 보상 쿠폰

예방:
- 이벤트 전 부하 테스트 (예상 트래픽의 2~3배)
- Scheduled Scaling으로 서버 미리 확보
- 모든 요청을 Kafka에 기록 → 유실 자체가 불가능
```

## 순서 보장의 현실

**"진짜 1ms 단위 순서를 보장해야 하나?"**

```
현실:
- 네트워크 지연은 사용자마다 다르다 (WiFi vs LTE vs 5G)
- 클릭 시점과 서버 도착 시점은 다르다
- "공정한 선착순"의 정의가 모호하다

실무적 판단:
- Kafka 도착 순서 = 서버 도착 순서 ≈ 합리적인 선착순
- 밀리초 단위 순서 분쟁은 사실상 불가능 (증명도 불가)
- 사용자 경험 관점: "접수됨 → 결과 알려줌"이면 충분
```

## 면접에서 이렇게 답하라

> "선착순 쿠폰은 **MQ 기반 대기열**로 설계합니다.
> API 서버는 요청을 Kafka에 적재하고 즉시 202 응답합니다. 순서 보장을 위해 **단일 파티션**을 사용하고, Consumer가 순서대로 처리합니다.
> **유실 방지**는 Kafka의 acks=all + replication으로 메시지 내구성을 보장하고, Consumer는 수동 커밋 + 멱등성으로 장애 시에도 안전합니다.
> **중복 방지**는 Redis Set + DB Unique Key 이중 안전망을 둡니다.
> 자정 장애 대비로 Scheduled Scaling, Pre-warming을 하고, 최악의 경우에도 Kafka 로그에 모든 요청이 남아있으니 사후 복구가 가능합니다.
> 핵심은 **API 서버는 접수만, 발급 판단은 Consumer에서** 분리하는 것입니다."
