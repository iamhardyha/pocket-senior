# Redis 장애 시 대응

> **핵심 질문**: Redis가 장애가 난다면 어떻게 대응하는가?

---

## 한 줄 요약

**"Redis가 죽었을 때 서비스가 같이 죽으면, Redis에 너무 의존한 것이다."**

---

## Redis 장애 유형

| 유형 | 증상 | 원인 |
|------|------|------|
| **완전 다운** | 연결 불가 | 프로세스 크래시, OOM Kill |
| **응답 지연** | 타임아웃 급증 | 메모리 부족, Slow Query, 네트워크 |
| **부분 장애** | 일부 노드만 불통 | Cluster 노드 장애, 네트워크 파티션 |
| **데이터 유실** | 키가 사라짐 | 메모리 초과 eviction, 재시작 |

---

## Redis 역할별 장애 영향과 대응

### 역할 1: 캐시 (Cache)

**장애 시 영향**: DB로 트래픽 직행 → DB 과부하 → **Cache Stampede**

```
정상:  Client → Redis(HIT) → 응답 (99% 캐시에서 해결)
장애:  Client → Redis(FAIL) → DB → 응답
       × 10만 요청이 동시에 DB로 → DB 사망
```

#### 대응 1: Local Cache 폴백

```java
// Caffeine (Local) → Redis → DB 순서
public Product getProduct(Long id) {
    // 1차: Local Cache (서버 메모리)
    Product cached = localCache.getIfPresent("product:" + id);
    if (cached != null) return cached;

    // 2차: Redis (장애 시 스킵)
    try {
        cached = redisTemplate.opsForValue().get("product:" + id);
        if (cached != null) {
            localCache.put("product:" + id, cached);
            return cached;
        }
    } catch (RedisConnectionFailureException e) {
        log.warn("Redis 연결 실패, DB 직접 조회: {}", e.getMessage());
        // Redis 장애 → 바로 DB로
    }

    // 3차: DB
    Product product = productRepository.findById(id).orElseThrow();
    localCache.put("product:" + id, product);  // Local에라도 캐싱
    return product;
}
```

#### 대응 2: Circuit Breaker on Redis

```java
@CircuitBreaker(name = "redis", fallbackMethod = "getFromDB")
public Product getProductFromCache(Long id) {
    return redisTemplate.opsForValue().get("product:" + id);
}

public Product getFromDB(Long id, Throwable t) {
    return productRepository.findById(id).orElseThrow();
}
```

- Redis 실패율 50% 초과 → Circuit OPEN → Redis 호출 자체를 안 함
- DB에 갑자기 몰리는 것도 방지 (Circuit Breaker가 빠르게 차단)

#### 대응 3: Cache Stampede 방지

Redis가 복구되더라도, 캐시가 비어있으면 동시 요청이 DB로 몰린다.

```java
// Mutex Lock으로 하나의 요청만 DB 조회
public Product getProductSafe(Long id) {
    String key = "product:" + id;
    Product cached = redis.get(key);
    if (cached != null) return cached;

    // Lock 시도: 하나만 DB 조회
    String lockKey = "lock:" + key;
    boolean locked = redis.setIfAbsent(lockKey, "1", Duration.ofSeconds(5));

    if (locked) {
        try {
            Product product = productRepository.findById(id).orElseThrow();
            redis.set(key, product, Duration.ofMinutes(5));
            return product;
        } finally {
            redis.delete(lockKey);
        }
    } else {
        // 다른 스레드가 조회 중 → 잠깐 대기 후 캐시 재조회
        Thread.sleep(100);
        return redis.get(key);  // 이제 캐시에 있을 것
    }
}
```

### 역할 2: 세션 저장소 (Session)

**장애 시 영향**: 모든 사용자 로그아웃됨

```
Redis 죽음 → 세션 조회 실패 → 인증 실패 → 전원 로그아웃 → CS 폭주
```

#### 대응

```java
// 1. 세션용 Redis를 별도 클러스터로 분리
// 캐시 Redis 장애가 세션에 영향 주지 않도록

// 2. JWT 기반으로 전환 (Redis 의존 제거)
// 세션: Redis에 저장 → Redis 장애 = 인증 불가
// JWT: 토큰 자체에 정보 → Redis 없어도 인증 가능

// 3. Redis Sentinel/Cluster로 고가용성
// Master 죽으면 → Sentinel이 Replica를 Master로 승격 (수초 내)
```

| 방식 | Redis 의존도 | 장애 영향 |
|------|-------------|-----------|
| Redis 세션 | 완전 의존 | 전원 로그아웃 |
| JWT + Redis 블랙리스트 | 부분 의존 | 로그아웃된 토큰만 문제 |
| JWT only | 없음 | 영향 없음 (단, 토큰 폐기 불가) |

### 역할 3: 분산 락 (Distributed Lock)

**장애 시 영향**: 동시성 제어 불가 → 데이터 정합성 깨짐

```
Redis 죽음 → Lock 획득 불가 → 쿠폰 중복 발급, 재고 마이너스
```

#### 대응

```java
// 1. DB Lock으로 폴백
public void issueCouponWithFallback(Long couponId, Long userId) {
    try {
        // 1차: Redis 분산 락
        issueCouponWithRedisLock(couponId, userId);
    } catch (RedisConnectionFailureException e) {
        // 2차: DB Pessimistic Lock으로 폴백
        log.warn("Redis 장애, DB Lock으로 전환");
        issueCouponWithDBLock(couponId, userId);
    }
}

// 2. DB Unique Key가 최종 안전망 (항상 작동)
// Redis 분산 락이 실패해도 → DB Unique(coupon_id, user_id)가 중복 방지
```

### 역할 4: 메시지 큐 / 카운터

선착순 쿠폰 수량 관리를 Redis로 할 때.

**장애 시 영향**: 수량 관리 불가 → 초과 발급 or 서비스 중단

#### 대응

```java
// Redis 수량 관리 + DB 수량 동기화
public boolean issueCoupon(Long couponId, Long userId) {
    try {
        // 1차: Redis에서 수량 관리 (빠름)
        return issueCouponViaRedis(couponId, userId);
    } catch (RedisConnectionFailureException e) {
        // 2차: DB에서 직접 수량 관리 (느리지만 안전)
        log.warn("Redis 장애, DB 직접 처리로 전환");
        return issueCouponViaDB(couponId, userId);
    }
}
```

## Redis 고가용성 구성

### Sentinel (센티널)

```
           ┌── Sentinel 1 ──┐
           │   Sentinel 2   │  ← Master 감시
           │   Sentinel 3   │
           └───────┬────────┘
                   │
        Master 장애 감지 (Quorum)
                   │
                   ▼
        Replica → Master 승격 (자동 Failover)
```

- Master 장애 시 자동으로 Replica를 Master로 승격
- Failover 시간: 보통 5~30초
- 그 사이 쓰기 불가 → 애플리케이션에서 재시도 로직 필요

### Cluster (클러스터)

```
Shard 1: Master A ↔ Replica A'
Shard 2: Master B ↔ Replica B'
Shard 3: Master C ↔ Replica C'

- 16384 슬롯을 3개 샤드에 분배
- Master 1개 죽으면 → 해당 Replica가 승격
- 나머지 샤드는 정상 동작
```

| 구성 | 용도 | 장애 영향 |
|------|------|-----------|
| 단일 Redis | 개발/소규모 | 전체 서비스 영향 |
| Sentinel | 중규모, 고가용성 | Failover 중 수초 불가 |
| Cluster | 대규모, 분산 | 장애 샤드만 영향 |

## 실전 체크리스트

```
□ 아키텍처
  ├─ [ ] 캐시/세션/락용 Redis가 분리되어 있는가?
  ├─ [ ] Sentinel 또는 Cluster로 고가용성 구성?
  └─ [ ] Redis 없이도 핵심 기능이 동작하는가?

□ 애플리케이션
  ├─ [ ] Redis 호출에 타임아웃 설정?
  ├─ [ ] Circuit Breaker 적용?
  ├─ [ ] 폴백 로직 (Local Cache, DB)
  └─ [ ] 커넥션 풀 설정 (Lettuce maxTotal, timeout)

□ 모니터링
  ├─ [ ] 메모리 사용률 알람 (80% 이상)
  ├─ [ ] 연결 수 알람
  ├─ [ ] Slow Log 감시
  └─ [ ] Eviction 발생 알람
```

## 면접에서 이렇게 답하라

> "Redis 장애 대응은 **역할별로 다르게** 접근합니다.
> **캐시**로 쓸 때는 Local Cache 폴백과 Circuit Breaker로 DB 직접 조회로 전환하고, Cache Stampede를 Mutex Lock으로 방지합니다.
> **세션**은 가능하면 JWT로 Redis 의존을 줄이고, 불가피하면 세션 전용 Redis Cluster를 별도로 둡니다.
> **분산 락**은 DB Pessimistic Lock으로 폴백하고, DB Unique Key를 최종 안전망으로 둡니다.
> 인프라 측면에서는 **Sentinel이나 Cluster**로 자동 Failover를 구성하고,
> 가장 중요한 원칙은 **Redis가 죽어도 서비스는 살아있어야 한다**는 것입니다."
