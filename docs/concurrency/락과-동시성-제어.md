# 락 — 데이터 정합성 및 동시성 제어

> **핵심 질문**: 동시에 같은 데이터에 접근할 때 어떻게 정합성을 보장하는가?

---

## 한 줄 요약

**"락은 '누가 먼저'를 정하는 규칙이다. 목적에 맞는 락을 선택하지 못하면 성능을 죽이거나 데이터를 깨뜨린다."**

---

## 왜 락이 필요한가

```
락 없이 동시 접근:

Thread A: READ balance=1000 → balance-500 → WRITE balance=500
Thread B:      READ balance=1000 → balance-300 → WRITE balance=700

기대값: 1000 - 500 - 300 = 200
실제값: 700 (Thread A의 차감이 사라짐 = Lost Update)
```

**동시성 문제 유형**:

| 문제 | 설명 | 예시 |
|------|------|------|
| **Lost Update** | 한쪽 변경이 사라짐 | 잔고 차감 누락 |
| **Dirty Read** | 커밋 안 된 데이터 읽음 | 롤백될 데이터를 보여줌 |
| **Non-Repeatable Read** | 같은 조회인데 값이 다름 | 조회 중 다른 트랜잭션이 수정 |
| **Phantom Read** | 같은 조회인데 행 수가 다름 | 조회 중 다른 트랜잭션이 INSERT |

---

## 락 전체 지도

```
락 (Lock)
├── DB 레벨
│   ├── Pessimistic Lock (비관적 락)
│   │   ├── Shared Lock (S Lock, 읽기 락)
│   │   └── Exclusive Lock (X Lock, 쓰기 락)
│   ├── Optimistic Lock (낙관적 락)
│   └── Named Lock (네임드 락)
│
├── Application 레벨
│   ├── synchronized (단일 JVM)
│   └── ReentrantLock (단일 JVM)
│
└── 분산 락
    ├── Redis (Redisson)
    ├── ZooKeeper
    └── DB Named Lock
```

## 1. DB Pessimistic Lock (비관적 락)

### "충돌이 일어날 것이라 가정하고, 미리 잠근다."

### Shared Lock (S Lock) — 읽기 락

```sql
SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE;
-- MySQL 8.0+: SELECT ... FOR SHARE;
```

```
Thread A: S Lock 획득 → 읽기 OK
Thread B: S Lock 획득 → 읽기 OK (S Lock끼리는 공존)
Thread C: X Lock 요청 → 대기 (S Lock이 풀릴 때까지)
```

- 여러 트랜잭션이 동시에 읽을 수 있다
- 하지만 쓰기는 못한다
- 용도: "내가 읽는 동안 아무도 수정하지 마라"

### Exclusive Lock (X Lock) — 쓰기 락

```sql
SELECT * FROM account WHERE id = 1 FOR UPDATE;
```

```
Thread A: X Lock 획득 → 읽기/쓰기 OK
Thread B: X Lock 요청 → 대기
Thread C: S Lock 요청 → 대기
```

- 하나의 트랜잭션만 접근 가능
- 용도: "내가 수정할 테니 아무도 건드리지 마라"

### JPA에서 사용

```java
// Exclusive Lock
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdForUpdate(@Param("id") Long id);

// Shared Lock
@Lock(LockModeType.PESSIMISTIC_READ)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdWithSharedLock(@Param("id") Long id);
```

```java
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    // 항상 작은 ID부터 잠그기 → Deadlock 방지
    Long firstId = Math.min(fromId, toId);
    Long secondId = Math.max(fromId, toId);

    Account first = accountRepository.findByIdForUpdate(firstId);
    Account second = accountRepository.findByIdForUpdate(secondId);

    Account from = fromId.equals(firstId) ? first : second;
    Account to = fromId.equals(firstId) ? second : first;

    from.withdraw(amount);
    to.deposit(amount);
}
```

### 주의: Deadlock

```
Deadlock 발생 조건:

Thread A: Lock(account:1) → Lock(account:2) 요청 → 대기
Thread B: Lock(account:2) → Lock(account:1) 요청 → 대기
→ 서로 상대방이 잡은 락을 기다림 → 영원히 대기
```

**Deadlock 방지 원칙**:
1. **항상 같은 순서로 락을 잡는다** (ID 오름차순 등)
2. **Lock Timeout을 설정한다** (무한 대기 방지)
3. **트랜잭션을 짧게 유지한다** (Lock 보유 시간 최소화)

```yaml
# Lock Timeout 설정
spring:
  jpa:
    properties:
      jakarta.persistence.lock.timeout: 3000  # 3초
```

## 2. DB Optimistic Lock (낙관적 락)

### "충돌이 안 일어날 것이라 가정하고, 충돌하면 다시 한다."

```java
@Entity
public class Product {
    @Id
    private Long id;
    private int stock;

    @Version
    private Long version;  // 핵심

    public void decreaseStock(int quantity) {
        if (this.stock < quantity) {
            throw new InsufficientStockException();
        }
        this.stock -= quantity;
    }
}
```

**실행되는 SQL**:
```sql
-- 읽기
SELECT id, stock, version FROM product WHERE id = 1;
-- stock=100, version=1

-- 쓰기
UPDATE product SET stock = 99, version = 2
WHERE id = 1 AND version = 1;
--                ^^^^^^^^^^^^ 핵심 조건

-- 다른 트랜잭션이 먼저 수정했으면?
-- version이 이미 2 → WHERE 조건 불일치 → 0 rows updated
-- → OptimisticLockingFailureException 발생
```

### 재시도 구현

```java
// 방법 1: Spring Retry
@Retryable(
    retryFor = OptimisticLockingFailureException.class,
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, multiplier = 2)  // 100ms, 200ms, 400ms
)
@Transactional
public void decreaseStock(Long productId, int quantity) {
    Product product = productRepository.findById(productId).orElseThrow();
    product.decreaseStock(quantity);
}

// 방법 2: 수동 재시도 (더 세밀한 제어)
public void decreaseStockWithRetry(Long productId, int quantity) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            decreaseStock(productId, quantity);
            return;  // 성공
        } catch (OptimisticLockingFailureException e) {
            if (i == maxRetries - 1) throw e;  // 마지막 시도도 실패
            // 잠깐 대기 후 재시도
        }
    }
}
```

### Version 없이 Optimistic Lock — CAS (Compare-And-Set)

```sql
-- version 컬럼 없이도 가능
UPDATE product SET stock = 99
WHERE id = 1 AND stock = 100;
--                ^^^^^^^^^^^ 이전에 읽은 값과 비교

-- 0 rows updated면 다른 누군가가 변경한 것
```

```java
// MyBatis/JDBC에서 자주 쓰는 패턴
int updatedRows = jdbcTemplate.update(
    "UPDATE product SET stock = stock - ? WHERE id = ? AND stock >= ?",
    quantity, productId, quantity
);
if (updatedRows == 0) {
    throw new InsufficientStockException();
}
```

이 방법이 실무에서 가장 많이 쓰인다. 간단하고 확실하다.

## 3. DB Named Lock

### "특정 이름의 자원을 잠근다."

```sql
-- MySQL Named Lock
SELECT GET_LOCK('coupon_issue_100', 3);  -- 'coupon_issue_100' 잠금, 3초 대기
-- ... 처리 ...
SELECT RELEASE_LOCK('coupon_issue_100');  -- 잠금 해제
```

```java
// JPA Native Query로 사용
@Query(value = "SELECT GET_LOCK(:key, :timeout)", nativeQuery = true)
Long getLock(@Param("key") String key, @Param("timeout") int timeout);

@Query(value = "SELECT RELEASE_LOCK(:key)", nativeQuery = true)
Long releaseLock(@Param("key") String key);
```

**특징**:
- Row Lock이 아니라 **이름 기반** 논리적 Lock
- 테이블/행과 무관하게 동작
- 별도 커넥션으로 관리해야 함 (Lock용 DataSource 분리)
- 분산 락의 간이 버전

**주의**: 커넥션을 점유하므로 Connection Pool 고갈 가능. 별도 DataSource 필수.

## 4. Application 레벨 락

### synchronized / ReentrantLock

```java
// synchronized — 단일 JVM에서만 동작
public synchronized void decreaseStock(Long productId, int quantity) {
    // 한 번에 하나의 스레드만 실행
}

// ReentrantLock — 더 세밀한 제어
private final Map<Long, ReentrantLock> locks = new ConcurrentHashMap<>();

public void decreaseStock(Long productId, int quantity) {
    ReentrantLock lock = locks.computeIfAbsent(productId, k -> new ReentrantLock());
    boolean acquired = lock.tryLock(3, TimeUnit.SECONDS);
    if (!acquired) throw new LockAcquisitionException();
    try {
        // 처리
    } finally {
        lock.unlock();
    }
}
```

**치명적 한계**: 서버가 2대 이상이면 안 된다.

```
Server 1: synchronized로 잠금 → Thread A 처리 중
Server 2: synchronized로 잠금 → Thread B 처리 중 ← 다른 JVM이라 Lock 공유 안 됨
→ 동시에 같은 데이터 수정 → 정합성 깨짐
```

**쓸 수 있는 상황**: 서버 1대인 경우, 또는 로컬 캐시 동시 접근 제어.

## 5. 분산 락 (Distributed Lock)

### "여러 서버에서 동일한 자원에 대한 접근을 제어한다."

### Redis 분산 락 (Redisson)

```java
// Redisson 설정
@Bean
public RedissonClient redissonClient() {
    Config config = new Config();
    config.useSingleServer().setAddress("redis://localhost:6379");
    return Redisson.create(config);
}
```

```java
// 사용
public void issueCoupon(Long couponId, Long userId) {
    RLock lock = redissonClient.getLock("lock:coupon:" + couponId);

    try {
        // 10초 대기, 획득 후 3초 후 자동 해제
        boolean acquired = lock.tryLock(10, 3, TimeUnit.SECONDS);
        if (!acquired) {
            throw new LockAcquisitionException();
        }

        // 크리티컬 섹션
        Coupon coupon = couponRepository.findById(couponId).orElseThrow();
        if (coupon.getRemaining() <= 0) throw new SoldOutException();
        coupon.decrease();
        couponIssueRepository.save(new CouponIssue(couponId, userId));

    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

### AOP로 깔끔하게

```java
// 어노테이션 정의
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DistributedLock {
    String key();          // SpEL 표현식
    long waitTime() default 5;
    long leaseTime() default 3;
    TimeUnit timeUnit() default TimeUnit.SECONDS;
}

// AOP 구현
@Aspect
@Component
public class DistributedLockAspect {
    private final RedissonClient redissonClient;

    @Around("@annotation(distributedLock)")
    public Object around(ProceedingJoinPoint joinPoint, DistributedLock distributedLock) throws Throwable {
        String key = parseKey(distributedLock.key(), joinPoint);
        RLock lock = redissonClient.getLock(key);

        boolean acquired = lock.tryLock(
            distributedLock.waitTime(),
            distributedLock.leaseTime(),
            distributedLock.timeUnit()
        );
        if (!acquired) throw new LockAcquisitionException(key);

        try {
            return joinPoint.proceed();
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}

// 사용 — 깔끔하다
@DistributedLock(key = "'coupon:' + #couponId")
@Transactional
public void issueCoupon(Long couponId, Long userId) {
    // Lock 획득된 상태에서 실행
    Coupon coupon = couponRepository.findById(couponId).orElseThrow();
    coupon.decrease();
    couponIssueRepository.save(new CouponIssue(couponId, userId));
}
```

### 분산 락 주의사항

**1. Lock과 Transaction 순서**

```
잘못된 순서:
@Transactional        ← 트랜잭션 시작
@DistributedLock      ← 락 획득
  처리...
  락 해제             ← 락 먼저 풀림
  트랜잭션 커밋       ← 커밋은 아직 안 됨!
                      ← 이 사이에 다른 스레드가 락 획득 → 커밋 안 된 데이터 읽음

올바른 순서:
@DistributedLock      ← 락 획득
  @Transactional      ← 트랜잭션 시작
    처리...
    트랜잭션 커밋     ← 먼저 커밋
  락 해제             ← 커밋 후 락 해제
```

**핵심: Lock이 Transaction을 감싸야 한다.** Lock 해제 전에 커밋이 완료되어야 한다.

**2. Lease Time (자동 해제 시간)**

```
문제: Lock 잡은 서버가 죽으면?
→ Lock이 영원히 풀리지 않음 → 전체 서비스 멈춤

해결: Lease Time 설정
→ 3초 후 자동 해제 (서버가 죽어도 3초 뒤 복구)

위험: 처리가 3초보다 오래 걸리면?
→ Lock이 풀려서 다른 스레드가 진입 → 정합성 깨짐

Redisson 해결: Watchdog
→ Lock을 잡고 있으면 자동으로 Lease Time 연장
→ leaseTime을 -1로 설정하면 Watchdog 활성화
```

→ [레디스-장애-대응](./failure/레디스-장애-대응.md)에서 상세 다룸

## 락 선택 가이드

```
"어떤 락을 써야 하나?"

서버가 1대인가? ──YES──→ synchronized / ReentrantLock
       │
       NO
       │
충돌이 자주 발생하나? ──NO──→ Optimistic Lock (@Version / CAS)
       │
      YES
       │
DB 기반이면 충분한가? ──YES──→ Pessimistic Lock (FOR UPDATE)
       │
       NO (여러 서비스, 여러 DB)
       │
       ▼
  분산 락 (Redisson)
```

### 성능 비교

| 락 종류 | TPS (대략) | 장점 | 단점 |
|---------|-----------|------|------|
| Optimistic | 수만 | 빠름, Lock 안 잡음 | 충돌 시 재시도 비용 |
| CAS (UPDATE WHERE) | 수만 | 가장 간단, 확실 | 복잡한 로직 어려움 |
| Pessimistic | 수천 | 확실한 정합성 | 대기 시간, Deadlock |
| Named Lock | 수천 | Row 무관 잠금 | 커넥션 점유 |
| Redis 분산 락 | 수천~만 | 서버 간 동기화 | Redis 의존, 복잡도 |
| synchronized | 수만 | 간단 | 단일 JVM만 |

## 실전 패턴 조합

### 재고 차감 — CAS + Pessimistic 전환

```java
// 1차: CAS로 시도 (빠름)
int updated = productRepository.decreaseStockCAS(productId, quantity);

if (updated == 0) {
    // 2차: 재고 부족인지, 동시 충돌인지 확인
    Product product = productRepository.findById(productId).orElseThrow();
    if (product.getStock() < quantity) {
        throw new InsufficientStockException();  // 진짜 재고 부족
    }
    // 동시 충돌 → Pessimistic Lock으로 재시도
    decreaseStockWithPessimisticLock(productId, quantity);
}
```

### 선착순 쿠폰 — Redis 분산 락 + DB Unique

```java
@DistributedLock(key = "'coupon:' + #couponId")
public void issueCoupon(Long couponId, Long userId) {
    // Redis Lock으로 동시성 제어
    // DB Unique Key로 최종 안전망
    try {
        couponIssueRepository.save(new CouponIssue(couponId, userId));
        couponRepository.decrease(couponId);
    } catch (DataIntegrityViolationException e) {
        // Unique 제약 위반 → 이미 발급됨
        throw new AlreadyIssuedException();
    }
}
```

## 면접에서 이렇게 답하라

> "락은 상황에 따라 선택합니다.
> **충돌이 드물면** Optimistic Lock이나 CAS로 성능을 살리고,
> **충돌이 잦으면** Pessimistic Lock으로 확실하게 직렬화합니다.
> **서버가 여러 대면** Redis 기반 분산 락(Redisson)을 사용하고,
> 이때 Lock이 Transaction을 감싸는 순서를 반드시 지킵니다.
> 실무에서는 단일 전략이 아니라 CAS를 먼저 시도하고 실패 시 Pessimistic으로 전환하는 **단계적 접근**을 많이 씁니다.
> 가장 중요한 건 Deadlock 방지를 위해 항상 같은 순서로 Lock을 잡고, Timeout을 설정하는 것입니다."
