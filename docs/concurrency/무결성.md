# 무결성 (Integrity)

> **핵심 질문**: 데이터가 비즈니스 규칙을 항상 만족하도록 어떻게 보장하는가?

---

## 한 줄 요약

**"잘못된 데이터가 들어가는 순간, 그 위에 쌓이는 모든 로직이 무너진다."**

무결성은 **방어선**이다. DB 제약 → 도메인 규칙 → 입력 검증, 3겹으로 쌓아야 한다.

---

## 무결성의 종류

| 종류 | 정의 | 예시 |
|------|------|------|
| **개체 무결성** | 모든 행은 고유하게 식별 가능 | PK는 NULL 불가, 중복 불가 |
| **참조 무결성** | 참조하는 데이터는 반드시 존재 | 주문의 user_id는 users 테이블에 존재 |
| **도메인 무결성** | 컬럼 값이 허용 범위 내 | 나이 > 0, 상태는 정해진 값만 |
| **비즈니스 무결성** | 비즈니스 규칙 만족 | 잔고 < 주문금액이면 주문 불가 |

---

## 계층별 무결성 방어

```
┌─────────────────────────────────┐
│  1층: 입력 검증 (Controller)     │  ← 비용 가장 낮음, 여기서 막으면 최선
│  - Bean Validation              │
│  - 형식, 범위, 필수값            │
├─────────────────────────────────┤
│  2층: 비즈니스 규칙 (Domain)     │  ← 핵심 방어선
│  - 도메인 객체 내 불변식          │
│  - 상태 머신                     │
├─────────────────────────────────┤
│  3층: DB 제약 조건               │  ← 최후의 방어선, 절대 뚫리지 않음
│  - NOT NULL, UNIQUE, CHECK      │
│  - FK, PK                       │
└─────────────────────────────────┘
```

### 1층: 입력 검증

```java
public record CreateOrderRequest(
    @NotNull(message = "상품 ID는 필수입니다")
    Long productId,

    @Min(value = 1, message = "수량은 1개 이상이어야 합니다")
    @Max(value = 9999, message = "수량은 9999개 이하여야 합니다")
    int quantity,

    @NotBlank(message = "배송 주소는 필수입니다")
    @Size(max = 200, message = "주소는 200자 이하여야 합니다")
    String address,

    @Pattern(regexp = "^\\d{5}$", message = "우편번호는 5자리 숫자입니다")
    String zipCode
) {}
```

**원칙**: Controller에서 형식과 범위만 검증. 비즈니스 로직은 넣지 않는다.

### 2층: 비즈니스 규칙 (도메인 불변식)

```java
public class Order {
    private OrderStatus status;
    private BigDecimal totalAmount;
    private List<OrderItem> items;

    // 불변식: 주문 금액은 항상 양수
    public void setTotalAmount(BigDecimal amount) {
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("주문 금액은 0보다 커야 합니다");
        }
        this.totalAmount = amount;
    }

    // 불변식: 빈 주문은 불가
    public void addItem(OrderItem item) {
        this.items.add(item);
        recalculateTotal();
    }

    // 상태 전이 규칙
    public void ship() {
        if (this.status != OrderStatus.PAID) {
            throw new IllegalStateException(
                "결제 완료 상태에서만 배송 가능합니다. 현재: " + this.status
            );
        }
        this.status = OrderStatus.SHIPPING;
    }
}
```

**상태 머신 — 허용된 전이만 가능**:

```
          결제
PENDING ──────→ PAID ──────→ SHIPPING ──────→ DELIVERED
   │              │               │
   │취소          │취소            │
   ↓              ↓               ↓
CANCELLED     REFUNDING ──→ REFUNDED

허용되지 않는 전이:
CANCELLED → PAID (X)
DELIVERED → PENDING (X)
SHIPPING → PENDING (X)
```

```java
public enum OrderStatus {
    PENDING,
    PAID,
    SHIPPING,
    DELIVERED,
    CANCELLED,
    REFUNDING,
    REFUNDED;

    // 허용된 전이를 명시적으로 정의
    private static final Map<OrderStatus, Set<OrderStatus>> TRANSITIONS = Map.of(
        PENDING,    Set.of(PAID, CANCELLED),
        PAID,       Set.of(SHIPPING, REFUNDING),
        SHIPPING,   Set.of(DELIVERED, REFUNDING),
        DELIVERED,  Set.of(REFUNDING),
        REFUNDING,  Set.of(REFUNDED),
        REFUNDED,   Set.of(),
        CANCELLED,  Set.of()
    );

    public boolean canTransitionTo(OrderStatus target) {
        return TRANSITIONS.getOrDefault(this, Set.of()).contains(target);
    }

    public OrderStatus transitionTo(OrderStatus target) {
        if (!canTransitionTo(target)) {
            throw new IllegalStateException(
                this + " → " + target + " 전이는 허용되지 않습니다"
            );
        }
        return target;
    }
}
```

### 3층: DB 제약 조건

```sql
CREATE TABLE orders (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,

    -- 참조 무결성
    user_id         BIGINT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id),

    -- 도메인 무결성
    status          VARCHAR(20) NOT NULL,
    CONSTRAINT chk_status CHECK (status IN (
        'PENDING','PAID','SHIPPING','DELIVERED',
        'CANCELLED','REFUNDING','REFUNDED'
    )),

    total_amount    DECIMAL(12,2) NOT NULL,
    CONSTRAINT chk_amount CHECK (total_amount > 0),

    -- 개체 무결성
    order_number    VARCHAR(32) NOT NULL UNIQUE,

    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW() ON UPDATE NOW()
);

-- 복합 유니크: 같은 주문에 같은 상품 중복 불가
CREATE TABLE order_items (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id    BIGINT NOT NULL,
    product_id  BIGINT NOT NULL,
    quantity    INT NOT NULL CHECK (quantity > 0),
    price       DECIMAL(10,2) NOT NULL CHECK (price >= 0),

    FOREIGN KEY (order_id) REFERENCES orders(id),
    UNIQUE KEY uk_order_product (order_id, product_id)
);
```

## 무결성이 깨지는 흔한 상황

### 1. 소프트 삭제 + FK

```sql
-- users 테이블에 소프트 삭제
UPDATE users SET deleted_at = NOW() WHERE id = 1;

-- orders에서 user_id = 1 참조 → FK는 통과 (행은 존재하니까)
-- 하지만 논리적으로 삭제된 사용자의 주문 → 무결성 깨짐
```

**해결**: 소프트 삭제된 사용자 참조 시 애플리케이션에서 체크.

```java
User user = userRepository.findActiveById(userId)
    .orElseThrow(() -> new UserNotFoundException());
```

### 2. Enum 불일치

```java
// Java에서는 4개
public enum Status { PENDING, PAID, SHIPPING, DELIVERED }

// DB에서는 6개 (마이그레이션 누락)
CHECK (status IN ('PENDING','PAID','SHIPPING','DELIVERED','CANCELLED','REFUNDED'))

// → Java에서 CANCELLED 저장 시도 → DB CHECK 통과 → Java에서 읽을 때 에러
```

**해결**: Enum 변경 시 DB 마이그레이션을 반드시 함께.

### 3. 숫자 정밀도

```java
// 위험: float/double로 금액 계산
double price = 0.1 + 0.2;  // 0.30000000000000004

// 안전: BigDecimal
BigDecimal price = new BigDecimal("0.1").add(new BigDecimal("0.2"));
// 0.3 정확
```

**원칙**: 금액은 **절대** float/double 쓰지 않는다. BigDecimal 또는 정수(원 단위) 사용.

### 4. NULL의 함정

```sql
-- NULL은 비교 불가
SELECT * FROM orders WHERE coupon_id = NULL;    -- 결과: 0건 (의도와 다름)
SELECT * FROM orders WHERE coupon_id IS NULL;   -- 올바름

-- NULL 포함 집계
SELECT AVG(rating) FROM reviews;
-- rating이 NULL인 행은 제외됨 → 평균이 왜곡될 수 있음
```

**원칙**: 의미 있는 기본값이 있으면 NOT NULL + DEFAULT. 정말 "없음"을 표현해야 할 때만 NULL.

## 무결성 모니터링

### 정기 검증 배치

```sql
-- 1. 고아 데이터 (참조 무결성 위반)
SELECT oi.* FROM order_items oi
LEFT JOIN orders o ON oi.order_id = o.id
WHERE o.id IS NULL;

-- 2. 상태 불일치
SELECT * FROM orders
WHERE status = 'DELIVERED' AND shipped_at IS NULL;

-- 3. 금액 불일치
SELECT o.id, o.total_amount, SUM(oi.price * oi.quantity) AS calculated
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
GROUP BY o.id
HAVING o.total_amount != calculated;

-- 4. 음수 데이터 (도메인 무결성)
SELECT * FROM wallet WHERE balance < 0;
SELECT * FROM product WHERE stock < 0;
```

이런 배치를 매일 돌리고, 결과가 1건이라도 있으면 알람.

## 면접에서 이렇게 답하라

> "무결성은 3계층 방어로 설계합니다.
> **입력 검증**으로 형식과 범위를 걸러내고,
> **도메인 객체**에서 불변식과 상태 전이 규칙을 강제하고,
> **DB 제약 조건**으로 최후의 방어선을 둡니다.
> 특히 금액은 반드시 BigDecimal, 상태 전이는 허용된 전이만 가능하도록 상태 머신을 구현합니다.
> 그리고 정기 검증 배치로 고아 데이터, 금액 불일치, 음수 값을 감시합니다.
> DB 제약이 최후의 방어선이지만, 거기까지 가기 전에 앞단에서 막는 것이 비용이 훨씬 싸다."
