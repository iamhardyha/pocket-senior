# 정합성, 무결성, 멱등성 설계

> **핵심 질문**: 평소 작업할 때 데이터 정합성, 무결성, 멱등성을 어떤 식으로 설계하는지? 상황에 따라 맞게 결정하는지?

---

## 한 줄 요약

**"정합성은 '데이터가 맞는가', 무결성은 '데이터가 올바른가', 멱등성은 '여러 번 해도 같은가'."**

이 세 가지는 별개가 아니라 서로 보완 관계다. 셋 다 지켜야 신뢰할 수 있는 시스템이다.

---

## 개념 정리 — 헷갈리기 쉬운 세 용어

### 비유로 이해하기

```
은행 계좌 이체: A → B에게 10만원

정합성 (Consistency):
  A가 -10만원 되면 B는 반드시 +10만원이어야 한다.
  총합이 달라지면 "정합성이 깨졌다"

무결성 (Integrity):
  잔고가 음수가 되면 안 된다.
  계좌번호는 반드시 존재해야 한다.
  "규칙에 맞는 데이터만 존재해야 한다"

멱등성 (Idempotency):
  같은 이체 요청을 2번 보내도 1번만 실행되어야 한다.
  "몇 번을 해도 결과가 같다"
```

### 정확한 정의

| 개념 | 정의 | 깨지면? |
|------|------|---------|
| **정합성** | 관련된 데이터 간의 일관성 | 주문은 있는데 결제가 없다 |
| **무결성** | 데이터가 비즈니스 규칙을 만족 | 잔고가 -500원 |
| **멱등성** | 같은 연산을 N번 해도 결과 동일 | 결제가 2번 됨 |

## 정합성 (Consistency) 설계

### 1. 단일 DB 내 정합성 — 트랜잭션

```java
@Transactional  // 이 안에서 전부 성공하거나, 전부 실패
public void placeOrder(OrderRequest request) {
    // 1. 재고 차감
    inventoryService.decrease(request.productId(), request.quantity());

    // 2. 주문 생성
    Order order = orderRepository.save(Order.from(request));

    // 3. 결제
    paymentService.pay(order.getId(), request.amount());

    // 3에서 실패하면? → 1, 2도 롤백 (정합성 유지)
}
```

**한계**: 트랜잭션은 **단일 DB** 안에서만 동작한다.

### 2. 여러 서비스 간 정합성 — 결과적 일관성 (Eventual Consistency)

MSA에서는 서비스마다 DB가 다르다. 분산 트랜잭션은 현실적으로 어렵다.

```
강한 일관성 (Strong Consistency):
  모든 시점에서 데이터가 일치 → 성능 포기

결과적 일관성 (Eventual Consistency):
  일시적으로 불일치할 수 있지만, 결국엔 일치 → 성능 유지
```

**실무에서 대부분 Eventual Consistency를 택한다.**

### 패턴: Outbox Pattern + CDC

```
주문 서비스:
┌──────────────────────────┐
│  @Transactional          │
│  1. orders 테이블 INSERT  │
│  2. outbox 테이블 INSERT  │  ← 같은 DB, 같은 트랜잭션
└──────────┬───────────────┘
           │
    CDC (Debezium) 또는 Polling
           │
           ▼
        Kafka  ──→  결제 서비스 (결제 처리)
                ──→  재고 서비스 (재고 차감)
                ──→  알림 서비스 (알림 발송)
```

- 주문 저장 + 이벤트 발행이 **같은 트랜잭션**으로 원자적
- 이벤트가 반드시 발행됨을 보장 (메시지 유실 방지)
- Consumer가 장애나도 Kafka에 메시지가 남아있으니 복구 가능

### 패턴: Saga Pattern

긴 프로세스를 단계별로 처리하고, 실패 시 보상 트랜잭션 실행.

```
성공 흐름:
주문 생성 → 결제 처리 → 재고 차감 → 배송 요청

결제에서 실패:
주문 생성 → 결제 실패 → 주문 취소 (보상 트랜잭션)

재고에서 실패:
주문 생성 → 결제 처리 → 재고 부족 → 결제 환불 → 주문 취소
```

| Saga 유형 | 특징 | 복잡도 |
|-----------|------|--------|
| Choreography | 각 서비스가 이벤트로 다음 단계 트리거 | 낮음 (소규모) |
| Orchestration | 중앙 오케스트레이터가 순서 제어 | 높음 (대규모) |

→ 자세한 내용: [MSA와-메시지큐](./architecture/MSA와-메시지큐.md)

### 정합성 검증 — 신뢰하되 검증하라

```sql
-- 주문-결제 정합성 체크 (매일 배치)
SELECT o.id AS order_id, o.status, p.status AS payment_status
FROM orders o
LEFT JOIN payments p ON o.id = p.order_id
WHERE o.status = 'COMPLETED' AND (p.id IS NULL OR p.status != 'PAID');
-- 결과가 있으면 → 정합성 깨진 것 → 알람 → 수동 보정
```

## 무결성 (Integrity) 설계

### 1. DB 레벨 무결성 — 제약 조건

```sql
CREATE TABLE orders (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT NOT NULL,                    -- NOT NULL 제약
    status      VARCHAR(20) NOT NULL
                CHECK (status IN ('PENDING','PAID','CANCELLED')),  -- CHECK 제약
    amount      DECIMAL(10,2) NOT NULL
                CHECK (amount > 0),                  -- 양수만 허용
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),

    FOREIGN KEY (user_id) REFERENCES users(id)       -- 참조 무결성
);
```

| 제약 유형 | 역할 | 예시 |
|-----------|------|------|
| NOT NULL | 필수값 보장 | 주문에 user_id 필수 |
| UNIQUE | 중복 방지 | 이메일 유일성 |
| CHECK | 값 범위 제한 | 금액 > 0 |
| FOREIGN KEY | 참조 무결성 | 존재하는 사용자만 |
| PRIMARY KEY | 행 식별 | 고유 ID |

### 2. 애플리케이션 레벨 무결성 — 비즈니스 규칙

DB 제약으로 표현 못하는 복잡한 규칙은 코드에서.

```java
public class Order {
    public void cancel() {
        // 비즈니스 규칙: 배송 시작 후에는 취소 불가
        if (this.status == OrderStatus.SHIPPING) {
            throw new IllegalStateException("배송 중인 주문은 취소할 수 없습니다");
        }
        // 비즈니스 규칙: 결제 완료 후 24시간 이내만 취소 가능
        if (this.paidAt.isBefore(Instant.now().minus(24, ChronoUnit.HOURS))) {
            throw new IllegalStateException("결제 후 24시간이 지나 취소할 수 없습니다");
        }
        this.status = OrderStatus.CANCELLED;
    }
}
```

### 3. 입력 레벨 무결성 — Validation

```java
// Controller에서 입력 검증
public record CreateOrderRequest(
    @NotNull Long productId,
    @Min(1) @Max(999) int quantity,
    @NotBlank String shippingAddress,
    @Pattern(regexp = "^\\d{5}$") String zipCode
) {}

// Service에서 비즈니스 검증
public void createOrder(CreateOrderRequest request) {
    Product product = productRepository.findById(request.productId())
        .orElseThrow(() -> new NotFoundException("상품이 존재하지 않습니다"));

    if (product.getStock() < request.quantity()) {
        throw new InsufficientStockException();
    }
}
```

**원칙**: 입력 → 비즈니스 → DB, 3단계 검증. 앞에서 막을수록 비용이 싸다.

## 멱등성 (Idempotency) 설계

→ 상세 내용: [멱등성](./concurrency/멱등성.md)

### 핵심 원리

```
멱등한 연산:   f(x) = f(f(x))
비멱등 연산:   f(x) ≠ f(f(x))

GET /orders/123      → 멱등 (몇 번 호출해도 같은 결과)
DELETE /orders/123   → 멱등 (이미 삭제된 걸 다시 삭제해도 결과 동일)
POST /orders         → 비멱등 (호출할 때마다 새 주문 생성)
POST /payments       → 비멱등 (호출할 때마다 결제 실행) ← 이게 위험
```

### Idempotency Key 패턴

```java
@PostMapping("/payments")
public ResponseEntity<PaymentResult> pay(
    @RequestHeader("Idempotency-Key") String idempotencyKey,  // 클라이언트가 생성
    @RequestBody PaymentRequest request
) {
    // 1. 이미 처리된 요청인지 확인
    Optional<PaymentResult> existing = idempotencyStore.get(idempotencyKey);
    if (existing.isPresent()) {
        return ResponseEntity.ok(existing.get());  // 저장된 결과 반환
    }

    // 2. 처리
    PaymentResult result = paymentService.pay(request);

    // 3. 결과 저장 (TTL 24시간)
    idempotencyStore.save(idempotencyKey, result, Duration.ofHours(24));

    return ResponseEntity.ok(result);
}
```

```
첫 번째 요청: Idempotency-Key: "abc-123" → 결제 실행 → 결과 저장
두 번째 요청: Idempotency-Key: "abc-123" → 이미 있음 → 저장된 결과 반환
세 번째 요청: Idempotency-Key: "abc-123" → 이미 있음 → 저장된 결과 반환
```

## 상황별 판단 가이드

### 어떤 것을 우선할 것인가?

```
                정합성
               /      \
              /        \
         무결성 ──── 멱등성

세 가지 다 지키면 가장 좋지만,
트레이드오프가 필요할 때:
```

| 상황 | 우선순위 | 이유 |
|------|----------|------|
| **결제/정산** | 정합성 > 멱등성 > 무결성 | 돈이 맞아야 한다 |
| **재고 관리** | 무결성 > 정합성 > 멱등성 | 음수 재고는 불가 |
| **외부 API 연동** | 멱등성 > 정합성 > 무결성 | 재시도 시 중복 실행 방지 |
| **로그/분석** | 정합성은 낮춰도 OK | 약간의 중복/유실 허용 |
| **알림 발송** | 멱등성 > 나머지 | 중복 알림은 사용자 경험 파괴 |

### 실무 의사결정 플로우

```
새로운 기능 설계 시:

1. "이 데이터가 틀리면 어떤 일이 발생하나?"
   └── 금전적 손실 → 정합성/멱등성 최우선
   └── 사용자 불편 → 무결성 우선
   └── 큰 영향 없음 → 최소한의 보장

2. "이 API는 재시도될 수 있나?"
   └── YES → 멱등성 키 필수
   └── NO (내부 동기 호출) → 트랜잭션으로 충분

3. "여러 서비스에 걸쳐 있나?"
   └── YES → Eventual Consistency + Saga
   └── NO → DB 트랜잭션

4. "실시간 정합성이 필요한가?"
   └── YES → 동기 처리 + Lock
   └── NO → 비동기 + 배치 정합성 검증
```

## 설계 체크리스트

```
□ 정합성
  ├─ [ ] 관련 데이터 변경이 하나의 트랜잭션인가?
  ├─ [ ] 서비스 간 Eventual Consistency 전략이 있는가?
  ├─ [ ] 정합성 검증 배치가 있는가?
  └─ [ ] 불일치 발견 시 보정 프로세스가 있는가?

□ 무결성
  ├─ [ ] DB 제약 조건이 설정되어 있는가?
  ├─ [ ] 입력 검증이 3단계(입력/비즈니스/DB)로 되어 있는가?
  ├─ [ ] 상태 전이 규칙이 명확한가?
  └─ [ ] 음수, NULL, 빈 값 등 엣지 케이스를 처리했는가?

□ 멱등성
  ├─ [ ] 외부 API 호출에 Idempotency Key가 있는가?
  ├─ [ ] 메시지 Consumer가 중복 처리에 안전한가?
  ├─ [ ] 재시도 시 부작용이 없는가?
  └─ [ ] 결제/발송 등 비가역 작업에 중복 방어가 있는가?
```

## 면접에서 이렇게 답하라

> "세 가지를 항상 같이 고려하되, 상황에 따라 우선순위를 다르게 둡니다.
> **결제처럼 돈이 관련되면** 정합성과 멱등성을 최우선으로, Idempotency Key와 Outbox Pattern을 적용합니다.
> **재고처럼 비즈니스 규칙이 중요하면** 무결성을 최우선으로, DB 제약 조건과 상태 머신을 사용합니다.
> **외부 API 연동이면** 멱등성을 최우선으로, 재시도에 안전한 설계를 합니다.
> 여러 서비스에 걸치는 경우 Eventual Consistency를 받아들이되, 정합성 검증 배치를 반드시 두어서 '신뢰하되 검증'합니다."
