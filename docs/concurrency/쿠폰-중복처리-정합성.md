# 쿠폰, 중복처리 — 데이터 정합성

> **핵심 질문**: 쿠폰 발급/사용에서 중복 처리를 어떻게 막고, 데이터 정합성을 어떻게 지키는가?

---

## 한 줄 요약

**"동시에 같은 자원에 접근하면 반드시 문제가 생긴다. 이걸 막는 게 정합성이다."**

쿠폰 시스템은 동시성 문제의 교과서다. 여기서 배운 패턴은 재고, 포인트, 좋아요 등 **모든 숫자 차감/증가 로직**에 그대로 적용된다.

---

## 문제 상황 — 왜 중복이 발생하는가

### 시나리오: 선착순 100장 쿠폰

```
쿠폰 잔여: 1장

[User A] 조회 → 잔여 1장 → 발급 OK → 잔여 0장
[User B] 조회 → 잔여 1장 → 발급 OK → 잔여 0장 (???)

결과: 쿠폰 100장인데 101장 발급됨
```

**Race Condition** (경쟁 상태): 두 요청이 거의 동시에 "잔여 수량 확인 → 차감"을 실행하면, 둘 다 "1장 남음"을 읽고 둘 다 발급한다.

```
시간 →
User A: ──READ(1)────────WRITE(0)──
User B: ──────READ(1)────────WRITE(0)──  ← 같은 값을 읽음
```

**근본 원인**: "확인"과 "실행"이 원자적(Atomic)이지 않다.

---

## 해결 전략 총정리

| 전략 | 원리 | 적합 상황 | 성능 |
|------|------|-----------|------|
| DB Unique 제약 | 중복 INSERT 자체를 DB가 차단 | 1인 1쿠폰 | ★★★★ |
| Pessimistic Lock | 먼저 잡은 사람만 접근 | 정합성 최우선 | ★★ |
| Optimistic Lock | 충돌 감지 후 재시도 | 충돌 빈도 낮을 때 | ★★★★ |
| Redis Atomic 연산 | 단일 스레드 + 원자적 명령 | 고성능 선착순 | ★★★★★ |
| Redis + Lua Script | 복합 로직을 원자적으로 | 조건부 발급 | ★★★★★ |
| 분산 락 (Redisson) | 여러 서버 간 동기화 | MSA 환경 | ★★★ |
| MQ 직렬화 | 요청을 줄 세워서 순차 처리 | 대규모 선착순 | ★★★★ |

## 전략 1: DB Unique 제약 — 가장 단순하고 확실한 방법

**"1인 1쿠폰" 중복 발급 방지의 정석.**

```sql
-- 쿠폰 발급 테이블
CREATE TABLE coupon_issue (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    coupon_id   BIGINT NOT NULL,
    user_id     BIGINT NOT NULL,
    issued_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    UNIQUE KEY uk_coupon_user (coupon_id, user_id)  -- 핵심
);
```

```java
// 발급 시도
try {
    couponIssueRepository.save(new CouponIssue(couponId, userId));
    // 성공
} catch (DataIntegrityViolationException e) {
    // 이미 발급됨 → 중복 요청
    throw new AlreadyIssuedException();
}
```

**장점**: 애플리케이션 코드에 상관없이 DB가 보장한다. 서버가 몇 대든 안전하다.
**한계**: 수량 제한 체크는 별도로 필요하다. "1인 1쿠폰"은 막지만 "총 100장"은 안 막는다.

## 전략 2: Pessimistic Lock (비관적 락)

**"일단 잠그고 시작한다."**

```java
// Repository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT c FROM Coupon c WHERE c.id = :id")
Coupon findByIdForUpdate(@Param("id") Long id);

// Service
@Transactional
public void issueCoupon(Long couponId, Long userId) {
    // 1. 쿠폰을 SELECT ... FOR UPDATE로 잠금
    Coupon coupon = couponRepository.findByIdForUpdate(couponId);

    // 2. 수량 확인
    if (coupon.getRemaining() <= 0) {
        throw new SoldOutException();
    }

    // 3. 수량 차감
    coupon.decrease();  // remaining - 1

    // 4. 발급 기록
    couponIssueRepository.save(new CouponIssue(couponId, userId));

    // 트랜잭션 커밋 시 잠금 해제
}
```

```
시간 →
User A: ──LOCK────READ(1)──WRITE(0)──UNLOCK──
User B: ──WAIT──────────────────────LOCK──READ(0)──SOLD OUT
                                     ↑
                                  A가 풀어야 진입
```

**장점**: 확실하다. 데이터가 절대 꼬이지 않는다.
**단점**:
- 동시 요청이 많으면 **직렬화** → 처리량 급감
- 잠금 대기 시간 → Timeout, Deadlock 가능
- 1000명이 동시에 요청하면 999명이 대기

**적합 상황**: 동시 요청이 수십 수준, 정합성이 성능보다 중요할 때 (결제, 정산)

## 전략 3: Optimistic Lock (낙관적 락)

**"일단 실행하고, 충돌하면 다시 한다."**

```java
@Entity
public class Coupon {
    @Id
    private Long id;
    private int remaining;

    @Version  // 핵심: 버전 필드
    private Long version;
}
```

```java
// 동작 원리
@Transactional
public void issueCoupon(Long couponId, Long userId) {
    Coupon coupon = couponRepository.findById(couponId);

    if (coupon.getRemaining() <= 0) {
        throw new SoldOutException();
    }

    coupon.decrease();
    couponIssueRepository.save(new CouponIssue(couponId, userId));

    // 커밋 시 실행되는 SQL:
    // UPDATE coupon SET remaining = 99, version = 2
    //   WHERE id = 1 AND version = 1
    //                    ^^^^^^^^^ 이 조건이 핵심
    // 다른 요청이 먼저 업데이트했으면 version이 이미 2 → 0 rows updated → 예외
}
```

```
시간 →
User A: READ(v1) ──────── UPDATE(v1→v2) ✅ 성공
User B: READ(v1) ──────────── UPDATE(v1→v2) ❌ 실패 (v이 이미 2)
                                              └→ 재시도 or 포기
```

**재시도 로직**:
```java
@Retryable(
    value = OptimisticLockingFailureException.class,
    maxAttempts = 3,
    backoff = @Backoff(delay = 100)  // 100ms 대기 후 재시도
)
public void issueCouponWithRetry(Long couponId, Long userId) {
    issueCoupon(couponId, userId);
}
```

**장점**: Lock 안 잡아서 빠르다. 충돌이 적으면 Pessimistic보다 훨씬 좋다.
**단점**: 충돌이 많으면 재시도 폭발 → 오히려 더 느려진다.

### Pessimistic vs Optimistic 선택 기준

| 기준 | Pessimistic | Optimistic |
|------|-------------|------------|
| 충돌 빈도 | 높을 때 (선착순) | 낮을 때 (일반 주문) |
| 처리량 | 낮음 (직렬화) | 높음 (병렬) |
| 정합성 | 강력 보장 | 재시도 실패 시 유실 가능 |
| Deadlock | 가능 | 없음 |

**실무 판단법**: "동시에 같은 row를 수정하는 요청이 초당 몇 건인가?"
- 10건 이하 → Optimistic
- 100건 이상 → Pessimistic 또는 Redis

## 전략 4: Redis Atomic 연산 — 고성능 선착순의 정석

**Redis는 Single Thread. 개별 명령은 원자적이지만, 여러 명령의 조합은 원자적이지 않다.**

### 단순 수량 차감: DECR

```
// 쿠폰 수량 세팅
SET coupon:100:remaining 100

// 발급 요청
DECR coupon:100:remaining
→ 99 (양수면 발급 OK)
→ ...
→ 0 (0이면 마지막 1장 발급 OK)
→ -1 (음수면 발급 실패 → INCR로 롤백)
```

```java
public boolean issueCoupon(Long couponId, Long userId) {
    String key = "coupon:" + couponId + ":remaining";

    Long remaining = redisTemplate.opsForValue().decrement(key);

    if (remaining < 0) {
        // 수량 초과 → 롤백
        redisTemplate.opsForValue().increment(key);
        return false;
    }

    // 발급 기록 (DB에 비동기로)
    couponIssueEventPublisher.publish(new CouponIssueEvent(couponId, userId));
    return true;
}
```

**문제**: DECR → 음수 → INCR 사이에 다른 요청이 끼어들 수 있다.
**해결**: Lua Script.

### Redis + Lua Script — 완벽한 원자성

```lua
-- issue_coupon.lua
-- KEYS[1]: 쿠폰 수량 키 (coupon:100:remaining)
-- KEYS[2]: 발급 기록 Set (coupon:100:issued)
-- ARGV[1]: userId

-- 1. 이미 발급받았는지 확인 (중복 방지)
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -1  -- 이미 발급됨
end

-- 2. 수량 확인
local remaining = tonumber(redis.call('GET', KEYS[1]))
if remaining <= 0 then
    return 0  -- 소진
end

-- 3. 수량 차감 + 발급 기록 (원자적)
redis.call('DECR', KEYS[1])
redis.call('SADD', KEYS[2], ARGV[1])
return 1  -- 발급 성공
```

```java
// Java에서 실행
public int issueCoupon(Long couponId, Long userId) {
    DefaultRedisScript<Long> script = new DefaultRedisScript<>(LUA_SCRIPT, Long.class);

    Long result = redisTemplate.execute(
        script,
        List.of(
            "coupon:" + couponId + ":remaining",
            "coupon:" + couponId + ":issued"
        ),
        userId.toString()
    );

    // result: 1(성공), 0(소진), -1(중복)
    return result.intValue();
}
```

**왜 Lua Script인가?**
- Redis는 Lua Script를 **하나의 명령처럼** 원자적으로 실행한다
- Script 실행 중 다른 명령이 끼어들 수 없다
- "확인 → 차감 → 기록"이 한 방에 끝난다

## 전략 5: MQ 기반 직렬화 — 대규모 선착순

초당 10만 요청이 오는 선착순 이벤트에서는 Redis도 부담이 될 수 있다.

```
┌─────────────┐     ┌──────────┐     ┌──────────────┐
│  API Server │────→│  Kafka   │────→│  Consumer    │
│ (즉시 응답)  │     │ (버퍼)    │     │ (순차 처리)   │
└─────────────┘     └──────────┘     └──────────────┘
  "접수됨"           순서 보장          1건씩 처리
  202 Accepted                        DB + Redis 업데이트
```

```java
// API Server: 즉시 큐에 넣고 응답
@PostMapping("/coupon/{id}/issue")
public ResponseEntity<Void> issueCoupon(@PathVariable Long id) {
    kafkaTemplate.send("coupon-issue", new CouponIssueRequest(id, getCurrentUserId()));
    return ResponseEntity.accepted().build();  // 202
}

// Consumer: 순차적으로 처리
@KafkaListener(topics = "coupon-issue", concurrency = "1")  // 1개 스레드 = 직렬 처리
public void processCouponIssue(CouponIssueRequest request) {
    // 중복 체크
    if (couponIssueRepository.existsByCouponIdAndUserId(request.couponId(), request.userId())) {
        return;
    }

    // 수량 체크 + 발급
    Coupon coupon = couponRepository.findById(request.couponId());
    if (coupon.getRemaining() > 0) {
        coupon.decrease();
        couponIssueRepository.save(new CouponIssue(request.couponId(), request.userId()));
    }
}
```

**장점**: API 서버는 순간 트래픽을 버틸 수 있고, 실제 처리는 안정적 속도로.
**단점**: 사용자가 결과를 즉시 알 수 없다 (비동기). Polling이나 WebSocket으로 결과 전달 필요.

→ 자세한 내용: [Kafka](../architecture/Kafka.md), [선착순-쿠폰-유실-대응](../failure/선착순-쿠폰-유실-대응.md)

## 실전 조합 패턴 — 쿠폰 시스템 전체 흐름

```
사용자 요청
    │
    ▼
[API Server]
    │
    ├── 1차 필터: Redis Set으로 중복 체크 (SISMEMBER) → O(1)
    │     └── 이미 발급 → 즉시 거부
    │
    ├── 2차 필터: Redis로 수량 체크 (GET remaining) → O(1)
    │     └── 소진 → 즉시 거부
    │
    ├── 발급: Redis Lua Script (DECR + SADD) → 원자적
    │     └── 성공 → Kafka에 이벤트 발행
    │
    ▼
[Kafka Consumer]
    │
    ├── DB에 발급 기록 INSERT (Unique Key로 안전망)
    ├── 알림 발송
    └── 정산 데이터 적재
```

**3중 안전망**:
1. **Redis Lua** — 원자적 수량 차감 + 중복 방지 (1차)
2. **DB Unique Key** — 최종 안전망 (2차)
3. **Kafka Consumer 멱등성** — 메시지 중복 처리 방지 (3차)

## 쿠폰 사용 시 중복 처리

발급뿐 아니라 **사용**에서도 중복 방지가 필요하다.

```java
@Transactional
public void useCoupon(Long couponIssueId, Long orderId) {
    CouponIssue issue = couponIssueRepository.findByIdForUpdate(couponIssueId);

    // 상태 체크
    if (issue.getStatus() != CouponStatus.ISSUED) {
        throw new InvalidCouponStateException();
        // 이미 사용됨, 만료됨, 취소됨 등
    }

    // 상태 변경 (ISSUED → USED)
    issue.use(orderId);

    // 상태 머신으로 전이 제어
    // ISSUED → USED (O)
    // USED → USED (X) - 중복 사용 불가
    // EXPIRED → USED (X) - 만료된 쿠폰 사용 불가
}
```

```
쿠폰 상태 머신:
ISSUED ──사용──→ USED
  │                │
  만료             취소
  ↓                ↓
EXPIRED        CANCELLED
```

## 정합성 검증 — 운영에서 반드시 필요하다

```sql
-- 발급 수량 정합성 체크 (배치)
SELECT
    c.id AS coupon_id,
    c.total_quantity,
    c.remaining,
    COUNT(ci.id) AS issued_count,
    c.total_quantity - c.remaining AS expected_issued,
    CASE
        WHEN COUNT(ci.id) != (c.total_quantity - c.remaining)
        THEN 'MISMATCH'
        ELSE 'OK'
    END AS status
FROM coupon c
LEFT JOIN coupon_issue ci ON c.id = ci.coupon_id
GROUP BY c.id;
```

- 매일 배치로 돌려서 불일치 감지
- 불일치 발견 시 알람 → 수동 보정

## 면접에서 이렇게 답하라

> "쿠폰 중복 처리는 3가지 레벨로 방어합니다.
> 첫째, **Redis Lua Script**로 수량 확인과 차감을 원자적으로 처리합니다. SISMEMBER로 중복을 O(1)로 체크하고, DECR+SADD를 하나의 Lua Script로 묶어 Race Condition을 원천 차단합니다.
> 둘째, **DB Unique 제약**(coupon_id + user_id)으로 최종 안전망을 둡니다. Redis가 장애나도 DB가 막아줍니다.
> 셋째, **Kafka Consumer의 멱등성** 처리로 메시지 중복에도 안전하게 합니다.
> 대규모 선착순이면 API는 즉시 202 응답하고 Kafka로 직렬화 처리합니다.
> 핵심은 '확인과 실행을 원자적으로' 만드는 것이고, 단일 전략이 아닌 **다중 안전망**으로 설계합니다."
