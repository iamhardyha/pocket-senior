# 멱등성 (Idempotency)

> **핵심 질문**: 같은 요청이 여러 번 와도 결과가 같게 만드는 방법은?

---

## 한 줄 요약

**"한 번 하든 열 번 하든 결과가 같다. 이걸 보장 못하면 돈이 두 번 빠지고, 알림이 열 번 간다."**

---

## 왜 멱등성이 중요한가

현실 세계에서 요청은 **반드시** 중복된다:

```
1. 네트워크 타임아웃 → 클라이언트 재시도
2. 로드밸런서 retry
3. MQ 메시지 재전달 (at-least-once)
4. 사용자 더블 클릭
5. 모바일 네트워크 끊김 → 재전송
```

```
결제 API가 멱등하지 않으면:

Client → 결제 요청 → Server 처리 완료 → 응답 전송 중 네트워크 끊김
Client → 응답 못 받음 → "실패인가?" → 재시도
Server → 또 결제 실행 → 돈 2배 빠짐
```

---

## HTTP 메서드별 멱등성

| 메서드 | 멱등? | 안전? | 설명 |
|--------|-------|-------|------|
| GET | O | O | 조회. 몇 번이든 같은 결과 |
| HEAD | O | O | GET과 같지만 Body 없음 |
| PUT | O | X | 전체 교체. 같은 값으로 교체하면 결과 동일 |
| DELETE | O | X | 이미 삭제된 걸 삭제해도 결과 동일 |
| **POST** | **X** | X | 생성. 매번 새로운 리소스 |
| **PATCH** | **△** | X | 구현에 따라 다름. `stock += 1`은 비멱등, `status = 'CANCELLED'`은 멱등 |

**POST는 항상 비멱등, PATCH는 구현에 따라 다르다.** 이 메서드들에 멱등성 보장이 필요하다.

## 멱등성 구현 패턴

### 패턴 1: Idempotency Key — 가장 범용적

```
원리: 클라이언트가 고유 키를 보내고, 서버가 "이 키로 이미 처리했는지" 확인한다.
```

```java
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> pay(
    @RequestHeader("Idempotency-Key") String key,
    @RequestBody PaymentRequest request
) {
    // 1. 이미 처리된 요청?
    Optional<PaymentResponse> cached = idempotencyStore.find(key);
    if (cached.isPresent()) {
        return ResponseEntity.ok(cached.get());  // 저장된 결과 반환
    }

    // 2. 진행 중인 요청? (동시 중복 방어)
    boolean locked = idempotencyStore.tryLock(key);
    if (!locked) {
        return ResponseEntity.status(409).body("처리 중입니다");  // Conflict
    }

    try {
        // 3. 실제 처리
        PaymentResponse response = paymentService.pay(request);

        // 4. 결과 저장 (TTL: 24시간)
        idempotencyStore.save(key, response, Duration.ofHours(24));

        return ResponseEntity.ok(response);
    } catch (Exception e) {
        idempotencyStore.unlock(key);  // 실패 시 잠금 해제 → 재시도 허용
        throw e;
    }
}
```

**Idempotency Key 저장소**:

```java
// Redis 기반 구현
@Component
public class RedisIdempotencyStore {

    public Optional<PaymentResponse> find(String key) {
        String value = redis.get("idempotency:" + key);
        return Optional.ofNullable(deserialize(value));
    }

    public boolean tryLock(String key) {
        // SET NX — 없을 때만 저장 (Lock 역할)
        return redis.setIfAbsent(
            "idempotency:lock:" + key,
            "PROCESSING",
            Duration.ofSeconds(30)  // 30초 후 자동 해제
        );
    }

    public void save(String key, PaymentResponse response, Duration ttl) {
        redis.set("idempotency:" + key, serialize(response), ttl);
        redis.delete("idempotency:lock:" + key);
    }
}
```

**Key 생성 주체**:
- **클라이언트**: UUID v4 생성 → 재시도 시 같은 Key 사용
- **서버**: 요청 내용 해시 → `hash(userId + orderId + amount)` — 클라이언트 신뢰 불가 시

### 패턴 2: DB Unique 제약 — 가장 간단

```sql
CREATE TABLE payment (
    id              BIGINT PRIMARY KEY,
    order_id        BIGINT NOT NULL,
    amount          DECIMAL(10,2) NOT NULL,
    idempotency_key VARCHAR(64) NOT NULL UNIQUE,  -- 이것만으로 중복 방지
    created_at      TIMESTAMP DEFAULT NOW()
);
```

```java
try {
    paymentRepository.save(payment);
} catch (DataIntegrityViolationException e) {
    // 이미 처리됨 → 기존 결과 반환
    return paymentRepository.findByIdempotencyKey(key);
}
```

### 패턴 3: 상태 기반 멱등성 — 상태 머신

```java
public class Order {
    private OrderStatus status;

    public void pay() {
        if (this.status != OrderStatus.PENDING) {
            // 이미 결제됨 or 취소됨 → 무시 (멱등)
            return;
        }
        this.status = OrderStatus.PAID;
    }

    public void cancel() {
        if (this.status == OrderStatus.CANCELLED) {
            return;  // 이미 취소됨 → 멱등
        }
        if (this.status == OrderStatus.SHIPPING) {
            throw new IllegalStateException("배송 중 취소 불가");
        }
        this.status = OrderStatus.CANCELLED;
    }
}
```

상태 전이가 명확하면, 같은 요청이 와도 이미 전이된 상태이므로 안전하다.

### 패턴 4: CAS (Compare-And-Set) — DB 레벨 멱등

```sql
-- 포인트 차감: 멱등하게 만들기
-- 비멱등 버전:
UPDATE wallet SET balance = balance - 100 WHERE user_id = 1;
-- 2번 실행하면 200 차감됨

-- 멱등 버전:
UPDATE wallet SET balance = balance - 100
WHERE user_id = 1 AND balance >= 100
  AND NOT EXISTS (
    SELECT 1 FROM transaction_log
    WHERE transaction_id = 'txn-abc-123'
  );

INSERT INTO transaction_log (transaction_id, user_id, amount)
VALUES ('txn-abc-123', 1, -100);
-- 2번 실행해도 transaction_log에 이미 있으니 UPDATE 실행 안 됨
```

### 패턴 5: MQ Consumer 멱등성

Kafka의 at-least-once 보장 = 메시지가 **최소 1번** 전달 = 중복 가능.

```java
@KafkaListener(topics = "payment-events")
public void handlePayment(PaymentEvent event) {
    // 1. 이미 처리한 메시지인지 확인
    if (processedEventRepository.existsByEventId(event.eventId())) {
        log.info("이미 처리된 이벤트: {}", event.eventId());
        return;  // 무시 (멱등)
    }

    // 2. 처리
    paymentService.process(event);

    // 3. 처리 기록 저장
    processedEventRepository.save(new ProcessedEvent(event.eventId()));
}
```

**주의**: 2와 3이 원자적이지 않으면 구멍이 생긴다.

```java
// 안전한 버전: 같은 트랜잭션으로 묶기
@Transactional
public void handlePaymentSafely(PaymentEvent event) {
    // UPSERT or INSERT IGNORE로 원자적 처리
    int inserted = processedEventRepository.insertIgnore(event.eventId());
    if (inserted == 0) return;  // 이미 처리됨

    paymentService.process(event);
}
```

## 멱등성 적용 판단 기준

| 상황 | 멱등성 필수? | 패턴 |
|------|-------------|------|
| 결제 | **필수** | Idempotency Key + DB Unique |
| 주문 생성 | **필수** | Idempotency Key |
| 알림 발송 | **필수** | 발송 기록 체크 |
| 쿠폰 발급 | **필수** | DB Unique (user_id + coupon_id) |
| 포인트 적립/차감 | **필수** | Transaction ID + CAS |
| 조회 API | 이미 멱등 | 불필요 |
| 로그 적재 | 선택적 | 약간의 중복 허용 가능 |
| 좋아요 토글 | 자연스럽게 멱등 | 상태 기반 (ON/OFF) |

## 흔한 실수들

| 실수 | 문제 | 수정 |
|------|------|------|
| TTL 없는 Idempotency Key | 저장소 무한 증가 | 24~48시간 TTL |
| 실패한 요청도 캐싱 | 재시도 불가 | 성공한 결과만 저장 |
| Key를 서버에서 생성 | 클라이언트가 재시도 시 다른 Key | 클라이언트가 Key 생성 |
| 처리 + 기록이 비원자적 | 처리됐는데 기록 실패 → 중복 처리 | 같은 트랜잭션으로 |
| Consumer에 멱등성 없음 | MQ 재전달 시 중복 실행 | 반드시 EventId 체크 |

---

## 면접에서 이렇게 답하라

> "멱등성은 네트워크 재시도, MQ 재전달, 사용자 더블 클릭 등 현실의 중복 요청에 대비하는 설계입니다.
> 가장 범용적인 패턴은 **Idempotency Key**입니다. 클라이언트가 고유 키를 생성하고, 서버는 이 키로 처리 여부를 확인합니다.
> 결제처럼 비가역적 작업은 **DB Unique 제약 + Idempotency Key**를 이중으로 적용합니다.
> MQ Consumer는 **at-least-once** 전달이 기본이므로, EventId로 중복 체크하고 처리+기록을 같은 트랜잭션으로 묶습니다.
> 핵심은 '처리 확인'과 '실제 처리'를 **원자적**으로 만드는 것입니다."
